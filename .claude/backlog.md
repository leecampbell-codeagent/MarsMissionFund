## Priority Rationale

### P0 — Foundation (build first; nothing else works without these)

- **feat-001** Monorepo scaffold: zero application features are possible without the package structure, tooling, and Express/React entry points.
- **feat-002** Auth: every protected endpoint and authenticated UI depends on Clerk integration and the `users` table.
- **feat-003** Design system: every frontend feature consumes these primitives; building any UI without them creates technical debt.
- **feat-004** App shell: route structure, auth guard, and centralised API client are prerequisites for every page feature.
- **feat-005** DB schema: the data model underpins all campaign, contribution, and KYC domain logic.

### P1 — Core Value (the minimum viable demo loop)

The P1 features deliver the complete happy-path demo: a creator submits a campaign → reviewer approves → campaign goes live → donor contributes → donor sees history.

- **feat-006** + **feat-007** together unblock campaign submission (KYC gate must pass).
- **feat-008** delivers the reviewer workflow that approves campaigns.
- **feat-009** delivers the public campaign page and "Back This Mission" CTA.
- **feat-010** closes the contribution loop with the payment stub.
- **feat-011** completes the donor experience with history and impact stats.

### P2 — Core Enhancements (complete the platform)

- **feat-012** Milestone verification and settlement: demonstrates the staged disbursement model.
- **feat-013** Search: makes the platform discoverable beyond direct links.
- **feat-014** Notifications: closes the feedback loop for users.
- **feat-016** Refunds: demonstrates the financial safety model (campaign failure handling).
- **feat-017** Admin dashboard: exposes platform health and management tools.

### P3 — Integrations (wire up real external services)

- **feat-015** Stripe live integration: replaces the mock with real Stripe Elements and webhook handling.

## Out of Scope for All Phases (theatre per specs)

The following are explicitly marked as theatre in the domain specs and should not be built:

- Multi-environment deployment (staging, production) infrastructure.
- Real AWS SES email delivery (mock adapter only for local demo).
- Real Veriff KYC document verification (mock adapter only for local demo).
- PCI DSS compliance audit processes.
- GDPR operational workflows and automated right-to-erasure processing.
- Sanctions screening (OFAC, EU, UN, DFAT).
- Manual KYC review workflow (beyond the stub auto-approve).
- Disaster recovery, failover, on-call rotations.
- Recommendation engine (collaborative/content-based filtering).
- Re-engagement automation and frequency capping.
- Tax receipt PDF generation.
- Social sharing Open Graph meta tag generation.
- Stretch goal mechanics.
- Appeal process for rejected campaigns.
- Deadline extension requests (admin approval workflow).
- Milestone change requests (admin approval workflow).
- Interest calculation and disbursement.
- Disbursement approval expiry window.
- Data portability exports (GDPR Article 20).
- Account deactivation GDPR erasure workflow.
