## Warnings (Should Fix Before or During Implementation)

### WARN-1: Error response body missing `correlation_id`

**Location:** `packages/backend/src/server.ts` — global error handler middleware

**Issue:** L3-001 Section 6.1 defines the standard error response format as:
```json
{ "error": { "code": "...", "message": "...", "correlation_id": "...", "details": {} } }
```
The spec's error handler shape is:
```json
{ "error": { "code": "INTERNAL_ERROR", "message": "..." } }
```
`correlation_id` and `details` are absent. In feat-001 there is no correlation ID middleware yet, so `correlation_id` cannot be populated. However, the scaffold establishes the error shape — if it omits `correlation_id` now, the format drift must be corrected when correlation ID infrastructure is added.

**Recommendation for implementation agent:** Add `correlation_id: null` as a placeholder in the error response shape in feat-001 so the structure is established correctly from day one. When correlation ID middleware is introduced, replace `null` with the real value.

**Agent responsible:** Implementation agent for feat-001.

---

### WARN-2: Health endpoint not versioned (`/health` vs. `/v1/health`)

**Location:** `packages/backend/src/health/api/health-router.ts`, `packages/backend/src/server.ts`

**Issue:** L2-002 Section 5.1 states "All APIs are versioned from day one." L3-001 Section 6.1 specifies URL path versioning (`/v1/resource`). The spec defines the endpoint as `GET /health` (unversioned). The spec references this as consistent with "API conventions (L3-001 Section 6.1)" in the resolved decisions table, but L3-001 does not explicitly exempt health checks from versioning.

**Context:** L2-002 Section 6.3 and Section 5.4 treat health checks as a special operational endpoint (the one exception to authentication). Whether versioning also applies to operational endpoints is ambiguous in the specs. L3-001 Section 6.1 defines versioning for the domain API surface. The health endpoint is an operational endpoint consumed by orchestration layers (ECS, ALB), not domain API consumers. Unversioned health endpoints are the universal industry convention for orchestration.

**Recommendation:** The spec's choice of `/health` (unversioned) is appropriate for an operational liveness endpoint. The `start-dev-stack.sh` integration and the L2-002 health check exemption both support this. Implement as `/health`. A Spec Writer clarification or ADR note on whether operational endpoints are exempt from versioning would eliminate this ambiguity for future features.

---

### WARN-3: App component exported functions lack explicit return types

**Location:** `packages/frontend/src/App.tsx`

**Issue:** The coding standards in `.claude/rules/backend.md` require "Explicit return types on all exported functions." The scaffold `App()` function has no `: JSX.Element` or `: React.ReactElement` return type annotation.

**Recommendation for implementation agent:** Add `: JSX.Element` return type to `export function App(): JSX.Element`. Verify TypeScript strict mode does not flag this.

---

### WARN-4: Token-based design deferred — clear handoff required

**Location:** `packages/frontend/src/App.tsx`, `packages/frontend/src/index.css`

**Issue:** The scaffold uses raw Tailwind utilities (`bg-black text-white text-4xl font-bold`) instead of design system semantic tokens. The design spec correctly documents this as a transitional state and provides the intended final token specification. However, this creates a risk of the transitional state persisting if the first UI feature's implementing agent does not explicitly replace these utilities with semantic token references.

**Recommendation:** The first UI feature spec (whichever feat-00X introduces real UI) must include an explicit task to replace the scaffold's raw Tailwind utilities in `App.tsx` and to populate the `@theme` block in `index.css`. The design spec's token list (Section: "Tokens to Establish in `src/index.css` `@theme` Block") should be referenced as a required deliverable in that feature.
