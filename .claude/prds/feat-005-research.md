## 1. Domain Knowledge

### Key Concepts

**Contribution State Machine**

Per L4-004 Section 5.3, the spec defines five contribution states.
For feat-005 (immediate-capture stub), only three are in scope:

| State | Description | Trigger |
|-------|-------------|---------|
| `pending_capture` | Capture request sent to gateway adapter | POST /contributions received, basic validation passes |
| `captured` | Funds captured and escrow ledger entry created | Stub gateway returns `success: true` |
| `failed` | Capture failed, no funds held | Stub gateway returns `success: false` (tok_fail sentinel) |

States `refunded` and `partially_refunded` are out of scope for feat-005 (handled in feat-006).

The transition model is simple for the stub:
```
(new) â†’ pending_capture â†’ captured
(new) â†’ pending_capture â†’ failed
```

`pending_capture` is essential even for the stub because it creates an immutable record of intent before the gateway call.
If the server crashes after the gateway call but before the DB update, the `pending_capture` record enables idempotent retry and prevents double charging.

**Escrow Ledger (Double-Entry Design)**

The escrow ledger is an append-only, immutable log of all money movements for a campaign.
Double-entry means every transaction has two sides: a credit (money in) and a debit (money out).
For feat-005, only credit entries are created (contributions flowing in to escrow).

Each ledger entry records:
- Which campaign (campaign_id)
- Which contribution (contribution_id)
- Entry type: `credit` (money in from donor) or `debit` (money out for disbursement/refund â€” future)
- Amount in integer cents (amount_cents BIGINT)
- Running balance at time of entry (running_balance_cents)
- Timestamp (created_at)

The running_balance is a denormalised convenience computed at write time using a subquery or application-layer sum.
It must never be updated retroactively â€” each row is immutable.

"Total raised" can be computed from the ledger as:
```sql
SELECT SUM(amount_cents) FILTER (WHERE entry_type = 'credit')
     - SUM(amount_cents) FILTER (WHERE entry_type = 'debit')
AS escrow_balance_cents
FROM escrow_ledger
WHERE campaign_id = $1
```

For performance, a denormalised `total_raised_cents` column on the `campaigns` table is updated atomically when a contribution is captured.
The ledger is the source of truth; the denormalised column is a read-optimisation.
Both must be updated in the same DB transaction as the contribution status update.

**Why Append-Only Immutability Matters**

Per L4-004 Section 6.2, the ledger must be append-only and immutable.
This satisfies:
- Repudiation prevention (STRIDE): no one can claim a contribution was never made
- Financial audit trail (L3-006): regulators and internal compliance can reconstruct the escrow history
- Double-entry integrity: a debit cannot be "undone" â€” a correcting credit is created instead

**Immediate Capture Pattern**

Per L4-004 Section 5.1, the decision is "immediate capture" â€” authorisation and capture in one step.
No pre-auth hold that must be captured later.
This means:
- Contribution is either `captured` or `failed` â€” never stuck in a pre-auth limbo
- Simpler state machine for the stub
- Risk: if the campaign fails and all contributions must be refunded, the refund is a separate gateway call on the original charge reference (not a void of an auth)

**Duplicate Contribution Prevention**

AC-PAY-003 requires detecting "same donor, same campaign, same amount within a short time window."
The spec does not define the time window but the feature brief specifies 60 seconds.

Two implementation approaches:

1. **Unique constraint + window check (recommended for stub):**
   Add a unique constraint on `(donor_id, campaign_id, amount_cents)` within a time window.
   PostgreSQL does not support windowed unique constraints natively.
   Instead: at application layer, query for contributions in the last 60 seconds with matching (donor_id, campaign_id, amount_cents) before inserting.
   Race condition risk if two concurrent requests pass this check simultaneously â€” mitigated by an INSERT with a conditional check or a unique index with partial predicate.

2. **Idempotency key pattern:**
   Client generates a UUID idempotency key and sends it in the request.
   Server stores the key; if a duplicate key arrives within 24 hours, return the original response.
   More robust for retries; recommended for production Stripe integration.
   For the stub, the 60-second window check is simpler and sufficient.

For feat-005: implement the 60-second window check at the application service layer.
Check: `SELECT id FROM contributions WHERE donor_user_id = $1 AND campaign_id = $2 AND amount_cents = $3 AND created_at > NOW() - INTERVAL '60 seconds'`.
If a matching row exists, throw `DuplicateContributionError` â†’ HTTP 409.

**Payment Gateway Port Interface Design**

Modelled on the existing `KycVerificationPort` pattern:

```typescript
// packages/backend/src/payment/ports/payment-gateway.port.ts

export interface CaptureInput {
  readonly token: string;           // Payment method token (from Stripe Elements / sentinel)
  readonly amountCents: number;     // Integer cents
  readonly currency: 'USD';         // Single currency per L4-004 Section 9.1
  readonly idempotencyKey: string;  // UUID; prevents double charging on retry
  readonly metadata: {
    readonly campaignId: string;
    readonly contributionId: string;
    readonly donorId: string;       // MMF user UUID (not Clerk user ID)
  };
}

export interface CaptureResult {
  readonly success: boolean;
  readonly transactionRef: string;  // Gateway transaction ID (stub: 'stub_txn_XXXXXXXX')
  readonly failureReason?: string;  // Only present when success === false
}

export interface PaymentGatewayPort {
  capture(input: CaptureInput): Promise<CaptureResult>;
}
```

**Stub Adapter Sentinel Tokens**

| Token | Behaviour | Result |
|-------|-----------|--------|
| `tok_fail` | Simulated decline | `{ success: false, transactionRef: 'stub_txn_failed', failureReason: 'Card declined' }` |
| Any other value (including `tok_success`) | Simulated success | `{ success: true, transactionRef: 'stub_txn_' + nanoid() }` |

The "any other token succeeds" approach is deliberate: it makes the frontend demo frictionless
(testers don't need to remember a specific success token) while still providing a failure path.

**Monetary Amount Handling**

Per L4-004 Section 9.1 and the backend rules:
- Stored as `BIGINT` (integer cents) in PostgreSQL
- API accepts amount as string (per G-024 BIGINT pattern)
- API response returns amount as string
- Frontend never parses to floating-point Number
- Display via `Intl.NumberFormat` in frontend

Minimum contribution: $5.00 USD = 500 cents.
Zod schema: `z.string().regex(/^\d+$/).transform(v => BigInt(v))` then validate `>= 500n`.

Actually per the campaign domain, monetary fields are `string` at API boundary and `BIGINT` from DB.
The Zod schema should validate as a numeric string and the application layer parses to a JS number (safe since INT64 max is ~922 trillion cents, well beyond any single contribution).

**Campaign Funding Cap Enforcement**

Per L4-002 Section 3.3: "When the maximum funding cap is reached, the campaign stops accepting new contributions."
The contribution flow must check whether a new contribution would push total_raised_cents past funding_cap_cents.
If so: reject with HTTP 422 / `CAMPAIGN_FUNDING_CAP_REACHED`.

This check must be atomic. If two concurrent contributions would both be under the cap individually but combined would exceed it, both must not be accepted.
Approach: check and update `campaigns.total_raised_cents` with a single UPDATE:
```sql
UPDATE campaigns
SET total_raised_cents = total_raised_cents + $1,
    contributor_count = contributor_count + 1,
    status = CASE
      WHEN total_raised_cents + $1 >= funding_goal_cents AND status = 'live' THEN 'funded'
      ELSE status
    END,
    updated_at = NOW()
WHERE id = $2
  AND status IN ('live', 'funded')
  AND (funding_cap_cents IS NULL OR total_raised_cents + $1 <= funding_cap_cents)
RETURNING *
```
If 0 rows affected: either campaign not in live/funded state, or cap reached.

**Campaign live â†’ funded Transition**

When a contribution pushes `total_raised_cents >= funding_goal_cents` and the campaign status is `live`,
the system must transition the campaign to `funded`.
This should happen atomically in the same UPDATE that increments total_raised_cents.
The CASE expression in the UPDATE above handles this.
A campaign_audit_events entry must be written for the `live â†’ funded` transition.

### Calculation Methodology

**Total Raised**

Two sources of truth:
1. Ledger: `SUM(amount_cents) WHERE entry_type = 'credit'` per campaign (source of truth, slow)
2. Denormalised: `campaigns.total_raised_cents` (fast, updated atomically with contribution)

For feat-005: maintain both.
The denormalised column is updated in the contribution capture transaction.
The ledger entry is the immutable record.

**Funding Percentage**

```
funding_percentage = (total_raised_cents / funding_goal_cents) * 100
```

Display as percentage; cap at 100% for display even if campaign is overfunded (past goal, still accepting up to cap).

**Contributor Count**

A simple denormalised counter: `campaigns.contributor_count` incremented by 1 on each successful capture.
Unlike total_raised_cents, this counts contributions (not unique donors).
The spec doesn't distinguish unique donors from contribution count for the local demo â€” one integer counter suffices.

### Industry Conventions

**Kickstarter pattern:**
- All-or-nothing funding: if goal not reached by deadline, no money taken
- Simple checkout: enter amount, enter card details (Stripe Elements), confirm
- Confirmation page shows backer number and amount
- No stored payment methods â€” re-enter for each project
- No intermediate "pending" state shown to the backer

**Indiegogo pattern:**
- Flexible funding: keep what you raise even below goal
- More complex UX: perks/rewards tiers
- Immediate capture on contribution

**MMF specific:**
- Flexible with minimum threshold (per L4-002): keep if goal reached, refund if not
- No perks/rewards â€” pure financial contribution
- The "Back This Mission" CTA on the campaign detail page links to `/campaigns/:id/contribute`
- The contribute page must be a protected route (requires authentication)
- If the donor is not authenticated, clicking CTA redirects to sign-in then back to contribute

**Expected UX flow:**
1. Donor sees campaign detail page (public, unauthenticated)
2. Clicks "Back This Mission" CTA
3. If not signed in: redirected to /sign-in, then redirected back to /campaigns/:id/contribute
4. On /campaigns/:id/contribute: shows amount input + payment form
5. For the stub: amount input + token input (in place of Stripe Elements card form)
6. Submit â†’ POST /api/v1/contributions
7. Success: navigate to confirmation page or back to campaign with success toast
8. Failure (tok_fail): show clear error message, allow retry

### Regulatory Notes

- PCI DSS SAQ-A: backend never sees raw card data; stub uses sentinel tokens, no card data involved
- Payment tokens (`tok_fail`, `tok_success`) must never be logged (L2-002 Section 1.7)
- The `idempotency_key` stored in the DB is a UUID, not a payment token â€” safe to log
- `transaction_ref` from the stub is a synthetic reference â€” safe to log
- KYC requirement for donors: The spec (L4-004) gates KYC on disbursement, NOT on contribution.
  L4-005 does not list donor contributions as a KYC-gated action.
  Backers do not need KYC verified to make contributions â€” only Creators do for campaigns/disbursements.
  This is confirmed by L3-002 Section 4.2 which lists "financial actions (contributions, disbursements, refunds)" for MFA â€” but MFA is handled by Clerk session, not an explicit KYC gate.
  Conclusion: Donors can contribute without KYC verification. Only account activation is required.

**Proposed contributions table:**

```sql
CREATE TABLE IF NOT EXISTS contributions (
  id               UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  campaign_id      UUID        NOT NULL REFERENCES campaigns(id) ON DELETE RESTRICT,
  donor_user_id    UUID        NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
  amount_cents     BIGINT      NOT NULL CHECK (amount_cents >= 500),
  status           TEXT        NOT NULL DEFAULT 'pending_capture'
                               CHECK (status IN ('pending_capture', 'captured', 'failed',
                                                  'refunded', 'partially_refunded')),
  payment_token    TEXT        NOT NULL,  -- sentinel token; never logged
  idempotency_key  UUID        NOT NULL UNIQUE,  -- prevents double capture on retry
  transaction_ref  TEXT,       -- populated on capture or failure
  failure_reason   TEXT,       -- populated on failure
  captured_at      TIMESTAMPTZ,
  created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Indexes
CREATE INDEX idx_contributions_campaign_id   ON contributions (campaign_id);
CREATE INDEX idx_contributions_donor_user_id  ON contributions (donor_user_id);
CREATE INDEX idx_contributions_status         ON contributions (status);
CREATE INDEX idx_contributions_created_at     ON contributions (created_at);
-- Duplicate detection index: check for same donor + campaign + amount in last 60s
CREATE INDEX idx_contributions_dedup ON contributions (donor_user_id, campaign_id, amount_cents, created_at);

-- Updated_at trigger
CREATE TRIGGER contributions_updated_at
  BEFORE UPDATE ON contributions
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
```

**Note on payment_token storage:**
The `payment_token` column stores the sentinel value or Stripe token ID.
For the stub, this is `tok_success`, `tok_fail`, or any string.
In production with real Stripe, the token is single-use and consumed on first capture â€” storing it afterwards is not a PCI risk but is unnecessary.
Consider not storing the token after use (set to NULL or don't store after capture).

**Proposed escrow_ledger table:**

```sql
CREATE TABLE IF NOT EXISTS escrow_ledger (
  id                     UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  campaign_id            UUID        NOT NULL REFERENCES campaigns(id) ON DELETE RESTRICT,
  contribution_id        UUID        REFERENCES contributions(id) ON DELETE RESTRICT,
  entry_type             TEXT        NOT NULL CHECK (entry_type IN ('credit', 'debit', 'interest_credit')),
  amount_cents           BIGINT      NOT NULL CHECK (amount_cents > 0),
  running_balance_cents  BIGINT      NOT NULL,  -- cumulative balance for this campaign at this entry
  description            TEXT,                  -- human-readable reason (e.g. 'Contribution from donor')
  created_at             TIMESTAMPTZ NOT NULL DEFAULT NOW()
  -- No updated_at: ledger is append-only, never updated
);

CREATE INDEX idx_escrow_ledger_campaign_id     ON escrow_ledger (campaign_id);
CREATE INDEX idx_escrow_ledger_contribution_id ON escrow_ledger (contribution_id);
CREATE INDEX idx_escrow_ledger_created_at      ON escrow_ledger (created_at);
```

**Note:** No `updated_at` trigger on `escrow_ledger` â€” it is append-only by design.
The `running_balance_cents` is computed at insert time:
```sql
INSERT INTO escrow_ledger (campaign_id, contribution_id, entry_type, amount_cents, running_balance_cents, description)
VALUES ($1, $2, 'credit', $3,
        COALESCE((SELECT running_balance_cents FROM escrow_ledger WHERE campaign_id = $1 ORDER BY created_at DESC LIMIT 1), 0) + $3,
        'Contribution captured')
```

**Mock strategy for feat-005:**

The `StubPaymentGatewayAdapter` implements `PaymentGatewayPort` and ignores the token value except for the `tok_fail` sentinel:

```typescript
// packages/backend/src/payment/adapters/stub-payment-gateway.adapter.ts

export class StubPaymentGatewayAdapter implements PaymentGatewayPort {
  async capture(input: CaptureInput): Promise<CaptureResult> {
    // Never log input.token â€” it could be a real token in production
    if (input.token === 'tok_fail') {
      return {
        success: false,
        transactionRef: 'stub_txn_failed',
        failureReason: 'Card declined',
      };
    }
    // Any other token: simulate success with a unique transaction reference
    return {
      success: true,
      transactionRef: `stub_txn_${crypto.randomUUID().replace(/-/g, '').slice(0, 16)}`,
    };
  }
}
```

**Sentinel tokens:**
| Token | Behaviour |
|-------|-----------|
| `tok_fail` | Returns failure with 'Card declined' |
| `tok_success` | Returns success (convenience alias â€” not strictly needed) |
| Any other string | Returns success |

**Idempotency key usage:**
For real Stripe, the idempotency key is sent as a request header: `Idempotency-Key: <uuid>`.
For the stub, the `idempotencyKey` field in `CaptureInput` is stored in the `contributions` table but not used by the stub adapter itself.
The stub adapter always returns a new transaction reference per call.

**Rate limits (production Stripe):**
Stripe allows 100 read operations/sec and 100 write operations/sec per secret key.
The stub has no rate limits.

## 4. Competitor Patterns

### Kickstarter

**How they handle contributions:**
- Amount selection: predefined reward tiers + custom amount field with minimum
- Card entry: Stripe Elements iframe embedded in page (SAQ-A compliant)
- Process: single step â€” enter amount + card details, click "Back this project"
- Confirmation: immediate in-page success state with backer number and amount
- No intermediate pending state shown to the backer
- If declined: inline error "Your card was declined. Please try another card."

**What works well:**
- Single-screen UX: amount + payment in one step reduces abandonment
- Inline error messages on decline â€” user knows immediately without page navigation
- Clear "You're backing [Project Name]" heading sets context

**What doesn't work:**
- No "Back" without losing form state (mitigated by Stripe Elements which preserves field state)
- Contribution amount validation happens server-side â€” client-side validation would reduce round trips

### Indiegogo

**How they handle contributions:**
- Contribution flow: multi-step (select perk â†’ configure â†’ payment â†’ review â†’ confirm)
- Payment: Stripe Elements
- Status page: detailed breakdown of what you funded
- Email confirmation sent immediately

**What works well:**
- Review screen before final charge â€” reduces accidental contributions
- Clear breakdown of exactly what the contribution amount does

**What doesn't work for MMF:**
- Multi-step flow is appropriate for perk-based campaigns but overkill for pure financial contributions
- MMF has no perks/rewards â€” single-step is appropriate

### MMF-Specific UX Recommendation

For the local demo stub, the contribute page should be:
1. **One-screen form**: campaign summary at top, amount input, token input (in place of card form), submit button
2. **Immediate response**: success â†’ show confirmation details on same page or navigate to a confirmation route
3. **Failure path**: inline error with clear message and a retry option (pre-fills the form, clears token)
4. The token input has a helper text: "Enter `tok_fail` to simulate a card decline"

No multi-step â€” keep it simple for the workshop demo.

## 5. Edge Cases

### Data Edge Cases

- [ ] **Amount below minimum ($5 / 500 cents):**
  Input: `"amountCents": "499"`.
  Expected: HTTP 400, `VALIDATION_ERROR`, "Minimum contribution is $5.00".
  Handled by Zod schema validation before any DB or gateway call.

- [ ] **Amount exactly at minimum (500 cents):**
  Input: `"amountCents": "500"`.
  Expected: accept and process normally.
  This is a boundary test â€” ensure the `>= 500` check is inclusive.

- [ ] **Amount as integer instead of string:**
  Input: `{ "amountCents": 50000 }` (JSON number, not string).
  Expected: Zod schema uses `z.string()` for `amountCents`, so a number fails validation.
  HTTP 400, `VALIDATION_ERROR`.
  This is G-024 applied to contributions â€” amounts must be strings at the API boundary.

- [ ] **Zero amount:**
  Input: `"amountCents": "0"`.
  Expected: HTTP 400, `VALIDATION_ERROR`.
  Caught by Zod regex + minimum validation.

- [ ] **Negative amount:**
  Input: `"amountCents": "-500"`.
  Expected: HTTP 400, `VALIDATION_ERROR`.
  Caught by Zod regex (only digits) â€” negative sign fails the regex.

- [ ] **Extremely large contribution:**
  Input: `"amountCents": "999999999999"` (nearly $10B).
  The campaign's `funding_cap_cents` would prevent this from exceeding the cap.
  If no cap and amount is valid integer cents: process normally (or reject if exceeds campaign cap).
  Zod schema should apply a max: no more than the campaign's `funding_cap_cents` or a platform maximum.
  Recommendation: validate `amountCents <= 100_000_000_00` (max $10M per contribution) at API layer.

- [ ] **Missing required fields:**
  Input: `{}` or `{ "campaignId": "uuid" }` (partial body).
  Expected: HTTP 400, `VALIDATION_ERROR`. Caught by Zod `.strict()`.

- [ ] **Invalid campaign ID (not a UUID):**
  Input: `"campaignId": "not-a-uuid"`.
  Expected: HTTP 400, `VALIDATION_ERROR`. Zod `z.string().uuid()` rejects this.

- [ ] **Campaign ID does not exist:**
  Input: valid UUID that doesn't correspond to any campaign.
  Expected: HTTP 404, `CAMPAIGN_NOT_FOUND`.
  Application service checks campaign existence before proceeding.

### Payment Edge Cases

- [ ] **Payment token `tok_fail`:**
  Expected: contribution created with status `pending_capture`, then transitions to `failed`.
  Response: HTTP 422 (Unprocessable Entity), error code `PAYMENT_CAPTURE_FAILED`, message "Your payment could not be processed. Please try another payment method."
  Note: HTTP 422 is more accurate than 400 here â€” the request was valid but the payment failed.
  Alternatively, the contribution IS created (as `failed`) and the API returns 201 with `status: "failed"` â€” so the frontend knows to show a failure state.
  Recommended: return 201 with `status: "failed"` and let the frontend handle it. This maintains the audit trail of the attempt.
  Discussion: the feature brief says "contribution transitions to failed and the donor receives a clear error response". This implies the API should communicate failure clearly â€” either via a non-2xx status or a 2xx with explicit `failed` status. The spec writer must resolve this.

- [ ] **Payment gateway "timeout" (stub doesn't timeout, but architecture must handle):**
  In production: if Stripe times out after 30s, the contribution is left in `pending_capture`.
  For the stub: no timeout â€” this is theatre.
  The `pending_capture` state serves as the recovery mechanism: a background job would query for stuck `pending_capture` contributions and retry.
  For feat-005: document this as a known limitation.

- [ ] **Empty/null payment token:**
  Input: `"paymentToken": ""` or `"paymentToken": null`.
  Expected: HTTP 400, `VALIDATION_ERROR`. Zod `z.string().min(1)`.

### Concurrency Edge Cases

- [ ] **Duplicate contribution within 60 seconds (same donor, campaign, amount):**
  Donor accidentally double-clicks submit or two browser tabs fire simultaneously.
  Expected: first request succeeds; second request returns HTTP 409, `DUPLICATE_CONTRIBUTION`.
  Implementation: query for matching row in `contributions` table within 60s before creating new one.

- [ ] **Duplicate contribution 61 seconds later:**
  Same donor, same campaign, same amount, but `created_at > NOW() - INTERVAL '60 seconds'` is false.
  Expected: accepted as a new independent contribution â€” no duplicate error.
  The 60-second window is the spec-defined threshold.

- [ ] **Concurrent contributions from same donor to same campaign:**
  Two separate amounts (different `amount_cents`): both should succeed â€” no duplicate detection.
  Same amount but concurrent (race condition): duplicate detection must be atomic.
  The `idx_contributions_dedup` index alone is not enough (no uniqueness constraint) â€” need the application-level 60s check.
  Consider: a partial unique index `WHERE created_at > NOW() - INTERVAL '60 seconds'` is not natively supported in PostgreSQL.
  Use application-layer check inside a transaction for the local demo.

- [ ] **Race condition on funding cap:**
  Two donors simultaneously submit contributions that individually fit under the cap but together would exceed it.
  Expected: exactly one should succeed; the other should receive HTTP 422, `CAMPAIGN_FUNDING_CAP_REACHED`.
  Implementation: the atomic `UPDATE campaigns SET total_raised_cents = total_raised_cents + $1 WHERE ... AND total_raised_cents + $1 <= funding_cap_cents` pattern handles this correctly â€” the second update will affect 0 rows if the first update made the balance exceed the cap.

- [ ] **Race condition on live â†’ funded transition:**
  Multiple contributions arrive simultaneously, all pushing the total past the funding goal.
  Expected: all contributions captured; only one audit event for the live â†’ funded transition.
  The atomic SQL UPDATE handles the status transition idempotently via the CASE expression.

### Integration Edge Cases

- [ ] **Campaign not in `live` or `funded` state:**
  Campaign is in `draft`, `submitted`, `under_review`, `approved`, `rejected`, `suspended`, `failed`, `settlement`, `complete`, or `cancelled`.
  Expected: HTTP 422, `CAMPAIGN_NOT_ACCEPTING_CONTRIBUTIONS`.
  The acceptance criteria in the feature brief specifies this.
  Note: the feature brief says HTTP 422 for this case; the existing error patterns use 409 for state conflicts.
  Recommend 422 (Unprocessable Entity) as this is not a conflict â€” the resource exists but cannot process the operation in its current state.

- [ ] **Campaign `funded` but at exact funding cap:**
  `total_raised_cents == funding_cap_cents` â€” no room for any contribution.
  Expected: HTTP 422, `CAMPAIGN_FUNDING_CAP_REACHED`.
  The atomic UPDATE WHERE clause `total_raised_cents + $1 <= funding_cap_cents` catches this.

- [ ] **Campaign deadline has passed but status is still `live`:**
  This is a stale state â€” the campaign should have been auto-transitioned to `funded` or `failed` at deadline.
  For the local demo, deadline enforcement is "theatre" (not implemented as an automated job).
  The contribution flow should check: `deadline IS NULL OR deadline > NOW()`.
  If deadline has passed: reject with HTTP 422, `CAMPAIGN_DEADLINE_PASSED`.

- [ ] **Campaign `funded` â€” deadline passed, still accepting contributions:**
  Per L4-002 Section 3.2: "campaign remains live for additional contributions until deadline or maximum cap."
  A funded campaign with deadline in the future still accepts contributions.
  The `funded` status must be in the allowlist for the contribution flow.

- [ ] **Escrow ledger insert fails after successful gateway capture:**
  Scenario: gateway returns success, contribution status updated to `captured`, but escrow ledger INSERT fails (constraint violation, timeout).
  The contribution is marked `captured` but no ledger entry exists â€” inconsistent state.
  Fix: wrap the contribution status update + ledger insert in a single DB transaction.
  If the transaction fails after the gateway call: the contribution stays in `pending_capture` and a background recovery mechanism can identify it.
  For the local demo: use a transaction; document this limitation.

- [ ] **Campaign total_raised_cents update fails after capture:**
  Same scenario: capture succeeds, contribution status updated, ledger inserted, but campaign counter update fails.
  Fix: include all three operations in a single DB transaction.
  The atomic SQL UPDATE (shown in Section 1) handles contribution + campaign update together.

- [ ] **Donor's account is not active (suspended, deactivated, pending_verification):**
  Expected: HTTP 403, `ACCOUNT_NOT_ACTIVE`.
  The application service checks `user.accountStatus === 'active'` before proceeding.
  This follows the pattern from campaign creation.

- [ ] **Donor user record not found (Clerk user exists but MMF user record missing):**
  This can happen if the Clerk webhook for `user.created` hasn't arrived yet.
  Expected: HTTP 404, `USER_NOT_FOUND`.
  Or better: trigger lazy user creation (the same pattern as the auth sync endpoint).
  Recommendation: follow the existing pattern â€” the contribution endpoint should be protected by the same MMF user lookup that other authenticated endpoints use.

### User Behaviour Edge Cases

- [ ] **Unauthenticated user tries to contribute:**
  Clicks "Back This Mission", session is expired or no session.
  Expected: HTTP 401, `UNAUTHENTICATED` from the backend.
  Frontend: the ProtectedRoute wrapper redirects to `/sign-in`.
  The "Back This Mission" CTA target (`/campaigns/:id/contribute`) should be a ProtectedRoute.

- [ ] **Authenticated user submits contribution to campaign they created:**
  Nothing in the spec prevents a creator from contributing to their own campaign.
  Expected: allow (no self-contribution restriction mentioned in L4-002 or L4-004).
  Flag this as a potential open question for the Spec Writer.

- [ ] **Contributing to a campaign that just transitioned to `funded` (between load and submit):**
  Donor loads the campaign detail page when status is `live`.
  By the time they submit, the campaign is `funded` (another donor pushed it over the goal).
  Expected: contribute succeeds â€” `funded` campaigns still accept contributions.
  This is not an error case.

- [ ] **Contributing to a campaign that just reached its funding cap:**
  Donor loads campaign page, goes to contribute page.
  By the time they submit, the cap has been reached.
  Expected: HTTP 422, `CAMPAIGN_FUNDING_CAP_REACHED`.
  The frontend should show: "This campaign has reached its funding cap. Your contribution was not processed."

- [ ] **Retry after `failed` contribution (different token):**
  Donor submits with `tok_fail`, gets failure.
  Donor tries again with `tok_success`.
  Expected: the second attempt is treated as an independent contribution (different token, different idempotency key).
  No duplicate detection triggers â€” the failed attempt has status `failed`, not a valid prior contribution.
  Note: duplicate detection only applies to `pending_capture` or `captured` contributions, not `failed` ones.

- [ ] **Retry the exact same contribution after failure within 60 seconds:**
  Same amount, same campaign, but previous attempt is `failed`.
  Expected: allow â€” the 60-second window check should exclude `failed` contributions.
  SQL: `WHERE donor_user_id = $1 AND campaign_id = $2 AND amount_cents = $3 AND status IN ('pending_capture', 'captured') AND created_at > NOW() - INTERVAL '60 seconds'`

### Boundary Edge Cases

- [ ] **Contribution to a campaign where `funding_cap_cents IS NULL`:**
  Some campaigns may have no maximum cap (unlimited).
  Expected: the cap check must handle NULL gracefully: `funding_cap_cents IS NULL OR total_raised_cents + $1 <= funding_cap_cents`.
  The WHERE clause already handles this with `IS NULL OR` pattern.

- [ ] **Contribution ID collision (UUID generation):**
  Astronomically unlikely but: `gen_random_uuid()` produces a duplicate.
  Expected: PostgreSQL PK constraint violation â†’ application-level error â†’ HTTP 500.
  No special handling needed â€” UUID4 collision probability is negligible.

- [ ] **Campaign with `funding_goal_cents IS NULL`:**
  A campaign submitted without a funding goal (theoretically possible in draft state but required for submission).
  If a live campaign somehow has NULL funding_goal_cents: the live â†’ funded transition cannot trigger.
  Contribution still recorded; campaign stays `live`.
  Defensive check in the atomic SQL: only transition to `funded` if `funding_goal_cents IS NOT NULL`.

## 6. Recommendations

### Must-Haves for Spec

1. **Atomicity requirement:** Contribution status update, escrow ledger insert, and campaign counter update MUST be in a single DB transaction. Without this, any partial failure creates inconsistent state. The spec must explicitly state this.

2. **HTTP status for failed payment:** Spec must resolve whether to return HTTP 201 with `status: "failed"` (contribution created, gateway declined) or HTTP 422 (request failed). Recommend 201 with `status: "failed"` â€” this preserves the audit trail and gives the frontend a clear signal to retry.

3. **`CAMPAIGN_NOT_ACCEPTING_CONTRIBUTIONS` vs `CAMPAIGN_FUNDING_CAP_REACHED`:** These are distinct error cases that need distinct error codes. The spec must enumerate all HTTP error responses with codes and status codes.

4. **Duplicate detection scope:** Spec must clarify whether `failed` contributions count toward the 60-second window. Recommend: NO â€” only `pending_capture` and `captured` contributions trigger duplicate detection.

5. **Route for `/campaigns/:id/contribute`:** Must be a ProtectedRoute in `routes.tsx`. The "Back This Mission" CTA already links here. The spec must define this frontend route and its behaviour when unauthenticated (redirect to sign-in).

6. **`payment_token` logging prohibition:** The spec must explicitly state that `payment_token` is NEVER logged at any layer (backend or frontend). The audit event stores `transaction_ref`, not `payment_token`.

7. **New env var:** `MOCK_PAYMENT=true` must be added to `.env.example`. Default: `true`.

8. **Audit event field:** Spec must define the exact audit event schema for contribution state transitions: `contribution.initiated`, `contribution.captured`, `contribution.failed`.

9. **AuditLoggerPort extension:** The `resourceType` union in `audit-logger.port.ts` must be extended to include `'contribution'`.

10. **Campaign audit extension:** The `campaign_audit_events.action` CHECK constraint must include `'campaign.funded'` for the live â†’ funded transition.

### Watch-Outs

1. **G-024 BIGINT pattern:** `amountCents` must be validated as a numeric string at the API boundary. Never parse to `Number` â€” pass as string to the DB. The `pg` driver returns BIGINT as string. The stub adapter can receive the numeric string and convert to Number for arithmetic (safe for values up to $90T).

2. **Payment token must never be logged:** All pino loggers in the contribution flow must explicitly exclude `paymentToken` from log lines. Use structured logging with explicit field inclusion, not `JSON.stringify(body)`.

3. **Cross-context repository injection:** The `ContributionAppService` needs `UserRepository` and `CampaignRepository` from other bounded contexts. Follow pattern P-023 (cross-context read via shared repository injection). Do NOT import `AccountAppService` or `CampaignAppService` â€” only repositories.

4. **Campaign status check race condition:** The campaign status must be checked atomically with the funding update. A two-phase "check then update" approach has a TOCTOU race. Use the single atomic UPDATE with WHERE status IN ('live', 'funded') â€” if 0 rows affected, the status changed between check and update.

5. **Error handler extension required:** `createErrorHandler()` in `shared/middleware/error-handler.ts` must be extended with new error classes: `DuplicateContributionError`, `CampaignNotAcceptingContributionsError`, `PaymentCaptureFailedError`, `CampaignFundingCapReachedError`, `CampaignDeadlinePassedError`.

6. **`public-campaign-detail-page.tsx` CTA is already implemented:** The "Back This Mission" link to `/campaigns/${campaign.id}/contribute` is already in the codebase (line 311). The contribute route just needs to be created. Do NOT modify the campaign detail page â€” just add the route.

7. **`/campaigns/:id/contribute` vs `/campaigns/:id` route order:** React Router registers `/campaigns/:id` BEFORE `/campaigns/:id/contribute` in `routes.tsx`. Nested routes with specific paths must be registered before the param route, OR use a different structure. In React Router v6, more specific routes match first regardless of order â€” but this should be tested.

8. **Escrow ledger running_balance race condition:** If two concurrent contributions are captured simultaneously, the subquery for running_balance may read the same last entry and produce duplicate running balances. Fix: compute running_balance in the transaction that inserts the contribution â€” or use `SELECT ... FOR UPDATE` on the ledger. For the local demo, this race is acceptable to document but not fix.

9. **Funding progress stub in feat-004:** The `PublicCampaignListItem` and `PublicCampaignDetail` types in `campaign-app-service.ts` have hardcoded `totalRaisedCents: '0'` and `contributorCount: 0`. Once the campaigns table has `total_raised_cents` and `contributor_count` columns, the SQL in `pg-campaign-repository.adapter.ts` must be updated to return real values. This is a cross-feature dependency that the implementation agent must handle.

10. **`campaign_audit_events.action` CHECK constraint is immutable once data exists:** To add `'campaign.funded'` to the action CHECK constraint, a migration must drop and recreate the constraint (or use `ALTER TABLE ... DROP CONSTRAINT ... ADD CONSTRAINT`). This is non-trivial. Alternatively, use the `contribution_audit_events` table for all contribution-related events and a separate signal for the campaign status change. The spec writer must decide.
