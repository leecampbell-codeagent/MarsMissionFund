## Overview

feat-004 enables anonymous and authenticated users to discover live Mars mission campaigns via
full-text search, category browsing, and filter/sort controls, and to view a public campaign detail
page with funding progress, milestone plan, and a contribution call-to-action.
This is the primary donor entry point and a prerequisite for the contribution flow (feat-005).
The feature introduces a separate unauthenticated public API router at `/api/v1/public/campaigns`
and new PostgreSQL full-text search infrastructure on the `campaigns` table.

## Key Decisions (Resolved)

| Decision | Resolution | Rationale |
|----------|-----------|-----------|
| Anonymous access architecture | Separate `/api/v1/public/campaigns` router mounted without `requireAuth` (G-036) | Existing authenticated router has `requireAuth` applied at mount point in `app.ts` — cannot add unauthenticated routes to it |
| Full-text search engine | PostgreSQL `tsvector`/`tsquery` with GIN index on `search_vector` column | L3-001 specifies no external search provider; `tsvector` provides relevance ranking via `ts_rank()` |
| Search user input handling | `websearch_to_tsquery('english', $1)` with null guard (G-037) | Safe for arbitrary user input including operator chars; available since PostgreSQL 11 |
| Creator name in search | JOIN to `users.display_name` at query time, not in pre-computed tsvector (G-038) | Trigger cannot JOIN across tables; query-time JOIN is simpler and correct for demo dataset |
| Multi-value category filter | `WHERE category = ANY($1::TEXT[])` with validated Zod enum array (G-039) | Avoids dynamic SQL; PostgreSQL array operator is injection-safe |
| Pagination | Offset/limit, default `limit=20`, max `limit=100` | Simpler for workshop; cursor-based deferred to production |
| Funding progress stub | `totalRaisedCents: '0'`, `contributorCount: 0`, `fundingPercentage: 0.00` | `contributions` table does not exist until feat-005; explicitly documented as stub |
| Public-visible statuses | `['live', 'funded']` only | Brief AC specifies this; narrower than existing authenticated `getCampaign` access |
| Anonymous 404 vs 403 | Return `404 NOT_FOUND` for non-public or non-existent campaigns | Security-correct: do not reveal existence of non-public campaigns |
| Creator name null fallback | Return `creatorName: null` when `users.display_name` is null | Frontend renders "Creator" as fallback; avoids exposing email (PII) |
| Funded campaign CTA | "Back This Mission" CTA remains active for `funded` campaigns | Per L4-002 Section 7.4: funded campaigns continue accepting contributions until deadline or cap |
| Category stats stub | Return `totalRaisedCents: '0'` and `contributorCount: 0` in category stats | No contributions table yet; documented in API contract |
| Default sort (no query term) | `launched_at DESC` (newest) | Brief does not specify; newest is the most logical browsable default |
| Status filter API values | `active` → `live`, `funded` → `funded`, `ending_soon` → `live`/`funded` AND deadline within 7 days | API filter values decouple from internal domain status names |
| `search_vector` tsvector contents | `title \|\| short_description \|\| description` only (no creator name) | Creator name match done at query time via JOIN to avoid trigger JOIN complexity (G-038) |

## User Stories & Acceptance Criteria

### US-001: Browse and Search Campaigns

**As a** potential mission backer (anonymous or authenticated)
**I want to** search and browse live campaigns with full-text search, filters, and sorting
**So that** I can find Mars mission projects that match my interests

**Acceptance Criteria:**

- **Given** no authentication **When** `GET /api/v1/public/campaigns` is called **Then** the response is `200 OK` with paginated campaign list (anonymous access confirmed)
- **Given** an authenticated user **When** `GET /api/v1/public/campaigns` is called **Then** the response is `200 OK` (same as anonymous — auth not required nor checked)
- **Given** a request with `q=nuclear+propulsion` **When** processed **Then** results include campaigns whose title, short description, description, or creator name match the search terms, ordered by FTS relevance score descending, then `launched_at DESC` as tiebreaker
- **Given** a request with `q=%20%20` (whitespace only, trims to empty) **When** processed **Then** results are sorted by `launched_at DESC` (default sort, no FTS filtering)
- **Given** a request with no `q` parameter **When** processed **Then** all public campaigns are returned sorted by `launched_at DESC`
- **Given** a request with `category=propulsion` **When** processed **Then** only campaigns in the `propulsion` category are returned
- **Given** a request with `category=propulsion&category=power_energy` **When** processed **Then** campaigns in either the `propulsion` or `power_energy` categories are returned
- **Given** a request with `category=invalid_value` **When** processed **Then** the response is `400 VALIDATION_ERROR`
- **Given** a request with `status=active` **When** processed **Then** only campaigns with internal status `live` are returned
- **Given** a request with `status=funded` **When** processed **Then** only campaigns with internal status `funded` are returned
- **Given** a request with `status=ending_soon` **When** processed **Then** only campaigns with deadline within 7 days from server time AND status in `['live', 'funded']` are returned, ordered by deadline ASC
- **Given** a request with `sort=newest` **When** processed **Then** campaigns are ordered by `launched_at DESC`
- **Given** a request with `sort=ending_soon` **When** processed **Then** campaigns are ordered by `deadline ASC` (nearest deadline first); only campaigns with a non-null deadline are included; campaigns where deadline is null are excluded from this sort
- **Given** a request with `sort=most_funded` **When** processed **Then** campaigns are ordered by `funding_goal_cents DESC` (stub until contributions table exists in feat-005)
- **Given** a request with `sort=least_funded` **When** processed **Then** campaigns are ordered by `funding_goal_cents ASC` (stub until contributions table exists)
- **Given** a request with `sort=invalid_value` **When** processed **Then** the response is `400 VALIDATION_ERROR`
- **Given** a request with `limit=5&offset=0` **When** processed **Then** at most 5 campaigns are returned
- **Given** a request with `limit=101` **When** processed **Then** the response is `400 VALIDATION_ERROR` — max limit is 100
- **Given** a request with `limit=-1` **When** processed **Then** the response is `400 VALIDATION_ERROR` — limit must be a non-negative integer
- **Given** the results **Then** each item includes: `id`, `title`, `creatorName`, `category`, `heroImageUrl`, `shortDescription`, `status`, `fundingGoalCents`, `totalRaisedCents`, `fundingPercentage`, `contributorCount`, `deadline`, `daysRemaining`, `launchedAt`
- **Given** no campaigns match the filters **When** processed **Then** the response is `200 OK` with `{ "data": [], "pagination": { "total": 0, "limit": 20, "offset": 0 } }`
- **Given** campaigns in `draft`, `submitted`, `under_review`, `approved`, `rejected`, `suspended`, `failed`, `settlement`, `complete`, `cancelled`, `archived` statuses **When** the public endpoint is called **Then** those campaigns do NOT appear in the results

### US-002: View Public Campaign Detail

**As a** potential mission backer (anonymous or authenticated)
**I want to** view the full details of a live campaign
**So that** I can evaluate the project, read the team profile, milestone plan, and risk disclosures before contributing

**Acceptance Criteria:**

- **Given** no authentication **When** `GET /api/v1/public/campaigns/:id` is called for a `live` campaign **Then** the response is `200 OK` with full campaign data including milestones, team members, risk disclosures, budget breakdown
- **Given** no authentication **When** `GET /api/v1/public/campaigns/:id` is called for a `funded` campaign **Then** the response is `200 OK` — funded campaigns remain publicly accessible
- **Given** `GET /api/v1/public/campaigns/:id` for a campaign in any non-public status (e.g., `draft`, `suspended`, `rejected`) **When** processed **Then** the response is `404 NOT_FOUND` (do not reveal status or existence)
- **Given** `GET /api/v1/public/campaigns/:id` for a non-existent UUID **When** processed **Then** the response is `404 NOT_FOUND`
- **Given** `GET /api/v1/public/campaigns/:id` for a valid `live` campaign **Then** the response includes: all campaign fields from US-001 plus `description`, `milestones`, `teamMembers`, `riskDisclosures`, `budgetBreakdown`, `alignmentStatement`, `tags`, `fundingCapCents`
- **Given** a `funded` campaign **Then** the response includes `status: 'funded'` and the "Fully Funded" badge signal is conveyed by the status field; the `totalRaisedCents` field is `'0'` in feat-004 (stub)
- **Given** a `funded` campaign where `totalRaisedCents` (future feat-005) exceeds `fundingCapCents` **Then** the `totalRaisedCents` value exceeds `fundingCapCents` — backend returns the actual amount, no capping at 100%
- **Given** a `live` campaign with `fundingGoalCents: null` (data anomaly) **Then** the response includes `fundingPercentage: null` and `totalRaisedCents: '0'` — backend guards division by zero

### US-003: Browse by Category

**As a** potential mission backer
**I want to** see aggregate statistics for a category of campaigns
**So that** I can understand the breadth and momentum of missions in that area

**Acceptance Criteria:**

- **Given** `GET /api/v1/public/campaigns/stats?category=propulsion` **When** processed **Then** the response is `200 OK` with `{ "category": "propulsion", "campaignCount": N, "activeCampaignCount": M, "totalRaisedCents": "0", "contributorCount": 0 }`
- **Given** a valid category with no live or funded campaigns **When** stats are requested **Then** `campaignCount: 0`, `activeCampaignCount: 0`, `totalRaisedCents: '0'`, `contributorCount: 0`
- **Given** a `category` value not in the 10-category enum **When** stats are requested **Then** the response is `400 VALIDATION_ERROR`
- **Given** a request with no `category` parameter **When** stats are requested **Then** the response is `400 VALIDATION_ERROR` — category is required for stats endpoint
- **Given** the stats response **Then** `totalRaisedCents` is `'0'` and `contributorCount` is `0` for feat-004 (stub pending feat-005) — this is documented behaviour, not a bug

## Data Model

### Migration

**File:** `db/migrations/20260305160000_campaign_search_vector.sql`

This migration adds full-text search infrastructure and missing performance indexes to the
existing `campaigns` table.

#### `-- migrate:up` operations

```sql
BEGIN;

-- 1. Add search_vector column for pre-computed full-text search tokens
ALTER TABLE campaigns
  ADD COLUMN IF NOT EXISTS search_vector TSVECTOR;

-- 2. Backfill search_vector for existing rows
UPDATE campaigns
SET search_vector = to_tsvector(
  'english',
  COALESCE(title, '') || ' ' ||
  COALESCE(short_description, '') || ' ' ||
  COALESCE(description, '')
);

-- 3. GIN index for fast full-text search
CREATE INDEX IF NOT EXISTS idx_campaigns_search_vector
  ON campaigns USING GIN (search_vector);

-- 4. Index for "ending_soon" sort and filter
CREATE INDEX IF NOT EXISTS idx_campaigns_deadline
  ON campaigns (deadline)
  WHERE deadline IS NOT NULL;

-- 5. Index for "newest" sort
CREATE INDEX IF NOT EXISTS idx_campaigns_launched_at
  ON campaigns (launched_at)
  WHERE launched_at IS NOT NULL;

-- 6. Index for category browse and filter
CREATE INDEX IF NOT EXISTS idx_campaigns_category
  ON campaigns (category)
  WHERE category IS NOT NULL;

-- 7. Trigger function: auto-update search_vector on INSERT or UPDATE
CREATE OR REPLACE FUNCTION campaigns_search_vector_update()
RETURNS TRIGGER AS $$
BEGIN
  NEW.search_vector := to_tsvector(
    'english',
    COALESCE(NEW.title, '') || ' ' ||
    COALESCE(NEW.short_description, '') || ' ' ||
    COALESCE(NEW.description, '')
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- 8. Attach trigger
DROP TRIGGER IF EXISTS campaigns_search_vector_trigger ON campaigns;
CREATE TRIGGER campaigns_search_vector_trigger
  BEFORE INSERT OR UPDATE ON campaigns
  FOR EACH ROW EXECUTE FUNCTION campaigns_search_vector_update();

COMMIT;
```

#### `-- migrate:down` operations

```sql
BEGIN;
DROP TRIGGER IF EXISTS campaigns_search_vector_trigger ON campaigns;
DROP FUNCTION IF EXISTS campaigns_search_vector_update();
DROP INDEX IF EXISTS idx_campaigns_category;
DROP INDEX IF EXISTS idx_campaigns_launched_at;
DROP INDEX IF EXISTS idx_campaigns_deadline;
DROP INDEX IF EXISTS idx_campaigns_search_vector;
ALTER TABLE campaigns DROP COLUMN IF EXISTS search_vector;
COMMIT;
```

#### Notes on search_vector design

- The trigger fires on every INSERT/UPDATE to `campaigns`, including draft auto-saves (G-007 pattern).
  This adds a small write overhead on every PATCH but is negligible for the demo dataset.
- Creator name is NOT included in the pre-computed `search_vector`.
  Creator name matching is done at query time via `to_tsvector('english', COALESCE(u.display_name, ''))`.
  This is slower than an indexed column but avoids the trigger JOIN complexity (G-038).
- The `search_vector` column is nullable for backward compatibility with existing rows until backfill runs.
  The backfill UPDATE in migration step 2 populates it for all existing rows before the trigger takes over.
