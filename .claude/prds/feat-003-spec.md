# PRD: feat-003 --- Authentication with Clerk

> Implementation-ready specification.
> Generated by Spec Writer.
> Feature brief: feat-003-authentication.md
> Research: feat-003-research.md

## Overview

Integrate Clerk as the authentication provider for Mars Mission Fund, covering both frontend (React) and backend (Express).
This feature establishes the auth middleware stack, JWT verification, auth context extraction, protected route patterns, user sync via webhooks and just-in-time creation, and the centralised API client with automatic Bearer token injection.
Every subsequent feature that requires user identity depends on feat-003.

---

## User Stories & Acceptance Criteria

### US-001: Sign Up

**As a** new user
**I want to** create an account using Clerk's sign-up flow
**So that** I can access the Mars Mission Fund platform

**Acceptance Criteria:**

- **Given** an unauthenticated user **When** they navigate to `/sign-up` **Then** Clerk's `<SignUp />` component is rendered with the MMF branding context.
- **Given** a user completing sign-up **When** Clerk confirms the account is created **Then** the user is redirected to `/onboarding` (placeholder route for feat-004).
- **Given** a user completing sign-up **When** the first authenticated API request is made **Then** an `accounts` row is created via JIT (just-in-time) creation with `status = 'active'`, `roles = '{backer}'`, and `onboarding_completed = false`.
- **Given** a user who has already signed up with a given email **When** another user attempts to sign up with the same email **Then** Clerk handles the duplicate rejection (MMF does not reimplement this).

### US-002: Sign In

**As a** returning user
**I want to** sign in to my account
**So that** I can access my dashboard and platform features

**Acceptance Criteria:**

- **Given** an unauthenticated user **When** they navigate to `/sign-in` **Then** Clerk's `<SignIn />` component is rendered.
- **Given** a user who successfully signs in **When** they have `onboarding_completed = false` **Then** they are redirected to `/onboarding`.
- **Given** a user who successfully signs in **When** they have `onboarding_completed = true` **Then** they are redirected to `/` (app home page).
- **Given** a user who has been suspended or deactivated **When** they sign in with a valid Clerk session **Then** the backend returns `403 FORBIDDEN` with `{ error: { code: "ACCOUNT_SUSPENDED", message: "Your account has been suspended. Please contact support." } }` and the frontend displays the error and signs the user out.

### US-003: Protected Routes (Frontend)

**As a** platform operator
**I want** unauthenticated users to be redirected to the sign-in page when accessing protected routes
**So that** only authenticated users can interact with the platform

**Acceptance Criteria:**

- **Given** an unauthenticated user **When** they navigate to any route other than `/sign-in`, `/sign-up`, or `/` (public landing) **Then** they are redirected to `/sign-in`.
- **Given** an authenticated user **When** they navigate to `/sign-in` or `/sign-up` **Then** they are redirected to `/` (home).
- **Given** an authenticated user **When** the Clerk session expires or is revoked **Then** they are redirected to `/sign-in` on the next navigation or API call.

### US-004: Protected Endpoints (Backend)

**As a** platform operator
**I want** all API endpoints (except `/health`) to require a valid Clerk JWT
**So that** unauthenticated requests are rejected

**Acceptance Criteria:**

- **Given** a request without an `Authorization` header **When** it hits any endpoint except `GET /health` **Then** the response is `401` with `{ error: { code: "UNAUTHENTICATED", message: "Authentication required.", correlation_id: "<req.id>" } }`.
- **Given** a request with an expired or invalid JWT **When** it hits a protected endpoint **Then** the response is `401` with the same error format.
- **Given** a request with a valid JWT **When** the corresponding MMF account has status `suspended` or `deactivated` **Then** the response is `403` with `{ error: { code: "ACCOUNT_SUSPENDED", message: "Your account has been suspended. Please contact support.", correlation_id: "<req.id>" } }`.
- **Given** a request with a valid JWT **When** the corresponding MMF account has status `deleted` **Then** the response is `403` with `{ error: { code: "ACCOUNT_DELETED", message: "This account has been deleted.", correlation_id: "<req.id>" } }`.
- **Given** a request to `GET /health` **When** no auth token is present **Then** the response is `200` with `{ status: "ok" }`.

### US-005: Authenticated API Client (Frontend)

**As a** frontend developer
**I want** a centralised API client that automatically attaches the Clerk Bearer token
**So that** every API call is authenticated without manual token management

**Acceptance Criteria:**

- **Given** an authenticated user **When** any API call is made via the centralised client **Then** the `Authorization: Bearer <token>` header is included automatically.
- **Given** an API call that returns `401` **When** the frontend receives the response **Then** the user is redirected to `/sign-in`.
- **Given** a user whose token has expired **When** an API call is made **Then** Clerk's `getToken()` fetches a refreshed token before the request is sent.

### US-006: Webhook User Sync

**As a** platform operator
**I want** Clerk webhook events to sync user data to the MMF accounts table
**So that** account records stay consistent with Clerk's identity data

**Acceptance Criteria:**

- **Given** a `user.created` webhook event **When** the handler receives it **Then** an `accounts` row is created (or no-op if it already exists via JIT) with `clerk_user_id`, `email`, `display_name`, `status = 'active'`, and `roles = '{backer}'`.
- **Given** a `user.updated` webhook event **When** the handler receives it **Then** the `accounts` row is updated with the new `email` and `display_name` (upsert if not found).
- **Given** a `user.deleted` webhook event **When** the handler receives it **Then** the `accounts` row status is set to `'deleted'`.
- **Given** a webhook request with an invalid Svix signature **When** the handler receives it **Then** the response is `401` and no database changes occur.
- **Given** a duplicate webhook delivery **When** the same `user.created` event arrives twice **Then** the second delivery is a no-op (idempotent via `ON CONFLICT`).

### US-007: Mock Auth Adapter

**As a** developer running the local demo
**I want** a mock auth adapter activated by `MOCK_AUTH=true`
**So that** I can develop and test without a real Clerk account

**Acceptance Criteria:**

- **Given** `MOCK_AUTH=true` in the environment **When** the backend starts **Then** the Clerk middleware is bypassed and a mock middleware injects a fake auth context with `userId = 'user_mock_001'`.
- **Given** `MOCK_AUTH=true` **When** any protected endpoint is called without a token **Then** the request is treated as authenticated with the mock user.
- **Given** `MOCK_AUTH=false` or unset **When** the backend starts **Then** the real Clerk middleware is active.

#### AuthPort

**File:** `packages/backend/src/account/ports/auth-port.ts`

This port abstracts the authentication provider (Clerk in production, mock in development).

| Method | Params | Returns | Description |
|--------|--------|---------|-------------|
| verifyToken | token: string | Promise\<AuthPayload \| null\> | Verify a JWT and extract claims. Returns null if invalid. |
| getMiddleware | --- | express.RequestHandler | Returns the Express middleware for JWT parsing |

**Type:** `AuthPayload = { clerkUserId: string; sessionId: string }`

**Mock adapter behaviour (`MOCK_AUTH=true`):**

- `verifyToken(token)`: Always returns `{ clerkUserId: 'user_mock_001', sessionId: 'sess_mock_001' }` regardless of input.
- `getMiddleware()`: Returns a no-op middleware that attaches `req.auth = { userId: 'user_mock_001', sessionId: 'sess_mock_001' }` to every request.

**Clerk adapter behaviour (`MOCK_AUTH=false`):**

- `verifyToken(token)`: Delegates to Clerk SDK to verify JWT signature and expiry. Returns extracted claims or null.
- `getMiddleware()`: Returns `clerkMiddleware()` from `@clerk/express`.

#### WebhookVerificationPort

**File:** `packages/backend/src/account/ports/webhook-verification-port.ts`

| Method | Params | Returns | Description |
|--------|--------|---------|-------------|
| verifyWebhookSignature | payload: string, headers: Record\<string, string\> | WebhookEvent | Verify Svix signature and parse event. Throws on invalid signature. |

**Type:** `WebhookEvent = { type: 'user.created' | 'user.updated' | 'user.deleted'; data: WebhookUserData }`

**Type:** `WebhookUserData = { id: string; email_addresses: Array<{ email_address: string }>; first_name: string | null; last_name: string | null }`

**Mock adapter behaviour:**

- `verifyWebhookSignature(payload, headers)`: Parses the payload as JSON without signature verification. Useful for local testing.

**Clerk adapter behaviour:**

- `verifyWebhookSignature(payload, headers)`: Uses the `svix` package with `CLERK_WEBHOOK_SIGNING_SECRET` to verify HMAC-SHA256 signature. Throws `AuthenticationError` on invalid signature.

## Application Service

### AccountAppService

**File:** `packages/backend/src/account/application/account-app-service.ts`

**Dependencies (injected):**

- `AccountRepository`

**Methods:**

#### `findOrCreateAccount(clerkUserId: string, email: string, displayName: string | null): Promise<Account>`

1. Call `accountRepository.findByClerkUserId(clerkUserId)`.
2. If found, return the existing account.
3. If not found, create a new `Account` via `Account.create({ clerkUserId, email, displayName })`.
4. Call `accountRepository.save(account)`.
5. Handle `ON CONFLICT` race condition: if save fails due to unique constraint, retry `findByClerkUserId`.
6. Return the account.

**Error handling:**

- Database unique constraint violation on concurrent JIT creation --- retry with find (handled at repository level via `INSERT ... ON CONFLICT`).

#### `getAccountByClerkUserId(clerkUserId: string): Promise<Account | null>`

1. Call `accountRepository.findByClerkUserId(clerkUserId)`.
2. Return the account or null.

#### `handleWebhookEvent(event: WebhookEvent): Promise<void>`

1. Switch on `event.type`:
   - `user.created`: Extract primary email, first/last name. Call `accountRepository.upsertFromWebhook({ clerkUserId: event.data.id, email, displayName })`.
   - `user.updated`: Same as `user.created` (upsert handles both).
   - `user.deleted`: Call `accountRepository.updateStatusByClerkUserId(event.data.id, 'deleted')`.
2. Log the event type and clerk user ID at `info` level.

**Error handling:**

- Unknown event type --- log a warning and return (do not throw; Clerk may send events we do not handle yet).
- Database errors --- let them propagate (Clerk will retry delivery).

## API Endpoints

### Middleware Stack Order (in `app.ts`)

1. `pinoHttp` --- request logging (already present)
2. `express.json()` --- body parsing (already present)
3. `authPort.getMiddleware()` --- Clerk JWT parsing (attaches `req.auth`)
4. Routes:
   - `healthRouter` --- `/health` (public, no auth check)
   - `webhookRouter` --- `/api/webhooks/clerk` (public, verified via Svix signature, NOT via JWT)
   - All other routes --- protected via `requireAuthentication` middleware

### Middleware

#### `requireAuthentication`

**File:** `packages/backend/src/shared/middleware/require-authentication.ts`

**Description:** Checks that the request has a valid authenticated user. Applied to all routes except `/health` and `/api/webhooks/clerk`.

**Behaviour:**

1. Extract auth from request (via `getAuth(req)` from `@clerk/express` or from `req.auth` in mock mode).
2. If `userId` is null or undefined, return `401`:
   ```json
   {
     "error": {
       "code": "UNAUTHENTICATED",
       "message": "Authentication required.",
       "correlation_id": "<req.id>"
     }
   }
   ```
3. If `userId` is present, call `next()`.

#### `enrichAuthContext`

**File:** `packages/backend/src/shared/middleware/enrich-auth-context.ts`

**Description:** After `requireAuthentication`, looks up the MMF account by Clerk user ID. Creates the account via JIT if not found. Checks account status. Attaches `AuthContext` to the request.

**Behaviour:**

1. Extract `clerkUserId` from the auth object on the request.
2. Call `accountAppService.findOrCreateAccount(clerkUserId, emailFromClaims, displayNameFromClaims)`.
   - Note: For JIT creation, email and display name come from the Clerk JWT claims (`sessionClaims`). If claims do not contain email (unlikely but defensive), use a placeholder and let the webhook update it.
3. If the account status is `'suspended'` or `'deactivated'`, return `403`:
   ```json
   {
     "error": {
       "code": "ACCOUNT_SUSPENDED",
       "message": "Your account has been suspended. Please contact support.",
       "correlation_id": "<req.id>"
     }
   }
   ```
4. If the account status is `'deleted'`, return `403`:
   ```json
   {
     "error": {
       "code": "ACCOUNT_DELETED",
       "message": "This account has been deleted.",
       "correlation_id": "<req.id>"
     }
   }
   ```
5. Attach `AuthContext` to the request object:
   ```typescript
   interface AuthContext {
     readonly userId: string;        // MMF internal account UUID
     readonly clerkUserId: string;    // Clerk user ID
     readonly email: string;
     readonly roles: readonly string[];
     readonly accountStatus: AccountStatus;
   }
   ```
6. Call `next()`.

**Type augmentation:** Extend the Express `Request` type to include `authContext: AuthContext`.

**File:** `packages/backend/src/shared/types/express.d.ts`

```typescript
declare namespace Express {
  interface Request {
    authContext?: AuthContext;
  }
}
```

### Endpoints

#### `POST /api/webhooks/clerk`

**Description:** Receives Clerk webhook events (user.created, user.updated, user.deleted) and syncs to the accounts table.
**Auth:** NOT authenticated via JWT. Verified via Svix webhook signature.
**Rate limit:** Standard webhook rate limiting.

**Request headers (Svix):**

- `svix-id`: Webhook event ID
- `svix-timestamp`: Event timestamp
- `svix-signature`: HMAC-SHA256 signature

**Request body:** Raw JSON (Clerk webhook event payload).

**Success response:** `200 OK`

```json
{
  "received": true
}
```

**Error responses:**

| Status | Code | When |
|--------|------|------|
| 401 | WEBHOOK_SIGNATURE_INVALID | Svix signature verification fails |
| 500 | INTERNAL_ERROR | Unexpected server error |

**Implementation notes:**

- The webhook route must use `express.raw({ type: 'application/json' })` for signature verification (Svix needs the raw body).
- Alternatively, use a middleware that captures the raw body alongside `express.json()`.
- This route must be mounted BEFORE the `requireAuthentication` middleware in the stack, or exempted from it.

#### `GET /api/v1/auth/me`

**Description:** Returns the current authenticated user's account information. Used by the frontend after sign-in to determine redirect destination (onboarding vs home).
**Auth:** Required (Clerk JWT).

**Success response:** `200 OK`

```json
{
  "data": {
    "id": "uuid",
    "email": "user@example.com",
    "display_name": "Jane Doe",
    "status": "active",
    "roles": ["backer"],
    "onboarding_completed": false
  }
}
```

**Error responses:**

| Status | Code | When |
|--------|------|------|
| 401 | UNAUTHENTICATED | No valid Clerk JWT |
| 403 | ACCOUNT_SUSPENDED | Account is suspended or deactivated |
| 403 | ACCOUNT_DELETED | Account has been deleted |

**Implementation notes:**

- Returns the account data already loaded by `enrichAuthContext` middleware.
- No additional database query needed.

## Edge Cases

| # | Scenario | Expected Behaviour | Test Type |
|---|----------|-------------------|-----------|
| 1 | Clerk user ID format changes (not `user_xxxx` prefix) | `clerk_user_id` is `TEXT` with no format validation. Accepts any non-empty string. | Unit |
| 2 | Email casing mismatch | Normalise email to lowercase before INSERT. Clerk already normalises; this is a defensive measure. | Unit |
| 3 | Null display name (SSO users without name) | `display_name` is nullable. Frontend falls back to email prefix or "Anonymous". | Unit (frontend) |
| 4 | Race condition: webhook and JIT both try to create account | `INSERT ... ON CONFLICT (clerk_user_id) DO UPDATE` ensures idempotent upsert. Second INSERT becomes an update. | Integration |
| 5 | Empty roles array set accidentally | `Account.create()` always sets `['backer']`. `reconstitute()` trusts DB data which defaults to `'{backer}'`. | Unit |
| 6 | Account suspended but Clerk session valid | `enrichAuthContext` middleware checks account status and returns `403` even though JWT is valid. | Integration |
| 7 | Clerk JWKS endpoint unreachable on cold start | Requests fail with 401 until JWKS is cached. `/health` remains functional (no auth). Log error at `warn` level. | Manual |
| 8 | JWT clock skew | Use `clerkMiddleware()` default `clockSkewInMs: 5000`. Sufficient for NTP-synced hosts. | Manual |
| 9 | Webhook signature verification failure (misconfigured secret) | Return `401 WEBHOOK_SIGNATURE_INVALID`. Log at `error` level with correlation ID. No DB changes. | Integration |
| 10 | Duplicate webhook delivery (at-least-once) | `INSERT ... ON CONFLICT` makes handler idempotent. Second delivery is a no-op or harmless update. | Integration |
| 11 | `user.updated` arrives before `user.created` (webhook ordering) | Use upsert pattern (`INSERT ... ON CONFLICT DO UPDATE`). Handles both cases. | Integration |
| 12 | Stale Clerk session after MMF account deletion | `enrichAuthContext` checks status. Deleted accounts get `403 ACCOUNT_DELETED`. | Integration |
| 13 | Clerk publishable key mismatch frontend/backend | Tokens fail validation. 401 on all requests. Documented in `.env.example` that both must use same Clerk app. | Manual |
| 14 | Missing `CLERK_SECRET_KEY` environment variable | Backend fails fast at startup with clear error message: "CLERK_SECRET_KEY is required". | Integration |
| 15 | CORS with Bearer token | Backend CORS config must include `Authorization` in `Access-Control-Allow-Headers`. Already in scope for existing CORS setup. | Integration |
| 16 | Rapid sign-up then sign-in without email verification | Clerk handles this. MMF only creates account row on first authenticated request (JIT), which implies Clerk verification passed. | Manual |
| 17 | Multiple tabs, sign-out on one | Clerk session syncing handles cross-tab state. Other tabs detect sign-out and redirect to `/sign-in`. | Manual |
| 18 | Browser back button after sign-out | `Cache-Control: no-store` header prevents cached authenticated content from displaying. | Integration |
| 19 | Very long Clerk user ID or email | `TEXT` column has no practical limit. Application layer does not impose length validation beyond non-empty check. | Unit |
| 20 | Concurrent first requests for same user (JIT race) | `ON CONFLICT` at database level. Both requests succeed; one creates, one finds the existing row. | Integration |
| 21 | Token passed in URL query string | Never supported. API client always uses `Authorization` header. No server-side support for query string tokens. | Code review |
| 22 | User signs up but abandons before onboarding | Account exists with `onboarding_completed = false`. On next login, frontend redirects to `/onboarding`. | Integration (frontend) |
| 23 | `user.deleted` webhook for non-existent account | `UPDATE` with `WHERE clerk_user_id = $1` affects zero rows. No error, no-op. Log at `warn` level. | Integration |
| 24 | `MOCK_AUTH=true` set in production | This is a configuration error. Document in `.env.example` that `MOCK_AUTH` must ONLY be `true` in development. No runtime guard (workshop scope). | Documentation |

## Adapter Implementations

### ClerkAuthAdapter

**File:** `packages/backend/src/account/adapters/clerk-auth-adapter.ts`

**Implements:** `AuthPort`

- Uses `@clerk/express` `clerkMiddleware()` and `getAuth()`.
- `getMiddleware()`: Returns `clerkMiddleware()`.
- `verifyToken(token)`: Uses Clerk SDK to verify. Returns `AuthPayload` or null.

### MockAuthAdapter

**File:** `packages/backend/src/account/adapters/mock-auth-adapter.ts`

**Implements:** `AuthPort`

- `getMiddleware()`: Returns middleware that sets `req.auth = { userId: 'user_mock_001', sessionId: 'sess_mock_001', sessionClaims: { email: 'mock@example.com', firstName: 'Mock', lastName: 'User' } }`.
- `verifyToken(token)`: Returns `{ clerkUserId: 'user_mock_001', sessionId: 'sess_mock_001' }`.

### ClerkWebhookVerificationAdapter

**File:** `packages/backend/src/account/adapters/clerk-webhook-verification-adapter.ts`

**Implements:** `WebhookVerificationPort`

- Uses the `svix` npm package with `CLERK_WEBHOOK_SIGNING_SECRET`.
- `verifyWebhookSignature(payload, headers)`: Creates a `Webhook` instance from svix, calls `verify()`. Throws `AuthenticationError` on failure.

### MockWebhookVerificationAdapter

**File:** `packages/backend/src/account/adapters/mock-webhook-verification-adapter.ts`

**Implements:** `WebhookVerificationPort`

- `verifyWebhookSignature(payload, headers)`: Parses `payload` as JSON and returns it directly without signature verification.

### PostgresAccountRepository

**File:** `packages/backend/src/account/adapters/postgres-account-repository.ts`

**Implements:** `AccountRepository`

- Uses raw `pg` with parameterised queries (no ORM).
- `findByClerkUserId(clerkUserId)`:
  ```sql
  SELECT id, clerk_user_id, email, display_name, status, roles, onboarding_completed, created_at, updated_at
  FROM accounts
  WHERE clerk_user_id = $1
  ```
- `save(account)`:
  ```sql
  INSERT INTO accounts (id, clerk_user_id, email, display_name, status, roles, onboarding_completed)
  VALUES ($1, $2, $3, $4, $5, $6, $7)
  ON CONFLICT (clerk_user_id) DO UPDATE SET
    email = EXCLUDED.email,
    display_name = EXCLUDED.display_name,
    updated_at = NOW()
  ```
- `upsertFromWebhook(input)`:
  ```sql
  INSERT INTO accounts (clerk_user_id, email, display_name, status, roles)
  VALUES ($1, $2, $3, 'active', '{backer}')
  ON CONFLICT (clerk_user_id) DO UPDATE SET
    email = EXCLUDED.email,
    display_name = EXCLUDED.display_name,
    updated_at = NOW()
  ```
- `updateStatusByClerkUserId(clerkUserId, status)`:
  ```sql
  UPDATE accounts SET status = $2, updated_at = NOW() WHERE clerk_user_id = $1
  ```

## Open Questions

None.
All decisions have been resolved:

- **JIT vs webhook-only account creation**: JIT is the primary path; webhooks are supplementary for production reliability.
- **`@clerk/clerk-react` vs `@clerk/react-router`**: Use `@clerk/clerk-react` with manual routing via `react-router-dom`. Simpler, fewer dependencies, and sufficient for MMF's routing needs.
- **Cookie vs Bearer token auth**: Bearer token via `Authorization` header. Required for cross-origin dev setup (frontend port 3000, backend port 3001).
- **Mock adapter strategy**: `MOCK_AUTH=true` environment variable switches between real and mock auth adapters at the dependency injection level.
