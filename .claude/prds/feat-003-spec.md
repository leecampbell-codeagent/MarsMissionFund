## Overview

feat-003 implements the campaign lifecycle from Draft through to Live: a verified Creator builds a campaign proposal, submits it for review, a Reviewer approves or rejects it, and the Creator launches an approved campaign to the Live state. This feature establishes the `campaign` bounded context, the `campaigns` and `campaign_audit_events` tables, the Creator role self-designation endpoint, and all review pipeline API endpoints. Nothing can be funded or discovered on MMF until a campaign passes this review pipeline — feat-003 is the platform's curation gatekeeper.

## Key Decisions (Resolved)

| Decision | Resolution | Rationale |
|----------|-----------|-----------|
| Creator role self-designation | `POST /api/v1/me/roles/creator` — any active, KYC-verified user can self-assign | Explicit opt-in; consistent with L4-001 role assignment pattern |
| KYC gate scope | BOTH `POST /campaigns` (create) AND `POST /:id/submit` require `kycStatus='verified'` | Feature brief AC says "Verified KYC and Creator role can create"; consistent with access control matrix |
| Milestone precision | Basis points (integers, must sum to 10000) | Avoids 33+33+34 problem with equal splits (G-025) |
| Hero image | External `https://` URL only (TEXT, optional at draft time, required at submission) | No CDN/upload for feat-003; simplest viable approach |
| Campaign descriptions | Plain text only, max 10,000 characters | XSS protection (G-028); no rich text for local demo |
| Draft auto-save | `PATCH /api/v1/campaigns/:id` — partial update, lenient validation (G-026) | Separates auto-save semantics from submission semantics |
| Draft submission | `POST /api/v1/campaigns/:id/submit` — full strict validation | All required fields + business rules checked at submission |
| Milestone/team/budget storage | JSONB arrays on `campaigns` table | Sufficient for feat-003; separate tables deferred to milestone verification feature |
| `creator_user_id` ON DELETE | `ON DELETE RESTRICT` | Prevents account deletion while campaigns are active (G-027) |
| Review queue ordering | FIFO by `submitted_at ASC` | Oldest-submitted first; matches L4-002 Section 5.1 |
| Notifications | Audit events only — no email delivery (SES is theatre for feat-003) | Out of scope per feature brief |
| Approved campaign revision | Not allowed — creator must submit a new campaign or contact Admin | L4-002 does not define Approved → Draft; prevents review bypass |
| Draft expiry | None — drafts do not auto-delete | L4-002 Section 4.3: no expiry stated |
| Simultaneous draft limit | None — creators may have unlimited drafts | L4-002 Section 4.3: no limit stated |
| Reviewer voluntary recusal | **Out of scope for feat-003.** No `/recuse` endpoint. A Reviewer who needs to recuse must ask an Admin to use `POST /:id/reassign`. | L4-002 does not specify a self-service recusal endpoint; adding one is deferred to post-MVP admin tooling |

## Bounded Context

**Campaign** — new bounded context at `packages/backend/src/campaign/`.

Cross-context interactions:
- **Campaign → Account**: Campaign app service reads `User` via `UserRepository` port to check `roles` and `kycStatus`. Campaign app service writes to `users.roles` via `AccountAppService.assignCreatorRole()` for the Creator role endpoint.
- **Campaign → KYC**: Direct — reads `user.kycStatus` from `UserRepository`. Does NOT call `KycAppService`.

**Creator role endpoint** (`POST /api/v1/me/roles/creator`) lives in the **Account** bounded context (writes to `users.roles`). It is routed under `/me` alongside other account endpoints.

## User Stories & Acceptance Criteria

### US-001: Self-Designate as Creator

**As a** user with verified KYC and an active account
**I want to** designate myself as a project creator
**So that** I can create and submit campaign proposals

**Acceptance Criteria:**

- **Given** an authenticated user with `accountStatus='active'` and `kycStatus='verified'` **When** they call `POST /api/v1/me/roles/creator` **Then** the response is `200 OK` with the updated user profile containing `'creator'` in the `roles` array
- **Given** an authenticated user who already has the `creator` role **When** they call `POST /api/v1/me/roles/creator` **Then** the response is `200 OK` with unchanged profile (idempotent — no error, no duplicate role)
- **Given** an authenticated user with `kycStatus='not_started'` **When** they call `POST /api/v1/me/roles/creator` **Then** the response is `403 KYC_NOT_VERIFIED`
- **Given** an authenticated user with `kycStatus='pending'` **When** they call `POST /api/v1/me/roles/creator` **Then** the response is `403 KYC_NOT_VERIFIED`
- **Given** an authenticated user with `accountStatus='pending_verification'` **When** they call `POST /api/v1/me/roles/creator` **Then** the response is `403 ACCOUNT_NOT_ACTIVE`
- **Given** an authenticated user with `accountStatus='suspended'` **When** they call `POST /api/v1/me/roles/creator` **Then** the response is `403 ACCOUNT_SUSPENDED`
- **Given** the role is successfully assigned **Then** an `AuditAction.RoleAssigned` audit event is written with `resourceType: 'user'`, actor = the user themselves, `metadata: { role: 'creator' }`

### US-002: Create a Campaign Draft

**As a** Creator with verified KYC
**I want to** start a campaign proposal and save it as a draft
**So that** I can build it over multiple sessions before submitting

**Acceptance Criteria:**

- **Given** an authenticated user with `roles=['creator', 'backer']` and `kycStatus='verified'` **When** they call `POST /api/v1/campaigns` with a `title` **Then** the response is `201 Created` with a campaign record in `draft` status containing the provided title and a generated UUID `id`
- **Given** a valid `POST /api/v1/campaigns` request **Then** the created campaign has `creatorUserId` set to the authenticated user's MMF `users.id` (UUID) — never the Clerk user ID
- **Given** a user with `kycStatus='not_started'` and `roles=['creator']` **When** they call `POST /api/v1/campaigns` **Then** the response is `403 KYC_NOT_VERIFIED`
- **Given** a user with `roles=['backer']` (no creator role) and `kycStatus='verified'` **When** they call `POST /api/v1/campaigns` **Then** the response is `403 CREATOR_ROLE_REQUIRED`
- **Given** a valid draft creation **Then** all optional fields not provided default to `null` or `[]` as appropriate
- **Given** a title of `"   "` (whitespace only) **When** `POST /api/v1/campaigns` is called **Then** the response is `400 VALIDATION_ERROR` — title cannot be whitespace-only

### US-003: Auto-Save Campaign Draft

**As a** Creator
**I want** my campaign draft to be auto-saved as I fill in each field
**So that** I never lose work in progress

**Acceptance Criteria:**

- **Given** an authenticated Creator who owns a campaign in `draft` status **When** they call `PATCH /api/v1/campaigns/:id` with partial fields **Then** the response is `200 OK` with the updated campaign
- **Given** an authenticated Creator who owns a campaign in `rejected` status **When** they call `PATCH /api/v1/campaigns/:id` **Then** the response is `200 OK` (rejected campaigns can be revised)
- **Given** a PATCH request with `description` exceeding 10,000 characters **When** processed **Then** the response is `400 VALIDATION_ERROR` — structural limits apply at draft-save time (G-026)
- **Given** a PATCH request with a `heroImageUrl` using `http://` (not `https://`) **When** processed **Then** the response is `400 VALIDATION_ERROR` — URL must use `https://` protocol
- **Given** a PATCH request on a campaign in `submitted` status **When** processed **Then** the response is `409 CAMPAIGN_NOT_EDITABLE` — submitted campaigns cannot be auto-saved
- **Given** a PATCH request on a campaign in `under_review` status **When** processed **Then** the response is `409 CAMPAIGN_NOT_EDITABLE`
- **Given** a Creator requesting PATCH on a campaign owned by a different Creator **When** processed **Then** the response is `404 NOT_FOUND` (do not reveal existence)
- **Given** a PATCH request from a user who is not the campaign creator **When** processed **Then** the response is `404 NOT_FOUND`

### US-004: Submit Campaign for Review

**As a** Creator
**I want to** submit my completed campaign proposal for review
**So that** a Reviewer can evaluate it for approval

**Acceptance Criteria:**

- **Given** an authenticated Creator with `kycStatus='verified'` who owns a draft campaign with all required fields filled **When** they call `POST /api/v1/campaigns/:id/submit` **Then** the response is `200 OK` with the campaign in `submitted` status and `submittedAt` set to the current timestamp
- **Given** a draft campaign with milestones whose `fundingBasisPoints` sum to 9000 (not 10000) **When** the creator submits **Then** the response is `400 VALIDATION_ERROR` with `{ "field": "milestones", "message": "Milestone funding basis points must sum to 10000. Current sum: 9000." }`
- **Given** a draft campaign with only one milestone (even at basis points 10000) **When** the creator submits **Then** the response is `400 VALIDATION_ERROR` — at least two milestones are required
- **Given** a draft campaign with an empty `teamMembers` array **When** the creator submits **Then** the response is `400 VALIDATION_ERROR` — at least one team member is required
- **Given** a draft campaign with an empty `riskDisclosures` array **When** the creator submits **Then** the response is `400 VALIDATION_ERROR` — at least one risk disclosure is required
- **Given** a draft campaign with `deadline` less than 7 days from the submission timestamp **When** the creator submits **Then** the response is `400 VALIDATION_ERROR` — deadline must be at least 7 days from submission
- **Given** a draft campaign with `deadline` more than 365 days from the submission timestamp **When** the creator submits **Then** the response is `400 VALIDATION_ERROR` — deadline must be within 1 year of submission
- **Given** a draft campaign with `fundingGoalCents` of `"50000000"` (below $1M minimum) **When** the creator submits **Then** the response is `400 VALIDATION_ERROR` with a message stating the minimum is $1,000,000
- **Given** a draft campaign with `fundingCapCents` less than `fundingGoalCents` **When** the creator submits **Then** the response is `400 VALIDATION_ERROR` — funding cap must be >= funding goal
- **Given** a campaign already in `submitted` status (double-submit) **When** the creator submits again **Then** the response is `409 CAMPAIGN_ALREADY_SUBMITTED`
- **Given** a campaign in `draft` status **When** submitted successfully **Then** a `campaign.submitted` audit event is written with `campaignId`, `previousStatus: 'draft'`, `newStatus: 'submitted'`, `actorClerkUserId`, `createdAt`
- **Given** a Creator whose `creator` role was removed after draft creation **When** they attempt to submit **Then** the response is `403 CREATOR_ROLE_REQUIRED` (role check is point-in-time)
- **Given** a Creator whose `kycStatus` changed to `'expired'` after draft creation **When** they attempt to submit **Then** the response is `403 KYC_NOT_VERIFIED` (KYC check is point-in-time)

### US-005: View Campaign Detail

**As a** campaign stakeholder
**I want to** view the full details of a campaign
**So that** I can understand the proposal, its status, and review information

**Acceptance Criteria:**

- **Given** a Creator authenticated with a valid JWT **When** they call `GET /api/v1/campaigns/:id` for their own campaign in any status **Then** the response is `200 OK` with full campaign data
- **Given** a Reviewer or Admin **When** they call `GET /api/v1/campaigns/:id` for a campaign in `submitted`, `under_review`, `approved`, or `rejected` status **Then** the response is `200 OK` with full campaign data including `reviewNotes` and `rejectionReason`
- **Given** a Reviewer or Admin **When** they call `GET /api/v1/campaigns/:id` for a campaign in `draft` status **Then** the response is `404 NOT_FOUND` — Reviewers cannot see drafts
- **Given** a user with no special role **When** they call `GET /api/v1/campaigns/:id` for a campaign not yet `approved` **Then** the response is `404 NOT_FOUND`
- **Given** any authenticated user **When** they call `GET /api/v1/campaigns/:id` for a non-existent campaign ID **Then** the response is `404 NOT_FOUND`
- **Given** a Creator requesting their own rejected campaign **When** `GET /api/v1/campaigns/:id` is called **Then** both `rejectionReason` and `resubmissionGuidance` fields are included in the response (EC-035)

### US-006: List Own Campaigns

**As a** Creator
**I want to** see all of my campaigns and their statuses
**So that** I can track my portfolio and identify which need attention

**Acceptance Criteria:**

- **Given** an authenticated user **When** they call `GET /api/v1/me/campaigns` **Then** the response is `200 OK` with a list of all campaigns where `creatorUserId` equals the authenticated user's MMF UUID
- **Given** a user with no campaigns **When** `GET /api/v1/me/campaigns` is called **Then** the response is `200 OK` with `{ "data": [] }`
- **Given** the list response **Then** campaigns are ordered by `createdAt DESC` (newest first)
- **Given** the list response **Then** each campaign includes: `id`, `title`, `status`, `category`, `fundingGoalCents`, `submittedAt`, `createdAt`, `updatedAt`

### US-007: View Review Queue

**As a** Reviewer or Administrator
**I want to** see campaigns awaiting review in order of submission
**So that** I can process the oldest submissions first

**Acceptance Criteria:**

- **Given** an authenticated Reviewer **When** they call `GET /api/v1/campaigns/review-queue` **Then** the response is `200 OK` with campaigns in `submitted` status ordered by `submittedAt ASC` (FIFO)
- **Given** no campaigns are in `submitted` status **When** `GET /api/v1/campaigns/review-queue` is called **Then** the response is `200 OK` with `{ "data": [] }` (EC-031)
- **Given** a user with only `backer` role **When** they call `GET /api/v1/campaigns/review-queue` **Then** the response is `403 REVIEWER_ROLE_REQUIRED` (EC-032)
- **Given** the review queue response **Then** each campaign includes: `id`, `title`, `category`, `fundingGoalCents`, `submittedAt`, `creatorUserId` — NOT the full description or milestones (summary view only)
- **Given** a Reviewer with `administrator` role calls this endpoint **Then** the response is `200 OK` (Admins can view the queue)

### US-008: Claim Campaign for Review

**As a** Reviewer
**I want to** claim a submitted campaign for review
**So that** I can assess it exclusively and prevent duplicate review effort

**Acceptance Criteria:**

- **Given** an authenticated Reviewer **When** they call `POST /api/v1/campaigns/:id/claim` on a campaign in `submitted` status **Then** the response is `200 OK` with the campaign in `under_review` status and `reviewedByUserId` set to the reviewer's MMF UUID
- **Given** two Reviewers simultaneously claiming the same campaign **When** the second claim arrives **Then** the response is `409 CAMPAIGN_ALREADY_CLAIMED` (EC-018) — atomic conditional WHERE enforces this
- **Given** a Reviewer calling `POST /:id/claim` on a campaign already in `under_review` **Then** the response is `409 CAMPAIGN_ALREADY_CLAIMED`
- **Given** a user with only `backer` role **When** they call `POST /:id/claim` **Then** the response is `403 REVIEWER_ROLE_REQUIRED`
- **Given** a successful claim **Then** a `campaign.claimed` audit event is written with `previousStatus: 'submitted'`, `newStatus: 'under_review'`, `actorClerkUserId` (the reviewer)

### US-009: Approve Campaign

**As a** Reviewer
**I want to** approve a campaign after review
**So that** the creator can launch it and backers can begin contributing

**Acceptance Criteria:**

- **Given** an authenticated Reviewer who is the assigned reviewer for a campaign in `under_review` **When** they call `POST /api/v1/campaigns/:id/approve` with non-empty `reviewNotes` **Then** the response is `200 OK` with the campaign in `approved` status
- **Given** a Reviewer calling `POST /:id/approve` with empty or missing `reviewNotes` **Then** the response is `400 VALIDATION_ERROR` — approval notes are required (EC-020)
- **Given** a Reviewer who did NOT claim the campaign (different `reviewedByUserId`) **When** they call `POST /:id/approve` **Then** the response is `403 NOT_ASSIGNED_REVIEWER` (EC-019) — exception: Admins can approve any `under_review` campaign
- **Given** a successful approval **Then** a `campaign.approved` audit event is written with `previousStatus: 'under_review'`, `newStatus: 'approved'`, `rationale` containing the review notes
- **Given** a user with only `backer` role **When** they call `POST /:id/approve` **Then** the response is `403 REVIEWER_ROLE_REQUIRED`

### US-010: Reject Campaign

**As a** Reviewer
**I want to** reject a campaign proposal with written rationale and guidance
**So that** the creator can address the issues and resubmit

**Acceptance Criteria:**

- **Given** an authenticated Reviewer who is the assigned reviewer **When** they call `POST /api/v1/campaigns/:id/reject` with non-empty `rejectionReason` AND non-empty `resubmissionGuidance` **Then** the response is `200 OK` with the campaign in `rejected` status
- **Given** a Reviewer calling `POST /:id/reject` with missing or empty `rejectionReason` **Then** the response is `400 VALIDATION_ERROR` — rejection reason is required (EC-021)
- **Given** a Reviewer calling `POST /:id/reject` with missing or empty `resubmissionGuidance` **Then** the response is `400 VALIDATION_ERROR` — resubmission guidance is required (EC-021)
- **Given** a Reviewer who did NOT claim the campaign **When** they call `POST /:id/reject` **Then** the response is `403 NOT_ASSIGNED_REVIEWER` — exception: Admins can reject any `under_review` campaign
- **Given** a successful rejection **Then** a `campaign.rejected` audit event is written with `rationale` containing the rejection reason

### US-011: Revise Rejected Campaign

**As a** Creator
**I want to** revise my rejected campaign based on the reviewer's feedback
**So that** I can resubmit an improved proposal

**Acceptance Criteria:**

- **Given** an authenticated Creator who owns a campaign in `rejected` status **When** they call `PATCH /api/v1/campaigns/:id` with updated fields **Then** the response is `200 OK` and the campaign remains in `rejected` status (PATCH does not change status; status transitions via submit)
- **Given** a Creator revising a rejected campaign **When** they call `POST /api/v1/campaigns/:id/submit` after addressing feedback **Then** the submission logic revalidates all required fields and transitions from `rejected` to `submitted`
- **Given** a rejected campaign **Then** the prior submission data (milestones, team members, description) is preserved and the creator can modify individual fields via PATCH
- **Given** a rejected campaign resubmitted **Then** a `campaign.submitted` audit event is written with `previousStatus: 'rejected'`, `newStatus: 'submitted'`
- **Given** a creator attempting `POST /:id/submit` on a campaign in `under_review` **Then** the response is `409 CAMPAIGN_NOT_REVIZABLE` (EC-017)

### US-012: Launch Approved Campaign

**As a** Creator
**I want to** launch my approved campaign to the Live state
**So that** backers can discover and fund it

**Acceptance Criteria:**

- **Given** an authenticated Creator who owns a campaign in `approved` status **When** they call `POST /api/v1/campaigns/:id/launch` **Then** the response is `200 OK` with the campaign in `live` status and `launchedAt` set to current timestamp
- **Given** a Creator calling `POST /:id/launch` on a campaign in `draft` status **Then** the response is `409 CAMPAIGN_NOT_LAUNCHABLE` (EC-022)
- **Given** a Creator calling `POST /:id/launch` on a campaign in `submitted` or `under_review` status **Then** the response is `409 CAMPAIGN_NOT_LAUNCHABLE`
- **Given** a successful launch **Then** a `campaign.launched` audit event is written with `previousStatus: 'approved'`, `newStatus: 'live'`
- **Given** a Creator attempting to revise an `approved` campaign (before launch) via PATCH **Then** the response is `409 CAMPAIGN_NOT_EDITABLE` — approved campaigns cannot be auto-saved (EC-023)

### US-013: Archive Campaign

**As a** Creator
**I want to** archive a campaign I no longer wish to pursue
**So that** it no longer appears in my active campaign list

**Acceptance Criteria:**

- **Given** an authenticated Creator who owns a campaign in `draft` or `rejected` status **When** they call `POST /api/v1/campaigns/:id/archive` **Then** the response is `200 OK` with the campaign in `archived` status
- **Given** a Creator attempting to archive a campaign in `submitted` or `under_review` status **Then** the response is `409 CAMPAIGN_CANNOT_ARCHIVE` — must be in draft or rejected state to self-archive
- **Given** an Admin **When** they call `POST /api/v1/campaigns/:id/archive` on any campaign in any state **Then** the response is `200 OK` with the campaign in `archived` status

### US-014: Admin Reassign Reviewer

**As an** Administrator
**I want to** reassign a campaign under review to a different reviewer
**So that** review can continue if the original reviewer is unavailable

**Acceptance Criteria:**

- **Given** an authenticated Admin **When** they call `POST /api/v1/campaigns/:id/reassign` with a `reviewerUserId` that references an existing user with the `reviewer` role **Then** the response is `200 OK` with `reviewedByUserId` updated to the new reviewer's MMF UUID
- **Given** an Admin calling `POST /:id/reassign` on a campaign not in `under_review` status **Then** the response is `409 CAMPAIGN_INVALID_STATE` (EC-024)
- **Given** an Admin calling `POST /:id/reassign` with a `reviewerUserId` that does not have the `reviewer` role **Then** the response is `400 VALIDATION_ERROR` with code `REASSIGN_TARGET_NOT_REVIEWER`
- **Given** a non-Admin (including a Reviewer) calling `POST /:id/reassign` **Then** the response is `403 ADMIN_ROLE_REQUIRED`
- **Given** a successful reassignment **Then** a `campaign.reassigned` audit event is written with `metadata: { previousReviewerUserId, newReviewerUserId }`

## State Machine Summary

| From | To | Actor | Trigger Endpoint | Required Condition |
|------|----|-------|------------------|--------------------|
| (new) | `draft` | Creator | `POST /campaigns` | Creator role + KYC verified |
| `draft` | `draft` | Creator | `PATCH /campaigns/:id` | Own campaign |
| `draft` | `submitted` | Creator | `POST /campaigns/:id/submit` | All required fields; milestones sum; deadline valid |
| `draft` | `archived` | Creator | `POST /campaigns/:id/archive` | Own campaign |
| `submitted` | `under_review` | Reviewer/Admin | `POST /campaigns/:id/claim` | Campaign in submitted state (atomic) |
| `under_review` | `approved` | Reviewer/Admin | `POST /campaigns/:id/approve` | Assigned reviewer or Admin; reviewNotes non-empty |
| `under_review` | `rejected` | Reviewer/Admin | `POST /campaigns/:id/reject` | Assigned reviewer or Admin; both rejection fields non-empty |
| `rejected` | `submitted` | Creator | `POST /campaigns/:id/submit` | Same as draft submission rules |
| `rejected` | `draft` | Creator | `PATCH /campaigns/:id` | Status stays `rejected`; PATCH updates fields but doesn't change status |
| `rejected` | `archived` | Creator | `POST /campaigns/:id/archive` | Own campaign |
| `approved` | `live` | Creator | `POST /campaigns/:id/launch` | Own campaign |
| any | `archived` | Admin | `POST /campaigns/:id/archive` | Admin role |

**Terminal states for feat-003:** `live` (launch is the terminal action in this feature's scope), `archived`. Future states (`funded`, `failed`, `complete`, `cancelled`, `settlement`, `suspended`) are defined in the DB CHECK constraint but not implemented in this feature.
