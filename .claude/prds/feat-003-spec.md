# PRD: feat-003 — Authentication Integration (Clerk)

> Implementation-ready specification. Generated by Spec Writer.
> Feature brief: feat-003-authentication.md
> Research: feat-003-research.md

## Overview

This feature integrates Clerk as the authentication provider for Mars Mission Fund.
The frontend uses `@clerk/react` for sign-in/sign-up UI and JWT token management; the backend uses `@clerk/express` to verify Clerk JWTs on every protected route via middleware.
On the first authenticated request from a new Clerk user, the backend lazily upserts a row into the `users` table and assigns the default `backer` role — no webhook infrastructure required.

## User Stories & Acceptance Criteria

### US-001: Sign In / Sign Up

**As a** visitor
**I want to** sign in or create an account using Clerk's hosted UI
**So that** I can access the authenticated areas of the platform

**Acceptance Criteria:**

- **Given** I am unauthenticated **When** I navigate to `/sign-in` **Then** Clerk's `<SignIn />` component is rendered
- **Given** I am unauthenticated **When** I navigate to `/sign-up` **Then** Clerk's `<SignUp />` component is rendered
- **Given** I complete sign-in **When** Clerk redirects me **Then** I land on `/dashboard`
- **Given** I am authenticated **When** I visit `/sign-in` or `/sign-up` **Then** I am redirected to `/dashboard`
- **Given** `VITE_CLERK_PUBLISHABLE_KEY` is missing **When** the app loads **Then** the `<ClerkProvider>` throws a meaningful error (Clerk SDK behaviour)

### US-002: JWT Injection into API Requests

**As a** signed-in user
**I want** every API request I make to automatically carry my authentication token
**So that** the backend can verify my identity without me managing tokens manually

**Acceptance Criteria:**

- **Given** I am signed in **When** any API request is made via the `useApiClient` hook **Then** the request includes an `Authorization: Bearer <token>` header
- **Given** `getToken()` returns `null` (Clerk not loaded or user signed out) **When** an API request is attempted **Then** no `Authorization` header is sent and the request is not made; the hook redirects to `/sign-in`
- **Given** `getToken()` throws (session expired, refresh failed) **When** an API request is attempted **Then** the error is caught and the user is redirected to `/sign-in`
- **Given** the backend returns HTTP 401 on an in-flight request **When** the API client receives the response **Then** it redirects the user to `/sign-in`

### US-003: Backend JWT Verification

**As a** protected API route
**I want** every incoming request to be verified before the route handler executes
**So that** unauthenticated or tampered requests never reach business logic

**Acceptance Criteria:**

- **Given** a valid Clerk JWT in the `Authorization: Bearer` header **When** a protected route receives the request **Then** the request proceeds and `req.auth` is populated
- **Given** no `Authorization` header **When** a protected route receives the request **Then** `401` is returned with `{ error: { code: "UNAUTHORIZED", message: "Authentication required." } }`
- **Given** an expired JWT **When** a protected route receives the request **Then** `401` is returned with the same error shape
- **Given** a JWT with an invalid signature **When** a protected route receives the request **Then** `401` is returned with the same error shape
- **Given** `GET /health` is called without any token **When** the health route handler runs **Then** `200 {"status":"ok"}` is returned (health route is public, mounted before auth middleware)

### US-004: Lazy User Sync on First Login

**As a** new Clerk user
**I want** my account to be created automatically in the MMF database on my first API request
**So that** I can use the platform immediately after signing in without an explicit registration step

**Acceptance Criteria:**

- **Given** a valid Clerk JWT for a Clerk user not yet in `users` **When** any protected route is called **Then** a `users` row is upserted with `account_status = 'active'` and a `user_roles` row with `role = 'backer'` is inserted — both atomically
- **Given** the above sync completes **When** `req.auth` is populated **Then** it contains `{ userId: <MMF UUID>, clerkUserId: <Clerk ID string>, email: <email> }`
- **Given** a valid Clerk JWT for a user already in `users` **When** a protected route is called **Then** no new rows are created; `req.auth` is populated from the existing rows
- **Given** two simultaneous first-login requests for the same Clerk user **When** both hit the upsert **Then** only one `users` row results; neither request returns an error

### US-005: Account Status Gating

**As** the platform
**I want** to block access for suspended, deactivated, or deleted accounts even if their Clerk JWT is valid
**So that** account status decisions are enforced at the API layer, not just the Clerk layer

**Acceptance Criteria:**

- **Given** a user with `account_status = 'suspended'` **When** they make an authenticated request **Then** `403` is returned with `{ error: { code: "ACCOUNT_SUSPENDED", message: "Your account has been suspended." } }`
- **Given** a user with `account_status = 'deactivated'` **When** they make an authenticated request **Then** `403` is returned with `{ error: { code: "ACCOUNT_DEACTIVATED", message: "Your account has been deactivated." } }`
- **Given** a user with `account_status = 'deleted'` **When** they make an authenticated request **Then** `403` is returned with `{ error: { code: "ACCOUNT_DELETED", message: "This account no longer exists." } }`
- **Given** a user with `account_status = 'active'` or `'pending_verification'` **When** they make an authenticated request **Then** the request proceeds normally

### US-006: GET /api/v1/me

**As a** signed-in user
**I want** to retrieve my MMF profile and roles from the API
**So that** the frontend can display my account state and personalise my experience

**Acceptance Criteria:**

- **Given** a valid Clerk JWT **When** `GET /api/v1/me` is called **Then** `200` is returned with the user's `id`, `clerkUserId`, `email`, `displayName`, `avatarUrl`, `accountStatus`, `onboardingCompleted`, and `roles` array
- **Given** no token **When** `GET /api/v1/me` is called **Then** `401` is returned
- **Given** this is the user's first request **When** `GET /api/v1/me` is called **Then** lazy sync fires and the response includes the newly created profile

### US-007: Correlation ID on Every Response

**As** an operator debugging a request
**I want** every API response to include an `X-Request-Id` header
**So that** I can trace a specific request across backend logs

**Acceptance Criteria:**

- **Given** any request to the backend **When** a response is returned **Then** the response includes `X-Request-Id` with a UUID v4 value
- **Given** an incoming request already has `X-Request-Id` **When** the response is returned **Then** the same value is echoed back (the incoming value must be validated: max 128 characters, alphanumeric + hyphen only; otherwise a new UUID is generated)
- **Given** any request **When** pino-http logs it **Then** the `correlation_id` field in the log entry matches the `X-Request-Id` on the response

### US-008: Protected Dashboard Route

**As a** signed-in user
**I want** `/dashboard` to be accessible only when authenticated
**So that** unauthenticated users cannot access protected content

**Acceptance Criteria:**

- **Given** I am signed in **When** I navigate to `/dashboard` **Then** I see the dashboard placeholder page
- **Given** I am not signed in **When** I navigate to `/dashboard` **Then** I am redirected to `/sign-in`
- **Given** Clerk has not yet loaded (`isLoaded = false`) **When** I navigate to `/dashboard` **Then** a loading state is shown (not the page content, not a redirect)

### US-009: Sign Out

**As a** signed-in user
**I want** to sign out from the header navigation
**So that** I can end my session securely

**Acceptance Criteria:**

- **Given** I am signed in **When** I see the header **Then** a "Sign Out" button is visible
- **Given** I am not signed in **When** I see the header **Then** a "Sign In" button is visible that navigates to `/sign-in`
- **Given** I click "Sign Out" **When** Clerk signs me out **Then** I am redirected to `/sign-in`

## Data Model

No new tables are created by this feature.
The `users` and `user_roles` tables were created in feat-002 migrations.

### Tables Used

#### `users` (migration `20260306000001_create_accounts.sql`)

| Column | Type | Notes for auth sync |
|--------|------|----------------------|
| `id` | UUID NOT NULL | Application-generated via `crypto.randomUUID()` — no DB default |
| `clerk_id` | TEXT NOT NULL UNIQUE | Clerk user ID string (e.g. `user_abc123`); upsert key |
| `email` | TEXT NOT NULL UNIQUE | Populated from Clerk JWT custom claim on first sync |
| `display_name` | TEXT NULL | Not populated at auth sync; set during onboarding (feat-004) |
| `avatar_url` | TEXT NULL | Not populated at auth sync |
| `bio` | TEXT NULL | Not populated at auth sync |
| `onboarding_completed` | BOOLEAN NOT NULL DEFAULT false | Remains false until feat-004 |
| `account_status` | TEXT NOT NULL | Set to `'active'` on lazy sync creation |
| `created_at` | TIMESTAMPTZ NOT NULL | Auto-set by DB |
| `updated_at` | TIMESTAMPTZ NOT NULL | Auto-updated by trigger |

**Upsert SQL** (on first authenticated request):

```sql
INSERT INTO users (id, clerk_id, email, account_status)
VALUES ($1, $2, $3, 'active')
ON CONFLICT (clerk_id) DO UPDATE
  SET email = EXCLUDED.email,
      updated_at = NOW()
RETURNING id, clerk_id, email, display_name, avatar_url, account_status,
          onboarding_completed, created_at, updated_at;
```

Note: `ON CONFLICT (clerk_id) DO UPDATE SET email = EXCLUDED.email` keeps the email current if the Clerk user changes their primary email.
If the new email conflicts with `users_email_unique` (another user's email), the upsert will throw a PostgreSQL unique constraint violation.
The middleware must catch this, log it as an error (with `clerkId` only — not the conflicting email), and return `500` with `INTERNAL_ERROR`.
The email value must come from the Clerk JWT custom claim `email` (see Manual Tasks for Dashboard configuration).

#### `user_roles` (migration `20260306000002_create_roles.sql`)

| Column | Type | Notes for auth sync |
|--------|------|----------------------|
| `id` | UUID NOT NULL | Application-generated via `crypto.randomUUID()` |
| `user_id` | UUID NOT NULL | FK to `users.id` |
| `role` | TEXT NOT NULL | Set to `'backer'` on first login |
| `assigned_by` | UUID NULL | `NULL` for system-assigned roles |
| `created_at` | TIMESTAMPTZ NOT NULL | Auto-set by DB |

**Insert SQL** (idempotent backer assignment):

```sql
INSERT INTO user_roles (id, user_id, role, assigned_by)
VALUES ($1, $2, 'backer', NULL)
ON CONFLICT (user_id, role) DO NOTHING;
```

**Roles query** (after upsert):

```sql
SELECT role FROM user_roles WHERE user_id = $1;
```

The result is always an array (a user may accumulate multiple roles over time).
An empty array must not crash — treat as valid state (user has no roles, pass through).

### Transaction Requirement

Both the `users` upsert and the `user_roles` insert must execute within a single database transaction.
If the `user_roles` insert fails after the `users` upsert commits, the user will have a `users` row but no roles on subsequent requests.
The `user_roles` insert is idempotent (`ON CONFLICT DO NOTHING`), so a retry on the next request is safe.
The implementation must handle this partial-state case: if `user_roles` is empty after a successful `users` lookup, proceed without crashing (do not 500).

**File:** `packages/backend/src/account/ports/user-repository.ts`

```typescript
export interface UserSyncInput {
  readonly id: string;           // pre-generated UUID
  readonly clerkUserId: string;

export interface UserRepository {
```

**Method contracts:**

| Method | Params | Returns | Description |
|--------|--------|---------|-------------|
| `findByClerkId` | `clerkUserId: string` | `Promise<User \| null>` | Look up user by `clerk_id`; returns `null` if not found. Includes roles. |
| `upsertWithBackerRole` | `UserSyncInput` | `Promise<User>` | Upserts `users` row and inserts `backer` role (idempotent) in a single transaction. Returns the full user with roles. |
| `findById` | `userId: string` | `Promise<User \| null>` | Look up user by internal `id` for `GET /api/v1/me`. Includes roles. |

### `ClerkPort`

**File:** `packages/backend/src/account/ports/clerk-port.ts`

```typescript
export interface ClerkPort {
```

## Application Service

### `AuthSyncService`

**File:** `packages/backend/src/account/application/auth-sync-service.ts`

**Dependencies (injected):**
- `UserRepository` — for DB lookup and upsert
- `ClerkPort` — for email retrieval if not in JWT

**Method: `syncUser(clerkUserId: string, emailFromJwt: string | null): Promise<User>`**

Step-by-step orchestration:

1. Call `userRepository.findByClerkId(clerkUserId)`.
2. If a user is found, return it immediately (no upsert needed; `findByClerkId` already includes roles).
3. If no user is found:
   a. If `emailFromJwt` is non-null and non-empty, use it as the email.
   b. If `emailFromJwt` is null/empty, call `clerkPort.getUser(clerkUserId)` and use the returned email.
   c. Generate `id = crypto.randomUUID()` and `roleId = crypto.randomUUID()`.
   d. Call `userRepository.upsertWithBackerRole({ id, clerkUserId, email })`.
   e. Return the resulting `User`.

**Error handling:**

| Error | Cause | Handling |
|-------|-------|----------|
| `ClerkPort.getUser` throws | Clerk API unreachable or returned error | Propagate as `500 INTERNAL_ERROR`; do not expose Clerk error details |
| `userRepository.upsertWithBackerRole` throws unique constraint on `email` | Email conflicts with another user's row | Propagate as `500 INTERNAL_ERROR`; log `clerkUserId` for debugging; do not expose email in response |
| `userRepository.upsertWithBackerRole` throws any other DB error | DB failure | Propagate as `500 INTERNAL_ERROR` |

## API Endpoints

### Auth Middleware (not endpoints — applied globally)

**File:** `packages/backend/src/shared/middleware/auth.ts`

The middleware stack after feat-003 in `server.ts`:

```
app.use(pinoHttp({ logger, genReqId: ... }))      // logs + assigns correlationId
app.use(correlationIdMiddleware)                   // sets X-Request-Id header + req.correlationId
app.use('/health', healthRouter)                  // PUBLIC — mounted BEFORE Clerk middleware
app.use(clerkMiddleware())                        // @clerk/express — attaches Clerk auth state
app.use(mmfAuthMiddleware)                        // MMF custom — DB lookup, lazy sync, account gating
app.use('/api/v1', apiRouter)                     // all protected routes
app.use(errorHandler)                             // 4-param Express error handler
```

**`correlationIdMiddleware`** (inline or separate file):

- Reads `X-Request-Id` from incoming headers.
- Validates: if present, must match `/^[a-zA-Z0-9\-]{1,128}$/`; if invalid, ignore and generate fresh.
- If absent or invalid, generates `crypto.randomUUID()`.
- Assigns to `req.correlationId`.
- Sets `X-Request-Id` response header.
- Must run before pino-http (or integrate with `genReqId` in pino-http config) so the `correlation_id` appears in logs.

**`mmfAuthMiddleware`** logic:

```
1. Read Clerk auth state: const { userId: clerkUserId } = getAuth(req)
2. If clerkUserId is null/undefined:
     → return 401 { error: { code: "UNAUTHORIZED", message: "Authentication required.", correlation_id } }
3. Check MOCK_AUTH=true:
     If true AND clerkUserId === 'user_test_mock':
       → skip Clerk verification (already bypassed by mock clerkMiddleware setup)
4. Extract email claim: const email = getAuth(req).sessionClaims?.email ?? null
5. Call authSyncService.syncUser(clerkUserId, email)
6. Check user.accountStatus:
     'suspended'  → 403 { code: "ACCOUNT_SUSPENDED", message: "Your account has been suspended." }
     'deactivated'→ 403 { code: "ACCOUNT_DEACTIVATED", message: "Your account has been deactivated." }
     'deleted'    → 403 { code: "ACCOUNT_DELETED", message: "This account no longer exists." }
     'pending_verification' → 403 { code: "ACCOUNT_PENDING", message: "Account verification required." }
     'active'     → proceed
7. Assign req.auth = { userId: user.id, clerkUserId: user.clerkUserId, email: user.email, roles: user.roles }
8. Call next()
```

**MOCK_AUTH behaviour:**

When `MOCK_AUTH=true` in the environment:
- `clerkMiddleware()` is replaced by a mock middleware that sets a synthetic Clerk auth state with `userId = 'user_test_mock'`.
- `mmfAuthMiddleware` proceeds normally with the mock `clerkUserId` — it calls `authSyncService.syncUser` which will upsert a real `users` row with `clerk_id = 'user_test_mock'` and `email = 'test@marsmissionfund.test'`.
- This allows integration tests to test the full middleware chain against the real database without a real Clerk instance.
- The mock user's fixed properties: `clerkUserId = 'user_test_mock'`, `email = 'test@marsmissionfund.test'`.

**Security headers middleware** (inline in `server.ts` or via `helmet`):

Must set the headers defined in L3-002 Section 7.2 on all responses:

| Header | Value |
|--------|-------|
| `Content-Security-Policy` | `default-src 'self'; script-src 'self'; style-src 'self'; img-src 'self' data: https:; font-src 'self'; connect-src 'self' https://*.clerk.accounts.dev; frame-ancestors 'none'; base-uri 'self'; form-action 'self'` |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains; preload` |
| `X-Content-Type-Options` | `nosniff` |
| `X-Frame-Options` | `DENY` |
| `Referrer-Policy` | `strict-origin-when-cross-origin` |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=(), payment=()` |
| `Cache-Control` | `no-store` (authenticated responses) |

Use `helmet` package for this — configure it with custom CSP options rather than its defaults.

### `useApiClient` Hook

**File:** `packages/frontend/src/hooks/use-api-client.ts`

Provides a `fetchWithAuth` function that injects the Clerk JWT and enforces a base URL from `VITE_API_URL`:

```typescript
export function useApiClient() {
  const { getToken, isSignedIn } = useAuth();
  const navigate = useNavigate();

  const fetchWithAuth = useCallback(
    async (path: string, options?: RequestInit): Promise<Response> => {
      let token: string | null = null;
      try {
        token = await getToken();
      } catch {
        navigate('/sign-in');
        throw new Error('Session expired');
      }
      if (!token) {
        navigate('/sign-in');
        throw new Error('Not authenticated');
      }
      const response = await fetch(
        `${import.meta.env.VITE_API_URL}${path}`,
        {
          ...options,
          headers: {
            ...options?.headers,
            Authorization: `Bearer ${token}`,
            'Content-Type': 'application/json',
          },
        }
      );
      if (response.status === 401) {
        navigate('/sign-in');
        throw new Error('Unauthorized');
      }
      return response;
    },
    [getToken, isSignedIn, navigate]
  );

  return { fetchWithAuth };
}
```

`getToken()` is called per-request (not cached at module level) — this is mandatory to prevent stale token bugs.

## Open Questions

None. All decisions from the research document have been resolved:

| Question | Resolution |
|----------|------------|
| `@clerk/express` vs `@clerk/clerk-sdk-node` | Use `@clerk/express` — Express-native middleware, current recommended package |
| How to get email from Clerk | Configure custom JWT template in Clerk Dashboard to include `email` claim; fall back to `ClerkPort.getUser` if claim absent |
| Upsert SQL | `ON CONFLICT (clerk_id) DO UPDATE SET email = EXCLUDED.email` for users; `ON CONFLICT (user_id, role) DO NOTHING` for roles; both in single transaction |
| `req.auth` TypeScript type | Module augmentation in `packages/backend/src/shared/types/express.d.ts` |
| Account status gating for `pending_verification` | Treat as 403 ACCOUNT_PENDING (defensive; should not arise from Clerk sync) |
| `deleted` account error code | `ACCOUNT_DELETED` (reveal status — consistent with suspended/deactivated pattern; not a security risk since 403 confirms authentication) |
| Frontend API client pattern | `useApiClient` hook using `useAuth().getToken()` per-request; `useNavigate` for redirect; no module-level caching |
| Post-auth redirect | Redirect to `/dashboard` after sign-in and sign-up |
| MOCK_AUTH behaviour | Bypass Clerk JWT; inject `user_test_mock`; full sync pipeline runs against real DB |
