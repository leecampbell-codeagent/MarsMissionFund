### 8. Complexity Assessment

- **Estimated size:** M (Medium — consistent with backlog entry)
- **Backend complexity:** Medium — New bounded context (`kyc/`) with entity, domain errors, 3 ports, 3 adapters (mock KYC, local storage, pg repository), application service with 4 methods, and integration with account context for role grant.
- **Frontend complexity:** Medium — New page with multi-state status display, file upload with magic byte validation, conditional form fields, and polling.
- **Infrastructure changes:** Minor — New environment variables (`MOCK_KYC`, `MOCK_KYC_APPROVAL_DELAY_MS`, `DOCUMENT_STORAGE_PATH`); new tmp directory.
- **Cross-context dependencies:** 1 context — KYC context calls Account context (via injected `AccountRepository`) to grant creator role.
- **External service integrations:** Mock only — Veriff is fully stubbed; S3 is replaced by local filesystem.
- **Estimated test count:** 13 domain unit + 6 magic byte unit + 16 router integration + 9 app service integration + 9 frontend component = ~53 tests.
- **Risk factors:**
  - The mock adapter's `setTimeout`-based auto-approval creates async behavior that is harder to test synchronously. The spec correctly specifies that `processApproval` is always out-of-band, but integration tests will need to await the Promise chain carefully.
  - The UPSERT pattern on `account_id` requires the UNIQUE constraint migration to land before any `save()` call. If the migration runs out of order, the UPSERT will fail. (Mitigated by dbmate's sequential migration execution.)
  - File upload via `multipart/form-data` requires multer middleware. The spec specifies this but the composition root must wire multer to this router only (not all routes).

## Warnings (Should Fix)

### WARN-1: E2E Test Missing

**Issue:** The testing requirements enumerate unit and integration tests but do not include a full end-to-end browser test (e.g., Playwright) for the KYC submission flow.

**Recommendation:** The implementation agent should add a Playwright E2E test that exercises: navigate to `/settings/verification`, select passport, upload a mock JPEG file, submit, observe `pending` status, observe `verified` status after mock auto-approval. This is not a FAIL because the integration tests effectively cover the same flow, but a proper E2E test is best practice per the design agent spec.

**Agent:** Spec Writer (or implementation agent may add this during build).

### WARN-2: Explicit Return Types

**Issue:** The spec instructs the implementation agent to follow CLAUDE.md rules on TypeScript explicit return types, but does not repeat the rule in the spec itself for the exported class methods.

**Recommendation:** Implementation agents should be reminded to add explicit return types on all public methods of `KycAppService`, `KycVerification`, mock adapters, and the router factory function. This is already in CLAUDE.md (TypeScript strict mode) but bears emphasis for a new bounded context.

**Agent:** Implementation agent (self-reminder during implementation).

---

### WARN-3: `withProviderReference` Entity Method

**Issue:** The application service spec references setting `providerReference` on the `KycVerification` entity ("Update `providerReference` on the entity") but the domain model section does not explicitly define a `withProviderReference()` method on the entity. The implementation agent may be confused about whether to add a separate method or use `reconstitute()`.

**Recommendation:** The implementation agent should add a `withProviderReference(ref: string): KycVerification` business method to the entity that returns a new instance with the updated `providerReference` field. This follows the immutable entity pattern established in the `Account` entity.

**Agent:** Implementation agent (infer from existing Account entity patterns).
