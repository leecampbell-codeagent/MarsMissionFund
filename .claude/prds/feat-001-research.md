### 3.2 Docker Compose Configuration Details

**PostgreSQL health check:**

```yaml
healthcheck:
  test: ["CMD-SHELL", "pg_isready -U $POSTGRES_USER -d $POSTGRES_DB"]
  interval: 2s
  timeout: 5s
  retries: 10
```

**Backend hot reload in Docker:**
Two options exist.
Option A: volume-mount source code into the container and use `tsx watch` (simpler).
Option B: run the backend outside Docker for development and only Dockerise for CI/integration.
Recommendation: Option A for environment parity per L2-002 Section 7.1, but with clear documentation that developers *can* run outside Docker if preferred.

**Frontend in Docker:**
The Vite dev server must bind to `0.0.0.0` (not `localhost`) to be accessible from the host.
Configure `server.host: true` in `vite.config.ts`.
Volume-mount source code for HMR.

**dbmate service:**

```yaml
dbmate:
  image: ghcr.io/amacneil/dbmate
  command: ["--wait", "up"]
  environment:
    DATABASE_URL: "postgres://mmf:mmf@postgres:5432/mmf_dev?sslmode=disable"
  volumes:
    - ./db/migrations:/db/migrations
  depends_on:
    postgres:
      condition: service_healthy
```

## 4. Comparable Project Patterns

### 4.1 npm Workspaces Monorepo Pattern

The standard pattern for npm workspaces monorepos with TypeScript:

- Root `package.json` with `"workspaces": ["packages/*"]` and `"private": true`
- Shared `tsconfig.base.json` at root with `strict: true`, extended by each package
- Single ESLint flat config at root with file-pattern overrides for frontend vs backend
- Prettier at root (single style across all packages)
- Root-level scripts delegate to workspaces via `--workspace` or `--workspaces`

### 4.2 TypeScript Project References

TypeScript project references can be used for cross-package type checking, but for feat-001 with only two packages and no shared code yet, simple independent `tsconfig.json` files extending a base config are sufficient.
Project references should be introduced when a `packages/shared/` package is created.

### 4.3 Docker Compose Dev Pattern

The typical pattern for local development:

1. PostgreSQL runs in a container with a named volume for data persistence
1. Backend runs in a container with source code volume-mounted and uses `tsx watch` for hot reload
1. Frontend runs in a container (or on host) with Vite dev server
1. A one-shot dbmate container runs migrations on startup before backend starts
1. All services connect via a Docker bridge network
1. Health checks ensure startup ordering

### 4.4 GitHub Actions CI Pattern

Standard monorepo CI workflow:

```yaml
name: CI
on:
  pull_request:
    branches: [main]
jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'
      - run: npm ci
      - run: npm run lint
      - run: npm run format
      - run: npm run typecheck
      - run: npm test
```

No need for path filtering or workspace-specific builds for a project this size.
Run all checks on every PR for simplicity.

## 5. Edge Cases

### 5.1 Port Conflicts

**Edge case:** Developer has another service running on port 5432, 3001, or 5173.
**Expected behaviour:** Docker Compose should document the port mappings in `.env.example`.
Ports should be configurable via environment variables in `docker-compose.yml` (e.g., `${POSTGRES_PORT:-5432}:5432`).
Error message from Docker is clear enough; no custom handling needed.

### 5.2 Docker Not Installed

**Edge case:** Developer runs `npm run dev` without Docker installed.
**Expected behaviour:** Docker Compose fails with a clear error.
Document Docker as a prerequisite in the project README and CLAUDE.md development workflow section.

### 5.3 Database Not Ready on Backend Start

**Edge case:** Backend starts before PostgreSQL accepts connections.
**Expected behaviour:** Docker Compose `depends_on` with `condition: service_healthy` prevents this.
The `pg_isready` health check must pass before dependent services start.
dbmate uses `--wait` flag as additional safety.

### 5.4 Empty Migrations Directory

**Edge case:** dbmate runs with no migration files in `db/migrations/`.
**Expected behaviour:** dbmate exits cleanly with no error.
The `schema_migrations` table is created automatically on first migration.

### 5.5 Volume Permission Issues on Linux

**Edge case:** Docker volume mounts may have UID/GID mismatches on Linux, causing permission denied errors for node_modules or build artifacts.
**Expected behaviour:** Use `node` user in Dockerfile (UID 1000, which matches most Linux users).
Document the workaround: `docker compose run --user $(id -u):$(id -g) backend npm install`.

### 5.6 node_modules Conflicts Between Host and Container

**Edge case:** Developer runs `npm install` on host (macOS/Windows) then starts Docker container (Linux).
Native modules (e.g., `esbuild`, `@swc/core`) compiled for the wrong platform cause crashes.
**Expected behaviour:** Use an anonymous volume for `node_modules` inside the container (`- /app/node_modules`) to isolate container dependencies from host.
Document that `npm install` inside the container and on the host are independent.

### 5.7 .env File Missing

**Edge case:** Developer clones the repo and runs `npm run dev` without creating `.env`.
**Expected behaviour:** Docker Compose reads from `.env` by default.
If missing, the default values in `docker-compose.yml` (using `${VAR:-default}` syntax) should allow the stack to start.
The `.env.example` file documents all variables.

### 5.8 TypeScript Strict Mode Violations

**Edge case:** Developer adds code that violates strict mode (implicit `any`, unchecked nulls).
**Expected behaviour:** `npm run typecheck` fails in CI.
`tsconfig.base.json` has `"strict": true` and both packages inherit it.
No package may override `strict` to `false`.

### 5.9 ESLint Flat Config Compatibility

**Edge case:** Developer has a globally installed ESLint with legacy `.eslintrc` expectations, or their IDE plugin does not support flat config.
**Expected behaviour:** The project uses `eslint.config.js` (flat config).
Document recommended VS Code extensions and settings.
ESLint 9+ uses flat config by default.

### 5.10 Vite Proxy Not Working in Docker

**Edge case:** The Vite dev server runs inside Docker but the proxy target `http://backend:3001` is not reachable because the frontend container is not on the same Docker network, or the service name is wrong.
**Expected behaviour:** All services must be on the same Docker Compose network.
Proxy target in `vite.config.ts` must use the Docker service name as hostname (e.g., `http://backend:3001`), not `localhost`.

### 5.11 Docker Compose Build Cache Invalidation

**Edge case:** Changing `package.json` or `package-lock.json` invalidates the Docker build cache, causing a full `npm install` on every build.
**Expected behaviour:** Dockerfile must copy `package.json` and `package-lock.json` before source code (COPY in layers).
This ensures `npm ci` only re-runs when dependencies change.

### 5.12 Inconsistent Line Endings (Windows)

**Edge case:** Developer on Windows commits files with CRLF line endings, causing Docker scripts (especially shell scripts) to fail with `\r: not found`.
**Expected behaviour:** Add `.gitattributes` with `* text=auto` and `*.sh text eol=lf`.
Prettier normalises line endings in source code.

### 5.13 PostgreSQL Data Persistence Across Restarts

**Edge case:** Developer runs `docker compose down` and loses database data.
**Expected behaviour:** Use a named volume (e.g., `mmf-pgdata`) for PostgreSQL data.
`docker compose down` preserves named volumes.
`docker compose down -v` explicitly removes them (document this for clean reset).

### 5.14 npm Workspace Dependency Hoisting Conflicts

**Edge case:** Frontend and backend depend on different versions of the same package (e.g., different `zod` versions).
npm hoists the most common version; the other gets a nested `node_modules`.
This can cause type mismatches or runtime errors.
**Expected behaviour:** Pin shared dependencies to the same version.
Use `overrides` in root `package.json` if necessary.
CI should catch type errors; add a note about this in development docs.

### 5.15 GitHub Actions Cache Miss

**Edge case:** First CI run after `package-lock.json` changes misses the npm cache, resulting in slow dependency installation.
**Expected behaviour:** `actions/setup-node@v4` with `cache: 'npm'` handles caching.
Cache key is based on `package-lock.json` hash.
First run after dependency changes is slower; subsequent runs are fast.

### 5.16 Vite HMR WebSocket Connection in Docker

**Edge case:** Vite's HMR WebSocket fails to connect when running inside Docker because the WebSocket port or host is misconfigured.
**Expected behaviour:** Configure `server.hmr.host` and `server.hmr.port` in `vite.config.ts` if the default detection fails.
For Docker, `server.host: true` usually suffices, but `server.hmr.clientPort` may need to match the exposed port.

### 5.17 dbmate Connection String with Special Characters

**Edge case:** Database password contains special characters (e.g., `@`, `#`, `/`) that break the URL-encoded `DATABASE_URL`.
**Expected behaviour:** Document that the password in `DATABASE_URL` must be URL-encoded.
Use simple passwords in `.env.example` (e.g., `mmf`) to avoid this issue in local dev.

### 5.18 CI Fails on Markdown Lint but Not Locally

**Edge case:** Developer's local markdownlint version differs from CI, or OS-level line ending differences cause inconsistent results.
**Expected behaviour:** Pin `markdownlint-cli2` as a dev dependency in root `package.json`.
CI and local use the same version via `npx markdownlint-cli2` or `npm run lint`.

### 5.19 Express 5 Breaking Changes from v4

**Edge case:** Developer copies Express 4 patterns (e.g., `app.del()`, callback-style error handling) that no longer work in Express 5.
**Expected behaviour:** Express 5 auto-catches rejected promises in async route handlers.
No need for `express-async-errors`.
Document this in patterns file.

### 5.20 Docker Compose File Version Compatibility

**Edge case:** Developer has an older Docker Compose version that does not support `depends_on.condition: service_healthy`.
**Expected behaviour:** Require Docker Compose v2 (v2.0+, which is the default with Docker Desktop).
Document minimum Docker version requirement.

## 6. Recommendations

### 6.1 Must-Haves for the Spec

1. **Exact directory structure**: The spec must include the full file tree with every file that needs to be created.
No ambiguity about where files go.

1. **Complete `package.json` contents**: Root and both workspace packages.
List exact dependencies and scripts.
Use `"latest"` for unpinned items but call out which must be pinned (e.g., React 19, Express 5).

1. **TypeScript configuration**: Full `tsconfig.base.json` with all `compilerOptions`.
Each package's `tsconfig.json` extending it.
Strict mode is non-negotiable.

1. **Docker Compose file**: Complete `docker-compose.yml` with all services, health checks, volumes, networks, and environment variables.
Annotate each service with its purpose.

1. **Backend Dockerfile**: Multi-stage build.
Builder stage compiles TypeScript.
Runtime stage uses slim image with `node` user.
Layer ordering for cache efficiency.

1. **ESLint flat config**: Complete `eslint.config.js` with TypeScript and React rules.
Must work for both packages from root.

1. **CI workflow**: Complete `.github/workflows/ci.yml`.
Steps: checkout, setup-node, npm ci, lint, format check, typecheck, test.

1. **`.env.example`**: Every environment variable with placeholder values and comments.

1. **Health check endpoint**: `GET /health` returning `{ status: "ok" }` with 200.
Unauthenticated per L2-002 Section 6.3.
Include `pino-http` middleware.

1. **Initial passing test in each package**: Backend: test the health endpoint with Supertest.
Frontend: test that App component renders.

### 6.2 Watch-Outs

1. **Tailwind CSS v4 is a significant breaking change from v3**.
Do not use `tailwind.config.js` or PostCSS plugin.
Use `@tailwindcss/vite` and `@import "tailwindcss"` in CSS.

1. **Express 5 types**.
The `@types/express` package may lag behind Express 5.
Check if Express 5 ships its own types or if `@types/express@5` is needed.
If types are incomplete, create a minimal `.d.ts` extension.

1. **Vite proxy in Docker**.
The proxy target must use Docker service names, not `localhost`.
This is a frequent source of confusion.

1. **node_modules in Docker volumes**.
Must use an anonymous volume to prevent host/container conflicts for native modules.

1. **dbmate `--wait` flag**.
Without it, dbmate will fail if PostgreSQL is not ready, even with `depends_on: service_healthy`.
The health check confirms PostgreSQL *accepts connections* but dbmate may attempt a migration before the DB is fully initialised.

1. **Vitest workspace/projects deprecation**.
Vitest 3.2 deprecated the `vitest.workspace` file in favour of `projects` in the root config.
Use the newer `projects` approach.

### 6.3 Suggested Implementation Approach

1. **Start with root configuration**: `package.json`, `tsconfig.base.json`, `eslint.config.js`, `.prettierrc`
1. **Create backend package**: `packages/backend/` with Express 5, Pino, health endpoint, and test
1. **Create frontend package**: `packages/frontend/` with Vite, React 19, Tailwind v4, and test
1. **Create Docker Compose**: PostgreSQL, dbmate, backend, frontend
1. **Create CI workflow**: GitHub Actions with lint, format, typecheck, test
1. **Create `.env.example`**: Document all variables
1. **Verify**: `npm run dev` starts the full stack; `npm run lint`, `npm run format`, `npm run typecheck`, `npm test` all pass from root
