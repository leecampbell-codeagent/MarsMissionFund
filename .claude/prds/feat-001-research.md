# Research: feat-001 — Project Infrastructure and Monorepo Setup

> Research document for Spec Writer consumption. Generated by Spec Researcher.

## 1. Domain Knowledge

### Key Concepts

**npm workspaces** — A built-in npm feature (since npm 7) that allows a single root `package.json` to declare multiple packages, each in its own subdirectory, with shared `node_modules` hoisted to the root. Declaring `"workspaces": ["packages/*"]` in the root `package.json` is all that is needed; npm then installs all workspace packages' dependencies into the root `node_modules`, symlinks workspace packages to each other, and allows cross-package `npm run` invocations via `--workspace=` flag.

**TypeScript project references** — TypeScript's mechanism for splitting a large codebase into independently compilable units. A root `tsconfig.base.json` holds shared `compilerOptions`; each package extends it with its own `tsconfig.json` that adds package-specific settings (e.g., `lib: ["DOM"]` for frontend, `outDir`, `rootDir`). Project references (`references` array) allow `tsc --build` to compile incrementally and in dependency order.

**Hexagonal architecture file layout** — The backend rules mandate a specific directory structure within `packages/backend/src/`: each bounded context gets a folder with sub-folders for `domain/`, `ports/`, `adapters/`, `application/`, and `api/`. For feat-001 (infrastructure only), the skeleton needs to exist so later features can land in the right places, but no domain logic is created yet.

**Vite + React 19 scaffold** — The standard Vite scaffold for React + TypeScript uses `vite create` templates, but for a monorepo the scaffold is created manually: a `packages/frontend/index.html` entry, `packages/frontend/vite.config.ts`, and a `src/main.tsx` with React 19's `createRoot`. React 19 introduces significant changes (hooks, Actions, `use()`) but the scaffold itself is structurally identical to React 18.

**Express 5 scaffold** — Express 5 (currently stable as of 2025) changes the default error handling (promises are automatically caught in route handlers) and removes several deprecated APIs from v4. The scaffold requires `express@5` explicitly; `@types/express` is compatible with v5.

**pino-pretty** — Must be installed as a `devDependency` and loaded conditionally (e.g., `NODE_ENV !== 'production'`) to avoid shipping it in production. The pattern is `transport: process.env.NODE_ENV !== 'production' ? { target: 'pino-pretty' } : undefined`.

**concurrently** — The standard npm tool for running multiple npm scripts in parallel from a single root `npm run dev`. Alternative: `npm-run-all`. Needed for root `dev` script to start both backend and frontend simultaneously.

### Industry Conventions

**npm workspaces vs. alternatives**: Yarn workspaces and pnpm workspaces are popular alternatives, but the specs mandate npm 10.x (L3-008). npm workspaces are the correct choice. Unlike pnpm, npm hoists all dependencies to root `node_modules` by default, which can cause phantom dependency issues but is simpler for a two-package monorepo.

**Root scripts pattern**: Industry convention for npm workspaces is to define high-level scripts in the root `package.json` that delegate to workspaces:
- `npm run dev` — runs both dev servers concurrently
- `npm run build` — builds both packages
- `npm run lint` — already defined; needs to cover workspace TS files
- `npm run test` — runs workspace tests
- `npm run typecheck` — runs `tsc --noEmit` for both packages

**TypeScript strict mode conventions**: `"strict": true` alone enables: `strictNullChecks`, `strictFunctionTypes`, `strictBindCallApply`, `strictPropertyInitialization`, `noImplicitAny`, `noImplicitThis`, `alwaysStrict`. The existing `tsconfig.base.json` already enables `strict: true` plus additional flags (`noUnusedLocals`, `noUnusedParameters`, `noUncheckedIndexedAccess`). This is the base — packages extend it.

**Backend `moduleResolution: "node16"` vs `"bundler"`**: The base tsconfig uses `"bundler"`, which is appropriate for Vite-built frontend code. For the backend (Node.js running directly via tsx), `"node16"` or `"nodenext"` is more appropriate to match Node's actual module resolution. This is a meaningful distinction for ESM imports.

**Vite config for monorepo**: In a monorepo, Vite's `root` defaults to where `vite.config.ts` lives (i.e., `packages/frontend/`). `index.html` must be at that root. The `build.outDir` should be set explicitly (e.g., `../../dist/frontend`), or left at the default `dist/` relative to the package root.

## 4. Competitor Patterns

This section documents how comparable open-source monorepo projects structure their workspaces, which is directly applicable to the implementation approach.

### T3 Stack (create-t3-app with turbo)

- Uses Turborepo on top of npm/pnpm workspaces for build orchestration.
- Each package has its own `tsconfig.json` extending a root base.
- Separate `packages/` for shared types between frontend and backend.
- **What works**: Clean separation, type sharing via workspace package. **Consideration for MMF**: Turborepo adds complexity not warranted for a two-package setup. Plain npm workspaces are sufficient.

### Nx Monorepos

- Generates full project configuration including test runners, lint configs.
- **Anti-pattern for MMF**: Too much generated boilerplate obscures the architecture. MMF's specs mandate explicit understanding of each config file.

### Vite + Express Monorepos (community patterns)

- Standard pattern: `packages/frontend` with `vite.config.ts`, `packages/backend` with `tsconfig.json` using `"module": "NodeNext"`.
- Root `package.json` uses `concurrently` or `npm-run-all` for parallel dev scripts.
- Backend entry point at `packages/backend/src/server.ts` (or `index.ts`); started via `tsx watch src/server.ts`.
- Frontend entry at `packages/frontend/src/main.tsx` with `packages/frontend/index.html` at package root.
- **What works**: Simple, explicit, no magic. Exactly right for MMF.

## 5. Edge Cases

### Data Edge Cases

- [ ] **`packages/` directory collision**: If `packages/` already exists with stale files when the feature is implemented, the scaffold commands may fail or produce incorrect state. Expected behaviour: the spec must define a clean-slate assumption; the implementation agent must check before creating.
- [ ] **`tsconfig.base.json` field conflicts**: The existing base uses `"moduleResolution": "bundler"`, which is incorrect for Node.js direct execution. Backend `tsconfig.json` must override this to `"node16"` or `"nodenext"`. If the backend tsconfig merely extends base without overriding, `tsx` may behave unexpectedly with ESM imports.
- [ ] **Root `package.json` scripts clobbering**: The existing `lint` and `lint:fix` scripts must be preserved when adding workspace-aware scripts. Adding `"dev"`, `"build"`, `"test"`, `"typecheck"` must not conflict.
- [ ] **`node_modules` hoisting conflicts**: If `packages/backend` and `packages/frontend` depend on different versions of the same package (e.g., `typescript`), npm workspace hoisting may pick one version for both. This should be avoided by specifying exact matching versions.
- [ ] **`.env` file missing `PORT`**: The script `start-dev-stack.sh` does not explicitly export `PORT`; backend must read `process.env.PORT` with a sensible fallback (e.g., `3001`).

### Configuration Edge Cases

- [ ] **Biome covering workspace TypeScript**: The existing `biome.json` has no explicit workspace include rules. Biome will discover `packages/backend/src/**` and `packages/frontend/src/**` automatically since there are no exclude rules for those paths. This is the desired behaviour — verify no unintended exclusions.
- [ ] **`tsconfig.base.json` `isolatedModules: true` compatibility**: This setting requires every file to be a module (must have at least one import/export). Any pure declaration files or ambient modules must be aware of this constraint.
- [ ] **`noUncheckedIndexedAccess`** in base tsconfig: This flag makes array element access return `T | undefined` even for in-bounds indices. This is strict and correct, but implementation agents must be aware it affects patterns like `items[0].name` (needs null check). Flag this for implementation agents.
- [ ] **Vite env variable prefix**: Vite only exposes env vars prefixed with `VITE_` to client code. `DATABASE_URL`, `CLERK_SECRET_KEY`, and all backend-only vars must never be prefixed with `VITE_` or they will be exposed to the browser. `.env.example` correctly uses `VITE_` only for `VITE_API_URL` and `VITE_CLERK_PUBLISHABLE_KEY`.
- [ ] **`pino-pretty` in production bundle**: If `pino-pretty` is listed as a regular `dependency` instead of `devDependency`, it ships in the production Docker image unnecessarily. Must be `devDependency` with conditional loading.

### Integration Edge Cases

- [ ] **`start-dev-stack.sh` expects `npm run dev` in each package**: If the backend `package.json` names the script differently (e.g., `start:dev`), the dev stack startup will silently do nothing. Script name must be exactly `dev`.
- [ ] **`dbmate up` idempotency**: The start script runs `dbmate up` every time. This is safe because dbmate tracks applied migrations in `schema_migrations`. But if `DATABASE_URL` is not set or wrong, the script will fail with a non-zero exit and the `|| echo "Warning"` suppresses the error. Migrations must be validated separately.
- [ ] **Port conflicts**: Backend expected on `3001`, frontend on `5173`. If another process occupies these ports (e.g., a leftover process from a previous run), the dev servers will fail to start. `start-dev-stack.sh` already kills `tsx.*server` and `vite` processes, which handles the common case.
- [ ] **`DATABASE_URL` in agent runtime points to `postgres:5432`**: The `.env` file uses `postgres` as the hostname (Docker service name), not `localhost`. Any direct `psql` or `dbmate` invocation must use the env var, not a hardcoded connection string.

### User Behaviour Edge Cases

- [ ] **Developer runs `npm install` in a workspace package**: Running `npm install some-package` inside `packages/backend/` rather than from the root with `--workspace` will work, but may create a `node_modules` inside the package directory instead of hoisting to root. The spec should document that all installs should be run from the repo root with `--workspace=packages/backend`.
- [ ] **Developer runs `npm run dev` from root when only one package exists**: The root `dev` script must handle the case where one workspace doesn't exist yet (backend or frontend). Currently `start-dev-stack.sh` handles this with `if [ -f package.json ]` guards. The root npm script should mirror this or delegate to the shell script.

### Boundary Edge Cases

- [ ] **`GET /health` response format**: The acceptance criteria require a `200` response on `GET /health`. The spec must clarify whether the response body should be JSON (`{"status": "ok"}`) or plain text. Given that L3-001 defines a consistent error format and L2-002 Section 5 governs API contracts, a minimal JSON body is appropriate even for health checks.
- [ ] **Health check authentication exception**: L2-002 Section 5.4 explicitly exempts the health check endpoint from authentication requirements. The Express router must mount the `/health` route before auth middleware to prevent accidental auth-gating.
- [ ] **Backend Express 5 async error handling**: Express 5 automatically catches rejected promise from async route handlers and passes errors to the error middleware. This differs from Express 4. The scaffold must use Express 5's native async support rather than wrapping with `asyncHandler`.
- [ ] **Vite build output path for CI**: `npm run build` in `packages/frontend` will output to `packages/frontend/dist/` by default. If the spec expects a specific output path (e.g., for a CI artefact), the `build.outDir` in `vite.config.ts` must be set explicitly.
- [ ] **TypeScript `declaration` output for shared types**: The base tsconfig sets `"declaration": true`. For the backend, this generates `.d.ts` files into `outDir`. If no `outDir` is set in the package tsconfig, declarations land alongside source files. Each package tsconfig must set an explicit `outDir`.
- [ ] **`concurrently` vs `npm-run-all` for root `dev` script**: Both are valid; `concurrently` provides colour-coded output and named processes. The choice should be consistent with what's already in `devDependencies` (currently neither is installed). One must be added.

### Concurrency Edge Cases

- [ ] **Simultaneous `npm install` from two terminals**: In a workspace, running `npm install` concurrently from two terminals can corrupt the lock file. Not a runtime concern but a developer workflow concern.
- [ ] **Hot reload in backend during frontend changes**: `tsx watch` watches backend files only. Changes to frontend files must not trigger backend restarts. Since they are in separate directories, this should not be an issue.

## 6. Recommendations

### Must-Haves for Spec

1. **Define the exact workspace structure**: The spec must list every file that must exist after the feature is complete — `packages/backend/package.json`, `packages/backend/src/server.ts`, `packages/backend/tsconfig.json`, `packages/frontend/package.json`, `packages/frontend/src/main.tsx`, `packages/frontend/index.html`, `packages/frontend/vite.config.ts`, `packages/frontend/tsconfig.json`.

2. **Specify backend `moduleResolution`**: Must override the base tsconfig's `"bundler"` setting to `"node16"` (or `"nodenext"`) for the backend package. This is a correctness requirement, not a preference.

3. **Specify exact npm scripts**: The start-dev-stack.sh script requires `npm run dev` in both packages. The spec must mandate the script names exactly (`dev`, `build`, `test`).

4. **Clarify `GET /health` response**: Specify whether `200 OK` with `{"status": "ok"}` JSON body or plain text. Recommend JSON for consistency with API conventions.

5. **Clarify docker-compose.yml requirement**: The feature brief mentions `docker-compose.yml` for PostgreSQL and dbmate. In the agent runtime this is not applicable (PostgreSQL is pre-provisioned). The spec must:
   - Acknowledge the two contexts (agent runtime vs. human dev).
   - Require `docker-compose.yml` for human developer workflow (as documented in architecture spec Section 12.2).
   - Clarify that the agent runtime must not depend on docker-compose and uses `DATABASE_URL` directly.

6. **Specify `concurrently` or `npm-run-all`**: Must pick one for the root `dev` script. Recommend `concurrently` (more widely used, better output formatting).

7. **Specify `pino-pretty` as devDependency**: And document the conditional loading pattern.

8. **Specify `.nvmrc` file content**: Feature brief mentions pinning Node.js version. Must specify `.nvmrc` with `22` (or `lts/jod`) at repo root.

9. **Specify `engines` field in root `package.json`**: `"engines": { "node": ">=22.0.0", "npm": ">=10.0.0" }` to enforce version requirements.

10. **Define what the Vite scaffold renders**: The acceptance criteria say "builds successfully via `npm run build`" but don't specify what the app renders. Recommend a minimal placeholder page (e.g., "Mars Mission Fund" heading) to validate the React 19 scaffold is functional.

### Watch-Outs

1. **`tsconfig.base.json` already exists and is correct**: Do not replace it — only extend it from package-level tsconfigs. The `noUncheckedIndexedAccess: true` setting will affect all implementation agents downstream; mention it in the spec as a known constraint.

2. **`.env.example` is already comprehensive**: Do not duplicate or conflict. The existing `.env.example` covers all known env vars including `PORT=3001`. The spec should confirm the existing file satisfies the acceptance criterion rather than requiring a new file.

3. **`biome.json` `noConsole` rule**: The linter already bans `console.log` globally (set to `"error"`). Backend implementation agents must use Pino from day one — even in the scaffold's health check route. Any `console.log` in the scaffold will fail linting.

4. **Docker-compose not runnable from agent**: The spec must clearly state that `docker-compose.yml` is for human developer workflow only and must never be executed from within the agent container.

5. **Express 5 is meaningfully different from Express 4**: Error handling, removed APIs, and type definitions differ. Any scaffold code copied from Express 4 tutorials will be subtly wrong. The spec should reference Express 5 documentation explicitly.

6. **React 19 `StrictMode` double-invocation**: React 19's StrictMode (enabled in development) double-invokes `useEffect` cleanup. This doesn't affect a minimal scaffold but will affect future features. Mention it as a gotcha for downstream agents.

7. **Tailwind CSS v4 has breaking changes from v3**: The spec mandates Tailwind CSS v4 (L3-008). Tailwind v4 eliminates the `tailwind.config.js` file — configuration moves into CSS using `@theme` blocks. All future frontend implementation agents must be aware of this. The scaffold must install and configure Tailwind v4 correctly (not v3 patterns).

### Suggested Approach

**Phase 1 — Root setup** (no new packages yet):
1. Add `"workspaces": ["packages/*"]` to root `package.json`.
2. Add `dev`, `build`, `test`, `typecheck` scripts to root `package.json`.
3. Add `concurrently` as root devDependency.
4. Add `.nvmrc` with `22`.
5. Add `engines` field to root `package.json`.

**Phase 2 — Backend scaffold**:
1. Create `packages/backend/package.json` with `express@5`, `pino`, `pino-http`, `tsx` (devDep), `typescript`, `vitest`, `supertest`.
2. Create `packages/backend/tsconfig.json` extending `../../tsconfig.base.json` with `moduleResolution: "node16"`, `outDir: "dist"`, `lib: ["ES2022"]`.
3. Create `packages/backend/src/server.ts` with Express app, Pino logger, `/health` route.
4. Add `dev` script: `tsx watch src/server.ts`.

**Phase 3 — Frontend scaffold**:
1. Create `packages/frontend/package.json` with `react@19`, `react-dom@19`, `vite`, `@vitejs/plugin-react`, `tailwindcss@4`, `vitest`, `@testing-library/react`.
2. Create `packages/frontend/tsconfig.json` extending `../../tsconfig.base.json` with `lib: ["ES2022", "DOM"]`, `jsx: "react-jsx"`.
3. Create `packages/frontend/vite.config.ts`.
4. Create `packages/frontend/index.html` and `packages/frontend/src/main.tsx`.
5. Add `dev` script: `vite`, `build` script: `vite build`.

**Phase 4 — Validate**:
1. Run `npm install` from root.
2. Run `npm run lint` — must pass.
3. Run `npm run typecheck` — must pass.
4. Run `npm run build --workspace=packages/frontend` — must succeed.
5. Confirm `GET /health` returns `200`.

**Key package version decisions** (based on L3-008):
- `express`: `^5.0.0`
- `react` + `react-dom`: `^19.0.0`
- `vite`: latest stable (currently `^6.x`)
- `typescript`: latest stable (`^5.x`)
- `tailwindcss`: `^4.0.0` (v4 — breaking changes from v3; configure via CSS, not `tailwind.config.js`)
- `tsx`: latest stable (for backend dev server + ts-node alternative)
- `vitest`: latest stable
