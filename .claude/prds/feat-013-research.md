# Research: feat-013 — KYC Verification (Mock Provider)

> Research document for Spec Writer consumption. Generated by Spec Researcher.

## 1. Domain Knowledge

### Key Concepts

**Know Your Customer (KYC)**: A regulatory and risk management process used by financial platforms to verify the identity of their users. For crowdfunding platforms, KYC is a gate for creator features — the platform must be confident of who is receiving money. The "Know" part means confirming that a real human is behind an account, that their identity documents are authentic, and that they are not on any sanctions lists.

**Identity Verification Provider (IVP)**: A third-party service (in MMF's case, Veriff) that automates the document check and liveness check. The platform submits document images and a liveness session reference; the provider returns a pass/fail verdict. The platform never does the actual document reading — that is the provider's job.

**Liveness Check**: A video or selfie-based step that confirms the person submitting documents is actually present (not just submitting a photocopy of someone else's ID). Veriff implements this as a short video session. For MMF's mock adapter, this step is bypassed — the mock auto-approves.

**Sanctions Screening**: Checking the verified identity (name, date of birth, nationality) against government-issued sanctions lists (OFAC, EU, UN, Australian DFAT). If someone is on a sanctions list, the platform is legally prohibited from doing business with them. For the mock adapter, this always returns "clear".

**Verification Status / KYC Status**: A per-account state that records where the user is in the verification lifecycle. The L4-005 spec defines nine states. This status is the central data point that gates Creator features across the platform.

**Failure Count**: The number of times a user's KYC submission has been rejected or required resubmission. After 5 failures, the account is locked for KYC submissions and requires admin intervention. After 3 failures, a customer service alert is raised.

**Document Types**: Passport (all jurisdictions), National Identity Card (where applicable), Driver's Licence (AU/US/UK/EU). Front and back required for multi-sided documents; passport requires only the photo page.

### KYC Status Lifecycle (Per L4-005 Section 6)

The spec defines nine statuses. The feature brief uses slightly different names that need reconciling:

| L4-005 Spec Status | Brief Status | DB Schema Status | Description |
|---|---|---|---|
| Not Verified | not_started | `not_verified` | Default state, no submission yet |
| Pending | pending | `pending` | Documents submitted, verification in progress |
| Pending Resubmission | pending_resubmission | `pending_resubmission` | Failed — user must resubmit |
| In Manual Review | in_manual_review | `in_manual_review` | Escalated to human review |
| Verified | verified/approved | `verified` | Identity confirmed |
| Expired | — (out of scope for feat-013) | `expired` | Verification expired |
| Re-verification Required | — (out of scope for feat-013) | `reverification_required` | Trigger event requires re-verify |
| Rejected | rejected | `rejected` | Definitive rejection |
| Locked | locked | `locked` | 5 failures — admin unlock required |

The DB migration uses `not_verified` as the initial status. The feature brief uses `not_started`. The spec writer should standardise on `not_verified` to match the existing migration.

### Calculation Methodology

No monetary calculations involved in this feature. The key "calculation" is the failure count logic:
- Failure count increments when: status moves to `pending_resubmission` or `rejected`.
- At count = 3: customer service alert event emitted.
- At count = 5: status transitions to `locked`.
- When admin unlocks: failure count resets to 0, status → `pending_resubmission`.

### Industry Conventions

**Kickstarter**: Does not require KYC for campaign creation in most jurisdictions. Not directly comparable.

**GoFundMe**: Requires identity verification before withdrawals, not before campaign creation. Veriff is one of their providers.

**Indiegogo**: Uses third-party identity verification for campaign creators; status shown on campaign page.

**Stripe Connect** (most relevant): Requires identity verification for connected accounts (platform users who receive payouts). Stripe uses Persona or Stripe Identity. The pattern is: user submits, async verification, webhook callback, status update. This is the pattern MMF should follow — the submission is synchronous, but the verification result may arrive asynchronously (webhook or polling). For the mock adapter, the "async" delay is simulated with a configurable timeout that then auto-approves.

**Key UX conventions from industry**:
1. Clear status display in account/profile section.
2. Step-by-step wizard for document submission (not a single form).
3. Explicit feedback on what went wrong when submission fails (but sanitised — no internal review notes exposed to users).
4. Progress indicators during async processing.
5. Proactive notifications (email) when status changes.

### Regulatory Notes

- **AML/CTF Act (Australia)**: Mandates identity verification for financial services. Records must be retained for 7+ years.
- **GDPR Article 17(3)(b)**: Right to erasure can be overridden by AML/CTF legal obligations — identity records must be retained even if user requests deletion.
- **DPIA**: Data Protection Impact Assessment required before KYC processing goes live (out of scope for workshop but noted in L4-005 Section 2, inherited from L3-002 Section 8.2).
- For the local demo, none of these regulatory mechanisms are active — the mock adapter treats this as a simple status state machine.

## 2. Codebase Integration

### Current Data Model

The `kyc_verifications` table already exists from migration `db/migrations/20260304000008_create_kyc_verifications.sql`:

```sql
CREATE TABLE kyc_verifications (
  id                  UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  account_id          UUID         NOT NULL REFERENCES accounts(id) ON DELETE RESTRICT,
  status              TEXT         NOT NULL DEFAULT 'not_verified',
  document_type       TEXT,
  provider_reference  TEXT,
  failure_count       INTEGER      NOT NULL DEFAULT 0,
  verified_at         TIMESTAMPTZ,
  expires_at          TIMESTAMPTZ,
  created_at          TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
  updated_at          TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
  CONSTRAINT chk_kyc_verifications_status CHECK (
    status IN (
      'not_verified', 'pending', 'pending_resubmission', 'in_manual_review',
      'verified', 'expired', 'reverification_required', 'rejected', 'locked'
    )
  ),
  CONSTRAINT chk_kyc_verifications_failure_count CHECK (failure_count >= 0)
);
```

Indexes exist on: `account_id`, `status`, `expires_at`.

The `updated_at` trigger is already attached.

**Important**: The table does NOT have columns for uploaded document storage references (no `front_image_ref`, `back_image_ref`). The feature brief requires document upload — the spec writer must decide whether to add columns for document storage references or handle this through a separate `kyc_documents` table. Given the migration is append-only, a new migration will be needed if new columns are required.

The table also lacks a `submitted_at` column, which would be useful for tracking when the most recent submission occurred.

### Integration Points

**KYC → Account**: When KYC status transitions to `verified`, the account service must grant the `creator` role. This is a cross-context dependency.
- Option A: KYC service directly calls account service to update roles (tight coupling, simpler for local demo).
- Option B: KYC service emits an event (`kyc.status.change` → `verified`) and account service subscribes (event-driven, preferred in hexagonal architecture, but more complex).
- For the local demo, Option A (direct call or in the same application service) is pragmatic. The spec writer should document which approach is used.

**KYC → Campaign**: Campaign submission must check KYC status. Per L4-005 Section 10.3, Campaign spec queries KYC status synchronously at submission time. For feat-013, this means the KYC service must expose a method to check if an account is verified.

**KYC → Payments**: Disbursement eligibility requires KYC verified status. Similar synchronous check pattern.

## 3. Third-Party APIs

### Veriff (Identity Verification Provider)

For feat-013, the Veriff adapter is mocked. The real adapter interface will be defined but the mock implementation is what matters for the local demo.

**Real Veriff API (for interface contract design)**:
- REST API, base URL: `https://stationapi.veriff.com`
- Creates a "session" per verification attempt: `POST /v1/sessions` → returns `sessionId` and a `url` for the user to complete verification
- Session has a status: `created`, `started`, `submitted`, `approved`, `declined`, `resubmission_requested`, `expired`, `abandoned`, `review`
- Decision webhook: `POST <your_callback_url>` with HMAC-SHA256 signature — sends final decision
- The session URL is opened in an iframe or redirect — the user interacts with Veriff's hosted UI
- For MMF's local demo: the mock skips the hosted UI entirely and simulates an auto-approval after a configurable delay

**Mock strategy**:
- `submitVerification(input)` → immediately stores a `pending` status, schedules an auto-approval after configurable delay (e.g., `MOCK_KYC_APPROVAL_DELAY_MS=1000`, default 0 = synchronous)
- `getVerificationStatus(providerReference)` → returns the stored status
- Auto-approval: mock transitions status to `verified` and grants creator role
- Environment variable: `MOCK_KYC=true` (existing pattern from CLAUDE.md rules)

**Interface methods needed**:
```typescript
interface KycPort {
  submitVerification(input: KycSubmissionInput): Promise<KycSubmissionResult>;
  getVerificationStatus(providerReference: string): Promise<KycVerificationStatus>;
}
```

### Document Storage (S3 / Local Filesystem)

For document upload, a storage adapter is needed:
- Production: AWS S3 (isolated bucket, separate from application assets)
- Local demo: Local filesystem mock (`StoragePort`)

**Storage port interface**:
```typescript
interface StoragePort {
  store(key: string, buffer: Buffer, mimeType: string): Promise<{ reference: string }>;
  getSignedUrl(reference: string, expirySeconds: number): Promise<string>;
  delete(reference: string): Promise<void>;
}
```

The feature brief lists this as a dependency: "Uploaded documents stored in isolated storage (S3 adapter with local filesystem mock)."

## 4. Competitor Patterns

### GoFundMe / Stripe Connect KYC Flow

**How they handle it**:
- Clear status banner at top of account dashboard ("Complete identity verification to withdraw funds")
- Step-by-step: (1) select document type, (2) upload front, (3) upload back (if applicable), (4) liveness check, (5) submit and wait
- After submission: "Under review — typically takes a few minutes" messaging
- Status badge on account/profile page: Verified (green checkmark), Pending (clock icon), Not Verified (warning icon)

**What works well**:
- Persistent, non-intrusive status indicator on dashboard — creator always knows their status
- Clear messaging about what is needed to unlock features
- Non-blocking: user can do everything else (browse, back campaigns) while KYC is pending

**What doesn't work**:
- Some platforms overload the dashboard with KYC prompts for backers who never need to verify — MMF should only show prominent KYC prompts to users who have selected the Creator role

### Kickstarter Creator Verification

**How they handle it**:
- Only surfaces KYC at the moment of campaign submission, not before
- Modal/inline prompt: "Before we can publish your campaign, we need to verify your identity"
- This creates friction at a high-stakes moment

**MMF should avoid this**: The spec shows KYC is a separate step in the user journey, not a gate at campaign submission. The user should have already completed KYC before attempting to submit a campaign.

## 5. Edge Cases

### Data Edge Cases

- [ ] User submits KYC with no documents uploaded — endpoint should return 400 (document is required)
- [ ] User submits a document file with wrong MIME type but correct extension (e.g., `.jpg` file that is actually a PDF) — magic byte check must catch this
- [ ] User submits a file exactly at 20 MB limit — should succeed; user submits at 20 MB + 1 byte — should fail
- [ ] `provider_reference` is null when status is `pending` (submission in flight) — this is valid; the reference may arrive asynchronously
- [ ] Multiple simultaneous `POST /v1/kyc/submit` requests from same user — race condition could create multiple `pending` records; database should enforce one active submission per account
- [ ] User has no `kyc_verifications` row yet (never submitted) — `GET /v1/kyc/status` must handle this gracefully and return `not_verified`
- [ ] Account ID in auth context does not match any account — return 404, not 500

### State Machine Edge Cases

- [ ] User submits when already in `pending` state — should return 409 (submission already in progress)
- [ ] User submits when in `verified` state — should return 409 (already verified)
- [ ] User submits when in `locked` state — should return 409 (account locked, requires admin unlock)
- [ ] Admin attempts to unlock an account that is not in `locked` state — should return 409 with appropriate message
- [ ] Admin attempts to unlock their own account — should be allowed (admin privilege is role-based, not user-based)
- [ ] Status transitions that are not permitted by the state machine (e.g., `verified` → `pending`) — domain entity must reject these

### Failure Count Edge Cases

- [ ] Failure count at exactly 3 — customer service alert event must be emitted on this submission
- [ ] Failure count at exactly 5 — status must transition to `locked`, not `rejected`
- [ ] After admin unlock (count resets to 0), user fails again — count increments from 0, not from 5; at count 5 again, status locks again
- [ ] Concurrent failure increments (two requests simultaneously cause a failure) — database update must be atomic; use `UPDATE ... SET failure_count = failure_count + 1`

### Integration Edge Cases

- [ ] Mock KYC adapter auto-approves, but account service fails to grant creator role — the KYC status must still be `verified`; the role grant failure should be logged but not block the KYC transition
- [ ] Veriff webhook arrives for a session that has already been manually resolved — idempotency needed; if already `verified`, ignore further webhooks
- [ ] Mock approval delay is 0 (synchronous) — the `POST /v1/kyc/submit` response must not include the `verified` status (it should be `pending`); the auto-approval happens asynchronously even if immediately
- [ ] Document storage fails (disk full, S3 error) — submission should fail with 500; KYC status must not change
- [ ] Event store append fails after status update — transaction must ensure both succeed atomically

### User Behaviour Edge Cases

- [ ] User navigates away during document upload — upload is stateless; user must restart
- [ ] User on mobile uploads a HEIC/HEIF photo — this format is not in the accepted list (JPEG/PNG/PDF); should return a clear format error
- [ ] User uploads a PDF with multiple pages — only the first page should be used, or the upload should be rejected as ambiguous; the spec should clarify
- [ ] User with `not_verified` status tries to submit a campaign — campaign service should return a meaningful error (not just "forbidden"), directing the user to complete KYC
- [ ] Backer (not Creator-role-selected) tries to access KYC page — should be accessible (any user can verify their identity), but the page should explain why verification is needed for Creator features

### Boundary Edge Cases

- [ ] Maximum number of documents per submission — the brief mentions front and back images; some documents need only one side; the spec must define exactly what is required per document type
- [ ] File name injection — file names should not be stored as-is; only sanitised references should be stored
- [ ] Document stored reference format — should use UUID-based keys, not original filenames, to prevent enumeration
- [ ] Admin `unlock` endpoint: what if `:accountId` is not a valid UUID — should return 404, not 500
- [ ] `GET /v1/kyc/status` from a Backer who has never interacted with KYC — should return `not_verified` status, not 404

## 6. Recommendations

### Must-Haves for Spec

1. **Resolve the status name discrepancy**: The feature brief uses `not_started` but the DB schema uses `not_verified`. The spec must standardise on `not_verified` (matching the existing migration which cannot be modified).

2. **Define the role-grant integration**: When KYC status → `verified`, the creator role must be granted. The spec must define whether this is done in the KYC application service directly (calling the account repository/service) or via an event-driven mechanism. For the mock demo, a direct in-process call is recommended.

3. **Define document storage requirements precisely**: The migration has no document storage reference columns. Either add them via a new migration, or use a separate `kyc_documents` table. The spec must decide. Given the brief says "document upload form" is required, document storage references are needed.

4. **Clarify whether `POST /v1/kyc/submit` is synchronous or async**: The mock adapter can be configured for synchronous approval (delay = 0). But the API response after submit should always return `pending` — the caller polls `GET /v1/kyc/status` to see the result. This is the correct pattern regardless of mock/real adapter.

5. **Define idempotency strategy for submissions**: The DB must prevent two concurrent `pending` submissions for the same account. A UNIQUE constraint on `(account_id)` where status is one of the in-flight states, or application-layer locking, must be specified.

6. **Specify error codes for KYC-specific errors**: `KYC_ALREADY_PENDING`, `KYC_LOCKED`, `KYC_ALREADY_VERIFIED`, `KYC_NOT_FOUND` — these must be defined in the spec for the implementation agent.

7. **Specify the admin role requirement for unlock**: The `POST /v1/kyc/admin/unlock/:accountId` endpoint requires `administrator` or `super_administrator` role. The spec must define how this role check works (using `req.authContext` and checking the roles array).

8. **Address the `events` table usage**: KYC status transitions should emit events to the event store. The spec must list exactly which events are emitted and what their payloads contain (no PII, only resource identifiers per L4-005 Section 9).

9. **Specify the `GET /v1/kyc/status` response shape**: What fields are returned? Status, failure count, verified_at, document_type — the spec must define the exact response shape.

10. **Out-of-scope clarifications for mock implementation**: Video liveness is skipped (mock auto-approves), sanctions screening always returns clear, document storage uses local filesystem. These simplifications must be clearly stated.

### Watch-Outs

1. **The `kyc_verifications` table uses `not_verified` but the brief says `not_started`** — if the implementation agent uses `not_started`, the DB CHECK constraint will reject inserts.

2. **No unique constraint on `(account_id)` in the existing migration** — the current schema allows multiple rows per account. The application must handle this (use the most recent row, or enforce one row per account at the application layer). A new migration adding a UNIQUE constraint may be needed.

3. **Document upload size validation at middleware layer** — 20 MB files must be validated before the request body parser tries to parse them. Express body parser limits must be configured correctly, or the server will OOM before validation runs.

4. **The mock adapter's auto-approval timing** — if the mock synchronously approves during the `submitVerification` call, the `POST /v1/kyc/submit` response must still return `pending`. The status change to `verified` and the role grant must happen out-of-band (via a setTimeout or a polling mechanism).

5. **Role grant atomicity** — if KYC transitions to `verified` and then the role grant fails, the system is in an inconsistent state. The role grant should be part of the same transaction as the KYC status update, or the spec must define a compensating mechanism.

6. **The `failure_count` column** — the existing migration initialises it to 0 with a non-negative CHECK. Atomic increment (`UPDATE ... SET failure_count = failure_count + 1`) is needed to prevent race conditions.

### Suggested Approach

**Architecture**: Follow the hexagonal architecture pattern established in the account bounded context:
- `packages/backend/src/kyc/domain/kyc-verification.ts` — entity with private constructor, `create()`, `reconstitute()`, state machine methods
- `packages/backend/src/kyc/ports/kyc-port.ts` — `KycPort` interface
- `packages/backend/src/kyc/ports/storage-port.ts` — `StoragePort` interface
- `packages/backend/src/kyc/ports/kyc-repository.ts` — `KycRepository` interface
- `packages/backend/src/kyc/adapters/mock/mock-kyc-adapter.ts` — auto-approving mock
- `packages/backend/src/kyc/adapters/mock/local-storage-adapter.ts` — filesystem mock
- `packages/backend/src/kyc/adapters/pg/pg-kyc-repository.ts` — PostgreSQL adapter
- `packages/backend/src/kyc/application/kyc-app-service.ts` — orchestrates ports
- `packages/backend/src/kyc/api/kyc-router.ts` — HTTP layer

**Mock adapter auto-approval**: The mock adapter should call a callback or emit an internal event after a configurable delay. The application service can handle the transition in a `processProviderResult` method. For simplicity in the workshop, the `submitVerification` call can schedule a `setTimeout` that calls back into the application service to set the status to `verified` and grant the creator role.

**New migration required**: A new migration should add document storage reference columns to `kyc_verifications` (or create a `kyc_documents` table). Recommendation: add `front_document_ref TEXT` and `back_document_ref TEXT` columns to `kyc_verifications` — this is simpler for the local demo scope.
