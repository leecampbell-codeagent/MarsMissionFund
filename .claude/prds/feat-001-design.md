# Design Spec: feat-001 — Monorepo Scaffold & Dev Environment

> Component-level UI specification. Generated by Design Speccer.
> Date: 2026-03-04
> Depends On: L2-001 (Brand Application Standard), L3-005 (Frontend Standards), feat-001-spec.md

---

## Overview

feat-001 is the monorepo scaffold. The frontend portion establishes the foundational design system infrastructure that all future features build upon. This spec defines:

1. The complete CSS custom properties file implementing all Tier 2 semantic tokens from `specs/standards/brand.md`
2. Global styles (dark background, CSS reset, font loading)
3. A base page layout shell (PageShell component)
4. An initial placeholder landing page that proves the stack works
5. Reduced-motion and accessibility foundations

Since this is the first feature, there are no existing components or tokens to extend. Everything defined here is net-new foundation.

## Page Layouts

### Landing Page (Initial Placeholder)

The feat-001 landing page is a single full-viewport page that proves the frontend stack is working. It is intentionally minimal — not a production landing page. Future features will replace this content.

```
+----------------------------------------------------------+
|  [PageShell]                                              |
|  +------------------------------------------------------+|
|  | <header> — nav placeholder (logo area + spacer)      ||
|  +------------------------------------------------------+|
|  | <main>                                                ||
|  |                                                       ||
|  |   MARS MISSION FUND          (--type-hero, centered)  ||
|  |   Crowdfunding the Next      (--type-body, centered)  ||
|  |   Giant Leap                                          ||
|  |                                                       ||
|  +------------------------------------------------------+|
|  | <footer> — minimal footer placeholder                 ||
|  +------------------------------------------------------+|
+----------------------------------------------------------+
```

**Layout rules:**

- Full viewport height (`min-height: 100vh`)
- Content centered both horizontally and vertically in the `<main>` area
- `<main>` uses `flex: 1` to fill available space between header and footer
- Maximum content width: `1280px` (matches `--breakpoint-xl`) with auto margins
- Horizontal padding: `16px` (mobile), `24px` (sm+), `32px` (lg+)

**Responsive title sizing:**

| Breakpoint | `--type-hero` font-size |
| ---------- | ----------------------- |
| Base (mobile) | `40px` |
| sm (640px+) | `56px` |
| lg (1024px+) | `80px` |
| xl (1280px+) | `96px` (full hero size) |

The `--type-hero` spec defines 96px as the hero size, but this must scale down on smaller viewports to prevent horizontal overflow. - Single `<h1>` on the page (document outline)
- Text content is readable without images or JavaScript-dependent rendering
- Contrast: `--color-text-primary` on `--color-bg-page` = 14.8:1 (AAA)
- Contrast: `--color-text-secondary` on `--color-bg-page` = 10.7:1 (AAA)

## Design Tokens

### CSS Custom Properties File

**File:** `packages/frontend/src/styles/tokens.css`

This file defines ALL Tier 2 semantic tokens from `specs/standards/brand.md` as CSS custom properties. It is imported in the global stylesheet and provides the complete token vocabulary for all current and future components.

The file is structured in two layers within a single `:root` block:

1. Tier 1 identity tokens (internal use only — never referenced by components)
2. Tier 2 semantic tokens (the component consumption layer)

Components must only reference Tier 2 tokens. Tier 1 tokens are present in this file solely to provide the mapping chain. A build-time lint rule should flag any component CSS that references a Tier 1 token name.

## Font Files

**Directory:** `packages/frontend/public/fonts/`

The following WOFF2 font files must be present. They are self-hosted per L3-005 Section 9.1 (no third-party font CDN dependency). Files should be subset to Latin + extended Latin.

| File | Font | Weight |
| ---- | ---- | ------ |
| `bebas-neue-v14-latin-regular.woff2` | Bebas Neue | 400 |
| `dm-sans-v15-latin-regular.woff2` | DM Sans | 400 |
| `dm-sans-v15-latin-500.woff2` | DM Sans | 500 |
| `dm-sans-v15-latin-600.woff2` | DM Sans | 600 |
| `dm-sans-v15-latin-700.woff2` | DM Sans | 700 |
| `space-mono-v13-latin-regular.woff2` | Space Mono | 400 |
| `space-mono-v13-latin-700.woff2` | Space Mono | 700 |

**Font weight justification from the type scale:**

- Bebas Neue 400: All display uses (`--type-hero`, `--type-page-title`, `--type-section-heading`, `--type-stat-value`, `--type-stat-value-compact`)
- DM Sans 400: `--type-body`, `--type-body-small`
- DM Sans 500: Stat card labels (brand.md Section 3.4)
- DM Sans 600: `--type-button`, `--type-input-label`
- DM Sans 700: `--type-card-title`
- Space Mono 400: `--type-label`, `--type-section-label`, `--type-data`
- Space Mono 700: Future emphasis in data contexts

**Font preload hint** (in `index.html` `<head>`):

```html
<link rel="preload" href="/fonts/dm-sans-v15-latin-regular.woff2" as="font" type="font/woff2" crossorigin>
```

DM Sans regular is preloaded because it is the body font and critical for first contentful paint.

```html
    <noscript>
      <div style="background:#060A14;color:#E8EDF5;font-family:system-ui,sans-serif;min-height:100vh;display:flex;align-items:center;justify-content:center;padding:32px;text-align:center;">
        <p>Mars Mission Fund requires JavaScript to run. Please enable JavaScript in your browser settings.</p>
      </div>
    </noscript>
    <script type="module" src="/src/main.tsx"></script>
  </body>
</html>
```

Note: The `<noscript>` fallback uses inline styles with raw values (not tokens) because CSS custom properties are not available without the stylesheet loading. The colours match `--color-bg-page` (#060A14) and `--color-text-primary` (#E8EDF5) and the font falls back to `system-ui` since web fonts will not load without JS. This is the one acceptable exception to the "no raw values" rule.

## Responsive Behaviour

### Breakpoints

Per L3-005 Section 5.2, the application uses four named breakpoints with mobile-first (`min-width`) media queries:

| Token | Value | Context |
| ----- | ----- | ------- |
| `--breakpoint-sm` | `640px` | Large phones landscape, small tablets |
| `--breakpoint-md` | `768px` | Tablets portrait |
| `--breakpoint-lg` | `1024px` | Tablets landscape, small desktops |
| `--breakpoint-xl` | `1280px` | Standard desktops and above |

### feat-001 Responsive Rules

For the scaffold, responsive behaviour is limited to:

1. **Content width**: Max `1280px` with auto horizontal margins
2. **Horizontal padding**: `16px` (base) / `24px` (sm+) / `32px` (lg+)
3. **Hero title sizing**: Scales from `40px` (mobile) to `96px` (xl+) — see LandingPlaceholder component spec
4. **Header/footer**: Single-row layout at all breakpoints (no mobile navigation collapse needed yet)

### Touch Targets

All interactive elements must meet 44x44px minimum touch target (WCAG 2.1 SC 2.5.5). For feat-001 this applies to:

- The skip-to-content link
- Any future interactive elements added to header/footer

## Animation & Transitions

### Base Motion Tokens

All motion semantic tokens are defined in `tokens.css` (see Design Tokens section above). For feat-001 the only animation is:

| Element | Animation | Token |
| ------- | --------- | ----- |
| Skip link appearance on focus | `top` transition | `--motion-hover-duration` / `--motion-hover-easing` |

### Reduced Motion

When `prefers-reduced-motion: reduce` is active (defined in `tokens.css`):

- All duration tokens resolve to `0ms` (instant transitions)
- The skip link appears instantly without transition
- `--motion-enter-emphasis-duration` resolves to `--duration-fast` (150ms) with linear easing for a minimal fade

### Performance

Per L3-005 Section 6.2:

- Only `transform` and `opacity` should be animated where possible
- The skip link animates `top` which triggers layout — acceptable for a single element that only animates on focus
- No `will-change` needed for feat-001 (no continuous animations)

## New Patterns Introduced

feat-001 establishes the following foundational patterns that all future features inherit:

### 1. Two-Tier Token Architecture

The `tokens.css` file establishes the Tier 1 to Tier 2 mapping. All future components consume only Tier 2 semantic tokens. This is the single most important pattern in the frontend.

### 2. PageShell Layout Pattern

The header/main/footer flex layout with constrained content width and responsive padding. All future pages render inside this shell.

### 3. Global CSS Reset

The `global.css` reset normalises browser defaults. Future components do not need their own resets.

### 4. Self-Hosted Font Loading

Fonts are self-hosted in `public/fonts/` with appropriate `font-display` strategies per L3-005 Section 9.1. Future font weights or families follow the same pattern.

### 5. Skip-Link Accessibility Pattern

The skip-to-content link pattern is established in PageShell. It must be preserved in all future page layouts.

### 6. Focus-Visible Global Style

The `:focus-visible` outline style is set globally. Components should not override this unless they provide an equivalent visible alternative (per L2-001 Section 5.3).

### 7. Reduced Motion Foundation

The `@media (prefers-reduced-motion: reduce)` block in `tokens.css` overrides all motion tokens. Future animations automatically respect this if they use semantic motion tokens.
