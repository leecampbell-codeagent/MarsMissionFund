## feat-002: Core Database Schema

**Bounded Context(s):** Cross-cutting (data layer)
**Priority:** P0
**Dependencies:** feat-001
**Estimated Complexity:** M

### Summary

Creates all core database tables via dbmate migrations covering the five domain bounded contexts: accounts, KYC, campaigns, payments/escrow, and contributions. Establishes the data foundation that all domain features depend on. Monetary columns use BIGINT (integer cents). All tables include `created_at` and `updated_at` with auto-update triggers.

### User Story

As a developer, I want a well-defined database schema so that I can build domain features against a stable data model.

### Key Decisions / Open Questions

- All monetary amounts stored as BIGINT integer cents (never float/decimal)
- UUIDs generated by the application layer (not database sequences)
- Clerk ID stored as the external identity reference in `users.clerk_id`
- The `updated_at` auto-trigger from the existing migration `20260305120000` should be applied to all tables that have `updated_at`

### Out of Scope

- Audit log tables (separate concern per L3-006)
- Campaign team members, risk disclosures, media attachments (these can be added as subsequent migrations when features need them)
- Notification preferences table
- Disbursement table (part of feat-013)
