## 1. Domain Knowledge

### Key Concepts

**Campaign Discovery** is the primary entry point for the donor journey.
It encompasses two surfaces: the search/browse listing and the individual campaign detail page.
Both surfaces must work for anonymous users — no Clerk JWT is required.

**Full-Text Search (FTS)** in PostgreSQL means searching across text columns using linguistic
analysis (stemming, stop words) to find relevant results ranked by relevance.
Two PostgreSQL FTS approaches are relevant:

- `tsvector`/`tsquery`: PostgreSQL's native full-text search. Tokens are normalised via
  dictionary-based stemming. A `tsvector` column stores pre-computed lexemes; a `GIN` index
  makes search fast. `ts_rank()` or `ts_rank_cd()` provides relevance scoring. This is the
  production-grade approach — appropriate for a workshop demo that should show real patterns.
- `pg_trgm`: Trigram similarity extension. Splits text into 3-character sequences and finds
  matches by overlap. Supports fuzzy matching and LIKE queries. Simpler to set up but does not
  provide linguistic stemming or meaningful relevance ranking.

For MMF's workshop demo, `tsvector`/`tsquery` is the correct choice because:
1. It models the real-world pattern for full-text search on PostgreSQL.
2. It provides `ts_rank()` relevance scoring required by the acceptance criteria ("sorted by relevance").
3. A single GIN index on a pre-computed `tsvector` column is easy to explain in a workshop context.
4. The spec brief explicitly calls this out as the option to evaluate.

**Funding Progress** is the ratio of total confirmed contributions to the campaign's minimum
funding goal, expressed as a percentage and an absolute amount in cents.
Because feat-005 (contribution/payment flow) has not been implemented, there is no
`contributions` table at the time feat-004 is implemented.
This is the most significant gap to resolve in the spec.

**"Funded" State** is a distinct campaign status (`funded`) representing a campaign that has
already reached its minimum funding goal.
The `CampaignStatus.Funded` value exists in the domain model (`packages/backend/src/campaign/domain/value-objects/campaign-status.ts`).
The `funded` status is reached when the total confirmed contributions reach `funding_goal_cents`.
This transition is triggered by the payments domain (future feat-005).
For the feat-004 demo, campaigns must be seedable in both `live` and `funded` states to exercise the UI.

**"Ending Soon"** filter means deadline within 7 days from the current server time.
This is an interpreted filter — not a campaign status value.
The brief and L4-003 (AC-DONOR-003) agree: campaigns with deadline within 7 days, sorted
by nearest deadline first.

**Contributor Count** is the count of distinct confirmed contributors, not a dollar total.
Individual identities are never exposed on the public page (opt-in feature is out of scope).
With no contributions table in feat-004, this will be stubbed as zero or omitted.

**Category Aggregate Stats** (for category browse pages) require:
- Campaign count: `COUNT(*) WHERE category = $1 AND status IN ('live', 'funded')`
- Total raised: requires contributions table — stubbed as `'0'` for feat-004
- Active campaign count: `COUNT(*) WHERE category = $1 AND status = 'live'`

### Calculation Methodology

**Funding Progress Percentage**:
```
percentage = (total_raised_cents / funding_goal_cents) * 100
```
Where both values are strings from the API (BIGINT), convert using `BigInt` arithmetic to avoid
floating-point precision loss. Display is frontend responsibility.

**Days Remaining**:
```
days_remaining = ceil((deadline_unix_ms - now_unix_ms) / (1000 * 60 * 60 * 24))
```
Negative values mean the deadline has passed. Display as 0 if past deadline.

**"Funded" Badge Logic**:
A campaign is visually "Fully Funded" when its status is `funded`.
It is "Ending Soon" when its deadline is within 7 days AND status is `live` or `funded`.

### Industry Conventions

Kickstarter, Indiegogo, and Experiment.com all show:
- Campaign cards with progress bars (percentage + amount raised)
- Days remaining as an integer (not timestamp)
- Hero image prominently
- Creator/team identifier
- Category badge
- CTA button ("Back This Project", "Fund This Campaign")

Kickstarter uses a "Project We Love" badge — MMF does not have editorial collections in MVP.
Indiegogo shows "Funding" vs "InDemand" (past deadline but still accepting) — MMF does not
have "InDemand" semantics; post-deadline funded campaigns move to `settlement` or `complete`.

For search, Kickstarter and Indiegogo both use Elasticsearch externally.
MMF's spec brief correctly notes that PostgreSQL FTS is sufficient for the workshop demo.

### Regulatory Notes

No regulatory considerations specific to campaign discovery (viewing is passive, no financial action).
The contributor count display rule (count only, no identities) is a privacy consideration from L4-003.
GDPR does not restrict public viewing of campaign content since campaigns are opt-in public disclosures.

Existing indexes: `creator_user_id`, `status`, `submitted_at`, `reviewed_by_user_id`.

**Missing indexes for feat-004**:
- `deadline` — needed for "ending_soon" sort and filter
- `launched_at` — needed for "newest" sort
- `category` — needed for category browse
- A GIN index on a full-text search column (new column needed)

**`users` table** (from `db/migrations/20260305130000_create_users_table.sql`):
- `id UUID`, `clerk_user_id TEXT`, `display_name TEXT`, `email TEXT`
- Creator name for search results comes from `users.display_name` (or `email` as fallback)
- There is NO `contributions` table yet — feat-005 creates this

**No contributions table exists**: feat-004 cannot query real funding progress. The spec must
decide how to handle `total_raised_cents` and `contributor_count` fields.

### Public Status Enumeration

The campaign domain model defines all statuses. The public-visible statuses per the brief are:
- `live` — accepting contributions, publicly visible
- `funded` — goal reached, still accepting until deadline or cap (per L4-002 Section 7.4)

The brief says "Only campaigns in `Live` or `Funded` state are returned in search results."

Cross-checking against the existing `getCampaign` access control in `campaign-app-service.ts`:
The current `getCampaign` (authenticated) also allows public access to: `suspended`, `failed`,
`settlement`, `complete`, `cancelled`. The feat-004 public endpoint should be more restrictive —
only `live` and `funded` as specified in the brief's acceptance criteria.

Key finding: there is currently NO unauthenticated endpoint. The `GET /api/v1/campaigns/:id`
route requires auth (`requireAuth` is applied globally to `/api/v1/campaigns`). This is the
central architectural challenge for feat-004.

- `fundingGoalCents` and `fundingCapCents` are `string | null` (G-024 pattern)
- `status` is a `CampaignStatus` union type
- No `totalRaisedCents` or `contributorCount` fields — these come from the payments domain

**CampaignStatus value object** (`packages/backend/src/campaign/domain/value-objects/campaign-status.ts`):
All 13 statuses defined. `Funded: 'funded'` IS present. The brief's concern about a missing

Neither serializer includes `totalRaisedCents` or `contributorCount`. New serializers or

**Critical finding**: `requireAuth` is applied as a blanket middleware in `app.ts`:
```typescript
app.use('/api/v1/campaigns', requireAuth, createCampaignRouter(...));
```
This means ALL `/api/v1/campaigns/*` routes require auth. To allow anonymous access, the
spec must choose one of:
1. Mount a separate `/api/v1/public/campaigns` router WITHOUT requireAuth
2. Conditionally apply auth in `createCampaignRouter()` for specific routes
3. Refactor the middleware application in `app.ts`

Option 1 (separate `/api/v1/public/campaigns` prefix) is cleanest architecturally —
no changes to the existing authenticated router, no mixup of auth rules.

### Auth Middleware Architecture

From `packages/backend/src/shared/middleware/auth.ts`:
- `clerkMiddleware()` is applied globally (attaches `req.auth` to all requests)
- `createRequireAuth()` is the guard that returns 401 if no `auth.userId`
- `getClerkAuth(req)` returns `{ userId }` or `null` — used inside handlers

The `clerkMiddleware()` is safe to apply globally — it does NOT reject unauthenticated
requests, it just populates `req.auth`. The `createRequireAuth()` middleware is the
enforcement gate. Therefore, anonymous endpoints can be added under the same Express app
— just don't apply `requireAuth` to those routes.

## 4. Competitor Patterns

### Kickstarter

- Campaign cards show: hero image, title, creator, category, progress bar (% + dollar amount), days left
- Search uses Elasticsearch; results show relevance ranking with matched term highlighting
- "Projects We Love" editorial badge
- Category pages show total projects, total pledged, % success rate
- Anonymous browsing fully supported; login only required to back a project
- "Ending Soon" filter is prominent and popular

**What works well**: Progress bar is central visual — percentage AND amount raised both shown.
Days remaining displayed as integer ("12 days to go"). "Funding" status badge on active campaigns.

**Anti-patterns to avoid**: Kickstarter's project categories have drifted to hundreds of subcategories,
making navigation complex. MMF's 10-category taxonomy is deliberately simple — keep it.

### Indiegogo

- "Fixed" vs "Flexible" funding model displayed prominently
- "InDemand" mode for campaigns past their deadline but still accepting contributions
- Category aggregate stats on browse pages include campaign count and total raised
- "Trending" sort based on recent contribution velocity — not in MMF scope for MVP

**What works well**: "Fully Funded" green badge on funded campaigns. Amount raised shown above
the progress bar. "Back It" CTA button always visible.

### Experiment.com (Science Crowdfunding)

- Mission-driven framing: "Fund Science" instead of generic CTA
- Creator team credentials displayed prominently — matches MMF's team section
- Progress shown as "XX% of $XX,XXX goal"
- Anonymous browsing supported

**What works well**: Milestone plan displayed on the campaign page (matches MMF requirement).
Risk disclosures section on public page (matches MMF requirement).

## 5. Edge Cases

### Data Edge Cases

- [ ] Campaign with `funding_goal_cents = null` (draft that was launched incorrectly): Funding progress percentage division by zero — backend must guard against null `funding_goal_cents` before computing percentage; return `null` or `"0"` for progress when goal is null.
- [ ] Campaign with `deadline = null` (malformed live campaign): Days remaining calculation must handle null deadline; return `null` for `daysRemaining`.
- [ ] Campaign with `hero_image_url = null`: Frontend must render a placeholder image, not a broken `<img>` tag.
- [ ] Campaign with empty `milestones = []` array: Detail page must handle empty milestones gracefully (no milestone section, or "No milestones defined").
- [ ] Campaign with `short_description = null`: Search result card must render campaign title only; do not crash on null description.
- [ ] Campaign `title` at maximum length (200 chars): Card layout must handle text overflow without breaking layout.
- [ ] Search term with SQL special characters (`%`, `_`, `'`, `\`): Parameterised queries protect against SQL injection; `tsquery` must sanitise special tsquery operators (`&`, `|`, `!`, `:`). Raw search term must be sanitised before conversion to tsquery.
- [ ] Search term with only whitespace: Backend must trim and treat as empty search (return all campaigns, sorted by default).
- [ ] Search term that matches no campaigns: Return empty array with 200, not 404.
- [ ] Category value not in the 10-category enum: Return 400 VALIDATION_ERROR, not 500.

### Funding Progress Edge Cases (Future / Stub)

- [ ] `total_raised_cents` equals `funding_goal_cents`: Campaign is exactly 100% funded but may still be in `live` status (transition to `funded` happens when payment is confirmed). Status must be the source of truth for the "Fully Funded" badge, not the progress percentage.
- [ ] `total_raised_cents` exceeds `funding_goal_cents`: Possible when cap is higher than goal and contributions continue after goal is met. Progress percentage can exceed 100% — display should show actual amount raised, not cap at 100%.
- [ ] `total_raised_cents = '0'` (stubbed): Progress bar renders at 0%, not as an error state.
- [ ] `funding_goal_cents` is `null` on a live campaign (data anomaly): Backend must return `null` for percentage and amount; frontend must handle `null` progress gracefully.

### Concurrency Edge Cases

- [ ] Two concurrent search requests with different filter combinations hitting the same index: PostgreSQL handles concurrent reads safely; no application-level concern.
- [ ] Campaign transitions from `live` to `funded` during a search request: Search results may include the campaign in either status depending on timing; both statuses are publicly visible, so this is benign.
- [ ] Campaign transitions from `live` to `suspended` mid-page-view: The detail page data is a point-in-time snapshot. A suspended campaign should NOT be accessible via the public endpoint; a 404 should be returned on subsequent fetches.
- [ ] `pg_trgm` or `tsvector` index not yet built on first request: The migration must create the GIN index before the endpoint is first accessed. This is a migration ordering concern, not a runtime concern.

### Integration Edge Cases

- [ ] Clerk `clerkMiddleware()` applied globally already populates `req.auth` for all requests — the anonymous endpoint does not need to call `getAuth()` at all, but calling it and checking for `null` is safe (returns `{ userId: null }` for unauthenticated requests).
- [ ] `requireAuth` middleware currently wraps the entire `/api/v1/campaigns` mount point in `app.ts`. Adding new public routes to the existing `campaign-router.ts` will cause them to also be auth-required. The public routes MUST be mounted separately without the `requireAuth` middleware.
- [ ] Joining `users` table to get `display_name` for creator: `users.display_name` can be `null`. The search result response must include `creatorName: display_name || email` fallback logic — or accept `null` and let frontend display "Unknown Creator".
- [ ] Search query parameter `q` with non-UTF-8 characters: Express will reject malformed URLs at the HTTP layer; Zod should validate `q` as a string with reasonable max length (e.g., 200 chars).

### User Behaviour Edge Cases

- [ ] Anonymous user bookmarks a campaign URL and returns after the campaign enters a non-public status (e.g., `suspended`): The `GET /api/v1/public/campaigns/:id` endpoint must return 404 for non-`live`/`funded` statuses — same response as a non-existent campaign (do not reveal status).
- [ ] User navigates to search page with invalid filter state in URL (e.g., `?sort=invalid`): Zod validation on query params must return 400 VALIDATION_ERROR with a clear message about which filter is invalid.
- [ ] User applies "Ending Soon" filter when no campaigns have deadlines within 7 days: Return empty array with 200, not an error.
- [ ] User applies multiple category filters (multi-value): The acceptance criteria says `category` is "multi-value" — need `IN ($1, $2, ...)` SQL. The Zod schema must accept `category` as a string array or comma-separated string.
- [ ] User loads campaign detail page for a campaign in `funded` status: "Back This Mission" CTA should still appear (funded campaigns continue accepting contributions until deadline or cap per L4-002 Section 7.4). The CTA may be disabled only if the cap is reached.

### Boundary Edge Cases

- [ ] Offset-based pagination: `offset=0, limit=20` is the default. Max `limit` should be capped (e.g., 100) to prevent excessive DB reads. Zod should validate `limit <= 100`.
- [ ] Very large `offset` (e.g., `offset=10000`): PostgreSQL will skip 10,000 rows efficiently with the GIN index. For the workshop demo this is acceptable; in production, cursor-based pagination avoids this.
- [ ] `page` parameter instead of `offset`: The brief recommends offset/limit; do not accept `page` as an alias without documenting it.
- [ ] Category aggregate stats: `total_raised_cents` will always be `'0'` for feat-004 (no contributions table). The spec must explicitly state this is a known stub, not a bug.
- [ ] Empty database: All public endpoints must return empty arrays and aggregate stats of zero, not errors.
- [ ] Campaign with 10 milestones and 20 team members (maximum): The detail endpoint must return all nested JSONB data without truncation.
- [ ] Search with `q` parameter containing tsquery operators (`&`, `|`, `!`): These must be escaped or stripped before the value is passed to `to_tsquery()`. Use `plainto_tsquery()` or `websearch_to_tsquery()` to avoid injection into tsquery syntax.

## 6. Recommendations

### Must-Haves for Spec

1. **Anonymous access architecture**: The spec must explicitly define that public endpoints are mounted at a SEPARATE path (e.g., `/api/v1/public/campaigns`) in `app.ts` WITHOUT the `requireAuth` middleware. Reusing the existing authenticated router is not viable without refactoring `app.ts`'s middleware application.

2. **Contributions stub strategy**: The spec must explicitly state that `totalRaisedCents` returns `'0'` and `contributorCount` returns `0` in feat-004, because the contributions table does not exist until feat-005. This must be documented as a known limitation in the spec, not left ambiguous.

3. **Full-text search implementation**: The spec must prescribe `tsvector`/`tsquery` with a GIN index. A computed `tsvector` column (`search_vector`) should be added to the `campaigns` table via migration, populated by a trigger that concatenates `title`, `short_description`, `description`, and joins `users.display_name`. The search query uses `websearch_to_tsquery()` (available since PostgreSQL 11) which safely handles arbitrary user input including tsquery operators.

4. **Separate serializer for public response**: The public campaign list item response includes fields NOT in the existing `serializeCampaignSummary()` (e.g., `daysRemaining`, `heroImageUrl`, `totalRaisedCents`). A new `serializePublicCampaignSummary()` and `serializePublicCampaignDetail()` must be defined.

5. **New repository methods**: The `CampaignRepository` port must be extended with:
   - `searchPublicCampaigns(options: PublicSearchOptions): Promise<CampaignSearchResult[]>` — full-text search with filters and sorting
   - `findPublicById(id: string): Promise<Campaign | null>` — returns only `live`/`funded` campaigns
   - `getCategoryStats(category: CampaignCategory): Promise<CategoryStats>` — aggregate stats

6. **Status filter naming**: The brief says the API accepts `status` filter values: `active`, `funded`, `ending_soon`. These are API-level filter values, NOT the internal `CampaignStatus` enum values. The spec must define the mapping:
   - `active` → `status = 'live'`
   - `funded` → `status = 'funded'`
   - `ending_soon` → `status IN ('live', 'funded') AND deadline <= NOW() + INTERVAL '7 days'`

7. **New migration required**: The migration must:
   - Add `search_vector TSVECTOR` column to `campaigns`
   - Add a GIN index on `search_vector`
   - Add a trigger to update `search_vector` on INSERT/UPDATE
   - Add an index on `campaigns.deadline` (for "ending_soon" sort/filter)
   - Add an index on `campaigns.launched_at` (for "newest" sort)

8. **Route ordering with G-023**: The public search endpoint `GET /api/v1/public/campaigns` and the public detail endpoint `GET /api/v1/public/campaigns/:id` must follow the same route registration order rule — literal paths before parameterised paths.

9. **Funding progress calculation boundary**: When `funding_goal_cents` is not null, percentage = `(totalRaisedCents / fundingGoalCents) * 100`. For feat-004, `totalRaisedCents = '0'` always, so percentage = `0`. The spec must explicitly note this for the implementation agent.

10. **Creator name JOIN**: The public search query must JOIN `campaigns` with `users` on `campaigns.creator_user_id = users.id` to return `users.display_name` (with `email` fallback). This is a cross-context read at the repository level — acceptable per pattern P-023 since it is a read-only JOIN.

### Watch-Outs

**W-001: `websearch_to_tsquery` vs `plainto_tsquery` vs `to_tsquery`**
Use `websearch_to_tsquery('english', $1)` for user-supplied search terms. It:
- Handles operator characters safely (no tsquery injection)
- Supports quoted phrases (`"nuclear propulsion"`)
- Available since PostgreSQL 11 (Aurora PostgreSQL >= 11, which is the target)
- Returns `NULL` for empty/invalid queries — the query must guard: `WHERE ($1 = '' OR search_vector @@ websearch_to_tsquery('english', $1))`

**W-002: The `tsvector` trigger must JOIN the users table to include creator name**
The `search_vector` column must include creator display name for the acceptance criteria to be met ("full-text search across campaign title, description, and creator name"). A trigger on the `campaigns` table cannot JOIN `users` directly. Options:
- Store a denormalized `creator_display_name TEXT` column on `campaigns` (updated via trigger when user name changes — complex, cross-context coupling)
- Or include creator name at query time in the WHERE clause, not in the pre-computed tsvector: `WHERE search_vector @@ ... OR to_tsvector('english', u.display_name) @@ ...` (slower but simpler)
- Or populate a separate `campaign_search_index` read model table with denormalized creator name and search vector (CQRS pattern, cleanest for workshop demo)
The CQRS read model table approach is the cleanest but adds implementation complexity. The spec should prescribe the JOIN approach at query time for the workshop (simpler, still correct).

**W-003: Pagination offset performance**
Offset-based pagination degrades on large offsets (PostgreSQL must scan and skip N rows). For the workshop demo with a small seeded dataset this is fine. The spec should note this limitation and recommend cursor-based for production.

**W-004: Multi-value category filter SQL**
`category IN ($1, $2, $3)` requires dynamic query construction when the number of categories varies. Use an array parameter: `WHERE category = ANY($1::TEXT[])` and pass a PostgreSQL array. This avoids dynamic SQL while supporting multi-value.

**W-005: `funded` campaigns and the CTA**
The brief says funded campaigns display "Fully Funded" badge but the acceptance criteria implies the CTA is still present. Per L4-002 Section 7.4, funded campaigns continue accepting contributions until the deadline or cap. The CTA should remain active for `funded` campaigns, disabled only when `funding_cap_cents` is reached. Since `totalRaisedCents` is stubbed as `'0'`, the cap is never reached in feat-004 — CTA is always active.

**W-006: `Archived` status and public visibility**
The `getCampaign` app service currently allows public access to `suspended`, `failed`, `settlement`, `complete`, and `cancelled` campaigns. The feat-004 brief says only `live` and `funded` are publicly discoverable. This is a narrower restriction. The new public endpoint must enforce this explicitly. The existing authenticated endpoint's broader access remains unchanged.

**W-007: The `search_vector` tsvector trigger fires on every UPDATE to campaigns**
Every auto-save (PATCH) to a draft campaign will fire the trigger even though draft campaigns are not publicly discoverable. This is correct behaviour but adds write overhead. For the workshop demo dataset, this is negligible. The spec should note it.

### Suggested Approach

**Backend approach:**

1. Create a new migration that adds `search_vector TSVECTOR`, a GIN index, and a trigger to auto-populate `search_vector` from `title || ' ' || COALESCE(short_description, '') || ' ' || COALESCE(description, '')` when `INSERT` or `UPDATE` occurs on `campaigns`. Creator name is included at query time via JOIN.

2. Mount a new `/api/v1/public/campaigns` router in `app.ts` WITHOUT `requireAuth`. Apply `clerkMiddleware()` (already global — it runs on all requests). The router handler calls `getClerkAuth(req)` — this returns `null` for anonymous users — and proceeds without auth.

3. New port methods on `CampaignRepository`:
   - `searchPublicCampaigns(options)` — executes FTS + filter + sort + pagination query
   - `findPublicById(id)` — enforces `status IN ('live', 'funded')` filter
   - `getCategoryStats(category)` — count-based aggregation

4. New application service methods on `CampaignAppService`:
   - `searchPublicCampaigns(options)` — delegates to repository, no auth required
   - `getPublicCampaign(campaignId)` — delegates to `findPublicById`, returns 404 for non-public
   - `getCategoryStats(category)` — delegates to repository

5. New serializers:
   - `serializePublicCampaignListItem()` — includes `totalRaisedCents: '0'`, `contributorCount: 0`, `daysRemaining`, `heroImageUrl`, `creatorName`
   - `serializePublicCampaignDetail()` — full detail including milestones, team, risks, budget, plus funding progress fields

**Frontend approach:**
Campaign search page with URL state management for filters (TanStack Query + React Router search params). Campaign detail page accessible at `/campaigns/:id`. Category browse accessible at `/campaigns?category=propulsion`.

**Pagination recommendation:**
Offset/limit for the workshop demo (simpler, fewer edge cases to explain). Default: `limit=20, offset=0`. Max limit: 100. Both values must be Zod-validated as non-negative integers.

## 7. Open Questions for Spec Writer

These questions the brief leaves ambiguous and require product decisions:

1. **Anonymous 404 vs. 403**: When an anonymous user requests a campaign that exists but is in a non-public status (e.g., `draft`), should the response be 404 (security-correct — do not reveal existence) or 403? Recommendation: 404, consistent with how the existing authenticated endpoint handles ownership-without-access.

2. **Creator name field**: `users.display_name` can be null. What should `creatorName` be when null? Options: `null` (frontend handles), `email` (exposes PII), or a generic fallback like `"Anonymous Creator"`. Recommendation: `display_name || null` — let frontend render "Creator" as fallback.

3. **`funded` campaign CTA behaviour**: The brief says funded campaigns show "Fully Funded" badge. Does the "Back This Mission" CTA still appear? Per L4-002, yes. Spec should confirm this.

4. **Category stats `total_raised_cents` stub**: Should the category stats endpoint return `'0'` for `total_raised_cents` and `0` for contributor count, or omit these fields entirely? Recommendation: return `'0'` and `0` with a clear comment in the API contract that these are stubs pending feat-005.

5. **Search result sorting when `q` is empty**: If no search term is provided, what is the default sort? The brief says the endpoint sorts by relevance when `q` is provided, but does not specify the default sort when `q` is absent. Recommendation: default to `newest` (by `launched_at DESC`) when no search term.
