## Warnings (Should Fix)

### WARN-001: Explicit return types not mandated in spec
**Affects:** Feature spec — `auth-sync-service.ts` and all exported functions.
**Details:** The spec describes `syncUser` with its signature in prose (`syncUser(clerkUserId: string, emailFromJwt: string | null): Promise<User>`) but does not include a general mandate that all exported functions must have explicit TypeScript return types. This is required by `CLAUDE.md` ("Explicit return types on all exported functions"). Implementation agents should be reminded to add `: Promise<User>`, `: Promise<void>`, `: void` etc. on all exported functions.
**Recommendation:** No spec revision needed — implementation agents should follow `CLAUDE.md` directly on this point. Flagged for awareness.

### WARN-002: `pending_verification` inconsistency between US-005 and middleware logic
**Affects:** Feature spec — US-005 vs. `mmfAuthMiddleware` step 6.
**Details:** US-005 states: "Given a user with `account_status = 'active'` or `'pending_verification'` When they make an authenticated request Then the request proceeds normally." However, the `mmfAuthMiddleware` logic (Section "API Endpoints") explicitly returns `403 ACCOUNT_PENDING` for `pending_verification`. The middleware logic is correct per the resolution in the "Open Questions" section ("Account status gating for pending_verification: Treat as 403 ACCOUNT_PENDING"). US-005's acceptance criterion is inconsistent with the middleware and the open questions resolution.
**Recommendation:** Spec Writer should update US-005 to split the acceptance criterion for `pending_verification`:
- "Given a user with `account_status = 'active'` When they make an authenticated request Then the request proceeds normally"
- "Given a user with `account_status = 'pending_verification'` When they make an authenticated request Then `403` is returned with `{ error: { code: 'ACCOUNT_PENDING', message: 'Account verification required.' } }`"
This is a documentation inconsistency, not a logic error — the middleware logic and open questions section are correct. Marked WARN rather than FAIL because the authoritative logic is clear from the middleware spec and the open questions resolution.

### WARN-003: `--color-text-accent` at 11px section label may fail WCAG AA
**Affects:** Design spec — DashboardPage section label "01 — MISSION CONTROL".
**Details:** The design spec uses `--color-text-accent` for the 11px section label and acknowledges the potential contrast issue, suggesting `--color-action-primary-hover` as an alternative. Per WCAG 2.1, 11px text does not qualify as "large text" regardless of letter-spacing, so the 3:1 large text threshold does not apply — the 4.5:1 AA threshold for normal text applies. `--color-text-accent` (`--launchfire` #FF5C1A) on `--color-bg-page` (`--void` #060A14) = 4.8:1, which passes AA (4.5:1) with a small margin. This is technically compliant but fragile. The brand spec itself notes this limitation and the spec wisely flags it.
**Recommendation:** Implementation agents should use `--color-action-primary-hover` for section labels at 11px to guarantee AA compliance comfortably, as the design spec suggests. No spec revision required — the flag is already in the design spec.

### WARN-004: `useApiClient` includes `isSignedIn` in `useCallback` dependency without rationale
**Affects:** Feature spec — `useApiClient` hook.
**Details:** The `fetchWithAuth` function in `useApiClient` has `[getToken, isSignedIn, navigate]` as `useCallback` dependencies. `isSignedIn` is not used inside `fetchWithAuth` — the hook calls `getToken()` and checks if it returns `null`. Including `isSignedIn` in the dependency array is harmless but slightly odd, as changes to `isSignedIn` will trigger a new `fetchWithAuth` reference unnecessarily. The spec should either justify this inclusion or remove it.
**Recommendation:** Spec Writer should either remove `isSignedIn` from the dependency array (since it is not referenced inside `fetchWithAuth`) or add an explanatory comment in the spec justifying its inclusion (e.g., "included to force a fresh closure when auth state changes"). This is a minor hook correctness concern; it does not affect functionality.
