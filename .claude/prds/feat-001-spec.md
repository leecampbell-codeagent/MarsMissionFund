## 1. User Stories & Acceptance Criteria

### US-001: Working Local Development Environment

**As a** developer,
**I want** to run `npm run dev` from the repository root,
**so that** the full local stack (PostgreSQL, backend, frontend) starts with a single command.

**Acceptance Criteria:**

```gherkin
Given I have Docker and Docker Compose v2 installed
  And I have cloned the repository
  And I have run `npm install` at the root
When I run `npm run dev`
Then Docker Compose starts PostgreSQL, dbmate, backend, and frontend services
  And PostgreSQL is accessible on port 5432
  And the backend API responds at http://localhost:3001/health with {"status":"ok"}
  And the frontend dev server is accessible at http://localhost:5173
  And the frontend proxies /api requests to the backend
```

```gherkin
Given the local dev stack is running
When I modify a file in packages/backend/src/
Then the backend restarts automatically via tsx watch
  And the change is reflected without manual restart
```

```gherkin
Given the local dev stack is running
When I modify a file in packages/frontend/src/
Then Vite HMR updates the browser without full page reload
```

### US-002: Code Quality Toolchain

**As a** developer,
**I want** linting, formatting, and type checking to work from the repository root,
**so that** I can verify code quality before pushing.

**Acceptance Criteria:**

```gherkin
Given I am at the repository root
When I run `npm run lint`
Then ESLint checks all TypeScript files in both packages
  And markdownlint-cli2 checks all Markdown files
  And the command exits with 0 if no violations exist
```

```gherkin
Given I am at the repository root
When I run `npm run format`
Then Prettier checks formatting of all supported files
  And the command exits with 0 if all files are formatted correctly
```

```gherkin
Given I am at the repository root
When I run `npm run typecheck`
Then TypeScript compiler checks both packages with strict mode
  And the command exits with 0 if no type errors exist
```

### US-003: Test Runner

**As a** developer,
**I want** `npm test` to run all tests across both packages,
**so that** I can verify nothing is broken.

**Acceptance Criteria:**

```gherkin
Given I am at the repository root
When I run `npm test`
Then Vitest runs tests in both packages/backend and packages/frontend
  And at least one test passes in each package
  And the command exits with 0
```

### US-004: CI Pipeline

**As a** developer,
**I want** a GitHub Actions CI pipeline that runs on every PR to main,
**so that** code quality is enforced automatically.

**Acceptance Criteria:**

```gherkin
Given I open a pull request targeting main
When the CI workflow triggers
Then it runs lint, format check, typecheck, and tests
  And the PR status reflects pass or fail
```

### US-005: Database Migration Infrastructure

**As a** developer,
**I want** dbmate configured with a migrations directory,
**so that** I can add database migrations in future features.

**Acceptance Criteria:**

```gherkin
Given the local dev stack is running
When dbmate runs on startup
Then it connects to PostgreSQL successfully
  And it creates the schema_migrations table
  And it exits cleanly with no errors (even with zero migration files)
```

### US-006: Health Check Endpoint

**As a** developer (or orchestration layer),
**I want** a `GET /health` endpoint that returns 200,
**so that** I can verify the backend is running.

**Acceptance Criteria:**

```gherkin
Given the backend is running
When I send GET /health
Then the response status is 200
  And the response body is {"status":"ok"}
  And the response Content-Type is application/json
  And no authentication is required (per L2-002 Section 6.3)
```

## 9. Docker Configuration

### 9.1 docker-compose.yml (repository root)

```yaml
services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: ${POSTGRES_USER:-mmf}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:-mmf}
      POSTGRES_DB: ${POSTGRES_DB:-mmf_dev}
    ports:
      - "${POSTGRES_PORT:-5432}:5432"
    volumes:
      - mmf-pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER:-mmf} -d ${POSTGRES_DB:-mmf_dev}"]
      interval: 2s
      timeout: 5s
      retries: 10

  dbmate:
    image: ghcr.io/amacneil/dbmate
    command: ["--wait", "up"]
    environment:
      DATABASE_URL: "postgres://${POSTGRES_USER:-mmf}:${POSTGRES_PASSWORD:-mmf}@postgres:5432/${POSTGRES_DB:-mmf_dev}?sslmode=disable"
    volumes:
      - ./db/migrations:/db/migrations
    depends_on:
      postgres:
        condition: service_healthy

  backend:
    build:
      context: ./packages/backend
      dockerfile: Dockerfile
      target: development
    environment:
      PORT: ${PORT:-3001}
      NODE_ENV: development
      LOG_LEVEL: ${LOG_LEVEL:-debug}
      DATABASE_URL: "postgres://${POSTGRES_USER:-mmf}:${POSTGRES_PASSWORD:-mmf}@postgres:5432/${POSTGRES_DB:-mmf_dev}?sslmode=disable"
    ports:
      - "${PORT:-3001}:3001"
    volumes:
      - ./packages/backend/src:/app/src
      - /app/node_modules
    depends_on:
      dbmate:
        condition: service_completed_successfully
    healthcheck:
      test: ["CMD-SHELL", "wget --no-verbose --tries=1 --spider http://localhost:3001/health || exit 1"]
      interval: 5s
      timeout: 5s
      retries: 10
      start_period: 10s

  frontend:
    image: node:22-bookworm-slim
    working_dir: /app
    command: ["npm", "run", "dev", "--", "--host"]
    environment:
      VITE_API_URL: ${VITE_API_URL:-http://localhost:3001}
    ports:
      - "${FRONTEND_PORT:-5173}:5173"
    volumes:
      - ./packages/frontend:/app
      - /app/node_modules
    depends_on:
      backend:
        condition: service_healthy

volumes:
  mmf-pgdata:
```

**Key decisions:**

- **Port configurability**: All host-side ports use `${VAR:-default}` syntax so developers can override them via `.env` if they have port conflicts (addresses edge case 5.1).
- **Named volume `mmf-pgdata`**: PostgreSQL data persists across `docker compose down`.
  Use `docker compose down -v` for a clean reset (addresses edge case 5.13).
- **Anonymous volume `/app/node_modules`**: Isolates container dependencies from host-mounted source (addresses edge case 5.6 and gotcha: node_modules in Docker).
- **dbmate `--wait` flag**: Handles PostgreSQL startup timing beyond `depends_on` (addresses edge case 5.3 and gotcha: dbmate in Docker Compose).
- **Backend health check**: Uses `wget` (available in bookworm-slim) to check the health endpoint.
  Frontend depends on backend health so the proxy target is available.
- **Backend build target `development`**: The Dockerfile has a `development` stage for hot reload.
- **Frontend runs from `node:22-bookworm-slim`** with volume-mounted source and `npm run dev -- --host`.
  The `--host` flag is passed to Vite to bind to `0.0.0.0` (equivalent to `server.host: true`).

### 9.2 packages/backend/Dockerfile

```dockerfile
# ============================================================
# Stage 1: Base (shared between development and production)
# ============================================================
FROM node:22-bookworm-slim AS base

WORKDIR /app

# Copy dependency manifests first for cache efficiency (addresses edge case 5.11)
COPY package.json package-lock.json* ./

# ============================================================
# Stage 2: Development (hot reload with tsx watch)
# ============================================================
FROM base AS development

RUN npm install

COPY . .

# Use non-root user for security
USER node

EXPOSE 3001

CMD ["npx", "tsx", "watch", "src/index.ts"]

# ============================================================
# Stage 3: Builder (compile TypeScript for production)
# ============================================================
FROM base AS builder

RUN npm ci --ignore-scripts

COPY . .

RUN npx tsc

# ============================================================
# Stage 4: Production (slim runtime)
# ============================================================
FROM node:22-bookworm-slim AS production

WORKDIR /app

COPY package.json package-lock.json* ./

RUN npm ci --omit=dev --ignore-scripts

COPY --from=builder /app/dist ./dist

# Use non-root user (UID 1000, matches most Linux users - addresses edge case 5.5)
USER node

EXPOSE 3001

CMD ["node", "dist/index.js"]
```

**Notes:**

- Multi-stage build per acceptance criteria.
- `package.json` copied before source code for Docker layer caching (addresses edge case 5.11).
- `USER node` for security and Linux UID compatibility (addresses edge case 5.5).
- Development stage includes `tsx` for hot reload; production stage runs compiled JS.
- `package-lock.json*` with glob handles the case where lock file may not exist in the package directory (it is at the workspace root).

**Important:** Because npm workspaces hoist dependencies to the root, the backend Dockerfile needs the workspace's own `package.json` plus installed `node_modules`.
The `docker-compose.yml` mounts source code into the container and uses an anonymous volume for `node_modules`.
For the development target, `npm install` inside the container installs based on the package's `package.json`.
The production build should be done from the repo root context in a future CI/CD setup.
For feat-001, the development target is the primary concern.

### 10.2 packages/backend/src/app.ts

```typescript
import express from 'express';
import pinoHttp from 'pino-http';
import { logger } from './logger.js';
import { healthRouter } from './health/health.router.js';

function createApp(): express.Express {
  const app = express();

  // Structured request logging
  app.use(
    pinoHttp({
      logger,
      // Redact sensitive headers
      serializers: {
        req(req) {
          return {
            id: req.id,
            method: req.method,
            url: req.url,
          };
        },
      },
    }),
  );

  // Parse JSON request bodies
  app.use(express.json());

  // Routes
  app.use(healthRouter);

  return app;
}
```

### 16.3 dbmate Behaviour

- dbmate is configured via the `DATABASE_URL` environment variable.
- The `db/migrations/` directory is volume-mounted into the dbmate container.
- On startup, dbmate runs `--wait up`, which:
  1. Waits for PostgreSQL to accept connections.
  1. Creates the `schema_migrations` table if it does not exist.
  1. Applies any pending migrations (none initially).
  1. Exits cleanly.
- With zero migration files, dbmate exits with code 0 (addresses edge case 5.4).
- Migration files will be added in future features using the naming convention: `YYYYMMDDHHMMSS_description.sql` with `-- migrate:up` and `-- migrate:down` sections.

## 17. Edge Cases & Defined Behaviours

This section consolidates all edge cases from the research document with their defined behaviours.
Implementation agents must handle each of these.

| # | Edge Case | Defined Behaviour |
| --- | --- | --- |
| 5.1 | Port conflicts (5432, 3001, 5173 in use) | All host-side ports in `docker-compose.yml` use `${VAR:-default}` syntax. Developer overrides via `.env`. Documented in `.env.example`. |
| 5.2 | Docker not installed | `npm run dev` fails with Docker Compose error. Docker is documented as a prerequisite in CLAUDE.md development workflow section. |
| 5.3 | Database not ready on backend start | `depends_on` with `condition: service_healthy` on PostgreSQL. dbmate uses `--wait` flag. Backend depends on dbmate `service_completed_successfully`. |
| 5.4 | Empty migrations directory | dbmate exits cleanly with code 0. `schema_migrations` table created automatically on first migration. |
| 5.5 | Volume permission issues on Linux | Dockerfile uses `USER node` (UID 1000). Workaround documented: `docker compose run --user $(id -u):$(id -g) backend npm install`. |
| 5.6 | node_modules conflicts host/container | Anonymous volume `/app/node_modules` in both backend and frontend services. Host and container `npm install` are independent. |
| 5.7 | .env file missing | All variables in `docker-compose.yml` have `${VAR:-default}` fallbacks. Stack starts without `.env`. |
| 5.8 | TypeScript strict mode violations | `tsconfig.base.json` has `"strict": true`. No package may override. CI catches violations via `npm run typecheck`. |
| 5.9 | ESLint flat config IDE compatibility | Project uses `eslint.config.js`. ESLint 9+ uses flat config by default. Document recommended VS Code extensions. |
| 5.10 | Vite proxy not working in Docker | All services on same Docker Compose network. Proxy target uses Docker service name `http://backend:3001`. |
| 5.11 | Docker build cache invalidation on dependency change | Dockerfile copies `package.json` before source code. `npm ci` layer only re-runs when dependencies change. |
| 5.12 | Inconsistent line endings (Windows) | `.gitattributes` with `* text=auto` and `*.sh text eol=lf`. Prettier `endOfLine: "lf"`. |
| 5.13 | PostgreSQL data persistence | Named volume `mmf-pgdata`. `docker compose down` preserves it. `docker compose down -v` removes it. |
| 5.14 | npm workspace dependency hoisting conflicts | Pin shared dependencies to same version. Use `overrides` in root `package.json` if needed. CI catches type errors. |
| 5.15 | GitHub Actions cache miss | `actions/setup-node@v4` with `cache: 'npm'`. Cache key based on `package-lock.json` hash. First run after dep changes is slower. |
| 5.16 | Vite HMR WebSocket in Docker | `server.host: true` in Vite config. If HMR fails, configure `server.hmr.clientPort` to match the exposed Docker port. |
| 5.17 | dbmate connection string special characters | Simple passwords in `.env.example` (`mmf`). Document that passwords must be URL-encoded if special characters are used. |
| 5.18 | CI markdown lint differs from local | `markdownlint-cli2` pinned as root devDependency. CI and local use same version. |
| 5.19 | Express 5 breaking changes from v4 | Express 5 auto-catches promise rejections. No `express-async-errors`. No `app.del()`. Document in gotchas.md. |
| 5.20 | Docker Compose version compatibility | Require Docker Compose v2 (default with Docker Desktop). Document minimum Docker version. |
