# PRD: feat-001 — Project Infrastructure and Monorepo Setup

> Implementation-ready specification. Generated by Spec Writer.
> Feature brief: feat-001-project-infrastructure.md

## Overview

This feature establishes the npm workspace monorepo with `packages/backend` (Express 5 + Pino)
and `packages/frontend` (Vite 6 + React 19 + Tailwind CSS v4) as independently runnable packages.
It is pure scaffolding — no business logic, no authentication, no domain entities.
Nothing else can be built until this foundation exists.

---

## User Stories and Acceptance Criteria

### US-001: Developer can start the full dev stack from the repo root

**As a** developer
**I want to** run a single command from the repo root that starts both the backend and frontend dev servers
**So that** I can begin feature development without manual per-package setup

**Acceptance Criteria:**

- **Given** the repo is cloned and `npm install` has been run from the root
  **When** `npm run dev` is executed from `/workspace`
  **Then** both `packages/backend` (tsx watch) and `packages/frontend` (Vite) dev servers start
  concurrently, with their output labelled in the terminal

- **Given** the backend dev server is running
  **When** `GET http://localhost:3001/health` is requested
  **Then** the response is `200 OK` with JSON body `{"status":"ok"}`
  and `Content-Type: application/json`

- **Given** the frontend dev server is running
  **When** `http://localhost:5173` is opened in a browser
  **Then** a placeholder page is served with no JavaScript errors in the console

### US-002: Developer can build both packages for production

**As a** developer
**I want to** run `npm run build` from the repo root and have both packages compile successfully
**So that** I can verify the TypeScript and bundler configuration are correct before adding features

**Acceptance Criteria:**

- **Given** the repo root `npm run build` is executed
  **When** both workspace build scripts complete
  **Then** `packages/frontend/dist/` exists and contains `index.html` and bundled assets,
  and the command exits with code 0

- **Given** `npm run typecheck` is executed from the repo root
  **When** `tsc --noEmit` runs for both packages
  **Then** there are zero TypeScript errors and the command exits with code 0

### US-003: Developer can run lint across the full workspace

**As a** developer
**I want to** run `npm run lint` from the repo root and have it check all TypeScript files in both packages
**So that** linting is a single consistent command regardless of which package I am working in

**Acceptance Criteria:**

- **Given** `npm run lint` is executed from the repo root
  **When** Biome checks all files in `packages/backend/src/**` and `packages/frontend/src/**`
  **Then** the scaffold files pass with zero errors (Biome `noConsole` rule is active —
  all backend logging must use Pino, not `console.log`)

- **Given** a `console.log` is added to any backend source file
  **When** `npm run lint` is executed
  **Then** Biome reports a lint error and exits with a non-zero code

### US-004: Environment variables are documented for new developers

**As a** new developer joining the project
**I want** `.env.example` to document every required environment variable with a placeholder value
**So that** I can set up my local environment by copying the file without reading additional documentation

**Acceptance Criteria:**

- **Given** `.env.example` exists at the repo root
  **When** I review it
  **Then** it contains `DATABASE_URL`, `PORT`, `NODE_ENV`, `LOG_LEVEL`, `VITE_API_URL`,
  all Clerk keys, all mock adapter flags, and commented PostHog keys

- **Given** the existing `.env.example` already satisfies the above
  **When** feat-001 is implemented
  **Then** `.env.example` is not modified (it already meets the acceptance criterion)

## API Endpoints

### `GET /health`

**Description:** Liveness check — confirms the Express process is running and accepting requests.
**Auth:** None. This is the sole exception to the authentication requirement per L2-002 Section 5.4.
The route must be mounted before any auth middleware.
**Roles:** Public.

**Request:** No body, no query parameters.

**Success response:** `200 OK`

```json
{"status":"ok"}
```

**Headers on success:**

```
Content-Type: application/json
```

**Error responses:**

| Status | When |
|--------|------|
| 500 | Express process is running but an unhandled exception occurs in the handler (should never happen for this trivial handler) |

**Implementation notes:**

- The route handler must use `res.json({ status: 'ok' })` — no manual `Content-Type` header needed.
- The handler must be registered on the Express router **before** any authentication middleware
  (auth middleware is not added in this feature, but the mount order must be established correctly).
- Express 5 async error handling: if the handler were async, rejected promises are automatically
  forwarded to error middleware. This handler is synchronous; no `asyncHandler` wrapper is needed.
- No Pino log entry is required for health check requests (pino-http logs all requests by default;
  to suppress health check noise, configure pino-http's `autoLogging.ignore` in a follow-up feature).

### `packages/backend/tsconfig.json` (new)

```json
{
  "extends": "../../tsconfig.base.json",
  "compilerOptions": {
    "module": "Node16",
    "moduleResolution": "node16",
    "lib": ["ES2022"],
    "outDir": "dist",
    "rootDir": "src"
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}
```

**Critical notes:**

- `"moduleResolution": "node16"` **overrides** the base config's `"bundler"` setting.
  This is mandatory for Node.js direct execution via `tsx`. The `"bundler"` setting in
  `tsconfig.base.json` is correct for Vite (frontend) but incorrect for Node.js.
- `"noUncheckedIndexedAccess": true` is inherited from `tsconfig.base.json`. This makes
  array element access return `T | undefined`. All backend code must handle this — e.g.,
  `items[0]` is typed as `T | undefined` and requires a null check before property access.
- `"declaration": true` is inherited from base. With `outDir: "dist"` set, `.d.ts` files
  land in `dist/` alongside the compiled `.js` files — this is correct behaviour.
- `"isolatedModules": true` is inherited from base. Every `.ts` file must have at least one
  import or export statement.

### `packages/backend/src/server.ts` (new)

Responsibilities:

1. Create a Pino logger instance with conditional `pino-pretty` transport for development.
2. Create the Express application instance.
3. Register `pino-http` as the first middleware (HTTP request logging).
4. Mount the health router at `/health`.
5. Register a global error handler middleware (Express 5 signature: `(err, req, res, next)`).
6. Read `PORT` from `process.env.PORT` with fallback to `3001`.
7. Start the HTTP server and log the port at INFO level using the Pino logger.

**Pino logger configuration:**

```typescript
// Conditional pino-pretty for development only
const transport =
  process.env['NODE_ENV'] !== 'production'
    ? { target: 'pino-pretty' }
    : undefined;

const logger = pino({ level: process.env['LOG_LEVEL'] ?? 'info', transport });
```

**Error handler middleware shape (Express 5):**

```typescript
// Must have exactly four parameters for Express to recognise it as an error handler
app.use((err: unknown, _req: Request, res: Response, _next: NextFunction): void => {
  logger.error({ err }, 'Unhandled error');
  res.status(500).json({
    error: {
      code: 'INTERNAL_ERROR',
      message: 'An unexpected error occurred.',
    },
  });
});
```

**Module import note:** Import `pino` and `pino-http` as default imports using ESM syntax.
Do not use `require()`. Example: `import pino from 'pino';`

### `packages/backend/src/health/api/health-router.ts` (new)

```typescript
import { Router } from 'express';

export const healthRouter: Router = Router();

healthRouter.get('/', (_req, res) => {
  res.json({ status: 'ok' });
});
```

**Mount in `server.ts`:** `app.use('/health', healthRouter);`

This router must be mounted **before** the pino-http middleware if health check noise suppression
is desired, or after it if all requests should be logged uniformly.
For this feature, mount it **after** pino-http (log all requests including health checks).
Health check auth exemption is enforced by the absence of auth middleware, which is not added
in this feature.

## Edge Cases

| # | Scenario | Expected Behaviour | Test Type |
|---|----------|--------------------|-----------|
| 1 | `packages/` directory already exists with stale files when feat-001 is implemented | Implementation agent must check before creating. If files conflict, agent resolves by overwriting scaffold files only. Existing non-conflicting files are left in place. | Manual check |

| 4 | `packages/backend` and `packages/frontend` specify different `typescript` versions | npm workspace hoisting may pick one version for both. Both packages must specify `"typescript": "^5.8.0"` (matching versions). | Dependency audit |
| 5 | Backend reads `process.env.PORT` but the variable is not set | Backend must fall back to `3001`: `const port = Number(process.env['PORT'] ?? 3001)`. Never crash on missing optional env vars. | Integration test |
| 6 | `pino-pretty` is listed as a `dependency` instead of `devDependency` | Biome/CI does not catch this, but it ships unnecessarily to production. It **must** be in `devDependencies`. | Code review gate |
| 7 | Backend `npm run dev` script name differs from `dev` | `start-dev-stack.sh` calls `npm run dev` — any other script name causes silent failure. Script name must be exactly `dev`. | Manual integration test |
| 8 | Frontend `npm run dev` script name differs from `dev` | Same as above — `start-dev-stack.sh` requires exactly `dev`. | Manual integration test |
| 9 | Health check response body is plain text instead of JSON | `GET /health` must return `Content-Type: application/json` with body `{"status":"ok"}`. Use `res.json()` not `res.send()`. | Integration test |
| 10 | `console.log` appears in any backend scaffold file | Biome `noConsole: "error"` rule is active globally. Will fail `npm run lint`. All backend logging must use the Pino logger instance. | CI lint |
| 11 | Vite env vars without `VITE_` prefix are accidentally exposed to browser | Vite only exposes `VITE_`-prefixed vars to client code. `DATABASE_URL`, `CLERK_SECRET_KEY`, etc. must never be prefixed with `VITE_`. `.env.example` already uses correct prefixes — do not alter it. | Security review |
| 12 | `tailwind.config.js` or `postcss.config.js` is created for v4 | Tailwind v4 does not use these files. Creating them causes unexpected behaviour. Only `@import "tailwindcss"` in CSS and the `@tailwindcss/vite` Vite plugin are needed. | Code review gate |

| 14 | Array index access without null guard (e.g., `items[0].name`) | `noUncheckedIndexedAccess: true` makes `items[0]` return `T | undefined`. TypeScript will error if `.name` is accessed without a null check. All implementation agents must be aware of this. | CI typecheck |
| 15 | Developer runs `npm install some-pkg` inside a workspace package directory | This may create a local `node_modules` instead of hoisting to root. | 16 | Concurrent `npm install` calls from two terminals | Can corrupt `package-lock.json`. Not a runtime concern but a developer workflow concern — serialise installs. | Developer workflow |
| 17 | Backend starts before database is available | The `/health` endpoint does not touch the database. Backend startup must not block on database connectivity — database connection pools are established in later features. | Integration test |
