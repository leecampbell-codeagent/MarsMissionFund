# Research: feat-003 --- Authentication with Clerk

> Research document for Spec Writer consumption.
> Generated by Spec Researcher.

## 1. Domain Knowledge

### Key Concepts

- **Authentication vs Authorisation**: feat-003 covers authentication only --- proving "who you are."
Authorisation (which roles can do what) is defined in L3-002 and L4-001 but enforcement beyond basic "is the user signed in?" is deferred to feat-004.
- **Session token (JWT)**: Clerk issues short-lived JWTs (access tokens, 5-minute lifetime per L3-002 Section 4.3) that the frontend sends on every API request.
The backend validates the signature and expiry without calling Clerk on every request.
- **Clerk user ID**: An opaque string (`user_xxxx`) generated by Clerk.
This is the external identity.
Our `accounts.clerk_user_id` column maps this to our internal `accounts.id` UUID.
- **User sync**: Clerk owns user identity data (email, name, avatar, password, MFA).
Our database owns domain data (roles, onboarding status, account status).
Syncing between the two happens via webhooks (Clerk pushes) or just-in-time creation on first authenticated request.

### Industry Conventions

- **Delegated authentication**: SaaS platforms overwhelmingly use OAuth 2.0/OIDC via an identity provider (Auth0, Clerk, Firebase Auth, Cognito).
The platform never stores passwords directly.
Crowdfunding platforms like Kickstarter and Indiegogo use delegated auth with social login options.
- **JWT in Authorization header**: The standard pattern for SPAs with separate API backends is `Authorization: Bearer <token>`.
This is required for cross-origin requests; same-origin can use cookies.
- **Post-signup redirect**: Crowdfunding platforms typically redirect new users to an onboarding/profile setup flow after sign-up, not directly to the dashboard.
This aligns with feat-003's acceptance criteria (redirect to onboarding placeholder route).

### Regulatory Notes

- Clerk handles PCI DSS compliance for authentication credentials (password storage, MFA).
MMF stays in SAQ-A scope per L3-002 Section 8.1.
- GDPR: Clerk acts as a data processor for authentication data.
MMF must ensure data processing agreement (DPA) is in place with Clerk.
User deletion requests (GDPR Art. 17) must cascade to both Clerk and MMF's database.
- Session management policies (idle timeout 15 min, absolute timeout 8 hr) are defined in L3-002 Section 4.4.
Clerk's dashboard must be configured to match these values.

## 2. Codebase Integration

### Current Data Model

The `accounts` table already exists (migration `20260304000003_create_accounts.sql`):

```sql
CREATE TABLE accounts (
  id                    UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  clerk_user_id         TEXT         UNIQUE NOT NULL,
  email                 TEXT         UNIQUE NOT NULL,
  display_name          TEXT,
  status                TEXT         NOT NULL DEFAULT 'pending_verification',
  roles                 TEXT[]       NOT NULL DEFAULT '{backer}',
  onboarding_completed  BOOLEAN      NOT NULL DEFAULT FALSE,
  created_at            TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
  updated_at            TIMESTAMPTZ  NOT NULL DEFAULT NOW(),

  CONSTRAINT chk_accounts_status CHECK (
    status IN ('pending_verification', 'active', 'suspended', 'deactivated', 'deleted')
  )
);

CREATE INDEX idx_accounts_status ON accounts (status);
```

Key observations:

- `clerk_user_id` is `TEXT UNIQUE NOT NULL` --- this is the join key between Clerk and our domain.
- No index on `clerk_user_id` beyond the implicit UNIQUE index --- the UNIQUE constraint creates an index automatically, so lookups by `clerk_user_id` are efficient.
- `status` defaults to `pending_verification` --- Clerk handles email verification, so our status should transition to `active` when Clerk confirms the email is verified.
- `roles` defaults to `{backer}` --- aligns with L4-001 Section 3.1 (every account gets Backer role by default).

## 3. Third-Party APIs

### Clerk --- Frontend SDK (`@clerk/clerk-react` or `@clerk/react-router`)

- **Package**: `@clerk/clerk-react` (or `@clerk/react-router` for declarative mode with React Router integration).
- **Setup**: Wrap app in `<ClerkProvider publishableKey={key}>`.
- **Environment variable**: `VITE_CLERK_PUBLISHABLE_KEY` (Vite requires `VITE_` prefix for client exposure).
- **Key components**:
  - `<SignIn />` --- renders Clerk's sign-in UI.
  Can be placed on a dedicated `/sign-in` route.
  - `<SignUp />` --- renders Clerk's sign-up UI.
  Can be placed on a dedicated `/sign-up` route.
  - `<SignedIn>` / `<SignedOut>` (or `<Show when="signed-in">` / `<Show when="signed-out">` in newer versions) --- conditional rendering based on auth state.
  - `<UserButton />` --- user avatar dropdown with sign-out, manage account.
- **Key hooks**:
  - `useAuth()` --- returns `{ userId, sessionId, getToken, isLoaded, isSignedIn }`.
  - `useUser()` --- returns `{ user, isLoaded, isSignedIn }` with full user object.
  - `getToken()` --- async, returns the current session JWT for API calls.
- **Making authenticated requests**:

  ```typescript
  const { getToken } = useAuth();
  const token = await getToken();
  const response = await fetch('/api/resource', {
    headers: { Authorization: `Bearer ${token}` },
  });
  ```

- **Same-origin note**: If frontend proxies API requests through Vite dev server to the backend (same origin), Clerk can pass the token via cookies automatically.
However, since the frontend and backend run on different ports in Docker (3000 and 3001), they are cross-origin in local dev.
The Bearer token pattern is required.
- **Mock strategy**: For tests, mock the `useAuth` hook to return a fake token.
For E2E tests, Clerk provides test mode with pre-configured test users (or use Clerk's Testing Tokens feature).

### Clerk --- Backend SDK (`@clerk/express`)

- **Package**: `@clerk/express`
- **Environment variables**: `CLERK_PUBLISHABLE_KEY`, `CLERK_SECRET_KEY`.
- **Key exports**:
  - `clerkMiddleware()` --- global middleware that validates JWTs from request cookies/headers and attaches `req.auth` (Auth object) to the request.
  Must be applied before other middleware.
  - `getAuth(req)` --- extracts the Auth object from the request.
  Returns `{ userId, sessionId, sessionClaims, orgId, getToken, has }`.
  - `requireAuth()` --- middleware that redirects unauthenticated users.
  **Not suitable for API routes** --- use `clerkMiddleware()` + `getAuth()` + manual 401 for APIs.
  - `clerkClient` --- server-side SDK for Clerk API calls (e.g., `clerkClient.users.getUser(userId)`).

- **Auth object shape** (signed-in state):

  | Property | Type | Description |
  | --- | --- | --- |
  | `userId` | `string` | Clerk user ID (`user_xxxx`) |
  | `sessionId` | `string` | Current session ID |
  | `sessionStatus` | `'active' \| 'pending'` | Session state |
  | `orgId` | `string \| undefined` | Active organisation (not used by MMF) |
  | `sessionClaims` | `JwtPayload` | Full JWT claims |
  | `factorVerificationAge` | `[number, number] \| null` | Minutes since 1FA and 2FA verification |
  | `getToken()` | `() => Promise<string>` | Get session token |
  | `has()` | `(params) => boolean` | Check permissions/claims |

- **Auth object (signed-out state)**: `userId` is `null`, `sessionId` is `null`.

- **Error handling**: `clerkMiddleware()` does NOT automatically return 401 for unauthenticated requests.
It silently attaches a "signed-out" auth object.
The application must check `userId` and return 401 manually.
This is by design --- it allows public endpoints (like `/health`) to coexist with protected endpoints.

- **Custom auth guard middleware pattern**:

  ```typescript
  function requireAuthentication(req, res, next) {
    const { userId } = getAuth(req);
    if (!userId) {
      return res.status(401).json({
        error: {
          code: 'UNAUTHENTICATED',
          message: 'Authentication required.',
          correlation_id: req.id,
        },
      });
    }
    next();
  }
  ```

- **Rate limits**: Clerk's free tier has limits on monthly active users and API calls.
For a workshop/demo project this is not a concern.
- **Mock strategy**: For unit/integration tests, bypass `clerkMiddleware()` by not applying it in the test app factory, and instead manually set `req.auth` with a fake userId.
Alternatively, use Clerk's Testing Tokens.

### Clerk --- Webhooks (Svix)

- **Endpoint**: Configure in Clerk Dashboard, points to `POST /api/webhooks/clerk`.
- **Signing**: Svix HMAC-SHA256 signature in request headers.
Verify using `@clerk/express` `verifyWebhook()` or the `svix` npm package.
- **Environment variable**: `CLERK_WEBHOOK_SIGNING_SECRET` (from Clerk Dashboard).
- **Key events**:
  - `user.created` --- fired when a user signs up via Clerk.
  Payload includes `{ data: { id, email_addresses, first_name, last_name, ... }, type: "user.created" }`.
  - `user.updated` --- fired when user data changes (email, name, etc.).
  - `user.deleted` --- fired when user is deleted from Clerk.
- **Delivery**: At-least-once.
Retries for up to 3 days on failure.
Handler must be idempotent.
- **Payload envelope**:

  ```json
  {
    "data": { "id": "user_xxxx", "email_addresses": [...], ... },
    "object": "event",
    "type": "user.created",
    "timestamp": 1654012591835,
    "instance_id": "ins_123"
  }
  ```

- **Mock strategy**: For local dev without ngrok, use a just-in-time account creation pattern (create account row on first authenticated request if it does not exist) as a fallback.
Webhooks are for production reliability, not the critical path for the local demo.
- **Local development challenge**: Webhooks require a publicly accessible URL.
Options: ngrok tunnel, Clerk CLI, or skip webhooks for local dev and use JIT creation.

## 4. Competitor Patterns

### Kickstarter

- Uses email/password and social login (Facebook, Google, Apple).
- After sign-up, users land on a discovery page (no mandatory onboarding).
- Sign-in state persists via long-lived sessions with "remember me" option.
- No visible MFA enforcement for backers (only for creators with financial access).

### Indiegogo

- Similar to Kickstarter: email/password + social login.
- Post-signup: users go directly to browsing.
Onboarding is deferred to the first action that requires it (e.g., backing a project prompts address collection).
- Uses a dropdown user menu in the header for auth state.

### GoFundMe

- Heavy push toward Google/Apple SSO for reduced friction.
- Post-signup: lands on the home feed with a brief welcome banner.
- "Continue as guest" pattern for browsing, sign-in required only for financial actions.

### Common patterns across all

- **Auth state in header**: All show user avatar/name in the header when signed in, with a dropdown for account management and sign-out.
- **Minimal sign-up friction**: Social login prominently featured.
Email/password as a fallback.
- **Deferred onboarding**: No mandatory multi-step onboarding at sign-up.
Profile completion is prompted contextually.
- **What works well**: One-click social login, persistent sessions, clear sign-in/sign-up CTAs.
- **What does not work**: Kickstarter's email verification delay before first action can frustrate users.

## 5. Edge Cases

### Data Edge Cases

- [ ] **Clerk user ID format changes**: Clerk user IDs are currently `user_xxxx` strings.
If Clerk changes the format, our `TEXT` column handles this, but validation should not assume a specific prefix.
- [ ] **Email casing**: Clerk normalises emails to lowercase.
Our `accounts.email` column should also store lowercase.
Ensure case-insensitive comparison or normalise before INSERT.
- [ ] **Null display name**: Clerk users may not have a first/last name set (especially SSO users).
The `display_name` column is nullable, which is correct.
Frontend must handle null display names gracefully (fallback to email or "Anonymous").
- [ ] **Duplicate email across Clerk and MMF**: If account creation via webhook races with JIT creation, both could attempt to INSERT with the same `clerk_user_id`.
Use `INSERT ... ON CONFLICT (clerk_user_id) DO NOTHING` (or `DO UPDATE`) for idempotency.
- [ ] **Empty roles array**: The default `'{backer}'` handles this, but guard against code that might accidentally set an empty array.
- [ ] **Account status mismatch**: If a user is `suspended` or `deactivated` in MMF but their Clerk session is still valid, the backend must check account status after auth and return 403 (not 401).

### Integration Edge Cases

- [ ] **Clerk service outage**: If Clerk's JWKS endpoint is unreachable, `clerkMiddleware()` cannot validate tokens.
The SDK caches JWKS keys, but on cold start without cache, all requests will fail.
Consider: should `/health` still work? Yes, it is already exempt from auth.
- [ ] **JWT clock skew**: `clerkMiddleware()` has a `clockSkewInMs` option (default 5000ms).
Servers with drifted clocks may reject valid tokens.
Ensure NTP is configured on all hosts.
- [ ] **Token expiry during long request**: A 5-minute access token could expire mid-request if the request takes a long time.
The middleware validates at request start, so this is not a practical issue for normal API calls.
- [ ] **Webhook signature verification failure**: If the signing secret is misconfigured, all webhooks will be rejected.
Log these failures clearly for debugging.
- [ ] **Webhook replay/duplicate delivery**: Svix delivers at-least-once.
The handler must be idempotent.
Use `INSERT ... ON CONFLICT` to handle duplicate `user.created` events.
- [ ] **Webhook ordering**: `user.updated` may arrive before `user.created` if there is retry/redelivery.
The handler must handle an UPDATE for a non-existent row (upsert pattern).
- [ ] **Stale Clerk session after account deletion**: If an MMF account is deleted but the Clerk session is still active, the user can still make authenticated requests.
The auth middleware must check for a valid MMF account, not just a valid Clerk JWT.
- [ ] **Clerk publishable key mismatch between frontend and backend**: If different Clerk instances are configured, tokens will fail validation.
Both must use keys from the same Clerk application.
- [ ] **Missing environment variables**: If `CLERK_SECRET_KEY` is not set, `clerkMiddleware()` will fail at startup or on first request.
Fail fast with a clear error message.
- [ ] **CORS with Bearer token**: Cross-origin requests with `Authorization` header require proper CORS configuration.
The backend must include `Authorization` in `Access-Control-Allow-Headers`.

### User Behaviour Edge Cases

- [ ] **Rapid sign-up then sign-in**: User signs up, does not verify email, tries to sign in.
Clerk handles this (blocks or allows based on dashboard config).
MMF should not create an account row until verification is complete (use `user.created` webhook or JIT on first authenticated request, which implies verification passed).
- [ ] **Multiple tabs**: User signs in on one tab, second tab still shows signed-out state.
Clerk's session management handles this via session syncing.
- [ ] **Sign-out on one tab, API request from another**: The auth middleware will reject the request (JWT is no longer valid or session is revoked).
Frontend should handle 401 responses by redirecting to sign-in.
- [ ] **Browser back button after sign-out**: User signs out, presses back.
The page may show cached content.
`Cache-Control: no-store` header (per L3-002 Section 7.2) prevents this.
- [ ] **User signs up via Clerk but abandons before completing onboarding**: The MMF account row should exist but with `onboarding_completed = false`.
The frontend should redirect to onboarding on subsequent logins.
- [ ] **User deletes their Clerk account via Clerk dashboard**: `user.deleted` webhook fires.
MMF should transition the account to `deleted` status and trigger GDPR erasure per L4-001 Section 7.
- [ ] **Session token in URL**: Ensure tokens are never passed as URL parameters (query strings are logged by proxies/CDNs).
Always use `Authorization` header.

### Boundary Edge Cases

- [ ] **Very long Clerk user ID**: `TEXT` column has no practical limit, but validate reasonable length at the application layer.
- [ ] **Very long email address**: RFC 5321 allows up to 254 characters.
Clerk enforces its own limits.
Our `TEXT` column handles this.
- [ ] **Concurrent first requests (JIT account creation race)**: Two requests arrive simultaneously for a user with no MMF account row.
Both attempt INSERT.
`ON CONFLICT` handles this at the database level.
- [ ] **Rate limiting on auth endpoints**: L3-002 specifies 10 req/min on auth endpoints.
Clerk handles rate limiting for its own endpoints (sign-in, sign-up).
MMF's webhook endpoint should also be rate-limited.
- [ ] **Maximum active sessions**: L3-002 specifies unlimited concurrent sessions.
No enforcement needed on our side.

## 6. Recommendations

### Must-Haves for Spec

1. **Auth middleware placement**: The spec must define exactly where `clerkMiddleware()` is applied in the Express middleware stack (after Pino HTTP, before routes) and which routes are exempt (`GET /health`).
2. **Auth guard middleware**: Define a custom `requireAuthentication` middleware that checks `getAuth(req).userId` and returns the standard 401 error format `{ error: { code: "UNAUTHENTICATED", message, correlation_id } }`.
Do not use `requireAuth()` from `@clerk/express` for API routes (it redirects instead of returning 401).
3. **Account creation strategy**: Choose between webhook-driven creation and JIT (just-in-time) creation.
Recommendation: **JIT creation** as the primary path for the local demo (no ngrok dependency), with webhook support as a production enhancement.
JIT means: on the first authenticated API request, if no `accounts` row exists for `clerk_user_id`, create one.
4. **Frontend routing**: The spec must define the route structure: `/sign-in`, `/sign-up`, and protected routes.
Define what happens when an unauthenticated user hits a protected route (redirect to `/sign-in`).
5. **API client**: Define the centralised API client that injects the Bearer token via `useAuth().getToken()`.
This must integrate with TanStack Query.
6. **CORS configuration**: If frontend and backend are cross-origin (different ports in dev), the backend must set `Access-Control-Allow-Origin`, `Access-Control-Allow-Headers` (including `Authorization`), and `Access-Control-Allow-Methods`.
7. **Environment variables**: Document all required variables with the exact names: `CLERK_PUBLISHABLE_KEY`, `CLERK_SECRET_KEY`, `VITE_CLERK_PUBLISHABLE_KEY`, and optionally `CLERK_WEBHOOK_SIGNING_SECRET`.
8. **Account status check**: After JWT validation, the auth middleware or application layer must verify that the MMF account is in an `active` status.
Suspended/deactivated accounts should receive 403, not 401.

### Watch-Outs

1. **`requireAuth()` is wrong for APIs**: Clerk's `requireAuth()` middleware redirects unauthenticated users (HTTP 302).
For JSON API endpoints, this is incorrect --- use `clerkMiddleware()` + manual 401.
This is a very easy mistake to make.
2. **Webhook vs JIT trade-off**: Webhooks require ngrok or a tunnel for local dev, adding setup friction to a workshop project.
JIT creation is simpler but means the first API request for a new user is slower (INSERT before the actual operation).
3. **`@clerk/clerk-react` vs `@clerk/react-router`**: Clerk has a separate package for React Router integration (`@clerk/react-router`).
It provides `ClerkProvider` that is router-aware.
Evaluate whether this is needed or if `@clerk/clerk-react` with manual routing is sufficient.
The declarative mode guide suggests using `@clerk/react-router` when React Router is present.
4. **Vite proxy and cookies**: If the Vite dev server proxies API requests to the backend (making them same-origin from the browser's perspective), Clerk's cookie-based auth could work.
However, in Docker with separate containers, the proxy must be configured correctly.
The Bearer token approach is more explicit and avoids cookie complexity.
5. **Express 5 compatibility**: `@clerk/express` should work with Express 5, but verify.
Express 5 auto-catches promise rejections, which may interact with Clerk's middleware error handling.
6. **CSP headers**: L3-002 Section 7.2 defines a CSP that includes `connect-src 'self' https://*.clerk.accounts.dev`.
Ensure this is set correctly for Clerk's API calls from the frontend.

### Suggested Approach

**Backend**:

1. Install `@clerk/express`.
2. Apply `clerkMiddleware()` globally in `app.ts` after Pino HTTP and JSON parsing.
3. Create a custom `requireAuthentication` middleware that checks `getAuth(req).userId` and returns 401 with the standard error format.
4. Apply `requireAuthentication` to all routes except `/health`.
5. Create an `AuthContext` type: `{ userId: string; clerkUserId: string; accountId: string }`.
6. Create an `enrichAuthContext` middleware that looks up the MMF account by `clerk_user_id` (JIT creation if not found) and attaches the full auth context to the request.
7. Add webhook endpoint `POST /api/webhooks/clerk` (unauthenticated, verified via Svix signature) for production user sync.

**Frontend**:

1. Install `@clerk/clerk-react` (or `@clerk/react-router`).
2. Wrap app in `<ClerkProvider>` in `main.tsx`.
3. Add `<BrowserRouter>` and define routes: `/sign-in`, `/sign-up`, `/` (protected home), `/onboarding` (placeholder).
4. Create a `ProtectedRoute` wrapper that checks `useAuth().isSignedIn` and redirects to `/sign-in`.
5. Create a centralised API client module that uses `useAuth().getToken()` to inject the Bearer token.
6. Integrate with TanStack Query's `QueryClientProvider`.
