## 1. Domain Knowledge

### Key Concepts

**Campaign lifecycle (feat-003 scope)**: This feature covers the first segment of the full
L4-002 state machine: Draft → Submitted → Under Review → Approved/Rejected → Live. The later
phases (Funded, Failed, Settlement, Complete, Cancelled, Suspended) belong to later features.
feat-003 is the gatekeeper — nothing can be funded or discovered until a campaign passes this
review pipeline.

**Draft persistence vs. submission**: These are two distinct operations with different semantics.
A draft is a mutable in-progress record the creator assembles over multiple sessions. Submission
is a one-way state transition that freezes the content and puts it in the review queue. The
backend must support both a create/update draft endpoint (idempotent, partial data OK) and a
separate submit endpoint (strict validation, all required fields, triggers state change).

**Curation model**: MMF is not a self-serve platform. Unlike Kickstarter where creators publish
directly, MMF campaigns require human review. This is core to the mission — the platform
curates credible Mars mission projects. The review pipeline is not theatre; it is a product
differentiator.

**Funding amounts as integer cents**: All amounts (funding target, max cap, milestone amounts)
are stored as BIGINT in PostgreSQL (integer cents). API accepts and returns them as strings.
The minimum funding target ($1,000,000 USD) = 100,000,000 cents. The maximum
($1,000,000,000 USD) = 100,000,000,000 cents. These are large numbers — BIGINT is essential
(INT would overflow at ~21 billion cents, so BIGINT handles these amounts).

**Milestone funding percentages**: Each milestone is assigned a percentage of the total raised
amount that gets disbursed when that milestone is verified. These must sum to exactly 100%.
The spec must decide on precision: whole integer percentages (0–100) vs. basis points
(0–10000, for fractional percentages like 33.33%). Basis points give more flexibility (e.g.,
three equal milestones at 3333 + 3333 + 3334 basis points = 10000). Whole percentages force
round numbers (33 + 33 + 34 = 100). Both have precedent in crowdfunding platforms. The
acceptance criteria say "percentages summing to 100%" without specifying precision.

**Review queue (FIFO, pull-based)**: Reviewers self-assign from a queue rather than being
auto-assigned. This is a deliberate design decision from L4-002 Section 5.1 — reviewers pull
work when ready. The queue is ordered by submission timestamp (oldest first). A reviewer
"claiming" a campaign is itself a state transition (Submitted → Under Review) and must be
atomic to prevent two reviewers claiming the same campaign simultaneously.

**Campaign category taxonomy**: 10 fixed categories derived from NASA's Technology Taxonomy
(L4-002 Section 4.4). These should be stored as a TEXT column with a CHECK constraint (or a
separate reference table). The accepted values are fixed and do not change per-deployment,
making a CHECK constraint simpler than a reference table for the local demo.

**Hero image handling**: L4-002 Section 4.2 references image specifications per L2-001 (Brand).
The feature brief flags this as an open question. For the local demo, the simplest approach is
external URL only (no file upload). The Spec Writer must define this explicitly to prevent
implementation agents from building an AWS S3 upload flow that is out of scope.

**Creator self-designation**: L4-002 does not define the exact mechanism for a user to obtain
the Creator role. The feature brief notes "likely: any Verified KYC user can self-designate as
Creator." The role assignment mechanism must be clarified. The existing `assignRole()` method
on the `User` entity supports this. The Spec Writer must decide whether:
(a) Any verified-KYC active user can call an endpoint to self-assign the Creator role, or
(b) The Creator role is assigned automatically when a user's KYC status reaches `verified`.
Option (a) is consistent with L4-001's role assignment pattern. Option (b) creates a tighter
coupling between KYC and campaign that may not be desirable.

**Auto-save semantics**: L4-002 Section 4.3 says "Drafts are auto-saved." For the local demo,
this is implemented as a PATCH endpoint that accepts partial updates and saves them without
validation. The frontend calls this periodically or on field blur. This is different from
"submit" which requires full validation and all required fields.

### Calculation Methodology

**Milestone percentage validation**:
If using whole integer percentages: `SUM(milestone.percentage) = 100`
If using basis points: `SUM(milestone.basis_points) = 10000`
The validation must be applied at submission time (Draft → Submitted), not at draft save time.
Validation failure must identify the current sum vs. 100 for clear error messaging.

**Funding target range**:
- Minimum: 100_000_000 cents ($1,000,000 USD)
- Maximum: 100_000_000_000 cents ($1,000,000,000 USD)
These are stored as BIGINT. The API accepts them as numeric strings (e.g., `"100000000"`).

**Campaign deadline range**:
- Minimum: `submission_date + 7 days` (1 week from submission, not from draft creation)
- Maximum: `submission_date + 365 days` (1 year from submission)
The deadline must be re-validated at submission time, not at draft creation time. A creator
who drafts a campaign months before submitting must ensure the deadline is still valid.

### Industry Conventions

**Kickstarter**: Uses an "all or nothing" model with a single funding goal. No max cap, no
milestone-based fund release. Campaigns are reviewed but auto-approved in most categories.
Review is exception-based (flagged for violations), not mandatory pre-launch. MMF differs:
manual pre-launch review is mandatory.

**Indiegogo**: Offers both "fixed funding" (all-or-nothing) and "flexible funding" (keep what
you raise). MMF uses a hybrid: minimum threshold required to keep any funds, max cap to prevent
over-raising, no all-or-nothing below the minimum. This is the "Flexible with a minimum
threshold" model described in L4-002 Section 10.1.

**Experiment.com (science crowdfunding)**: Requires researcher affiliation verification and
expert peer review of proposals before campaigns launch. Milestone-based fund release is
standard. This most closely resembles MMF's model.

**GoFundMe**: No review pipeline, no milestones, personal campaigns. Not comparable.

**Republic / SeedInvest (equity crowdfunding)**: SEC-regulated platforms use mandatory review
because they are handling securities offerings. Their review pipelines (legal documents, team
diligence, financial projections) are more intensive than MMF's but have the same structural
pattern: submission → review → approval/rejection → live.

**Reviewer UX pattern**: Review dashboards typically show a queue sorted by submission time,
with basic campaign metadata (title, category, funding target, submission date). Reviewers
click into a detail view to read the full proposal. Approval/rejection is a form with a required
notes/rationale field.

### Regulatory Notes

For the local demo, the primary regulatory driver is **KYC gating** (already implemented in
feat-002). Additional considerations:

- The campaign is a solicitation of funds. In Australia, this may trigger the Australian
  Financial Services regime if the platform is not structured correctly. For the demo, this is
  theatre — the spec only needs to note that KYC verification of creators is the gating check.
- Risk disclosures (required field in the campaign submission) serve as the legal disclaimer
  mechanism. The Spec Writer should note that the review pipeline includes checking that risk
  disclosures are substantive (not placeholder text).
- Budget breakdown and milestone plan serve as financial accountability mechanisms. These are
  compliance-adjacent features.

**Proposed `campaigns` table structure** (derived from L4-002 Section 4.2 and acceptance criteria):

```sql
CREATE TABLE IF NOT EXISTS campaigns (
  id                    UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  creator_user_id       UUID        NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
  status                TEXT        NOT NULL DEFAULT 'draft'
                                    CHECK (status IN (
                                      'draft', 'submitted', 'under_review',
                                      'approved', 'rejected', 'live', 'funded',
                                      'suspended', 'failed', 'settlement', 'complete', 'cancelled'
                                    )),
  title                 TEXT,
  summary               TEXT,              -- ≤280 chars, validated at submission
  description           TEXT,
  alignment_statement   TEXT,              -- Mars-mission alignment statement
  category              TEXT,
  hero_image_url        TEXT,              -- external URL, no file upload
  funding_goal_cents    BIGINT,            -- minimum funding target in cents
  funding_cap_cents     BIGINT,            -- maximum funding cap in cents
  deadline              TIMESTAMPTZ,
  budget_breakdown      JSONB,             -- structured breakdown
  team_members          JSONB,             -- array of {name, role, bio}
  risk_disclosures      JSONB,             -- array of {risk, mitigation}
  milestones            JSONB,             -- array of milestone objects
  tags                  TEXT[],            -- free-form discoverable tags
  -- Review fields (set during review phase)
  reviewer_user_id      UUID        REFERENCES users(id) ON DELETE SET NULL,
  review_notes          TEXT,              -- approval notes or rejection rationale
  resubmission_guidance TEXT,              -- populated on rejection
  reviewed_at           TIMESTAMPTZ,
  -- Submission tracking
  submitted_at          TIMESTAMPTZ,
  launched_at           TIMESTAMPTZ,
  -- Audit timestamps
  created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at            TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

The exact column set, nullable constraints, and JSONB structures must be finalised by the
Spec Writer. Key decisions:
- Is `milestones` stored as a JSONB array on `campaigns` or as a separate `milestones` table?
- Is `team_members` JSONB or a separate `campaign_team_members` table?
- Is `budget_breakdown` JSONB or a separate `campaign_budget_items` table?

**Recommendation**: For feat-003 (creation through review), JSONB arrays on `campaigns` are
simpler and sufficient. Separate tables become necessary when milestones need their own lifecycle
(evidence submission, verification status) — that belongs in a later milestone verification
feature. Start with JSONB; migrate to separate tables in a later feature when needed.

**Campaign audit events table**: Analogous to `kyc_audit_events`, a `campaign_audit_events`
table is needed to log state transitions:

```sql
CREATE TABLE IF NOT EXISTS campaign_audit_events (
  id                    UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  campaign_id           UUID        NOT NULL REFERENCES campaigns(id) ON DELETE RESTRICT,
  actor_user_id         UUID        REFERENCES users(id) ON DELETE SET NULL,
  actor_clerk_user_id   TEXT        NOT NULL,
  action                TEXT        NOT NULL,  -- e.g., 'campaign.status.change'
  previous_status       TEXT,
  new_status            TEXT        NOT NULL,
  rationale             TEXT,
  metadata              JSONB,
  created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### Integration Points

**Campaign → Account (user identity)**:
- Campaign creation: check `user.roles.includes('creator')` and `user.kycStatus === 'verified'`.
- Review actions: check `user.roles.includes('reviewer')`.
- Admin actions: check `user.roles.includes('administrator')`.
- The campaign application service will call `UserRepository.findByClerkUserId()` to look up
  the requesting user, then perform role/KYC checks before proceeding.

**Campaign → KYC (gating)**:
- The campaign application service checks `user.kycStatus` directly (reading from the
  `users` table via `UserRepository`). It does not call the KYC application service directly.
- This is the established pattern — the `kycStatus` field is on `User`, accessible from
  `UserRepository.findByClerkUserId()`.
- Error code for unverified KYC: `KYC_NOT_VERIFIED` (HTTP 403) — established in feat-002.

**Campaign → Payments (future)**:
- When a campaign goes `Live`, it needs to trigger escrow account creation via the Payments
  domain. This is out of scope for feat-003 but the Launch endpoint must be designed to
  accommodate this future integration.

**Campaign → Notifications (established pattern)**:
- Creator notified when: campaign submitted (confirmation), campaign claimed for review,
  campaign approved, campaign rejected.
- The notification pattern is currently via the `notification_prefs` preference flags on `User`.
  The actual notification delivery (AWS SES) is theatre — the spec must define the audit event
  and any in-app notification, but not the email delivery.

**App wiring (`composition-root.ts`)**:
- A `CampaignAppService` must be instantiated and returned from `createServices()`.
- The campaign router must be mounted in `app.ts` under `/api/v1/campaigns`.
- Role checking should be done at the application service layer, not via a separate middleware
  (consistent with the existing pattern where KYC account checks are in `KycAppService`).

## 4. Competitor Patterns

### Kickstarter

- **Campaign creation**: Multi-step form with sections: Basics, Funding, Story, Rewards, People,
  Account. Creators can save drafts and return later. No review queue for standard campaigns.
- **What works well**: Section-by-section save (auto-save per section, not full form). Progress
  indicator showing completeness. Inline validation as creator fills each field.
- **What doesn't work**: No mandatory human review before launch means low-quality campaigns go
  live. MMF's mandatory review pipeline is a deliberate improvement.

### Experiment.com

- **Campaign creation**: Long-form with sections including: Summary (1 paragraph), Hypothesis,
  Experimental Design, Budget breakdown, Team, FAQ. Peer review by experts in the field before
  publishing.
- **What works well**: The peer review model is closest to MMF. Reviewers are domain experts,
  not platform staff. The review criteria are specific and checklist-driven.
- **Learnings for MMF**: The rejection feedback mechanism matters. Experiment.com reviewers
  provide structured feedback under each review criterion, not a free-form paragraph. MMF
  should require reviewers to address each curation criterion (L4-002 Section 5.2) when
  rejecting a proposal.

### Republic (equity crowdfunding)

- **Review pipeline**: All campaigns go through a structured diligence process: application
  submitted → legal review → diligence questionnaire → approval. Takes 2–6 weeks.
- **What works well**: Clear SLA communication. Creators know the expected timeline. Escalation
  path if review is delayed.
- **Learnings for MMF**: The 5-day SLA from L4-002 Section 5.4 should be surfaced in the
  review queue UI. The reviewer dashboard should show how many days each campaign has been
  in Submitted state. An overdue indicator helps reviewers prioritise.

### Patreon (creator relationship)

- **Not directly comparable**: Patreon has no campaign review. But their campaign creation UX
  (multi-step guided flow with rich text editor, image upload, preview mode) is a useful UX
  reference for the campaign creation form.
- **Learnings**: Preview mode (see your campaign as a backer would) is a high-value feature
  for creators before submitting. For MMF, this could be a future enhancement.

## 5. Edge Cases

### Data Edge Cases

- [ ] **EC-001: Summary exceeds 280 characters at submission**. A creator saves a draft with a
  282-character summary (allowed at draft save time). When they submit, the submission
  validation must catch this and return `400 VALIDATION_ERROR` with the specific field and
  character count. Expected: `{ "field": "summary", "message": "Summary must be 280 characters or fewer. Provided: 282." }`.

- [ ] **EC-002: Milestone percentages sum to 99% (off-by-one)**. If a creator uses whole
  integer percentages and three milestones at 33/33/33 = 99%, submission must fail with the
  specific discrepancy. Expected: `{ "field": "milestones", "message": "Milestone funding percentages must sum to 100%. Current sum: 99%." }`.

- [ ] **EC-003: Milestone percentages sum to 101% (rounding up)**. If three milestones at
  34/34/34 = 102%, or 34/33/34 = 101%, submission must fail. Same error pattern as EC-002.

- [ ] **EC-004: Zero milestones submitted**. Creator submits with an empty milestones array.
  Expected: `400 VALIDATION_ERROR` — at least two milestones are required per L4-002 Section 4.2.

- [ ] **EC-005: Only one milestone submitted**. Creator submits with exactly one milestone at
  100%. Expected: `400 VALIDATION_ERROR` — minimum two milestones required. The 100% check
  passes but the count check fails.

- [ ] **EC-006: Individual milestone percentage is zero**. A milestone with `percentage: 0`
  is meaningless (no funds released). Expected: `400 VALIDATION_ERROR` — each milestone
  funding percentage must be greater than zero.

- [ ] **EC-007: Funding goal below minimum ($1M)**. Creator submits with `fundingGoalCents:
  "50000000"` (= $500,000 USD). Expected: `400 VALIDATION_ERROR` — minimum is 100,000,000
  cents. Error must state the minimum in a human-readable way.

- [ ] **EC-008: Funding cap below funding goal**. Creator submits with `fundingGoalCents:
  "100000000"` and `fundingCapCents: "90000000"` (cap < goal). Expected: `400 VALIDATION_ERROR`
  — the funding cap must be greater than or equal to the funding goal.

- [ ] **EC-009: Funding goal and cap are equal**. `fundingGoalCents === fundingCapCents`. This
  is valid — the campaign stops accepting contributions as soon as the goal is met.

- [ ] **EC-010: Deadline is in the past at submission time**. A creator drafts a campaign with
  a deadline far in the future, but the draft sits for a long time. When they submit, the
  deadline might be less than 7 days away or even in the past. Expected: `400 VALIDATION_ERROR`
  — deadline must be at least 7 days from the submission timestamp (not draft creation time).

- [ ] **EC-011: Deadline is exactly 7 days from submission (boundary)**. A deadline exactly
  `submission_timestamp + 7 * 24 * 60 * 60 seconds` is valid. The boundary must be inclusive.
  The implementation must compare timestamps consistently (UTC) to avoid timezone edge cases
  where "7 days" could be 6 days and 23 hours depending on DST.

- [ ] **EC-012: No team members provided**. Creator submits with an empty `teamMembers` array.
  Expected: `400 VALIDATION_ERROR` — at least one team member is required.

- [ ] **EC-013: No risk disclosures provided**. Creator submits with an empty `riskDisclosures`
  array. Expected: `400 VALIDATION_ERROR` — at least one risk disclosure is required.

- [ ] **EC-014: Title is empty or whitespace-only**. A title of `"   "` (spaces only) must be
  treated as absent. Expected: `400 VALIDATION_ERROR`. Zod's `.trim()` transform catches this.

- [ ] **EC-015: Funding amounts provided as numbers, not strings**. The API accepts monetary
  amounts as strings (to avoid precision loss). If a client sends `fundingGoalCents: 100000000`
  (a number), the server must reject it with `400 VALIDATION_ERROR`. Zod schema must use
  `z.string()` (not `z.number()`) for monetary fields.

### State Machine Edge Cases

- [ ] **EC-016: Creator attempts to submit a campaign already in Submitted state**. Double-click
  or network retry sends two submit requests. Expected: `409 CAMPAIGN_ALREADY_SUBMITTED`. The
  `POST /:id/submit` endpoint must use a conditional WHERE transition (`status = 'draft'`) to
  make this atomic.

- [ ] **EC-017: Creator attempts to resubmit a campaign in Under Review state**. The campaign
  has been claimed by a reviewer. Creator cannot force it back to draft. Expected: `409
  CAMPAIGN_NOT_REVIZABLE` (or equivalent). Only a `Rejected` campaign can be revised.

- [ ] **EC-018: Two reviewers claim the same campaign simultaneously**. Both see it in the
  queue, both click Claim. Expected: first claim succeeds (Submitted → Under Review with
  reviewer_user_id set); second claim fails with `409 CAMPAIGN_ALREADY_CLAIMED`. Must use a
  conditional WHERE: `UPDATE campaigns SET status = 'under_review', reviewer_user_id = $1
  WHERE id = $2 AND status = 'submitted'`.

- [ ] **EC-019: Reviewer tries to approve a campaign they did not claim**. A reviewer who is
  not the `reviewer_user_id` on the campaign tries to approve it. Expected: `403 FORBIDDEN` —
  only the assigned reviewer (or an Admin) can take review actions on a specific campaign.

- [ ] **EC-020: Reviewer approves with empty notes**. Notes are required per L4-002 Section 5.3.
  Expected: `400 VALIDATION_ERROR` — approval notes are required, must be non-empty.

- [ ] **EC-021: Reviewer rejects without resubmission guidance**. Both rationale AND guidance
  are required per L4-002 Section 6.2. Expected: `400 VALIDATION_ERROR` if either is missing
  or empty.

- [ ] **EC-022: Creator launches a campaign that is not in Approved state**. A creator calls
  `POST /:id/launch` on a Draft or Submitted campaign. Expected: `409 CAMPAIGN_NOT_LAUNCHABLE`
  — only Approved campaigns can be launched.

- [ ] **EC-023: Creator revises a campaign in Approved state** (before launching). L4-002
  does not define this transition. If a creator decides not to launch and wants to revise,
  is that allowed? Expected behaviour is undefined by the spec — the Spec Writer must decide.
  Recommendation: Do not allow revision of an Approved campaign without going through
  rejection first. Creator should submit a new campaign or ask an Admin to return it to Draft.

- [ ] **EC-024: Admin reassigns a reviewer on a campaign already approved or rejected**.
  Reassignment is only valid for `Under Review` campaigns (L4-002 Section 5.1). Expected:
  `409 CAMPAIGN_INVALID_STATE` if the campaign is not in `under_review` state.

- [ ] **EC-025: Creator deletes/deactivates account while campaign is in Under Review**.
  The `users` table uses `ON DELETE RESTRICT` is not yet defined for `campaigns`. If the FK
  is `ON DELETE RESTRICT`, the user deletion fails. If `ON DELETE SET NULL`, the campaign
  record becomes orphaned. This is a cross-context concern. The Spec Writer must define the
  ON DELETE rule on `campaigns.creator_user_id`.
  Recommendation: `ON DELETE RESTRICT` — do not allow account deletion while a campaign is
  active (not in a terminal state: complete, cancelled, failed).

- [ ] **EC-026: Campaign in Draft state — no time limit**. Drafts have no expiry per L4-002
  Section 4.3. The spec must be explicit that old drafts do not auto-delete.

### Concurrency Edge Cases

- [ ] **EC-027: Concurrent PATCH (auto-save) and submit request**. A creator's browser auto-saves
  a draft while they simultaneously click Submit. The auto-save PATCH arrives first, updates the
  draft. The submit arrives simultaneously. Expected: both operations succeed without data
  corruption because the submit reads the latest draft state after all field updates. Or: the
  submit fails with a conflict if it checks a version/ETag. Recommendation: The submit endpoint
  should read the latest campaign from DB (not from the request body) and validate what is stored,
  not what was passed. The PATCH and POST are independent operations on the DB record.

- [ ] **EC-028: Two creators attempt to create campaigns with the same title simultaneously**.
  There is no uniqueness constraint on `campaigns.title`. Duplicate titles are allowed — MMF is
  a curation platform, not a namespace. Expected: both campaigns are created as separate drafts.

### Permission Edge Cases

- [ ] **EC-029: User with Backer role (no Creator role) attempts to create a campaign draft**.
  Expected: `403 FORBIDDEN` with error code `ROLE_REQUIRED` (or `INSUFFICIENT_ROLE`). The
  user must hold the `creator` role.

- [ ] **EC-030: User with Creator role but KYC not verified attempts to create a draft**.
  The acceptance criteria say both "create" and "submit" require KYC. The Spec Writer must
  decide: does draft creation (before submission) also require KYC? L4-002 Section 4.1 says
  "submission requires KYC" — not draft creation. But the acceptance criteria in the feature
  brief say "A user with `Verified` KYC status and `Creator` role can create a campaign draft."
  This implies both create AND submit require KYC. Expected: `403 KYC_NOT_VERIFIED` on both
  `POST /campaigns` (create) and `POST /:id/submit`. The Spec Writer must resolve this ambiguity.

- [ ] **EC-031: Reviewer role views the review queue with no campaigns in Submitted state**.
  The queue is empty. Expected: `200 OK` with `{ "data": [] }` — empty array, not 404.

- [ ] **EC-032: Non-reviewer views the review queue endpoint**. A user with only the `backer`
  role calls `GET /campaigns/review-queue`. Expected: `403 FORBIDDEN`.

- [ ] **EC-033: Creator views another creator's draft**. `GET /campaigns/:id` on a draft
  campaign owned by a different creator. Expected: `404 NOT_FOUND` (not 403) — do not reveal
  the existence of other creators' drafts.

- [ ] **EC-034: Reviewer or Admin views a draft campaign**. Reviewers and Admins should not be
  able to see Draft campaigns — only Submitted and above. Expected: `403 FORBIDDEN` or `404
  NOT_FOUND` for reviewers reading a Draft. Admin access policy must be defined explicitly.

- [ ] **EC-035: Creator views their own campaign's review notes after rejection**.
  The rejection rationale and resubmission guidance must be visible to the creator (they need
  it to revise). Expected: `GET /campaigns/:id` for the creator returns `reviewNotes` and
  `resubmissionGuidance` fields when status is `rejected`.

### Boundary Edge Cases

- [ ] **EC-036: Maximum number of simultaneous draft campaigns per creator**. L4-002 Section 4.3
  states creators may have "multiple drafts simultaneously" with no stated limit. For the local
  demo, no limit is imposed. The Spec Writer should document this explicitly so there is no
  ambiguity for implementation agents.

- [ ] **EC-037: Maximum number of team members per campaign**. Not stated in L4-002. The Spec
  Writer must define a reasonable maximum (e.g., 20 team members). Without a limit, a malicious
  creator could send gigabytes of team member data.

- [ ] **EC-038: Maximum number of milestones per campaign**. Not stated in L4-002. A reasonable
  limit prevents both data bloat and unusable milestone plans (e.g., 100 milestones at 1% each
  is technically valid but impractical). Recommendation: maximum 10 milestones.

- [ ] **EC-039: Maximum number of risk disclosures per campaign**. Not stated. Recommendation:
  maximum 10 risk disclosures.

- [ ] **EC-040: Very long description text (rich text, possible XSS injection)**. The description
  field is rich text (per L4-002 Section 4.2). The backend must sanitise this field to prevent
  XSS when the content is rendered on the public campaign page. For the local demo, Zod string
  validation with a max length (e.g., 10,000 characters) is a minimum safeguard. A proper
  sanitisation library (e.g., DOMPurify on the frontend, or a server-side sanitiser) should
  be specified.

- [ ] **EC-041: Hero image URL that is actually a JavaScript URL or data URI**.
  A `heroImageUrl` of `javascript:alert('xss')` or `data:text/html,...` would be XSS if
  rendered as an `<img src>`. The Zod schema must validate that the URL uses `https://`
  protocol only.

- [ ] **EC-042: Funding goal amount with leading zeros in the string**. `"0100000000"` is not
  a valid monetary string. The Zod schema must validate that the string represents a valid
  positive integer (no leading zeros, no decimals, no commas).

### Integration Edge Cases

- [ ] **EC-043: Creator role gets removed between draft creation and submission**. A creator
  creates a draft, then an Admin removes their Creator role. When they try to submit, they no
  longer have the role. Expected: `403 ROLE_REQUIRED` at submission time. The role check is
  always point-in-time at the application service, not cached from draft creation.

- [ ] **EC-044: KYC expires between draft creation and submission**. The `expired` KYC state
  is in the CHECK constraint. If KYC expires (future feature), a creator might have a draft
  but be unable to submit. Expected: `403 KYC_NOT_VERIFIED` at submission time.

- [ ] **EC-045: Reviewer's role is removed while a campaign is in Under Review**.
  An Admin removes the Reviewer role from the user who has a campaign claimed. The campaign
  remains in `Under Review` with an orphaned `reviewer_user_id`. Expected: the Admin must
  reassign the campaign to another reviewer. The spec should require an Admin-only endpoint
  for reassignment (already in the acceptance criteria), and should note that role removal
  does not automatically return the campaign to the queue.

---

## 6. Recommendations

### Must-Haves for Spec

1. **Define the Creator role self-designation mechanism explicitly.** The most pragmatic approach
   for the local demo: add a `POST /api/v1/me/roles/creator` endpoint that allows any active,
   verified-KYC user to self-assign the Creator role. This mirrors the onboarding flow pattern.
   Alternative: auto-assign Creator role when KYC reaches `verified`. The Spec Writer must
   choose one and document it clearly.

2. **Resolve the KYC gate for draft creation vs. submission.** The feature brief acceptance
   criteria imply KYC is required for both. L4-002 Section 4.1 says "submission requires KYC."
   These are contradictory. The Spec Writer must align them. Recommendation: require KYC for
   both create and submit, matching the acceptance criteria.

3. **Define milestone precision (whole percentages vs. basis points).** Recommend basis points
   (integers 0–10000 summing to 10000) to avoid the 33+33+34 edge case with three equal
   milestones. Document the precision decision explicitly so frontend and backend implement the
   same unit.

4. **Define hero image handling.** Recommend external URL only (no file upload) for feat-003.
   The URL must be `https://` only. Maximum length: 2048 characters. Accepted image formats
   are validated at render time by the browser (`<img src>`), not at submission time.

5. **Define the JSONB vs. separate table decision for milestones, team members, and budget**.
   Recommendation: JSONB arrays on `campaigns` for feat-003. Document the migration path to
   separate tables in a later feature.

6. **Define ON DELETE rules for `campaigns.creator_user_id`**. Recommend `ON DELETE RESTRICT`
   to prevent account deletion while campaigns are in a non-terminal state. Define what states
   are "terminal" for this purpose: `complete`, `cancelled`, `failed`.

7. **Define maximum field lengths and counts** for all JSONB array fields (team members,
   milestones, risk disclosures) to prevent unbounded data. Recommendation:
   - Team members: max 20
   - Milestones: max 10
   - Risk disclosures: max 10
   - Tags: max 20, each tag max 50 characters
   - Title: max 100 characters
   - Summary: max 280 characters (per spec)
   - Description: max 10,000 characters (rich text, sanitised)
   - Alignment statement: max 1,000 characters
   - Budget breakdown: max 20 line items, each item title max 100 chars

8. **Specify the audit event schema for campaign state transitions**. The audit table
   `campaign_audit_events` must be created in a migration. The schema should follow the same
   pattern as `kyc_audit_events` (append-only, no `updated_at`, UUID PK).

9. **Specify concurrent-safe state transitions**. Every status transition must use a conditional
   WHERE clause on the current status to prevent race conditions. This is the same pattern as
   `updateKycStatus()` in `PgUserRepository`.

10. **Define the `GET /campaigns/:id` access control matrix explicitly**:
    - Draft: creator only (not even reviewers)
    - Submitted, Under Review, Approved, Rejected: creator + reviewer + admin
    - Live and beyond: public (no auth required)
    This prevents information leakage between creators.

11. **Notifications are in-app audit events only** (no email delivery for feat-003). The spec
    should record that a notification was triggered (as an audit event) but not implement actual
    email delivery via AWS SES. Define what "notified" means concretely.

12. **Register new domain errors in the error handler**. `createErrorHandler` in
    `packages/backend/src/shared/middleware/error-handler.ts` must be extended with campaign-
    specific domain errors and their HTTP status codes.

### Watch-Outs

1. **The campaigns router must be mounted correctly in `app.ts`**. Following the pattern in the
   existing `app.ts`, add: `app.use('/api/v1/campaigns', requireAuth, createCampaignRouter(...))`.
   The campaign router needs both `CampaignAppService` and `logger` parameters. The campaign
   app service must be instantiated in `composition-root.ts`.

2. **`review-queue` is a sub-path that could clash with `/:id`**. Express router ordering
   matters: `GET /campaigns/review-queue` must be registered BEFORE `GET /campaigns/:id`,
   otherwise Express treats `review-queue` as a campaign ID parameter. Use `router.get(
   '/review-queue', ...)` before `router.get('/:id', ...)`.

3. **BIGINT from PostgreSQL comes back as a string in `pg`**. The `pg` library returns BIGINT
   columns as JavaScript strings (not numbers) because JavaScript numbers cannot safely represent
   all 64-bit integers. This is actually desired behaviour for monetary amounts — the backend
   should pass these strings directly to the API response without parsing to number.

4. **JSONB ordering is not guaranteed**. When storing milestones, team members, or risk
   disclosures as JSONB arrays, PostgreSQL preserves the array order within the JSONB value.
   However, if the array is reconstructed from separate queries (not applicable for JSONB), order
   could be lost. For JSONB arrays, order is preserved.

5. **Milestone percentage validation requires the full milestones array**. The validation
   logic (sum = 100 or sum = 10000) cannot be applied to a single milestone in isolation. The
   application service must receive the complete milestones array and validate the sum. This
   means the submit endpoint must receive all milestones (or read them from DB), not a partial
   update. Auto-save can update individual fields; submit validates the whole.

6. **Rich text description field is an XSS risk**. If the frontend renders `description` as
   raw HTML (which rich text implies), unvalidated user input becomes an XSS vector. For the
   local demo, define the description as a plain text area with newline-to-paragraph rendering
   (not a full rich text editor), OR specify a whitelist of allowed HTML tags and use a
   sanitisation library. The spec must be explicit about this.

7. **The `AuditActions` const in `audit-logger.port.ts` needs campaign actions added**.
   Following the pattern in `packages/backend/src/account/ports/audit-logger.port.ts`, the
   `AuditActions` object must be extended and the `resourceType` union must include `'campaign'`.
   Currently: `resourceType: 'user' | 'kyc'`. Must become: `resourceType: 'user' | 'kyc' | 'campaign'`.
   The campaign-specific audit goes to `campaign_audit_events`, not the general audit logger port.

8. **Creator role self-designation is a write to `users.roles` in the account context**.
   If a `POST /me/roles/creator` endpoint is created, it writes to `users.roles`. This operation
   belongs in the Account bounded context (not Campaign), following the same pattern as the
   existing `assignRole()` method on the `User` entity. The campaign application service should
   NOT be writing to `users.roles`.

- `20260305150000_create_campaigns_table.sql`

**State transition pattern** (following KYC precedent):

```typescript
// In PgCampaignRepository
async updateStatus(campaignId: string, fromStatus: CampaignStatus, toStatus: CampaignStatus, reviewerUserId?: string): Promise<Campaign> {
  const result = await this.pool.query<CampaignRow>(
    `UPDATE campaigns
     SET status = $2, reviewer_user_id = COALESCE($3, reviewer_user_id), updated_at = NOW()
     WHERE id = $1 AND status = $4
     RETURNING *`,
    [campaignId, toStatus, reviewerUserId ?? null, fromStatus],
  );
  if (result.rowCount === 0) throw new CampaignTransitionConflictError();
  return rowToDomain(result.rows[0]);
}
```

**KYC + role gate pattern** (in campaign app service):

```typescript
// In CampaignAppService.createDraft()
const user = await this.userRepository.findByClerkUserId(clerkUserId);
if (!user) throw new UserNotFoundError(clerkUserId);
if (!user.roles.includes(Role.Creator)) throw new CreatorRoleRequiredError();
if (user.kycStatus !== KycStatus.Verified) throw new KycNotVerifiedError();
```
