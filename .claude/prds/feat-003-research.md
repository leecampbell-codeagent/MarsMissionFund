## 1. Domain Knowledge

### Key Concepts

**Clerk** is a managed authentication and user management service.
It handles the full identity lifecycle: sign-up, sign-in, session management, MFA, and OAuth provider linking.
MMF delegates all of this to Clerk so it never stores passwords, manages tokens, or owns session state directly.

**Clerk JWT** is a short-lived JSON Web Token (RS256) issued by Clerk after successful authentication.
It is the credential the frontend passes to the MMF backend on every API request, via `Authorization: Bearer <token>`.
The backend verifies the JWT signature using Clerk's public key (fetched from Clerk's JWKS endpoint), then extracts claims.

**Clerk User ID (`clerkId`)** is a stable, Clerk-assigned string identifier of the form `user_<alphanumeric>`.
It is not a UUID.
It is the foreign key between Clerk's identity store and MMF's `users` table (`users.clerk_id`).

**Lazy user sync** (the chosen strategy per feat-003) means MMF does not proactively push Clerk events into the database.
Instead, on the first authenticated request, the backend checks whether a `users` row exists for that `clerkId`.
If not, it creates one (upsert).
This avoids running a webhook ingestion infrastructure for the local demo.

**`req.auth` context** is a TypeScript object attached to the Express request by the auth middleware after successful JWT verification.
Downstream route handlers and application services consume it â€” they never decode JWT themselves.
The shape the feature brief mandates is: `{ userId: string, clerkId: string, roles: string[] }`.
`userId` is the MMF internal UUID from `users.id`.
`clerkId` is the Clerk-issued string from `users.clerk_id`.
`roles` is the array of role strings from `user_roles` for that user.

**`GET /api/v1/me`** is the authenticated self-profile endpoint.
It is the canonical way for the frontend to discover the current user's MMF identity after initial sign-in.
It doubles as a first-login trigger â€” calling it with a valid Clerk JWT for a user not yet in the database causes the lazy sync upsert.

**`X-Request-Id` correlation ID** is a UUID generated per request (or forwarded from an incoming header if present).
It is attached to all authenticated responses and logged alongside every request.
This enables tracing a single request across logs.

### Auth Flow Summary

```text
User authenticates in Clerk (frontend) â†’ Clerk issues JWT
Frontend attaches JWT as Bearer token â†’ API request hits backend
Backend auth middleware verifies JWT with Clerk SDK â†’ extracts clerkId and email from claims
Middleware queries users table for clerkId
  If found: populate req.auth from users + user_roles rows
  If not found: upsert users row (account_status='active') + insert user_roles row (role='backer') â†’ populate req.auth
Route handler receives request with populated req.auth
```

### Role Assignment on First Login

Per L4-001 Section 3.1: every new account gets the Backer role by default.
The lazy sync must insert both the `users` row and a `user_roles` row for `backer`.
The `user_roles.assigned_by` field can be NULL for system-assigned roles.
The `user_roles.id` must be a UUID generated at the application layer (per domain-knowledge.md â€” UUIDs are application-generated, not `DEFAULT gen_random_uuid()`).

### Industry Conventions

Most production applications using Clerk use webhooks (`user.created`) to sync users into their database in real-time.
This is more reliable (handles sign-ups that never trigger an API call) but requires a publicly reachable webhook endpoint.
The feat-003 brief explicitly chooses lazy sync to keep demo infrastructure minimal.
The trade-off: a Clerk user who registers but never hits a protected endpoint will never appear in the MMF `users` table â€” acceptable for the demo context.

The `@clerk/express` package (published by Clerk, released ~2024) is the current recommended backend integration for Express.
The older `@clerk/clerk-sdk-node` is the legacy package.
The feature brief mentions `@clerk/clerk-sdk-node` in Key Decisions but the Spec Writer should evaluate whether to recommend `@clerk/express` instead â€” it provides `clerkMiddleware()` and `requireAuth()` as Express-native middleware, which aligns more cleanly with the established middleware pattern.

For the frontend, `@clerk/react` provides `<ClerkProvider>`, `<SignIn>`, `<SignUp>`, `useUser()`, `useAuth()`, and `getToken()`.
The `VITE_CLERK_PUBLISHABLE_KEY` env var is the frontend's credential â€” it is public and safe to expose in client code.

### Regulatory Notes

Authentication data (session tokens, Clerk user IDs) falls under GDPR Article 6 as technically necessary processing.
Per L4-001 and L3-002, account deletion (GDPR right-to-erasure) sets `account_status = 'deleted'` and nulls PII fields â€” it does not delete the `users` row.
The `clerk_id` column should be treated as PII-adjacent and nulled on GDPR erasure.
Clerk separately holds user data in their own infrastructure; GDPR deletion from MMF does not automatically delete the Clerk account.
This is a known gap per the demo scope.

## 2. Codebase Integration

### Current Data Model

**`users` table** (migration `20260306000001_create_accounts.sql`):

```sql
id               UUID        NOT NULL  -- application-generated UUID (no DEFAULT)
clerk_id         TEXT        NOT NULL  -- Clerk user ID, e.g. "user_abc123"
email            TEXT        NOT NULL
display_name     TEXT        NULL
avatar_url       TEXT        NULL
bio              TEXT        NULL
onboarding_completed BOOLEAN NOT NULL DEFAULT false
account_status   TEXT        NOT NULL  -- CHECK: pending_verification | active | suspended | deactivated | deleted
created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()

-- Unique indexes on both clerk_id and email
-- Trigger: set_users_updated_at â†’ update_updated_at_column()
```

Key observations:
- `clerk_id` has a `UNIQUE` constraint â€” the upsert can use `ON CONFLICT (clerk_id)` safely.
- `id` has no `DEFAULT gen_random_uuid()` â€” the application must generate the UUID before insert.
- `account_status` valid values: `pending_verification`, `active`, `suspended`, `deactivated`, `deleted`.
  The lazy sync should set `active` (Clerk already verified the email for most flows).
- `email` has a `UNIQUE` constraint â€” if a Clerk user changes their email and tries to log in, a conflict is possible if the new email matches another MMF row.

**`user_roles` table** (migration `20260306000002_create_roles.sql`):

```sql
id          UUID        NOT NULL  -- application-generated UUID
user_id     UUID        NOT NULL  -- FK â†’ users.id ON DELETE CASCADE
role        TEXT        NOT NULL  -- CHECK: backer | creator | reviewer | administrator | super_administrator
assigned_by UUID        NULL      -- FK â†’ users.id ON DELETE SET NULL (NULL = system-assigned)
created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()

-- UNIQUE (user_id, role) â€” no duplicate role assignment
-- No updated_at, no trigger
```

Key observations:
- `(user_id, role)` unique constraint means the upsert for `backer` role on first login can use `ON CONFLICT (user_id, role) DO NOTHING`.
- `assigned_by` is NULL for system-assigned roles â€” correct for the first-login backer assignment.
- No `account_status` equivalent â€” role revocation is by deletion, not status update.

## 3. Third-Party APIs

### Clerk â€” Backend SDK

**Package options:**
- `@clerk/express` â€” newer, Express-native middleware (`clerkMiddleware()`, `requireAuth()`), recommended by Clerk as of 2024.
- `@clerk/clerk-sdk-node` â€” legacy, lower-level, requires manual JWT verification.

The feature brief mentions `@clerk/clerk-sdk-node` but `@clerk/express` is likely the better choice for Express 5.
The Spec Writer should resolve this.

**JWT Verification with `@clerk/express`:**

```typescript
import { clerkMiddleware, getAuth } from '@clerk/express';

// Global middleware â€” attaches Clerk auth state to req
app.use(clerkMiddleware());

// Route-level â€” enforces authentication (returns 401 if not authed)
app.get('/api/v1/me', requireAuth(), handler);

// Extracting auth claims inside a handler
const { userId: clerkId } = getAuth(req);  // clerkId is the Clerk user ID string
```

**JWT Verification with `@clerk/clerk-sdk-node` (legacy):**

```typescript
import { createClerkClient } from '@clerk/clerk-sdk-node';
const clerk = createClerkClient({ secretKey: process.env.CLERK_SECRET_KEY });

// Manual verification
const token = req.headers.authorization?.split(' ')[1];
const payload = await clerk.verifyToken(token);
// payload.sub === clerkId
```

**JWT Claims available in the token:**
- `sub` â€” Clerk user ID (e.g., `user_abc123`)
- `email` â€” primary email address (may require configuring a custom claim in Clerk dashboard, or use Sessions API)
- `iss` â€” issuer (`https://clerk.your-app.accounts.dev`)
- `exp` / `iat` / `nbf` â€” standard JWT timing claims
- `azp` â€” authorised party (frontend app's publishable key domain)
- `sid` â€” Clerk session ID

**Important**: The Clerk JWT does not always include the user's email by default.
The `sub` claim is always present, but email may require either:
1. Configuring a custom JWT template in the Clerk Dashboard to include `email` as a claim, or
2. Using the Clerk Backend API (`clerk.users.getUser(clerkId)`) to fetch the user's primary email during lazy sync.

This is a significant integration detail the Spec Writer must address.

**Clerk Backend API â€” Get User:**

```
GET https://api.clerk.com/v1/users/{user_id}
Authorization: Bearer <CLERK_SECRET_KEY>
```

Response includes: `id`, `email_addresses[]`, `primary_email_address_id`, `first_name`, `last_name`, `image_url`, etc.

Rate limits: 20 requests/second on free tier. Not a concern for the lazy-sync pattern since it only fires on first login per user.

**Environment Variables (backend):**
- `CLERK_SECRET_KEY` â€” authenticates backend SDK and API calls to Clerk. Keep server-side only. Never expose to frontend.
- `CLERK_PUBLISHABLE_KEY` â€” used for domain validation in token verification. Can be backend-side only if using SDK verification.
- `CLERK_WEBHOOK_SIGNING_SECRET` â€” for webhook verification; not used in feat-003 (lazy sync chosen), but already documented in `.env.example`.

**JWKS endpoint:** `https://<your-clerk-domain>.clerk.accounts.dev/.well-known/jwks.json`
The `@clerk/express` SDK fetches and caches this automatically.
No manual JWKS handling is needed.

**Token lifetime:** Clerk access tokens have a default lifetime of 60 seconds.
Refresh is handled by the Clerk frontend SDK automatically.
The backend verifies the token on each request â€” no caching of auth decisions.

**Error codes from Clerk SDK verification:**
- Token expired â†’ `TokenVerificationError: Token has expired`
- Invalid signature â†’ `TokenVerificationError: Token signature is invalid`
- Wrong issuer â†’ `TokenVerificationError: Token issuer is invalid`
- Token not yet valid â†’ `TokenVerificationError: Token is not yet valid`

All of these should map to HTTP 401 with `{ error: { code: "UNAUTHORIZED", message: "..." } }`.

```tsx

// They must be rendered at their configured "paths" in Clerk Dashboard

// Routes:
// /sign-in â†’ <SignIn />

```

**JWT injection into API requests:**

```typescript
// api-client.ts â€” centralised fetch wrapper
import { useAuth } from '@clerk/react';

// Inside a hook/component:
const { getToken } = useAuth();

async function apiFetch(path: string, options?: RequestInit) {
  const token = await getToken();
  return fetch(`${import.meta.env.VITE_API_URL}${path}`, {
    ...options,
    headers: {
      ...options?.headers,
      Authorization: `Bearer ${token}`,
      'Content-Type': 'application/json',
    },
  });
}
```

Note: `getToken()` is async and returns the current active token, automatically refreshing if needed.
The API client must be a hook or accept `getToken` as an argument â€” it cannot be a plain module-level function because `getToken` requires React context.
The Spec Writer must define the API client pattern carefully given this constraint.

**Key hooks:**
- `useAuth()` â€” `{ isLoaded, isSignedIn, userId, getToken, signOut }` â€” the primary hook for auth state
- `useUser()` â€” `{ isLoaded, isSignedIn, user }` â€” full user object including email, name, avatar
- `useClerk()` â€” lower-level SDK access

**Redirect flow:**
After sign-in, Clerk redirects to a configured URL (set in Clerk Dashboard, typically `/` or `/dashboard`).
After sign-up, similarly configured.
The Spec Writer must decide the post-auth redirect target for this demo.

## 4. Competitor Patterns

### How Auth-Gated Apps Typically Structure This

**Kickstarter / GoFundMe pattern:**
These platforms use their own auth, but the pattern they follow for API auth is standard:
- Unauthenticated browsing is allowed for campaign discovery.
- Contribution and account actions require authentication.
- Auth state is checked at the route level on the frontend and enforced at the API level on the backend.

For MMF's demo scope (all routes currently require auth), the simpler approach is to gate everything except `/health` at the middleware level, and add the `/sign-in` / `/sign-up` routes as exceptions in the frontend router.

**Next.js + Clerk pattern (widely used reference):**
The standard pattern is:
1. `<ClerkProvider>` wraps the entire app.
2. Middleware checks JWT on every request.
3. Public routes are defined in a config allowlist.
4. Protected routes redirect to `/sign-in` if not authenticated.

MMF's Express backend mimics this: mount auth middleware globally, mount `/health` before it as the only public endpoint.

**JWT injection â€” common mistake to avoid:**
Many implementations inject the token in individual component fetch calls instead of a centralised client.
This leads to stale tokens (if `getToken()` is called once at module load time rather than per-request).
`getToken()` must be called per request, not cached.

## 5. Edge Cases

### Authentication / JWT Edge Cases

- [ ] **JWT expired**: Token has a 60-second lifetime. `exp` claim in the past. Backend should return 401 with `UNAUTHORIZED` code. Frontend Clerk SDK should auto-refresh; if refresh fails, redirect to `/sign-in`. This is the most common auth failure in production.

- [ ] **JWT invalid signature**: Token tampered with or issued by wrong Clerk instance. Return 401. Do not leak details about why verification failed â€” generic `UNAUTHORIZED` message.

- [ ] **JWT wrong issuer**: Token from a different Clerk application or environment (e.g., test env token used against prod). Clerk SDK checks `iss` claim. Return 401.

- [ ] **JWT not yet valid**: `nbf` (not-before) claim in the future â€” clock skew. Rare but possible. Return 401.

- [ ] **No Authorization header**: Request to protected endpoint with no `Authorization` header at all. Return 401. Do not return a different error code that reveals the auth mechanism.

- [ ] **Malformed Bearer token**: `Authorization: Bearer ` (empty), `Authorization: not-a-jwt`, `Authorization: Basic ...`. Return 401. Do not attempt to parse a clearly non-JWT value.

- [ ] **Authorization header wrong format**: `Authorization: <token>` without `Bearer ` prefix. Return 401.

- [ ] **Token for deactivated/deleted Clerk user**: Clerk may issue a token for a user whose Clerk account was deleted between token issuance and API call. The Clerk SDK may still verify the token successfully (signature valid, not expired). The MMF middleware must check `account_status` after the DB lookup and return 401 (or 403) if status is `deleted` or `deactivated`.

### User Sync / Lazy Upsert Edge Cases

- [ ] **First login â€” Clerk user not in MMF DB**: The standard happy path. Upsert creates `users` row with `account_status='active'` and `user_roles` row with `role='backer'`. Must succeed and return `req.auth` with the new internal userId.

- [ ] **Race condition â€” two simultaneous first-login requests**: Two requests arrive concurrently for a Clerk user with no existing `users` row. Both attempt the upsert. The `ON CONFLICT (clerk_id) DO UPDATE` (or `DO NOTHING`) PostgreSQL upsert is atomic. The second request must not fail or create a duplicate row. The unique constraint on `clerk_id` is the safety net. The upsert query must be written to handle this gracefully without throwing a unique constraint violation at the application level.

- [ ] **Race condition â€” `users` upsert succeeds but `user_roles` insert fails**: Partial state â€” user row exists, backer role row does not. On the next request, the middleware finds the `users` row but may find zero roles. The roles insert should be idempotent: `ON CONFLICT (user_id, role) DO NOTHING`. The middleware should also handle the case where `user_roles` is empty (user has no roles) without crashing.

- [ ] **Email conflict on upsert**: A Clerk user signs in with an email that already exists in `users` for a different `clerk_id`. This can happen if data was seeded manually or if the same email was used in multiple Clerk environments. The `users_email_unique` constraint will cause the insert to fail. The middleware must catch this and return a meaningful error (500 internal, log the conflict, do not expose to client).

- [ ] **Clerk user's email changes between logins**: The email stored in `users.email` may become stale if the user updates their email in Clerk. The upsert `ON CONFLICT (clerk_id) DO UPDATE SET email = EXCLUDED.email` would keep it current, but the `users_email_unique` constraint could cause a conflict if the new email belongs to another user. The Spec Writer must decide whether the upsert updates email on conflict.

- [ ] **Clerk user suspended/deleted in Clerk but still in MMF DB with `active` status**: Clerk will not issue new tokens for suspended/deleted users, so this scenario is only reachable if an existing valid token (up to 60 seconds old) is used after Clerk-side deletion. The token will verify successfully. The backend relies on `account_status` in the MMF DB to gate access. If Clerk deletes a user without an MMF `account_status` update (no webhook), the user remains `active` in MMF for the lifetime of their last issued token.

- [ ] **MMF user with `account_status = 'suspended'`**: Clerk JWT verifies successfully, DB lookup finds the user, but `account_status` is `suspended`. Must return 403 Forbidden (not 401 â€” the user is authenticated but not authorised to access the platform). Error code: `ACCOUNT_SUSPENDED`.

- [ ] **MMF user with `account_status = 'deleted'`**: Same as suspended. Must return 403. Error code: `ACCOUNT_DELETED` or `ACCOUNT_NOT_FOUND` (the Spec Writer should decide whether to reveal the deletion status or use a generic 403).

- [ ] **MMF user with `account_status = 'deactivated'`**: User self-deactivated. Must return 403. Error code: `ACCOUNT_DEACTIVATED`. The frontend should redirect to a reactivation prompt.

- [ ] **`user_roles` query returns multiple roles**: A user can have multiple roles (Backer + Creator, etc.). The middleware must collect all roles into an array â€” not just the first one. An error in `req.auth.roles` being a scalar instead of an array would silently break all role-based feature gating.

### Webhook Edge Cases (out of scope for feat-003, but noted for future)

- [ ] **Clerk `user.deleted` webhook arrives**: The webhook payload contains the Clerk user ID of the deleted user. MMF should set `account_status = 'deleted'` and null PII fields. This is not implemented in feat-003 (lazy sync only) but the Spec Writer should note that GDPR compliance requires this eventually.

### Correlation ID Edge Cases

- [ ] **Incoming `X-Request-Id` header already present**: A reverse proxy or load balancer may forward a correlation ID. The middleware should forward it rather than generate a new one, to preserve end-to-end tracing. Must validate the incoming value (length limit, safe characters) before echoing it.

- [ ] **`X-Request-Id` collision**: UUID v4 collision probability is negligible (2^122 space) but the pattern of generating a new UUID per request if none is provided is correct.

### Token Injection Edge Cases (Frontend)

- [ ] **`getToken()` returns null**: Can happen if the user is not signed in, or Clerk is still initialising (`isLoaded = false`). The API client must handle this and not send an `Authorization: Bearer null` header. If `getToken()` returns null and the request requires auth, redirect to `/sign-in`.

- [ ] **Clerk not yet loaded**: `isLoaded` is false on initial render. Components that depend on auth state must handle the loading state â€” do not render auth-dependent UI or fire API requests until `isLoaded` is true.

- [ ] **`getToken()` throws**: Can happen if the Clerk session has expired and refresh fails. The API client must catch this and redirect to `/sign-in`.

- [ ] **Token refresh during in-flight request**: `getToken()` always returns the current valid token, handling refresh transparently. However, if a request is made with a token that expires mid-flight, the backend will return 401. The frontend API client should handle 401 responses by triggering a re-authentication flow, not silently failing.

### Data Integrity Edge Cases

- [ ] **UUID generation produces duplicate**: `crypto.randomUUID()` output collision is negligible but the `id` column's PRIMARY KEY constraint catches it. Application-layer UUID generation for `users.id` and `user_roles.id` must use a cryptographically random source. `crypto.randomUUID()` (Node.js built-in) is correct.

- [ ] **`/api/v1/me` called before `users` row is written**: Theoretically impossible in lazy-sync pattern (the middleware creates the row before the route handler fires), but if the upsert throws and the error handler does not catch it correctly, the route handler might execute with no `req.auth`. TypeScript non-null handling and strict mode will surface this at compile time if `req.auth` is typed as potentially undefined.

## 6. Recommendations

### Must-Haves for Spec

1. **Resolve `@clerk/express` vs `@clerk/clerk-sdk-node`.**
   The feature brief names the legacy `@clerk/clerk-sdk-node`.
   `@clerk/express` is the current package with Express-native middleware (`clerkMiddleware()`, `requireAuth()`).
   The Spec Writer should specify one and explain the rationale.
   Recommendation: use `@clerk/express` for cleaner Express integration.

2. **Specify how email is obtained from Clerk during lazy sync.**
   The Clerk JWT `sub` claim gives us `clerkId`.
   Getting `email` requires either a custom JWT template in the Clerk Dashboard (adding `email` as a claim) OR a Clerk Backend API call to `GET /v1/users/{userId}`.
   The Spec Writer must specify which approach and document the Dashboard configuration step if the JWT template approach is chosen.
   Recommendation: custom JWT template is simpler (no extra API call on each first-login), but requires a one-time Clerk Dashboard setup step.

3. **Define `req.auth` TypeScript type precisely.**
   The type must be declared and used via Express's `Request` type augmentation (module augmentation pattern).
   Without this, TypeScript will not know `req.auth` exists and every route handler will have a type error.

4. **Specify the upsert SQL precisely.**
   The lazy sync upsert must handle:
   - `users`: `INSERT ... ON CONFLICT (clerk_id) DO UPDATE SET email = EXCLUDED.email, updated_at = NOW()` (to keep email current).
   - `user_roles`: `INSERT ... ON CONFLICT (user_id, role) DO NOTHING` (idempotent backer assignment).
   Both must be in a transaction to prevent partial state from the race condition.

5. **Specify the API client pattern for token injection.**
   `getToken()` requires React context and is async.
   A TanStack Query-based approach (pass `getToken` as a dependency to query functions) or a custom hook wrapping `fetch` is needed.
   A plain module-level function will not work.

6. **Specify 403 vs 401 for suspended/deleted accounts.**
   HTTP 401 means "not authenticated" (no valid credentials).
   HTTP 403 means "authenticated but not authorised".
   Suspended/deleted accounts have valid JWTs but are blocked by business rules â†’ 403 is correct.
   The spec must define the response body format and error codes for each account status.

7. **Document the CSP header requirement for Clerk.**
   The security spec (L3-002, Section 7.2) defines a CSP that includes `connect-src 'self' https://*.clerk.accounts.dev`.
   The backend must add `helmet` (or manual header middleware) to set the CSP and other security headers.
   Clerk's frontend components make connections to `*.clerk.accounts.dev` â€” without this in `connect-src`, they will fail under strict CSP.

8. **Specify `X-Request-Id` generation and propagation.**
   Every authenticated response must include this header.
   Specify whether to use `crypto.randomUUID()` or a third-party package.
   Specify the middleware position (before auth, after pino-http, so correlation IDs appear in pino-http logs).

9. **Define the directory structure for the Account bounded context.**
   The spec should define where the auth middleware lives in the hexagonal structure.
   Recommendation: `src/account/api/auth-middleware.ts` (HTTP concern, belongs in the api layer) with `src/account/application/user-sync-service.ts` (orchestration of the upsert).

### Watch-Outs

- **`@clerk/express` vs `@clerk/clerk-sdk-node` version mismatch**: The `.env.example` and feature brief were written before `@clerk/express` was widely adopted. Installing the wrong package will not cause runtime errors but will result in a worse API surface and more boilerplate.

- **Email uniqueness conflict during upsert**: The `users_email_unique` constraint is a real risk if email changes are not handled in the upsert. Failing to `DO UPDATE SET email = ...` means the email in MMF will silently go stale, but attempting to update email when the new email conflicts will throw a constraint violation. This needs explicit handling.

- **`req.auth` typing in Express**: Express does not have a built-in `auth` property on `Request`. Without a module augmentation declaration (`declare global { namespace Express { interface Request { auth?: AuthContext } } }`), TypeScript will error on every `req.auth` access. This is a compile-time gotcha, not a runtime one, but will block the build.

- **React 19 StrictMode + Clerk**: React 19 StrictMode double-invokes effects. If the API client calls `getToken()` inside a `useEffect` without proper cleanup, the token may be fetched twice on mount. Use TanStack Query to manage API calls â€” it is idempotent and handles this correctly.

- **Vite environment variable prefix**: `CLERK_SECRET_KEY` must NEVER be prefixed with `VITE_`. Only `VITE_CLERK_PUBLISHABLE_KEY` gets the `VITE_` prefix. A common mistake when setting up Clerk is putting the secret key in frontend code. This would expose the key in the built JS bundle.

- **The `MOCK_AUTH=true` flag in `.env.example`**: The `.env.example` already has `MOCK_AUTH=true`. This implies the Spec Writer should define how the mock auth adapter works â€” the auth middleware should bypass Clerk JWT verification when `MOCK_AUTH=true` and instead use a pre-configured test user identity. This is critical for integration tests that cannot use a real Clerk instance.

- **No `pg` dependency in backend yet**: The backend `package.json` has `express`, `pino`, and `pino-http` but no `pg` or `@types/pg`. The user sync upsert requires direct PostgreSQL access. The implementation agent will need to add `pg` as a dependency.

- **The `users.id` UUID must be generated at application layer**: Per `domain-knowledge.md`, UUIDs are application-generated (no `DEFAULT gen_random_uuid()` in the schema). The implementation must call `crypto.randomUUID()` before the SQL insert, not rely on the DB to generate it. Failing this will cause a NOT NULL violation on the `id` column.

### Suggested Approach

1. **Backend auth middleware** lives at `src/account/api/auth-middleware.ts`.
   It uses `@clerk/express`'s `clerkMiddleware()` globally and a custom MMF middleware for the DB lookup and lazy sync.
   The custom middleware runs after Clerk's middleware â€” it can trust that `getAuth(req).userId` is present.

2. **User sync service** lives at `src/account/application/user-sync-service.ts`.
   It accepts a port interface for the user repository and encapsulates the upsert transaction logic.
   The middleware calls the sync service, which returns the populated `AuthContext`.

3. **User repository** lives at `src/account/adapters/user-repository.ts`.
   It contains the `pg`-based SQL queries for finding and upserting users and roles.

4. **`GET /api/v1/me` handler** lives at `src/account/api/me-router.ts`.
   It reads `req.auth` (populated by middleware) and returns the user profile.
   For full profile data (display_name, avatar_url, etc.), it queries the `users` table by `req.auth.userId`.

5. **Frontend API client** is a custom hook (`useApiClient`) wrapping `fetch` with `getToken()` injection.
   TanStack Query (`@tanstack/react-query`) should be used for all data fetching to handle loading/error states correctly.
   `<QueryClientProvider>` wraps the app inside `<ClerkProvider>`.
