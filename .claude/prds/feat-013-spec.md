## Overview

Implements the KYC (Know Your Customer) identity verification workflow with a mock Veriff adapter that auto-approves submissions for the local demo. Establishes the `KycVerification` domain entity, the nine-state KYC status lifecycle, document upload validation, the `KycPort` and `StoragePort` interfaces, and the HTTP endpoints for submitting, querying, and administratively unlocking KYC verifications. On mock-approval, the creator role is granted to the account. The mock adapter is controlled by `MOCK_KYC=true` (default for local demo).

## User Stories & Acceptance Criteria

### US-001: Submit KYC Verification

**As a** registered user (any role)
**I want to** upload identity documents and submit them for verification
**So that** I can become KYC-verified and unlock Creator features

**Acceptance Criteria:**

- **Given** an authenticated user with `not_verified` status **When** they POST a valid document type and file metadata to `POST /api/v1/kyc/submit` **Then** a `kyc_verifications` row is created (or updated if one exists) with status `pending`, and the response returns `{ data: { status: "pending" } }`
- **Given** an authenticated user with `pending_resubmission` status **When** they POST a valid submission **Then** the status transitions to `pending` and failure count is preserved (not reset until admin unlock)
- **Given** an authenticated user with `rejected` status (failure count < 5) **When** they POST a new submission **Then** the status transitions to `pending`
- **Given** an authenticated user with `pending` status **When** they attempt to submit again **Then** the API returns 409 with code `KYC_ALREADY_PENDING`
- **Given** an authenticated user with `verified` status **When** they attempt to submit **Then** the API returns 409 with code `KYC_ALREADY_VERIFIED`
- **Given** an authenticated user with `locked` status **When** they attempt to submit **Then** the API returns 409 with code `KYC_ACCOUNT_LOCKED`
- **Given** a submission with an invalid document type **When** the request is processed **Then** the API returns 400 with code `VALIDATION_ERROR`
- **Given** a submission where the front document file is larger than 20 MB **When** the request is processed **Then** the API returns 400 with code `VALIDATION_ERROR` and message "Document file must be 20 MB or smaller."
- **Given** a submission with an unsupported file type (not JPEG, PNG, or PDF per magic bytes) **When** the request is processed **Then** the API returns 400 with code `INVALID_DOCUMENT_FORMAT`
- **Given** a valid submission to the mock adapter **When** processed **Then** the mock schedules an auto-approval (configurable delay via `MOCK_KYC_APPROVAL_DELAY_MS`, default 0 ms), transitions status to `verified`, and grants the `creator` role to the account

### US-002: Check KYC Status

**As a** registered user
**I want to** query my current KYC verification status
**So that** I know whether I need to take action to unlock Creator features

**Acceptance Criteria:**

- **Given** an authenticated user with an existing `kyc_verifications` row **When** they call `GET /api/v1/kyc/status` **Then** the response returns `{ data: { status, failure_count, verified_at, document_type } }`
- **Given** an authenticated user with no `kyc_verifications` row **When** they call `GET /api/v1/kyc/status` **Then** the response returns `{ data: { status: "not_verified", failure_count: 0, verified_at: null, document_type: null } }`
- **Given** an unauthenticated request **When** the endpoint is called **Then** the API returns 401 with code `UNAUTHENTICATED`

### US-003: Admin Unlock Locked Account

**As an** administrator or super_administrator
**I want to** unlock a user's KYC account that has been locked after 5 failures
**So that** the user can resubmit their identity documents

**Acceptance Criteria:**

- **Given** an admin-role user **When** they POST to `POST /api/v1/kyc/admin/unlock/:accountId` for an account with `locked` status **Then** the status transitions to `pending_resubmission`, failure count resets to 0, an audit event is emitted, and the response returns `{ data: { status: "pending_resubmission", failure_count: 0 } }`
- **Given** an admin user **When** they attempt to unlock an account that is NOT in `locked` status **Then** the API returns 409 with code `KYC_NOT_LOCKED`
- **Given** a non-admin user **When** they call the admin unlock endpoint **Then** the API returns 403 with code `UNAUTHORIZED`
- **Given** an invalid or non-existent `:accountId` **When** the admin unlock endpoint is called **Then** the API returns 404 with code `ACCOUNT_NOT_FOUND`

### US-004: Mock Auto-Approval with Failure Count Escalation

**As a** system
**I want to** track and escalate repeated KYC failures
**So that** fraudulent or persistently failing accounts can be locked and reviewed

**Acceptance Criteria:**

- **Given** the mock KYC adapter is configured to simulate failure **When** a submission is processed as failed **Then** status transitions to `pending_resubmission` and failure count increments by 1
- **Given** a failure count reaches exactly 3 **When** the third failure is processed **Then** a `kyc.resubmission.cs_alert` event is appended to the event store
- **Given** a failure count reaches exactly 5 **When** the fifth failure is processed **Then** status transitions to `locked` (not `pending_resubmission`) and a `kyc.resubmission.account_locked` event is emitted
- **Given** an admin unlocks an account **When** the unlock is processed **Then** failure count resets to 0 and a `kyc.resubmission.account_unlocked` event is emitted

**Types:**

```typescript
  | 'not_verified'
  | 'pending'
  | 'pending_resubmission'
  | 'in_manual_review'
  | 'verified'
  | 'expired'
  | 'reverification_required'
  | 'rejected'
  | 'locked';

export type KycDocumentType = 'passport' | 'national_id' | 'drivers_licence';
```

**Factory method:**
- `static create(input: CreateKycVerificationInput): KycVerification`
  - Input: `{ accountId: string }`
  - Sets status to `not_verified`, failure count to 0, all nullable fields to null
  - Does NOT validate accountId format â€” that is a repository concern
  - Returns new `KycVerification` instance

**Reconstitution:**
- `static reconstitute(data: KycVerificationProps): KycVerification`
- No validation â€” data is from trusted DB source

**Business methods (all return new `KycVerification`, entity is immutable):**

- `submit(documentType: KycDocumentType, frontDocumentRef: string, backDocumentRef: string | null): KycVerification`
  - Permitted from: `not_verified`, `pending_resubmission`, `rejected` (when failure count < 5)
  - Sets status to `pending`, updates `documentType`, `frontDocumentRef`, `backDocumentRef`, `submittedAt = new Date()`
  - Throws `KycAlreadyPendingError` if current status is `pending`
  - Throws `KycAlreadyVerifiedError` if current status is `verified`
  - Throws `KycAccountLockedError` if current status is `locked`
  - Throws `KycInvalidTransitionError` for any other non-permitted source status

- `approve(): KycVerification`
  - Permitted from: `pending`, `in_manual_review`
  - Sets status to `verified`, `verifiedAt = new Date()`
  - Throws `KycInvalidTransitionError` if transition is not permitted

- `fail(): KycVerification`
  - Permitted from: `pending`, `in_manual_review`
  - Increments `failureCount`
  - If new `failureCount >= 5`: sets status to `locked`
  - Else: sets status to `pending_resubmission`
  - Throws `KycInvalidTransitionError` if transition is not permitted

- `unlock(): KycVerification`
  - Permitted from: `locked` only
  - Sets status to `pending_resubmission`, resets `failureCount` to 0
  - Throws `KycNotLockedError` if current status is not `locked`

**Computed properties:**
- `isVerified(): boolean` â€” returns `status === 'verified'`
- `canSubmit(): boolean` â€” returns `status` is one of `not_verified`, `pending_resubmission`, `rejected` (and failure count < 5 for rejected)

#### `submitVerification(input: SubmitKycInput): Promise<KycVerification>`

Input type:
```typescript
interface SubmitKycInput {
  readonly accountId: string;
  readonly documentType: KycDocumentType;
  readonly frontDocumentBuffer: Buffer;
  readonly frontDocumentMimeType: 'image/jpeg' | 'image/png' | 'application/pdf';
  readonly backDocumentBuffer: Buffer | null;
  readonly backDocumentMimeType: ('image/jpeg' | 'image/png' | 'application/pdf') | null;
}
```

Steps:
1. Look up existing `KycVerification` for `accountId` via `kycRepository.findByAccountId`. If none exists, create one with `KycVerification.create({ accountId })`.
2. Call `kycVerification.submit(documentType, '', null)` first to validate state transition (will throw domain error if invalid state). If this throws, propagate immediately.
3. Store front document via `storagePort.store(frontKey, frontDocumentBuffer, frontDocumentMimeType)` where `frontKey = kyc/${accountId}/${crypto.randomUUID()}`. Capture the returned reference.
4. If `backDocumentBuffer` is provided, store back document via `storagePort.store(backKey, backDocumentBuffer, backDocumentMimeType)`. Capture the returned reference.
5. Call `kycPort.submitVerification({ accountId, documentType, frontDocumentRef, backDocumentRef })`. Capture `providerReference`.
6. Apply the full `submit()` call on the entity with actual refs: `kycVerification.submit(documentType, frontDocumentRef, backDocumentRef)`.
7. Update `providerReference` on the entity (via a `withProviderReference` method or by rebuilding from reconstitute â€” see entity spec note).
8. In a transaction: `kycRepository.save(updatedKycVerification)`, then append event `kyc.verification.start` to event store.
9. The mock adapter calls `onApproval(accountId, providerReference)` after the configured delay. The `onApproval` callback is `this.processApproval.bind(this)`.

**Error handling:**
- Domain errors (`KycAlreadyPendingError`, `KycAlreadyVerifiedError`, `KycAccountLockedError`) â†’ propagate as-is (controller maps to 409)
- Storage failures â†’ log error, return 500 (do NOT change KYC status)
- Event store failure â†’ log error, return 500 (transactional rollback ensures KYC status not persisted)

#### `processApproval(accountId: string, providerReference: string): Promise<void>`

Called by the mock adapter's auto-approval callback (or by a real webhook handler in future).

Steps:
1. Fetch `KycVerification` by `accountId` from `kycRepository`.
2. If not found or status is not `pending`, log warning and return (idempotency: already resolved).
3. Apply `kycVerification.approve()`.
4. Fetch the `Account` from `accountRepository.findById(accountId)`.
5. If account does not have `creator` role, apply `account.withRoles([...account.roles, 'creator'])`.
6. In a transaction:
   a. `kycRepository.save(approvedKycVerification)`
   b. `accountRepository.update(updatedAccount, txClient)`
   c. Append event `kyc.status.change` to event store (payload: `{ previous_status: 'pending', new_status: 'verified', account_id: accountId }`)
7. Log: `{ accountId, action: 'kyc_approved' }` â€” no PII

#### `processFailure(accountId: string): Promise<void>`

For future use by real Veriff webhook or admin manual review rejection. Used internally for testing failure escalation.

Steps:
1. Fetch `KycVerification` by `accountId`.
2. Apply `kycVerification.fail()`.
3. Capture new `failureCount` and `status` from the returned entity.
4. In a transaction:
   a. `kycRepository.save(failedKycVerification)`
   b. Append `kyc.status.change` event
   c. If `failureCount === 3`, also append `kyc.resubmission.cs_alert` event
   d. If `new status === 'locked'`, also append `kyc.resubmission.account_locked` event

#### `getKycStatus(accountId: string): Promise<KycStatusResult>`

```typescript
interface KycStatusResult {
  readonly status: KycStatus;
  readonly failureCount: number;
  readonly verifiedAt: Date | null;
  readonly documentType: KycDocumentType | null;
}
```

Steps:
1. Fetch `KycVerification` by `accountId` from `kycRepository`.
2. If none found, return `{ status: 'not_verified', failureCount: 0, verifiedAt: null, documentType: null }`.
3. Return the relevant fields from the entity.

#### `adminUnlock(adminAccountId: string, targetAccountId: string): Promise<KycVerification>`

Steps:
1. Fetch `KycVerification` by `targetAccountId`.
2. If not found, throw `KycVerificationNotFoundError`.
3. Apply `kycVerification.unlock()` â€” throws `KycNotLockedError` if not in `locked` status.
4. In a transaction:
   a. `kycRepository.save(unlockedKycVerification)`
   b. Append `kyc.resubmission.account_unlocked` event (payload: `{ account_id: targetAccountId, admin_actor_id: adminAccountId }`)
5. Return the updated entity.

**Note on magic byte validation**: After multer parses the upload, validate each file's buffer against magic bytes:
- JPEG: `FF D8 FF`
- PNG: `89 50 4E 47`
- PDF: `25 50 44 46`

### `POST /api/v1/kyc/submit`

**Description:** Initiates KYC verification by uploading identity documents.
**Auth:** Required (Clerk JWT)
**Roles:** Any authenticated user
**Content-Type:** `multipart/form-data`

**Request fields:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `document_type` | string | Yes | One of: `passport`, `national_id`, `drivers_licence` |
| `front_document` | file | Yes | Front of document. JPEG, PNG, or PDF. Max 20 MB. |
| `back_document` | file | No | Back of document. Required for `national_id` and `drivers_licence`. JPEG, PNG, or PDF. Max 20 MB. |

**Validation rules:**
- `document_type`: must be one of `passport`, `national_id`, `drivers_licence`
- `front_document`: required, max 20 MB, magic bytes must match JPEG/PNG/PDF
- `back_document`: required when `document_type` is `national_id` or `drivers_licence`, max 20 MB, magic bytes must match JPEG/PNG/PDF
- `back_document`: must be null/absent when `document_type` is `passport`

**Success response:** `202 Accepted`
```json
{
  "data": {
    "status": "pending"
  }
}
```

**Error responses:**

| Status | Code | When |
|--------|------|------|
| 400 | `VALIDATION_ERROR` | `document_type` missing or invalid |
| 400 | `INVALID_DOCUMENT_FORMAT` | File magic bytes do not match accepted formats |
| 400 | `VALIDATION_ERROR` | File exceeds 20 MB |
| 400 | `VALIDATION_ERROR` | `back_document` required but not provided |
| 401 | `UNAUTHENTICATED` | No valid Clerk JWT |
| 409 | `KYC_ALREADY_PENDING` | User already has a pending submission |
| 409 | `KYC_ALREADY_VERIFIED` | User is already verified |
| 409 | `KYC_ACCOUNT_LOCKED` | Account is locked â€” admin must unlock first |
| 500 | `INTERNAL_ERROR` | Storage failure or event store failure |

### `GET /api/v1/kyc/status`

**Description:** Returns the current KYC verification status for the authenticated user.
**Auth:** Required (Clerk JWT)
**Roles:** Any authenticated user

**Request:** No body, no query parameters.

**Success response:** `200 OK`
```json
{
  "data": {
    "status": "pending",
    "failure_count": 0,
    "verified_at": null,
    "document_type": "passport"
  }
}
```

Field descriptions:
- `status`: one of the nine `KycStatus` values
- `failure_count`: integer, 0 or greater
- `verified_at`: ISO 8601 string or null
- `document_type`: one of `KycDocumentType` or null

If no record exists, returns the default `not_verified` response (same shape, all nulls for optional fields).

**Error responses:**

| Status | Code | When |
|--------|------|------|
| 401 | `UNAUTHENTICATED` | No valid Clerk JWT |

### `POST /api/v1/kyc/admin/unlock/:accountId`

**Description:** Admin-only endpoint to unlock a KYC-locked account, allowing the user to resubmit.
**Auth:** Required (Clerk JWT)
**Roles:** `administrator` or `super_administrator` only

**Request:** No body.

**Role check:** Fetch the calling user's account and check `account.roles.includes('administrator') || account.roles.includes('super_administrator')`. Return 403 if check fails.

**Success response:** `200 OK`
```json
{
  "data": {
    "status": "pending_resubmission",
    "failure_count": 0
  }
}
```

**Error responses:**

| Status | Code | When |
|--------|------|------|
| 401 | `UNAUTHENTICATED` | No valid Clerk JWT |
| 403 | `UNAUTHORIZED` | Calling user does not have administrator or super_administrator role |
| 404 | `ACCOUNT_NOT_FOUND` | `:accountId` does not match any account |
| 409 | `KYC_NOT_LOCKED` | Account exists but KYC status is not `locked` |

## Event Store Usage

All KYC state transitions emit events to the `events` table. All events follow the existing `AppendEventInput` interface.

| Event Type | Trigger | Payload (no PII) |
|---|---|---|
| `kyc.verification.start` | `submitVerification` | `{ account_id, document_type, provider_reference }` |
| `kyc.status.change` | Any status transition | `{ account_id, previous_status, new_status }` |
| `kyc.resubmission.cs_alert` | failure_count reaches 3 | `{ account_id, failure_count }` |
| `kyc.resubmission.account_locked` | failure_count reaches 5 | `{ account_id, failure_count }` |
| `kyc.resubmission.account_unlocked` | Admin unlock | `{ account_id, admin_actor_id }` |

All events use `aggregate_type: 'kyc_verification'`, `aggregate_id: kyc_verification.id`, `source_service: 'kyc-service'`.

## Open Questions

None. All decisions are resolved:
- Status naming: uses `not_verified` (matching DB migration, not `not_started` from brief)
- Role grant: done in `processApproval` as part of same transaction as KYC status update (in-process call to account repository)
- Document storage: columns added to `kyc_verifications` table via new migration
- Idempotency: UNIQUE constraint on `account_id` + UPSERT + domain entity state checks
- Admin role check: account roles checked from `req.authContext` and fetched account
- `POST /api/v1/kyc/submit` response: always returns `pending` (never `verified` directly)
- Mock auto-approval: uses `setTimeout` + callback, approval is always out-of-band from the submit response
