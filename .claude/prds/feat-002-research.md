# Research: feat-002 — Core Database Schema

> Research document for Spec Writer consumption. Generated by Spec Researcher.

## 1. Domain Knowledge

### Key Concepts

**Migration-first database design**: feat-002 is purely a schema exercise — no application code, no API endpoints, no frontend. The deliverable is seven dbmate migration files that establish the data foundation all subsequent features depend on. Getting the schema right here is critical because all downstream features build on it; schema mistakes are expensive to fix later.

**dbmate format**: Each migration file is SQL with `-- migrate:up` and `-- migrate:down` comment markers. The tool tracks applied migrations in a `schema_migrations` table it manages automatically. Migrations are applied in filename timestamp order. Idempotency of the `up` migration is NOT required (dbmate tracks which have run), but the `down` must cleanly reverse the `up`.

**The `updated_at` trigger pattern**: Migration `20260305120000_add_updated_at_trigger.sql` already exists and creates the `update_updated_at_column()` PL/pgSQL function. Each table that has an `updated_at` column must separately CREATE TRIGGER calling this function. The function itself is already created — only the per-table trigger bindings need to be added in each subsequent migration.

**UUID generation strategy**: The feature brief specifies UUIDs are generated at the application layer, not by the database. This means columns should be `UUID PRIMARY KEY` with no `DEFAULT gen_random_uuid()`. The application domain layer creates UUIDs before inserting. This is intentional — it allows domain entities to know their own ID before persistence, enabling the hexagonal architecture pattern where the domain doesn't depend on database sequences.

**BIGINT for money**: All monetary amounts — `funding_goal_cents`, `funding_cap_cents`, `amount_raised_cents`, `amount_cents` — are stored as BIGINT (integer cents). The maximum Stripe charge is approximately $999,999.99 USD. The campaign maximum is $1,000,000,000 USD (1 billion), which is 100,000,000,000 cents — within BIGINT range (max ~9.2 × 10^18). NEVER use NUMERIC, FLOAT, DOUBLE, or REAL for any monetary column.

**Append-only escrow ledger**: The `escrow_ledger` table has no `updated_at` column because it is append-only and immutable. Once a ledger entry is written, it must never be modified. This is consistent with the audit trail requirements in L3-006 and the escrow double-entry design in L4-004. Any correction to the ledger is done via a new offsetting entry, not an update.

**One-to-one KYC per user**: `kyc_verifications` has a UNIQUE constraint on `user_id` — a user has at most one KYC record. The status column tracks the lifecycle; the record is updated in place (with `updated_at` reflecting changes). This differs from an event-sourced KYC approach; the current spec design is a single row per user with mutable status.

**Bounded context cross-cutting**: feat-002 creates schema for all five bounded contexts (Account, KYC, Campaign, Payments/Escrow, Contributions) in a single feature. All seven tables are in a single PostgreSQL database with a single schema — no schema-per-domain separation. Foreign keys cross bounded contexts (e.g., `contributions.donor_id` → `users`, `campaigns.creator_id` → `users`).

### Calculation Methodology

**Funding progress**: Calculated as `amount_raised_cents / funding_goal_cents * 100`. The `amount_raised_cents` column on the `campaigns` table is a denormalised counter — it accumulates as contributions are captured. This is a read-optimisation; the authoritative source of truth is the sum of `captured` contributions in the `contributions` table.

**Milestone funding percentages**: Sum must equal exactly 100 across all milestones for a given campaign. The `funding_percentage` CHECK constraint on individual milestones ensures each is > 0 AND <= 100, but the sum-to-100 invariant is enforced at the application level, not the database level (no feasible CHECK constraint for aggregate sums in PostgreSQL without a trigger).

**Escrow balance calculation**: Sum of all `escrow_ledger` entries for a campaign: `SUM(CASE WHEN entry_type IN ('contribution', 'interest') THEN amount_cents ELSE -amount_cents END)`. Disbursement and refund entries reduce the balance; contribution and interest entries increase it. The `amount_cents` column on escrow_ledger allows negative values (the CHECK constraint needs careful thought — see Edge Cases).

**Campaign funding cap**: The `funding_cap_cents` column is the maximum amount the campaign can raise. Contributions that would push `amount_raised_cents` above `funding_cap_cents` are rejected at the application layer (see AC-CAMP-022 in L4-002). The schema does not enforce this cap — it is a business rule in the application service.

### Industry Conventions

**Crowdfunding platform schema patterns**:

- Kickstarter and Indiegogo separate the campaign entity from financial tracking, with contribution tables linking donors to campaigns.
- Escrow-backed platforms like those using Stripe Connect typically maintain a separate ledger table (as specified here) rather than relying solely on Stripe's records — the internal ledger is the source of truth; the gateway reference is the external verification.
- All major platforms use integer cents for monetary storage internally and format for display in the frontend.

**PostgreSQL UUID patterns**: The `gen_random_uuid()` function is available natively in PostgreSQL 13+. However, per the feature brief, UUID generation is the application's responsibility. Some teams add `DEFAULT gen_random_uuid()` as a safety net even when the application generates UUIDs — this is a valid option but the feature brief explicitly states "UUIDs generated by the application layer (not database sequences)", so no DEFAULT should be set.

**Cascade delete strategy**: The schema uses `ON DELETE CASCADE` for child records that are meaningless without the parent (e.g., `user_roles` cascades when user is deleted, `kyc_verifications` cascades when user is deleted, `milestones` cascade when campaign is deleted). Contributions do NOT cascade delete when a user is deleted — financial records must be retained per retention policies in L3-004 (7 years from transaction date). Contributions also do NOT cascade when a campaign is deleted (campaigns should not be deletable once they have contributions; the `cancelled` status handles this).

**Review queue pattern**: The `campaigns` table stores all lifecycle states in a single `status` column (text enum pattern). This is the standard approach for state machines in PostgreSQL — a text CHECK constraint on the status column with all valid values, plus an index on status for efficient queue queries.

### Regulatory Notes

**Financial data retention**: Transaction records (contributions, escrow ledger) must be retained for 7 years per L3-004. This means contributions and escrow ledger rows cannot be hard-deleted — the schema design must support soft retention. No `deleted_at` column is required in the initial schema (feat-002 is scoped to creating tables), but the design must not use `ON DELETE CASCADE` on financial tables.

**KYC data retention**: KYC records must be retained for ~8+ years per L3-004 (AML/CTF Act). The `kyc_verifications` table should NOT cascade-delete when the user is deleted in the GDPR sense — the spec says to cascade on `ON DELETE CASCADE` (per the feature brief), but this conflicts with the AML/CTF retention requirement. This is flagged as an open question in the Recommendations section.

**PII in schema**: Email addresses in `users.email` are PII and classified Confidential. The schema does not enforce encryption — that is a storage-layer concern. No plaintext passwords are stored (authentication is delegated to Clerk via `clerk_id`).

This migration creates the `update_updated_at_column()` trigger function:

```sql
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();

END;
```

## 3. Third-Party APIs

### dbmate

- **Purpose**: Database migration tool, used here in the agent runtime to apply/rollback migrations.
- **Docker image**: `amacneil/dbmate`
- **In agent runtime**: dbmate binary is available directly (not via Docker — the agent cannot run Docker commands per gotchas.md).
- **Key commands**:
  - `dbmate up` — apply all pending migrations
  - `dbmate down` — roll back the most recent migration
  - `dbmate status` — show which migrations have run
- **Environment**: Reads `DATABASE_URL` from environment. In agent runtime: `postgres://mmf:mmf@postgres:5432/mmf?sslmode=disable`
- **Migration format**: Files must be named `YYYYMMDDHHMMSS_description.sql`. Files must contain exactly one `-- migrate:up` and one `-- migrate:down` comment (these are literal SQL comments that dbmate uses as section markers).
- **`schema_migrations` table**: dbmate creates and manages this table itself. Do not create or modify it manually.
- **Idempotency**: dbmate will not re-run a migration it has already applied. The `-- migrate:up` section does NOT need to be idempotent (no `IF NOT EXISTS` required for table creation — though using it is not wrong).

### PostgreSQL (via `pg` library)

- **Connection**: `postgres://mmf:mmf@postgres:5432/mmf?sslmode=disable` in agent runtime
- **Version**: Aurora PostgreSQL compatible — functions like `gen_random_uuid()`, `NOW()`, `TIMESTAMPTZ` are all available.
- **UUID type**: The `uuid` type is native in PostgreSQL. No extension needed (the `uuid-ossp` extension is NOT required when using `gen_random_uuid()` from pgcrypto, which is built-in since PostgreSQL 13).
- **BIGINT range**: -9,223,372,036,854,775,808 to 9,223,372,036,854,775,807. Maximum campaign cap of $1B USD = 100,000,000,000 cents, well within range.

## 4. Competitor Patterns

### Kickstarter Schema Patterns

- Uses integer cents for all monetary values.
- Project status is a text enum column.
- Milestones and rewards are separate tables with FK to the project.
- Backer records (contributions) have a status column for payment states (succeeded, failed, refunded).
- No separate escrow ledger table is public knowledge — their escrow is managed by Stripe Connect.

**Learnings**: The separate `escrow_ledger` table (append-only) is architecturally superior to Kickstarter's approach because it gives MMF a complete audit trail of fund movements independent of the gateway.

### Indiegogo Schema Patterns

- Supports both fixed and flexible funding models. MMF uses a hybrid (flexible with minimum threshold).
- Campaign table includes both a `goal_amount` and a `collected_amount` (denormalised counter, as in MMF).
- Contribution states align closely with MMF's: pending, captured, refunded.

**Learnings**: Denormalised `amount_raised_cents` counter on campaigns is the right call for performance on campaign listing queries (avoids SUM across contributions table on every page load).

### GoFundMe / Charity platforms

- Donations are typically simpler (no escrow, no milestones).
- Contribution tables are straightforward with gateway reference for reconciliation.

**What doesn't work**: GoFundMe's single-step "contribute and it goes directly to creator" model is inappropriate for MMF's escrow + milestone pattern.

### Stripe Connect Marketplace Patterns

- Platform creates a `PaymentIntent` per contribution, capturing immediately.
- The `gateway_reference` column in `contributions` maps to the Stripe `PaymentIntent.id` or `Charge.id`.
- Refunds reference the original charge by ID — not the original token.
- For the escrow ledger, Stripe Connect uses "destination charges" or "separate charges + transfers" — but the local demo stubs this. The internal ledger is the authoritative record.

**Learnings**: `gateway_reference` in `contributions` should store the Stripe PaymentIntent ID or Charge ID (not the token). The `reference` in `escrow_ledger` can store gateway transfer IDs for disbursements.

## 5. Edge Cases

### Data Edge Cases

- [ ] **UUID PK collision**: Application generates UUIDs — if two requests generate the same UUID (astronomically unlikely with UUID v4 but theoretically possible), the INSERT fails with a unique constraint violation. The application layer must handle this and retry with a new UUID. The schema should use `UUID PRIMARY KEY` without a DEFAULT.
- [ ] **NULL vs empty string**: PostgreSQL treats `''` (empty string) and `NULL` differently. `display_name`, `avatar_url`, `bio` should be nullable (not `NOT NULL DEFAULT ''`). If the application inserts `''` instead of `NULL`, queries filtering for "no display name" may behave unexpectedly. The spec must clarify the contract.
- [ ] **`amount_raised_cents` negative**: The `campaigns.amount_raised_cents` starts at 0. If a refund is processed for a campaign with no contributions (data inconsistency), this could go negative. A `CHECK (amount_raised_cents >= 0)` constraint should be applied to prevent this.
- [ ] **`escrow_ledger.amount_cents` sign convention**: The entry_type determines whether this is a credit or debit. Should `amount_cents` for refund/disbursement entries be stored as positive (with sign determined by entry_type) or negative? The spec must be explicit. Storing all values as positive (with entry_type determining direction) is safer and avoids negative BIGINT confusion. A `CHECK (amount_cents > 0)` constraint would enforce this — but then balance calculation requires CASE WHEN logic rather than SUM.
- [ ] **`funding_percentage` precision**: Stored as INT (0-100). If a creator wants milestones at 33.3%, 33.3%, 33.4%, the INT constraint forces rounding. The spec should clarify if INTEGER is sufficient or if a NUMERIC is needed. The feature brief specifies INT — this is confirmed.
- [ ] **`account_status` and `status` constraint violation**: Text columns for status enums have no PostgreSQL-level CHECK constraint in the feature brief. An invalid status value (e.g., a typo in application code) would be silently inserted. The spec should mandate CHECK constraints listing all valid values for every status column.
- [ ] **`users.clerk_id` uniqueness race**: Two simultaneous registrations with the same Clerk ID (shouldn't happen but could if a retry storm occurs) would both attempt INSERT. The UNIQUE constraint on `clerk_id` ensures only one succeeds; the other gets a constraint violation that the application must handle gracefully with a 409 response.
- [ ] **`user_roles` duplicate role insertion**: The UNIQUE constraint on `(user_id, role)` prevents duplicate roles. An upsert pattern (`INSERT ... ON CONFLICT DO NOTHING`) should be used by the application to make role assignment idempotent. Without this, a retry of a role assignment would return a 409 that must be handled.

### Schema Migration Edge Cases

- [ ] **Migration order dependency**: `user_roles` (migration 2) references `users` (migration 1). If migration 1 fails or is not applied, migration 2 will fail with a "table does not exist" error. The FK constraint creates a hard ordering dependency. dbmate applies migrations in timestamp order — the timestamps `20260306000001` through `20260306000007` ensure correct ordering.
- [ ] **Down migration ordering**: Running `dbmate down` multiple times must be done in reverse order. If the `contributions` table is dropped (migration 6 down) but the `escrow_ledger` (migration 7) still exists with its FK to `contributions`, the drop will fail unless escrow_ledger is dropped first (migration 7 down). The down migrations must be applied in reverse order — dbmate handles one-at-a-time down migrations, so this should work correctly as long as the spec tests running down migrations in reverse sequence.
- [ ] **Trigger creation in down migration**: Each migration that creates a trigger must also drop that trigger in the `-- migrate:down` section. For example, migration 1 (users) creates a trigger `set_users_updated_at` — the down migration must `DROP TRIGGER IF EXISTS set_users_updated_at ON users` before `DROP TABLE IF EXISTS users`. If the down just drops the table, the trigger is dropped implicitly, but explicit drops are safer.
- [ ] **Trigger function dependency**: Migration `20260305120000` creates the `update_updated_at_column()` function. Running its `-- migrate:down` (which drops the function) while any trigger still references it will fail with "cannot drop function because other objects depend on it". The existing migration's down is therefore only safely runnable if all subsequent migrations have been rolled back first. This is a key ordering constraint.
- [ ] **`BEGIN/COMMIT` wrapping**: Per infra.md rules, `-- migrate:up` sections should be wrapped in `BEGIN; ... COMMIT;` to make each migration atomic. If any statement fails mid-migration (e.g., index creation fails), the transaction rolls back and no partial schema is left. This is critical for the seven-table schema.
- [ ] **`CREATE TABLE IF NOT EXISTS`**: Per infra.md, tables should be created with `IF NOT EXISTS`. This prevents failure if a migration is somehow applied twice (though dbmate prevents this via `schema_migrations`). It also makes the migrations more robust against manual schema changes.

### Concurrency Edge Cases

- [ ] **Simultaneous contribution + `amount_raised_cents` update**: The `campaigns.amount_raised_cents` is a denormalised counter updated by each contribution capture. With multiple simultaneous contributions, an optimistic update (`UPDATE campaigns SET amount_raised_cents = amount_raised_cents + $1`) must be used, not a read-modify-write cycle. The schema itself does not enforce this — it is an application concern — but the schema must allow atomic updates (no column-level lock required; PostgreSQL row-level locking handles this).
- [ ] **`amount_raised_cents` exceeding `funding_cap_cents` under concurrency**: Two contributions submitted simultaneously could both pass the "is the campaign below cap?" check before either is committed, resulting in `amount_raised_cents` exceeding `funding_cap_cents`. A `CHECK (amount_raised_cents <= funding_cap_cents)` constraint on the `campaigns` table would enforce this at the database level — but would block legitimate updates. The alternative is a database-level constraint or an advisory lock in the application. A CHECK constraint here is worth considering even though it makes the campaign UPDATE logic more complex.
- [ ] **Escrow ledger entry race**: Two simultaneous disbursements for different milestones could race to insert into `escrow_ledger`. Since the ledger is append-only and each row is independent, this is safe — PostgreSQL's row-level locking means INSERT never blocks on other INSERTs to the same table (no unique constraint conflicts expected for distinct entries).
- [ ] **`kyc_verifications` simultaneous update**: If a provider webhook and an admin action both attempt to update a `kyc_verifications` row simultaneously, the last writer wins (no optimistic locking in the schema). An `updated_at` column provides a mechanism for optimistic concurrency at the application layer — compare `updated_at` before updating.
- [ ] **Campaign status transition race**: Two requests attempting to transition a campaign from `submitted` to `under_review` simultaneously (two reviewers claiming the same campaign) could both succeed if not protected. The application must use `UPDATE campaigns SET status = 'under_review' WHERE id = $1 AND status = 'submitted'` and check `rowCount = 1` to detect the race. The schema must support this pattern — a simple text status column with no optimistic lock token means the application must use conditional updates.

### Boundary Edge Cases

- [ ] **Maximum `funding_goal_cents`**: The campaign spec (L4-002) sets a maximum funding target of $1,000,000,000 USD. In cents: 100,000,000,000 (100 billion). This fits in BIGINT (max ~9.2 × 10^18). A CHECK constraint `CHECK (funding_goal_cents >= 100000000)` (min $1M = 100,000,000 cents) and `CHECK (funding_cap_cents <= 100000000000)` (max $1B = 100 billion cents) should be added to enforce the business rules at the database level.
- [ ] **Minimum `amount_cents` on contributions**: The feature brief specifies `CHECK > 0` on contributions.amount_cents. This must be `CHECK (amount_cents > 0)`, not `CHECK (amount_cents >= 0)`. Zero-value contributions must be rejected.
- [ ] **`milestones.order_index` uniqueness**: Two milestones for the same campaign with the same `order_index` would create an ambiguous ordering. A UNIQUE constraint on `(campaign_id, order_index)` should be added — the feature brief does not mention this, but it is implied by the "ordered milestones" requirement.
- [ ] **`failure_count` on `kyc_verifications`**: Stored as INT. The spec defines a maximum of 5 attempts before the account locks. A CHECK constraint `CHECK (failure_count >= 0)` prevents negative counts. No upper bound CHECK is needed as the `locked` status gate is enforced at the application layer.
- [ ] **Index performance on `campaigns.status`**: The status column will be heavily queried (review queue, live campaigns listing, admin dashboard). An index on `status` alone is useful, but a composite index on `(status, created_at)` or `(status, deadline)` would better support typical query patterns. The feature brief specifies an index on `creator_id, status` — this should be two separate indexes (one on `creator_id`, one on `status`) OR a composite index depending on query patterns. A composite index `(creator_id, status)` is only efficient for queries filtering on both columns.
- [ ] **Escrow ledger `entry_type` exhaustiveness**: The feature brief lists `contribution, disbursement, refund, interest` as valid entry types. A CHECK constraint should enumerate all valid values. Future entry types (e.g., `fee`, `adjustment`) would require a migration to update the CHECK constraint — this is the cost of using CHECK over a lookup table.

### User Behaviour Edge Cases

- [ ] **Deactivated user with active KYC record**: If a user is deleted (`account_status = 'deleted'`) but the `kyc_verifications` row cascades-deletes (per `ON DELETE CASCADE` in the feature brief), the KYC record is lost — violating AML/CTF retention requirements in L3-004. This is a conflict between the feature brief spec (ON DELETE CASCADE on kyc) and the regulatory requirement. The Spec Writer must resolve this: the down-migration should retain KYC records even after user deactivation. Options: (a) no cascade on kyc_verifications (FK with `ON DELETE RESTRICT` or `ON DELETE SET NULL`), (b) soft-delete users rather than hard-delete, (c) cascade is fine because "deleted" is a status, not a hard delete (physical row remains).
- [ ] **Creator with no roles record**: If `user_roles` assignment fails after `users` INSERT (e.g., application crash between the two inserts), the user exists with no role. Queries that JOIN users to user_roles would exclude this user from "backer" queries. The application must handle this atomically (both in one transaction) or tolerate users without roles. A database-level constraint cannot enforce "every user must have at least one role" without a deferred trigger.
- [ ] **Campaign with zero milestones**: The `milestones` table is a separate table. A campaign could be inserted with no corresponding milestone rows (the milestone validation is enforced at submission time in the application, not at the database level). The schema does not prevent zero milestones for a campaign — this is acceptable for Draft campaigns being iteratively assembled.
- [ ] **`assigned_by` self-reference on `user_roles`**: The `assigned_by` column references `users.id`. If a Super Administrator assigns their own role (which the spec says is impossible via the UI), the FK would not prevent it — it only enforces that the referenced user exists. The UNIQUE constraint on `(user_id, role)` doesn't help here. Business rule enforcement is at the application layer.

### Integration Edge Cases

- [ ] **`dbmate up` then immediate `dbmate down` and `dbmate up` again**: The acceptance criteria require this to leave the schema in the same state. The idempotency test means each migration's down must exactly reverse the up — no data loss, no orphaned triggers, no missing indexes. Specifically: triggers must be DROP-ped in down migrations before the table is dropped; indexes are automatically dropped when their table is dropped (no explicit DROP INDEX needed in down migrations for table-bound indexes).
- [ ] **Trigger function dropped before table triggers**: If `20260305120000` down migration runs while any of migrations 1-7 have been applied, PostgreSQL will refuse to drop the function because triggers depend on it. The down migrations for all seven table migrations must be applied first. This is an operational risk — developers must run `dbmate down` seven times before running it an eighth time to reach the trigger function migration.

## 6. Recommendations

### Must-Haves for Spec

1. **CHECK constraints on all status/enum columns**: Every `status`, `role`, `entry_type`, `category` column must have a CHECK constraint enumerating valid values. The feature brief does not specify CHECK constraints explicitly — this is a critical omission that the Spec Writer must address. Without CHECK constraints, invalid status values will be silently accepted.

2. **CHECK constraints on monetary amounts**: `funding_goal_cents CHECK >= 100000000` (min $1M), `funding_cap_cents CHECK <= 100000000000` (max $1B), `funding_cap_cents >= funding_goal_cents` (cap must be at or above goal), `amount_cents > 0` on contributions, `amount_raised_cents >= 0` on campaigns.

3. **`BEGIN; ... COMMIT;` wrapping**: Per infra.md rules, all `-- migrate:up` sections must be wrapped in a transaction. Verify that all seven migrations follow this pattern.

4. **Trigger binding per table**: The migration for each table that has `updated_at` must include a `CREATE TRIGGER` statement calling `update_updated_at_column()`. The trigger name should follow a consistent pattern: `set_<table>_updated_at`. The down migration must `DROP TRIGGER IF EXISTS set_<table>_updated_at ON <table>` before dropping the table (though dropping the table implicitly drops its triggers — the explicit DROP is cleaner but redundant).

5. **Resolve the KYC cascade-delete vs. retention conflict**: `kyc_verifications.user_id FK ON DELETE CASCADE` conflicts with the AML/CTF retention requirement. Options: use `ON DELETE RESTRICT`, use `ON DELETE SET NULL` (would require making user_id nullable), or accept CASCADE on the basis that the application never hard-deletes users (uses `account_status = 'deleted'` instead). The feature brief says ON DELETE CASCADE — the Spec Writer should flag this conflict and confirm the interpretation. Physical user row deletion should probably never happen (GDPR erasure is done by nulling PII fields, not dropping rows). CASCADE is likely safe in practice.

6. **Unique constraint on `(campaign_id, order_index)` for milestones**: The feature brief does not specify this but it is implied. Without it, duplicate order_index values for the same campaign create ambiguous ordering.

7. **Index strategy clarification**: The feature brief specifies "index on creator_id, status" for campaigns. Clarify if this means a single composite index `(creator_id, status)` or two separate indexes. For the review queue (query by status alone), a separate index on `status` is needed. For creator-specific campaign listing (query by creator_id), an index on `creator_id` is needed. These are different access patterns.

8. **`users.email` case sensitivity**: Email addresses should be stored in lowercase normalised form. A case-sensitive UNIQUE constraint on `email` would allow `User@Example.com` and `user@example.com` as different users. The Spec Writer should confirm that the application layer normalises emails to lowercase before insert, and optionally specify `LOWER(email)` functional index for case-insensitive uniqueness.

### Watch-Outs

- **Trigger function dependency order**: The `20260305120000` migration's down drops the trigger function. Running that down migration without first rolling back all seven table migrations will fail. Document this operational constraint.

- **`amount_raised_cents` as a denormalised counter**: This column can drift from the sum of captured contributions if a contribution state transition is not properly reflected (e.g., a refund is processed in Stripe but the `amount_raised_cents` counter is not decremented). The Spec Writer should flag that `amount_raised_cents` is not automatically consistent — it is maintained by application logic.

- **No `deleted_at` soft-delete column**: The schema uses `account_status` to represent logical deletion. Physical row deletion never happens (confirmed by the GDPR erasure pattern of nulling PII). The migrations should not add `deleted_at` columns — status-based soft delete is the pattern.

- **`provider_reference` on `kyc_verifications` length**: Veriff session IDs are UUID-format strings (~36 chars). The column is `text` (unbounded) — this is fine, just confirming no truncation risk.

- **Timestamps as TIMESTAMPTZ (not TIMESTAMP)**: All timestamp columns must use `TIMESTAMPTZ` to store timezone-aware timestamps. Plain `TIMESTAMP` stores local time without timezone, leading to subtle bugs when the server timezone changes. Infra.md explicitly mandates `TIMESTAMPTZ`.

- **`NOT NULL DEFAULT NOW()` on `created_at` and `updated_at`**: Per infra.md rules, all tables must have `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()` and `updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`. The `DEFAULT NOW()` ensures the columns are populated even if the application does not explicitly provide them. The trigger updates `updated_at` on every UPDATE.

- **No `NOT NULL` on optional FK columns**: `assigned_by` on `user_roles` is nullable (the first Super Administrator role has no assigner). `contribution_id` on `escrow_ledger` is nullable (interest entries are not associated with a specific contribution). `provider_reference` on `kyc_verifications` is nullable until verification is initiated with the provider. The spec must be explicit about which FK columns are nullable.

### Suggested Approach

The seven migrations should be written in this order, each building on the previous:

1. `users` — base identity table, no FK dependencies
2. `user_roles` — depends on `users`
3. `kyc_verifications` — depends on `users`
4. `campaigns` — depends on `users` (creator_id FK)
5. `milestones` — depends on `campaigns`
6. `contributions` — depends on `users` and `campaigns`
7. `escrow_ledger` — depends on `campaigns` and `contributions`

Each migration should follow the pattern:
- `BEGIN;`
- `CREATE TABLE IF NOT EXISTS ...` with all columns, constraints, and comments
- `CREATE INDEX IF NOT EXISTS ...` for all secondary indexes
- `CREATE TRIGGER set_<table>_updated_at ...` (where applicable)
- `COMMIT;`

The `-- migrate:down` should:
- `DROP TABLE IF EXISTS <table> CASCADE;` (CASCADE handles dependent triggers and indexes automatically)

Note: `DROP TABLE CASCADE` is safe in down migrations because the dependent objects (triggers, indexes) are table-owned. It would cascade to FK-referencing tables only if CASCADE were applied to a referenced table — but since down migrations are applied in reverse order (escrow_ledger dropped before contributions, contributions before campaigns, etc.), no FK violation occurs.

Test plan for completeness:
1. Run `dbmate up` — all seven migrations applied cleanly.
2. Run `dbmate down` seven times — each table dropped cleanly in reverse order.
3. Run `dbmate up` again — schema is identical to step 1.
4. Verify `schema_migrations` table tracks all applied migrations.
