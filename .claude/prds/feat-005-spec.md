## Overview

feat-005 implements the end-to-end contribution flow: an authenticated backer navigates to a live
campaign's contribute page, enters an amount and payment token, and the backend captures the
payment through the stubbed payment gateway adapter, atomically records it to the escrow ledger,
updates the campaign totals, and returns a confirmation with a transaction reference.

The payment gateway is stubbed — no real money moves — but the full architectural pattern
(port/adapter abstraction, escrow ledger, contribution state machine, audit trail) is implemented
exactly as it would be for a real Stripe integration. The stub uses the sentinel value `tok_fail`
to trigger a payment failure path; all other tokens succeed.

## Key Decisions (Resolved)

| Decision | Resolution | Rationale |
|----------|-----------|-----------|
| Contribution state machine | `pending_capture` → `captured` or `failed` | Creates immutable intent record before gateway call; enables crash-recovery |
| Atomic success transaction | `UPDATE contribution` + `INSERT escrow_ledger` + `UPDATE campaigns` in a single `BEGIN/COMMIT` | Data integrity is non-negotiable — partial success states are impossible |
| Payment gateway port | `PaymentGatewayPort.capture(CaptureInput): Promise<CaptureResult>` | Standard port/adapter pattern; stub and real Stripe adapter both satisfy interface |
| Stub sentinel | `tok_fail` → failure; any other non-empty string → success | Enables deterministic test of both paths |
| Duplicate detection | SQL query: same `donor_user_id` + `campaign_id` + `amount_cents`, `created_at > NOW() - 60s`, status != `failed` | Application-layer check before gateway call; no DB unique index to avoid blocking retries |
| HTTP status for payment failure | `201 Created` with `status: "failed"` | Preserves audit trail; contribution record exists regardless of outcome |
| Creator contribution to own campaign | Allowed | No restriction needed for demo; simplifies logic |
| KYC requirement | NOT required for contributions | Only required for creator role and disbursements per L4-004 |
| Campaign `funded` auto-transition | When `total_raised_cents >= funding_goal_cents`, transition campaign to `funded` in same DB transaction | Atomic; prevents race conditions; funded campaigns still visible but stop accepting new contributions |
| Funded campaigns accepting contributions | Reject with `422 CAMPAIGN_NOT_ACCEPTING_CONTRIBUTIONS` | Once funded, no further contributions accepted (per final decision vs. brief ambiguity) |
| Concurrent contribution race | Last-write-wins on campaign totals | Acceptable for demo; production would use `SELECT FOR UPDATE` or row-level advisory locks |
| amountCents serialisation | String in both request and response (per G-024 BIGINT rule) | Frontend never receives number for monetary amounts |
| `payment_token` logging | NEVER logged at any layer | Security invariant; treat like a credential |
| Ledger insert fails after capture | Cannot happen — single transaction | Migration + application design prevent partial states |
| Campaign `funded` vs. `live` routing | Both checked at validation — only `live` accepts contributions | After auto-transition to `funded`, the campaign no longer accepts new contributions |

## Bounded Context

**New bounded context**: `payments` at `packages/backend/src/payments/`.

This is a new top-level bounded context (distinct from `campaign` and `account`) per the
hexagonal architecture rule that payment processing is a separate domain concern.

The `payments` context:
- Owns `Contribution` entity, `EscrowLedger` value (append-only), `ContributionAuditEvent`
- Reads `Campaign` via injected `CampaignRepository` port (cross-context read, P-023)
- Reads `User` via injected `UserRepository` port (cross-context read)
- Calls `PaymentGatewayPort` for capture
- Writes to `contributions`, `escrow_ledger`, `contribution_audit_events`, and `campaigns` tables

## User Stories

### US-001: Make a contribution
As a mission backer,
I want to contribute money to a live campaign I believe in,
So that I can help fund the mission and receive a confirmation of my contribution.

**Acceptance Criteria:**
- AC-001: An unauthenticated attempt to POST /contributions returns `401 UNAUTHENTICATED`
- AC-002: An authenticated user can submit `{ campaignId, amountCents, paymentToken }` to `POST /api/v1/contributions`
- AC-003: `amountCents` must be at least `500` (representing $5.00 USD); below this returns `400 VALIDATION_ERROR` with message "Minimum contribution is $5.00"
- AC-004: On success, the response is `201` with `{ data: { id, campaignId, amountCents, status: "captured", transactionRef, createdAt } }`
- AC-005: The campaign's `total_raised_cents` and `contributor_count` are updated immediately after successful capture

### US-002: Stub payment failure testing
As a developer testing the contribution flow,
I want to trigger a payment failure using a known sentinel token,
So that I can verify the failure path is handled correctly.

**Acceptance Criteria:**
- AC-006: Submitting `paymentToken: "tok_fail"` returns `201` with `status: "failed"` and a clear `failureReason`
- AC-007: A failed contribution does NOT update campaign totals or create an escrow ledger entry
- AC-008: Both success and failure paths create a `contribution_audit_events` record

### US-003: Duplicate contribution protection
As a platform operator,
I want to prevent accidental duplicate contributions within a short window,
So that backers are not charged twice for the same action.

**Acceptance Criteria:**
- AC-009: Submitting the same `{ donorUserId, campaignId, amountCents }` within 60 seconds returns `409 DUPLICATE_CONTRIBUTION`
- AC-010: A duplicate check does NOT count `failed` contributions
- AC-011: The same amount submitted 61 seconds later is accepted as a new contribution

### US-004: Campaign funding goal reached
As a platform operator,
I want the campaign to automatically transition to `funded` when its goal is reached,
So that the campaign status accurately reflects real-time funding progress.

**Acceptance Criteria:**
- AC-012: When a successful capture causes `total_raised_cents >= funding_goal_cents`, the campaign status transitions to `funded` in the same DB transaction
- AC-013: When a campaign is `funded`, further contributions return `422 CAMPAIGN_NOT_ACCEPTING_CONTRIBUTIONS`
- AC-014: If `fundingGoalCents` is null (no goal set), the campaign never auto-transitions to `funded`

### US-005: View contribution status
As a backer,
I want to retrieve the status of a specific contribution,
So that I can confirm my contribution was processed.

**Acceptance Criteria:**
- AC-015: `GET /api/v1/contributions/:id` returns the contribution if it belongs to the authenticated user
- AC-016: `GET /api/v1/contributions/:id` returns `404 NOT_FOUND` if the contribution belongs to another user (security: do not reveal existence)
- AC-017: `GET /api/v1/campaigns/:id/contributions` returns the authenticated user's contributions to that campaign, paginated

## Out of Scope

- Real Stripe gateway integration with live credentials — stub only
- Multi-approval disbursement workflow — deferred to feat-006+
- Milestone-based fund release — deferred
- Campaign failure refund automation — deferred
- Donor-initiated refund requests — deferred to feat-006
- Tax receipt generation — deferred
- Stored payment methods — explicitly excluded per L4-004 Section 4.3
- `refunded` and `partially_refunded` contribution states — feat-006
- Interest accrual on escrowed funds — theatre; schema supports it, calculation not implemented
- Daily reconciliation and financial reporting dashboards — theatre
- Stripe Elements client-side tokenisation — stub uses a plain text input; real tokenisation is feat-007+
