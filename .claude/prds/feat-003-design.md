## Overview

feat-003 introduces authentication UI using Clerk's pre-built components. The design challenge is not building auth forms from scratch but theming Clerk's components to feel native within the MMF dark-first design system, and defining the auth-related UI patterns (header auth state, loading states, route protection) that all future features depend on.

Key design decisions:
- Clerk's `<SignIn />` and `<SignUp />` components are used directly with appearance customisation via Clerk's `appearance` prop
- Auth pages use a centred single-column layout within PageShell
- Header gains auth-aware state (sign-in button or user menu)
- Loading states during auth checks use a branded spinner, not skeleton screens (there is no content shape to skeleton)

**Design token mapping rationale:**
- Clerk card maps to `--color-bg-surface` / `--color-border-subtle` / `--radius-card` — matching the standard card spec from brand.md Section 3.2
- Primary button uses `--gradient-action-primary` values with `--radius-button` and `--color-action-primary-shadow` — matching the Primary button spec from brand.md Section 3.1
- Form inputs match the Form Inputs spec from brand.md Section 3.6
- Labels use `--type-input-label` properties
- Clerk headings use `--font-display` with `--type-section-heading` sizing (not `--type-page-title`, since the Clerk card heading is contextually a section within the page, not the page title)

**Note:** Clerk's `variables` API requires resolved values (hex/rgba), not CSS variable references. The `elements` API accepts CSS variable references. This is why `variables` uses raw values matching our tokens while `elements` uses `var()` references.

### Component: SignInPage

**File:** `packages/frontend/src/pages/sign-in.tsx`
**Reuses:** AuthCentreLayout, clerkAppearance

#### Props

None (page component).

#### Visual Specification

Renders `<AuthCentreLayout>` containing Clerk's `<SignIn />` component with the MMF appearance overrides.

```tsx
<PageShell>
  <AuthCentreLayout>
    <SignIn
      routing="path"
      path="/sign-in"
      signUpUrl="/sign-up"
      forceRedirectUrl="/dashboard"
      appearance={clerkAppearance}
    />
  </AuthCentreLayout>
</PageShell>
```

#### States

**Default:** Clerk sign-in form rendered within the centred layout.

**Loading state:** Clerk handles its own internal loading states. The page renders the layout immediately; the Clerk component manages its own spinner.

**Error state:** Clerk handles form-level errors (invalid credentials, account locked). These are styled via the `formFieldErrorText` element in the appearance config: `--color-text-error` (red-planet), `--type-body-small`.

#### Accessibility

- **Landmark:** Content is within `<main>` via PageShell.
- **Heading structure:** Clerk renders its own `<h1>` for "Sign in". The MMF logo text above is purely decorative branding and should not be a heading element (use `<p>` or `<span>`).
- **Keyboard navigation:** Clerk components are keyboard accessible by default.
- **Focus states:** Handled by Clerk, overridden via `formFieldInput__focused` to use MMF's focus ring (`--color-action-primary` border + shadow).

### Component: HeaderAuthSection

**File:** `packages/frontend/src/components/layout/header-auth-section.tsx`
**Reuses:** New component (added to PageShell's header)

This component renders the auth-aware portion of the page header. It shows different content based on whether the user is signed in or not.

#### Props

None (uses Clerk hooks internally).

#### Visual Specification

**Signed-out state:**

A single "Sign In" link/button in the header, right-aligned.

- **Element:** `<a>` styled as a ghost button
- **Typography:** `--type-button` (DM Sans, 14px, 600, letter-spacing 0.01em)
- **Colour:** `--color-action-ghost-text` (ignition)
- **Border:** `1px solid var(--color-action-ghost-border)`
- **Border radius:** `--radius-button`
- **Padding:** `8px 20px`
- **Hover:** Background transitions to `--color-bg-elevated`, `--motion-hover-duration` / `--motion-hover-easing`

**Signed-in state:**

Clerk's `<UserButton />` component, right-aligned in the header.

- **Size:** 32px (matching the 32px header logo height for visual balance)
- **Appearance override:** The `<UserButton />` accepts an `appearance` prop to style the avatar and dropdown.

```typescript
const userButtonAppearance = {
  elements: {
    avatarBox: {
      width: '32px',
      height: '32px',
    },
    userButtonPopoverCard: {
      backgroundColor: 'var(--color-bg-surface)',
      border: '1px solid var(--color-border-subtle)',
      borderRadius: 'var(--radius-card)',
      boxShadow: '0 8px 32px rgba(0, 0, 0, 0.4)',
    },
    userButtonPopoverActionButton: {
      color: 'var(--color-text-secondary)',
      fontFamily: 'var(--font-body)',
      fontSize: '14px',
    },
    userButtonPopoverActionButton__hover: {
      backgroundColor: 'var(--color-bg-elevated)',
      color: 'var(--color-text-primary)',
    },
    userButtonPopoverFooter: {
      display: 'none',
    },
  },
};
```

**Loading state (auth loading):**

While `isLoaded` is false from `useAuth()`, render nothing (no flash of sign-in button before user is confirmed). The header logo is still visible; only the right-side auth section is empty during the brief load.

**Layout within header:**

```
┌────────────────────────────────────────────────────────┐
│  MMF (logo)                          [Sign In] / (●)  │
└────────────────────────────────────────────────────────┘
```

- Header inner container: `display: flex`, `justify-content: space-between`, `align-items: center`
- The auth section is right-aligned via flex layout

#### States

| State | Display |
|-------|---------|
| Auth loading (`isLoaded === false`) | Empty space (no content rendered) |
| Signed out | Ghost-style "Sign In" link button |
| Signed in | Clerk `<UserButton />` with 32px avatar |

#### Interactions

- **Click "Sign In" (signed-out):** Navigate to `/sign-in`
- **Click avatar (signed-in):** Clerk's `<UserButton />` opens its dropdown menu (manage account, sign out)
- **Hover "Sign In":** Background `--color-bg-elevated`, transition `--motion-hover-duration`

#### Accessibility

- **Role:** The "Sign In" link uses an `<a>` element with `href="/sign-in"` — no ARIA role needed.
- **Label:** "Sign In" text serves as the accessible name.
- **Keyboard navigation:** Tab-focusable. Enter activates navigation.
- **Screen reader (signed-in):** Clerk's `<UserButton />` announces "User menu" or similar (Clerk handles this).
- **Screen reader (loading):** No announcement needed for the brief auth loading state.

## Design Tokens

### Tokens Used

|-------|----------------------|
| `--color-bg-page` | Primary page background (void) |
| `--color-bg-surface` | Clerk card background, UserButton popover background |
| `--color-bg-elevated` | Hover states on buttons, UserButton popover actions hover |
| `--color-bg-input` | Clerk form input backgrounds |
| `--color-text-primary` | Page headings, Clerk card headings, form input text |
| `--color-text-secondary` | Body text, user info, Clerk card subtitles |
| `--color-text-tertiary` | Form labels, loading screen text, divider text |
| `--color-text-error` | Form validation errors, API error messages |
| `--color-action-primary` | Clerk primary button base, focus ring colour, spinner accent |
| `--color-action-primary-hover` | Focus ring outline, link hover states |
| `--color-action-primary-text` | Text on primary buttons |
| `--color-action-primary-shadow` | Primary button drop shadow |
| `--color-action-secondary-bg` | Secondary button background, social login buttons |
| `--color-action-secondary-text` | Secondary button text |

## Responsive Behaviour

### Breakpoints

| Breakpoint | Width | Layout Changes |
|------------|-------|---------------|
| Desktop | >=1024px | Auth pages have generous top padding (80px). Clerk component at natural width (~400px). |
| Tablet | 768-1023px | Reduced top padding (48px). Clerk component still centred at natural width. |
| Mobile | <768px | Minimal top padding (32px). Clerk component fills available width minus 16px horizontal padding. |

### Component-specific responsive notes

- **AuthCentreLayout:** Top padding scales down at smaller breakpoints. Clerk component is always centred.
- **DashboardPlaceholder heading:** Scales from 40px (mobile) to 56px (sm+) per the type scale.
- **OnboardingPlaceholder heading:** Same scaling as DashboardPlaceholder.
- **HeaderAuthSection:** "Sign In" button and `<UserButton />` remain in the header at all breakpoints. No mobile navigation collapse needed for feat-003 (only one auth action in the header).
- **Clerk components:** Clerk's `<SignIn />` and `<SignUp />` are responsive by default. The `maxWidth` in their container ensures they do not exceed the card width, while `width: 100%` ensures they fill available space on small screens.

### Touch Targets

- "Sign In" header link: Minimum 44px height via padding (8px vertical + 14px font + line-height buffer exceeds 44px). If computed height is below 44px, increase vertical padding to `12px`.
- "Skip for now" button: 12px vertical padding + 14px font ensures minimum 44px touch target.
- Clerk form inputs: Clerk's default input height meets 44px minimum.
- Clerk buttons: Clerk's default button height meets 44px minimum.

## New Patterns Introduced

### 1. Clerk Appearance Theming Pattern

A centralised `clerkAppearance` configuration object that maps MMF semantic tokens to Clerk's theming API. This pattern should be reused for any future Clerk components (e.g., `<UserProfile />`, `<OrganizationSwitcher />`). The configuration lives in a single module to ensure consistency.

### 2. AuthCentreLayout Pattern

A centred layout wrapper for auth-related pages. Used for sign-in and sign-up. Could be reused for password reset, email verification, or other auth flows added in the future.

### 3. Auth Loading Screen Pattern

A branded full-page loading screen for auth state initialisation. This pattern is distinct from content skeleton loading (which mimics content shape). Auth loading uses a spinner because there is no predictable content shape to skeleton.

### 4. Header Auth State Pattern

The header conditionally renders auth controls based on sign-in state. This establishes the pattern for the header growing with auth-related controls. Future features (notifications, account switcher) will extend this section.

### 5. Route Protection Pattern

`ProtectedRoute` and `PublicOnlyRoute` wrappers establish the pattern for all future route definitions. Every new protected route wraps in `ProtectedRoute`; public-only routes (that should redirect when signed in) wrap in `PublicOnlyRoute`.
