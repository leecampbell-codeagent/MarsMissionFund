## Overview

feat-003 has minimal UI surface area. The authentication form itself is hosted by Clerk ŌĆö not custom designed by MMF. The MMF design work here covers three things:

1. **The `Header` component** ŌĆö navigation bar with conditional auth controls (Sign In / Sign Out).
2. **The `DashboardPage`** ŌĆö a protected placeholder page that proves the auth shell works.
3. **The `ProtectedRoute` wrapper** ŌĆö a loading/redirect shell, not a visual design in its own right, but its loading state requires a spinner spec.

All three are scaffolding for the features that follow. The header and dashboard must use the full MMF token system ŌĆö they will evolve in later features, so every decision here sets a precedent.

### Sign In / Sign Up Pages

**Route:** `/sign-in/*`, `/sign-up/*`
**Layout:** Clerk-hosted UI ŌĆö no MMF design needed

Clerk renders its own `<SignIn />` and `<SignUp />` components. MMF does not wrap, style, or override these. The pages are Clerk's responsibility.

The only MMF concern: when a user lands on `/sign-in`, Clerk's component renders against whatever background shows behind it. The `<ClerkProvider>` will use Clerk's default theme. No `appearance` prop customisation is specified for feat-003 ŌĆö Clerk's built-in UI is acceptable for the auth scaffold. Theme customisation for Clerk components is deferred to a future design iteration.

## Components

### Component: Header

**File:** `packages/frontend/src/components/layout/header.tsx`
**Reuses:** New component
**Export:** Named export ŌĆö `export function Header()`

#### Props

None. The component reads auth state internally via Clerk hooks (`useAuth()` and `useClerk()`).

#### Visual Specification

**Container:**
- Position: `fixed`, `top: 0`, `left: 0`, `right: 0`
- Height: `64px`
- Background: `--color-bg-surface` (deep-space ŌĆö slightly lifted from the void page background to create a subtle depth separation)
- Border bottom: `1px solid --color-border-subtle`
- `backdrop-filter: blur(8px)` ŌĆö adds atmospheric depth consistent with the space theme; acceptable decorative effect
- `z-index: 50`
- `padding: 0 24px` (mobile); `0 32px` (Ōēź768px); `0 48px` (Ōēź1024px)

**Inner layout:**
- `display: flex; align-items: center; justify-content: space-between`
- Max-width: `1280px`; `margin: 0 auto` (centres the content at very wide viewports)
- Left slot: MMF logo/wordmark
- Right slot: auth controls

**Logo / wordmark (left slot):**
- Coin icon mark: 32px height, per brand spec Section 6.1 (nav bar context)
- Minimum 8px clear space on all sides (per brand spec Section 6.2)
- Wordmark text: "Mars Mission Fund"
  - Font: `--type-button` (DM Sans 600, 14px)
  - Colour: `--color-text-primary`
  - Hidden on mobile (<640px) ŌĆö coin icon only to preserve space
  - Visible from 640px up
- The coin icon and wordmark are wrapped in a `<Link to="/">` (React Router) ŌĆö clicking navigates to `/`
- Icon: decorative `aria-hidden="true"`; the `<Link>` has `aria-label="Mars Mission Fund ŌĆö home"`

**Auth controls (right slot):**
- Three mutually exclusive states: loading, signed-out, signed-in (see States below)
- Minimum 44x44px touch target on all interactive elements (WCAG 2.5.5)

#### Typography in Header

| Element | Token | Size | Weight |
|---------|-------|------|--------|
| Wordmark text | `--type-button` | 14px | 600 |
| Sign In button label | `--type-button` | 14px | 600 |
| Sign Out button label | `--type-button` | 14px | 600 |
| User email | `--type-body-small` | 13px | 400 |

#### States

**Loading state (`isLoaded = false`):**
- Right slot renders nothing ŌĆö no button, no text
- This avoids a flash of wrong state (showing Sign In before auth resolves, then swapping to Sign Out)
- No skeleton placeholder ŌĆö the slot is simply empty during this brief moment
- Duration is typically under 300ms; a placeholder would flash before being useful

**Signed-out state (`isSignedIn = false`, `isLoaded = true`):**
- Right slot renders: a single Ghost button labelled "Sign In"
- Ghost button spec (per brand spec Section 3.1):
  - Background: transparent
  - Text: `--color-action-ghost-text` (`--ignition`)
  - Border: `1px solid --color-action-ghost-border` (`--launchfire / 30%`)
  - Border radius: `--radius-button` (100px pill)
  - Padding: `10px 20px` (slightly tighter than the 12px/24px standard ŌĆö header is compact at 64px)
  - Font: `--type-button`
- The button is rendered as a `<Link to="/sign-in">` styled as a button (React Router link, not `<button>` ŌĆö it navigates, not submits)
- Accessible name: "Sign In"

**Signed-in state (`isSignedIn = true`, `isLoaded = true`):**
- Right slot renders: user identifier + Sign Out button
- Layout: `display: flex; align-items: center; gap: 16px`
- User identifier:
  - Displays the user's primary email address from `useUser().user.primaryEmailAddress?.emailAddress`
  - Font: `--type-body-small` (DM Sans 400, 13px)
  - Colour: `--color-text-tertiary`
  - Hidden on mobile (<640px) ŌĆö screen space is limited; Sign Out button alone is sufficient
  - Visible from 640px up
  - Not a link; decorative display only
  - `aria-hidden="true"` ŌĆö the Sign Out button's accessible name is sufficient context; the email is redundant for screen readers and potentially a privacy leak in announcements
- Sign Out button:
  - Ghost button variant (same visual spec as Sign In button above)
  - Label: "Sign Out"
  - `<button>` element (not a link ŌĆö it performs an action, not navigation)
  - On click: calls `signOut()` from `useClerk()`, then navigates to `/sign-in` via `useNavigate()`
  - Accessible name: "Sign Out"

**Hover state (Sign In / Sign Out buttons):**
- Text colour: `--color-action-primary-hover` (`--ignition`) ŌĆö already at this value for ghost; no change needed for text
- Border colour: transitions to `--color-action-ghost-text` (fully opaque ignition orange)
- Background: `--color-action-secondary-bg` (`--white / 6%`) ŌĆö a subtle surface lift
- Transition: `background-color var(--motion-hover), border-color var(--motion-hover)` (`150ms, cubic-bezier(0.25, 1, 0.5, 1)`)

**Focus state (Sign In / Sign Out buttons):**
- `outline: 2px solid --color-action-primary-hover; outline-offset: 2px`
- Per brand spec Section 5.3

#### Interactions

- **Click Sign In button:** Navigates to `/sign-in` (React Router `<Link>`)
- **Click Sign Out button:** Calls `signOut()` from Clerk, then navigates to `/sign-in`
- **Click logo/wordmark:** Navigates to `/` (root ŌĆö redirects to `/dashboard` if authenticated)
- **Tab key:** Logo link ŌåÆ Sign In button (or Sign Out button when signed in); logical left-to-right order
- **Enter/Space on Sign Out button:** Same as click

#### Accessibility

- **Role:** `<header>` element with `role="banner"` (implicit for `<header>` at the top level)
- **Label:** No additional `aria-label` needed ŌĆö `role="banner"` is self-describing
- **Logo link:** `aria-label="Mars Mission Fund ŌĆö home"` on the `<Link>` wrapper; coin icon is `aria-hidden="true"`
- **Keyboard navigation:** Tab order: logo link, then auth button (Sign In or Sign Out). Tab count: 2 interactive elements maximum.
- **Screen reader announcement (signed-out):** "Sign In, link" (navigates to sign-in page)
- **Screen reader announcement (signed-in):** "Sign Out, button"
- **Auth state change:** When `isLoaded` transitions from false to true, no live region announcement needed ŌĆö the page already reflects the correct state. Navigation-induced announcements handle page context.

### Component: ProtectedRoute

**File:** `packages/frontend/src/components/layout/protected-route.tsx`
**Reuses:** New component
**Export:** Named export ŌĆö `export function ProtectedRoute()`

This is a routing wrapper, not a visually designed component. Its only visual output is the loading state ŌĆö the spinner rendered while Clerk auth state resolves.

#### Props

| Prop | Type | Required | Default | Description |
|------|------|----------|---------|-------------|
| `children` | `React.ReactNode` | Yes | ŌĆö | Page content to render when authenticated |

#### States

**Loading state (`isLoaded = false`):**
- Renders `<LoadingSpinner />` component (see below)
- Fills the full viewport below the header
- `min-height: calc(100vh - 64px)`, flexbox centred

**Unauthenticated state (`isSignedIn = false`, `isLoaded = true`):**
- Renders `<Navigate to="/sign-in" replace />`
- No visual output ŌĆö instant redirect
- `replace` prevents the protected route appearing in browser history

**Authenticated state (`isSignedIn = true`, `isLoaded = true`):**
- Renders `children` with no visual wrapper

#### Accessibility

- Loading state must set `aria-busy="true"` on the spinner container
- Screen reader: spinner is labelled "Loading" (see LoadingSpinner spec below)

### Component: LoadingSpinner

**File:** `packages/frontend/src/components/ui/loading-spinner.tsx`
**Reuses:** New component
**Export:** Named export ŌĆö `export function LoadingSpinner()`

This component is used by `ProtectedRoute` during Clerk's initialisation. It is the first loading state a user ever sees.

#### Props

| Prop | Type | Required | Default | Description |
|------|------|----------|---------|-------------|
| `label` | `string` | No | `"Loading"` | Screen reader label for the spinner |
| `size` | `'sm' \| 'md' \| 'lg'` | No | `'md'` | Controls spinner diameter |

#### Visual Specification

**Container:**
- `display: flex; flex-direction: column; align-items: center; justify-content: center; gap: 16px`
- When used inside `ProtectedRoute`: `min-height: calc(100vh - 64px)` applied by the parent

**Spinner element:**
- A circular SVG ring: 40px diameter (`md`), 24px (`sm`), 56px (`lg`)
- Stroke: `--color-action-primary` (`--launchfire`)
- Track (full circle, dim): `--color-progress-track` (`--white / 6%`)
- Animated arc: `stroke-dasharray` + `stroke-dashoffset` CSS animation ŌĆö one arc sweeping around the ring
- Animation: `360deg` rotation over `800ms` linear infinite (uses `--duration-slow` for the orbit speed)
- `prefers-reduced-motion: reduce` ŌĆö stops rotation animation; the arc renders static at 25% position

**Label text (visible):**
- Not shown by default ŌĆö spinner only
- `aria-label` on the SVG provides the accessible name

#### Accessibility

- SVG has `role="status"` and `aria-label={label}` (defaults to "Loading")
- Container has `aria-busy="true"`
- Screen reader: announces "Loading" when the spinner mounts

## New Patterns Introduced

### Ghost Button Hover Surface

The brand spec Section 3.1 defines Ghost button colours but does not explicitly define the hover background. This spec introduces:

- Hover background: `--color-action-secondary-bg` (`--white / 6%`)
- Hover border: `--color-action-ghost-text` (fully opaque ignition orange ŌĆö `--ignition`)

This is a natural extension of the existing token system ŌĆö Secondary button background applied to Ghost button hover ŌĆö not a new token. It is consistent with the surface elevation hierarchy (`bg-page` ŌåÆ `bg-surface` ŌåÆ `bg-elevated` ŌåÆ `action-secondary-bg` as the lightest interactive surface).

### Fixed Header with Backdrop Blur

The brand spec does not define a specific header treatment. This spec establishes:

- Background: `--color-bg-surface` with `backdrop-filter: blur(8px)` ŌĆö gives the header a frosted-glass depth over the page content as the user scrolls
- This is purely decorative and does not affect content legibility
- The blur is a CSS property, not a design token ŌĆö it is a one-off treatment for the header only, acceptable given the header is a singleton component

### LoadingSpinner

The brand spec does not define a loading spinner component. This spec introduces one using:

- Stroke colour: `--color-action-primary` ŌĆö the primary interactive colour signals "the app is working"
- Track colour: `--color-progress-track` ŌĆö consistent with the progress bar system's track token
- Animation timing: `--duration-slow` ŌĆö orbits slowly, like a satellite; consistent with the space theme and semantic motion vocabulary

This pattern should be added to the shared component library for reuse across loading states in future features.
