## Page Layouts

### 1. Campaign Create Page (`/campaigns/new`)

**Purpose:** Multi-section form for creating a campaign draft. Campaign ID is created server-side on mount; all subsequent sections auto-save via PATCH.

**Auth gate:** Creator role + KYC verified. If KYC not verified: show KYC gate panel (see KYC Gate Pattern below). If no creator role: show Creator Role Gate panel.

**Overall layout:**

```
┌────────────────────────────────────────────────────────┐
│  Page background: --color-bg-page                      │
│  Max-width: 900px, centered, padding: 48px 24px        │
│                                                        │
│  ┌──────────────────────────────────────────────────┐  │
│  │  Section label: "01 — NEW MISSION PROPOSAL"      │  │
│  │  Page title (--type-page-title): "BUILD YOUR     │  │
│  │  CAMPAIGN"                                       │  │
│  │  Sub-label (--type-body-small, --color-text-     │  │
│  │  secondary): "Draft auto-saves as you go."       │  │
│  └──────────────────────────────────────────────────┘  │
│                                                        │
│  ┌──── Auto-save status bar ───────────────────────┐  │
│  │  [Saving... | Saved | Failed to save]            │  │
│  └──────────────────────────────────────────────────┘  │
│                                                        │
│  ┌──── Section nav (vertical stepper, sticky left) ┐  │
│  │  1. Basics          ● complete                   │  │
│  │  2. Details         ○ incomplete                 │  │
│  │  3. Funding         ○ incomplete                 │  │
│  │  4. Team            ○ incomplete                 │  │
│  │  5. Milestones      ○ incomplete                 │  │
│  │  6. Risk            ○ incomplete                 │  │
│  │  7. Budget          ○ optional                   │  │
│  │  8. Review & Submit ○ pending                    │  │
│  └──────────────────────────────────────────────────┘  │
│                                                        │
│  ┌──── Active form section ──────────────────────── ┐  │
│  │  (section component rendered here)               │  │
│  └──────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────┘
```

**Desktop layout (≥1024px):**
- Two-column: 220px left nav (sticky top: 24px) + remaining content area
- Left nav shows all 8 sections as a vertical stepper with completion indicators
- Content area: single column, max-width 640px

**Section nav stepper item states:**
- Complete: filled circle `--color-status-success`, section label `--color-text-primary`
- Incomplete (visited): empty circle `--color-border-emphasis`, section label `--color-text-secondary`
- Active: filled circle `--color-action-primary`, section label `--color-text-primary`, left accent bar 2px `--color-action-primary`
- Incomplete (not visited): empty circle `--color-border-subtle`, section label `--color-text-tertiary`
- Optional: empty circle `--color-border-subtle`, section label `--color-text-tertiary`, "(optional)" suffix in `--type-body-small`

**Loading state on mount:**
- Full page area replaced by centered `LoadingSpinner` (existing component) with `aria-label="Creating your campaign draft..."`
- Displayed while the initial `POST /campaigns` call completes

**KYC Gate Panel (shown instead of form when KYC not verified):**
```
┌──────────────────────────────────────────────────────────┐
│  Background: --color-bg-surface                          │
│  Border: 1px solid color-mix(in srgb,                    │
│          --color-status-warning 25%, transparent)        │
│  Top accent: 2px --color-status-warning                  │
│  Padding: 32px, --radius-card                            │
│                                                          │
│  Icon: warning symbol (aria-hidden)                      │
│  Heading (--type-card-title, --color-text-primary):      │
│    "Identity Verification Required"                      │
│  Body (--type-body, --color-text-secondary):             │
│    "You must complete identity verification before       │
│     creating a campaign. This keeps the platform safe    │
│     for all backers."                                    │
│  Button (secondary): "Verify Your Identity →"           │
│    → links to /settings/profile                         │
└──────────────────────────────────────────────────────────┘
```

**Creator Role Gate Panel (shown when no creator role):**
Same structure as KYC Gate Panel but with:
- Heading: "Creator Role Required"
- Body: "Designate yourself as a project creator to start building campaigns."
- Button (primary): "Become a Creator" → calls `assignCreatorRole()` inline, no redirect

| Status | Buttons shown | Button variants |

| `rejected` | "Edit Campaign" (primary), "Archive" (ghost) | One primary |

| `under_review` | Read-only view with claimed-by info | — |

- Shown in right sidebar below funding info

- `under_review` + assigned to different reviewer: "Claimed by [reviewer id truncated]" info text

**Admin extras:**
- `under_review`: "Reassign Reviewer" button (secondary) below ReviewActionPanel
- Any status: "Archive Campaign" button (ghost with error color)

### 4. Review Queue Page (`/review-queue`)

**Purpose:** Admin/Reviewer view of all submitted campaigns awaiting review, in FIFO order.

**Auth gate:** Reviewer or Administrator role required. If backer only: redirect to home with error toast "You don't have access to the review queue."

**Overall layout:**

```
┌────────────────────────────────────────────────────────┐
│  Page background: --color-bg-page                      │
│  Max-width: 960px, centered, padding: 48px 24px        │
│                                                        │
│  ┌──── Page header ─────────────────────────────────┐  │
│  │  Section label: "ADMIN — REVIEW QUEUE"           │  │
│  │  Page title: "MISSION REVIEW QUEUE"              │  │
│  │  Sub-text: "Oldest submissions appear first.     │  │
│  │    Review and process in order."                 │  │
│  │  Right: count badge "X awaiting review"          │  │
│  └──────────────────────────────────────────────────┘  │
│                                                        │
│  ┌──── SLA banner (conditional) ───────────────────┐  │
│  │  Shown when any campaign submitted > 5 days ago  │  │
│  │  Background: color-mix(in srgb,                  │  │
│  │    --color-status-warning 10%, transparent)      │  │
│  │  Border: 1px solid color-mix(in srgb,            │  │
│  │    --color-status-warning 25%, transparent)      │  │
│  │  Text: "X campaigns have been waiting more than  │  │
│  │    5 days. Review these first."                  │  │
│  └──────────────────────────────────────────────────┘  │
│                                                        │
│  ┌──── Queue list ──────────────────────────────────┐  │
│  │  [ ReviewQueueCard ] (overdue indicator if > 5d) │  │
│  │  [ ReviewQueueCard ]                             │  │
│  │  ...                                             │  │
│  └──────────────────────────────────────────────────┘  │
│                                                        │
│  ┌──── Empty state (conditional) ──────────────────┐  │
│  │  "No campaigns awaiting review." (not an error)  │  │
│  └──────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────┘
```

**ReviewQueueCard** is a variant of `CampaignCard` with an additional overdue indicator. Uses the same card component with an `isOverdue` prop.

**Empty state:**
```
  Icon: checkmark circle (aria-hidden), 48px, --color-status-success
  Heading (--type-card-title, --color-text-secondary): "Queue is clear."
  Body (--type-body, --color-text-tertiary):
    "No campaigns are awaiting review right now."
```

**Loading state:** 3× CampaignCard skeletons.

**Auto-refresh:** Page refetches every 60 seconds. No visible countdown — silent background refresh. On new data, list updates in place.

### `CampaignStatusBadge`

**File:** `packages/frontend/src/components/campaign/campaign-status-badge/CampaignStatusBadge.tsx`

**Description:** Compact inline status chip for displaying campaign status. Follows the same structural pattern as `KycStatusBadge` (existing component).

**Props:**
```typescript
interface CampaignStatusBadgeProps {
  readonly status: CampaignStatus;
}
```

**Visual spec per status:**

| Status | Background | Text Color | Border | Dot Color | Dot Pulse | Label |
|--------|-----------|-----------|--------|-----------|-----------|-------|
| `draft` | `--color-bg-elevated` | `--color-text-tertiary` | `1px --color-border-subtle` | `--color-text-tertiary` | No | DRAFT |
| `submitted` | `color-mix(in srgb, --color-status-warning 10%, transparent)` | `--color-text-warning` | `1px solid color-mix(in srgb, --color-status-warning 25%, transparent)` | `--color-status-warning` | Yes | SUBMITTED |
| `under_review` | `color-mix(in srgb, --color-status-warning 10%, transparent)` | `--color-text-warning` | `1px solid color-mix(in srgb, --color-status-warning 25%, transparent)` | `--color-status-warning` | Yes | UNDER REVIEW |
| `approved` | `--color-status-success-bg` | `--color-text-success` | `1px --color-status-success-border` | `--color-status-success` | No | APPROVED |
| `rejected` | `color-mix(in srgb, --color-status-error 10%, transparent)` | `--color-text-error` | `1px solid color-mix(in srgb, --color-status-error 25%, transparent)` | `--color-status-error` | No | REJECTED |
| `live` | `--color-status-active-bg` | `--color-action-primary-hover` | `1px --color-status-active-border` | `--color-status-active` | Yes | LIVE |
| `archived` | `--color-bg-elevated` | `--color-text-tertiary` | `1px --color-border-subtle` | `--color-text-tertiary` | No | ARCHIVED |

**Shared badge styles:**
- `display: inline-flex`, `align-items: center`, `gap: 6px`
- `border-radius: var(--radius-badge)` (8px)
- `padding: 6px 12px`
- `font-family: var(--font-data)` — Space Mono
- `font-size: 11px`, `font-weight: 400`
- `letter-spacing: 0.2em`, `text-transform: uppercase`

**Dot:** 6px × 6px, `border-radius: 50%`, `flex-shrink: 0`, `aria-hidden="true"`

**Pulse animation** (for `submitted`, `under_review`, `live`):
```css
@keyframes status-pulse {
  0%, 100% { opacity: 1; }
  50% { opacity: 0.3; }
}
/* Applied via class to avoid conflict with kyc-pulse */
.campaign-status-dot--pulse {
  animation: status-pulse var(--motion-urgency) ease-in-out infinite;
}
@media (prefers-reduced-motion: reduce) {
  .campaign-status-dot--pulse {
    animation: none;
    opacity: 0.7;
  }
}
```

**ARIA:** `<span role="status">` wraps the badge. Dot is `aria-hidden="true"`. Full status text is visually and programmatically present.

## Animation & Transitions

### Page Entry

All pages use `--motion-enter` (300ms ease-out) for the primary content block entry:
```css
@keyframes page-enter {
  from { opacity: 0; transform: translateY(12px); }
  to { opacity: 1; transform: translateY(0); }
}
.page-content {
  animation: page-enter var(--motion-enter);
}
@media (prefers-reduced-motion: reduce) {
  .page-content { animation: none; }
}
```

### Auto-Save Status Indicator

- "Saving..." appearance: `opacity 0 → 1`, `--motion-hover`
- "Saved" → fade out: `opacity 1 → 0` after 2s delay, `--motion-hover`
- "Failed" appearance: `opacity 0 → 1`, `--motion-hover`, persists until next save attempt

### CampaignCard Hover

- `transform: translateY(-2px)`, duration `--motion-hover`
- `border-color` transition, duration `--motion-hover`
- Disabled under `prefers-reduced-motion: reduce`

### ReviewActionPanel Accordion

- Panel height: `0 → auto` via `--motion-panel` (500ms ease-out)
- Implementation: CSS Grid row height trick (`grid-template-rows: 0fr → 1fr`)
- Chevron rotation: 0° → 180°, `--motion-hover`
- Reduced motion: instant toggle, no transition

### Status Badge Dot Pulse

- `submitted`, `under_review`, `live`: dot opacity 1 → 0.3 → 1, `--motion-urgency` (1.5s)
- Reduced motion: static dot at 70% opacity, no animation

### Skeleton Loading

- `skeletonPulse`: opacity 0.5 → 1 → 0.5, 2s ease-in-out infinite (matches existing `ProfileCard` pattern)
- Reduced motion: static opacity 0.7

### Form Section Transition (section nav click)

- Previous section: `opacity 1 → 0`, `--motion-hover`
- New section: `opacity 0 → 1`, `--motion-enter`
- Reduced motion: instant switch

### Submit Success Redirect

- Brief `--motion-enter-emphasis` on success message before redirect
- "Campaign submitted — redirecting..." toast-style notification
- `role="status"`, `aria-live="assertive"`
- Auto-redirect to `/campaigns/:id` after 1.5s (or immediate on click if user acts first)

### Milestone Sum Progress Bar Fill

- Fill width transition: `--motion-base` (300ms) on value change
- Reduced motion: instant fill

## New Patterns Introduced

The following patterns do not exist in the current codebase and are established by feat-003.

### 1. Multi-Section Form with Vertical Stepper Nav

**Pattern:** Large form split into named sections with a persistent left-sidebar navigation showing completion state. Used in Campaign Create.

**Implementation notes:**
- Section state tracked in parent `CampaignForm` state: `{ [sectionKey]: 'complete' | 'incomplete' | 'error' | 'optional' }`
- Active section stored in URL hash (`/campaigns/new#milestones`) to support browser back
- Section completion derived from: required fields populated AND no validation errors
- Stepper uses `<nav aria-label="Form sections">` with `<ol>` items; active item `aria-current="step"`

### 2. Debounced Auto-Save with Per-Field Status

**Pattern:** Field-level PATCH on blur with 500ms debounce. Single request queue — concurrent PATCHes for the same campaign are prevented by queuing.

**Implementation notes:**
- Global auto-save state: `'idle' | 'saving' | 'saved' | 'error'`
- Debounce: `useRef` timer cleared on each keystroke, fired on blur
- Queue: `useRef` to track in-flight request; pending requests wait for completion
- Status indicator: `role="status"`, `aria-live="polite"` — announces "Saved" to screen readers without interrupting

### 3. Accordion with Mutual Exclusion

**Pattern:** Two accordion sections (Approve / Reject) where opening one automatically closes the other. Used in `ReviewActionPanel`.

**Implementation notes:**
- State: `'approve' | 'reject' | null` — only one open at a time
- Transition via CSS Grid row height (no `max-height` hacks that cause clipping)
- `aria-expanded` on trigger buttons, panels connected via `aria-controls`

### 4. Running Basis Points Allocation Indicator

**Pattern:** Real-time total display with progress bar and colour-coded feedback as user adds milestone percentages. Sticky within the milestones section.

**Implementation notes:**
- Derived from form state, not from API
- Progress bar fill changes gradient/colour based on total vs 10000
- `aria-live="polite"` on total display to announce changes

### 5. Role-Conditional Action Areas

**Pattern:** The same campaign detail page renders different action areas based on the viewer's role and the campaign status. Creator, Reviewer, and Admin each see different CTAs.

**Implementation notes:**
- Role determined from `useCurrentUser()` hook
- Ownership determined by comparing `campaign.creatorUserId` with current user id
- Conditional rendering is explicit — no CSS `display: none` hacks
- All role checks are UI-only; server enforces permissions

### 6. KYC / Role Gate Panel

**Pattern:** Full-bleed gate panel that replaces page content when the user lacks a required role or KYC status. Used on Campaign Create.

**Implementation notes:**
- Check performed on page mount before creating campaign draft
- Gate panel is a standalone component, not a modal
- Includes actionable CTA to remedy the gate condition inline (Creator Role gate: calls API without redirect)

### 7. Hero Image with Graceful Fallback

**Pattern:** Campaign hero displays an `<img>` from an external HTTPS URL, with a gradient fallback when no URL is set or the image fails to load.

**Implementation notes:**
- `onError` handler replaces `<img>` with gradient background
- Image is never from a `data:` or `javascript:` URL (server-validated)
- `alt` attribute: campaign title (not empty, not decorative — it conveys the campaign's visual identity)

## Accessibility Checklist

### Keyboard Navigation

- [ ] All interactive elements reachable by Tab in logical DOM order
- [ ] Multi-section form stepper: arrow keys navigate between sections in left nav (desktop); Enter/Space activates
- [ ] Status filter tabs (My Campaigns): arrow keys navigate, Enter/Space activates; `role="tablist"` with `role="tab"` children
- [ ] Accordion (ReviewActionPanel): Enter/Space toggles; Escape collapses active panel
- [ ] Campaign card (when `onClick` present): Enter/Space activates navigation
- [ ] Category dropdown: keyboard-navigable options; Escape closes; Enter selects
- [ ] Add/remove row buttons (team, milestones, risk, budget): reachable by Tab; after remove, focus moves to next row or the Add button
- [ ] Form submit: Enter on focused Submit button fires submission
- [ ] Auto-save does not steal focus

### Focus States

- [ ] All buttons: `outline: 2px solid var(--color-action-primary-hover); outline-offset: 2px`
- [ ] All form inputs: `border-color: var(--color-action-primary); box-shadow: 0 0 0 3px rgba(255,92,26,0.25)` on focus
- [ ] All interactive cards: `border-color: var(--color-action-primary); box-shadow: 0 0 0 3px rgba(255,92,26,0.15)` on focus
- [ ] No `outline: none` without visible equivalent replacement
- [ ] Focus indicators meet WCAG 2.1 AA minimum 3:1 contrast against adjacent colour

### Screen Reader Support

- [ ] Page `<h1>` present on every page (page title in `--type-page-title`)
- [ ] Section headings follow correct hierarchy (h1 → h2 → h3)
- [ ] `CampaignStatusBadge`: `role="status"`, dot is `aria-hidden="true"`, status text is visible
- [ ] `CampaignCard`: `<article aria-label="{title} — {status}">`
- [ ] Progress bar (milestone allocation): `role="progressbar"`, `aria-valuenow`, `aria-valuemin="0"`, `aria-valuemax="10000"`, `aria-label="Milestone funding allocation"`
- [ ] Auto-save indicator: `role="status"`, `aria-live="polite"`
- [ ] Submit success message: `role="status"`, `aria-live="assertive"`
- [ ] Rejection feedback panel: `role="alert"` (read immediately on mount)
- [ ] Form field labels: all inputs have `<label for="...">` or `aria-labelledby`
- [ ] Character counters: associated with their input via `aria-describedby`
- [ ] Error messages: `aria-describedby` linking input to error element; `aria-invalid="true"` on invalid inputs
- [ ] Loading states: `aria-busy="true"` on loading containers; `aria-label="Loading..."` on spinners
- [ ] Skeleton cards: `aria-busy="true"`, `aria-label="Loading campaign"`
- [ ] Monetary amounts: screen reader text includes currency denomination (e.g., "1,500,000 US dollars")
- [ ] Date inputs: associated label includes timezone context
- [ ] Icon-only buttons: `aria-label` required (e.g., remove button: `aria-label="Remove milestone 2"`)
- [ ] Accordion panels: `aria-expanded` on trigger, `aria-controls` pointing to panel `id`, panel has `role="region"` with `aria-labelledby`
- [ ] Section stepper nav: `<nav aria-label="Form sections">`, active step `aria-current="step"`
- [ ] Status filter tabs: `role="tablist"`, `role="tab"`, `aria-selected`, `aria-controls` for tab panels
- [ ] Hero image `<img>`: `alt="{campaign title}"` — descriptive, not empty
- [ ] Image preview in form: `alt="Hero image preview"`
- [ ] Load error message for image: live region `aria-live="polite"`

### Colour Contrast

- [ ] All primary text (`--color-text-primary`) on page bg: 14.8:1 — AAA pass
- [ ] All secondary text (`--color-text-secondary`) on page bg: 10.7:1 — AA pass
- [ ] All tertiary text (`--color-text-tertiary`) on page bg: 5.4:1 — AA pass (metadata only; never body text)
- [ ] `--color-text-accent` used only at ≥18px bold or ≥24px regular — WCAG large text rule
- [ ] `--color-text-warning` on warning badge bg: verified against WCAG AA (minimum 4.5:1)
- [ ] `--color-text-error` on error bg: verified against WCAG AA
- [ ] `--color-text-success` on success bg: verified against WCAG AA
- [ ] Section label text (`--color-text-accent`) is at 11px — confirm all section labels use `--type-section-label` (≥18px bold or upgrade to `--color-action-primary-hover` for the 11px instances)

  > Note: `--color-text-accent` at 11px fails WCAG large text rule. Section labels at `--type-section-label` (11px) MUST use `--color-action-primary-hover` (`--ignition`) which provides higher contrast, not `--color-text-accent` directly. This matches the brand spec rule: "For smaller accent text, use `--color-action-primary-hover`."
