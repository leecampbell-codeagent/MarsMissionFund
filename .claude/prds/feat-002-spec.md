# PRD: feat-002 — Core Database Schema

> Implementation-ready specification. Generated by Spec Writer.
> Feature brief: feat-002-database-schema.md

## Overview

feat-002 creates the foundational PostgreSQL schema for Mars Mission Fund via seven ordered dbmate
migration files covering five bounded contexts: Account, KYC, Campaign, Payments/Escrow, and
Contributions.
This is a pure database infrastructure feature — no API endpoints, no domain TypeScript code, no
frontend changes.
All subsequent domain features (feat-003 through feat-006+) depend on this schema.

---

## 1. User Stories & Acceptance Criteria

### US-001: Developer Foundation

**As a** developer building domain features for Mars Mission Fund,
**I want** a complete, correctly-typed PostgreSQL schema installed via idempotent migrations,
**So that** I can build domain entities, repositories, and application services against a stable data model without guessing column types or constraints.

**Acceptance Criteria:**

- **Given** a clean database, **When** `dbmate up` is run, **Then** all seven migrations apply in
  order with no errors and the schema is complete.
- **Given** the schema has been applied, **When** `dbmate down` is run seven times, **Then** all
  tables are dropped in reverse order with no FK violation errors.
- **Given** the schema has been fully rolled back, **When** `dbmate up` is run again, **Then** the
  resulting schema is structurally identical to the first application (verified by `dbmate status`
  showing all migrations applied).
- **Given** the schema is applied, **When** a SQL client inspects each table, **Then** every
  monetary column is `BIGINT`, every timestamp column is `TIMESTAMPTZ`, every status/enum column
  has a CHECK constraint, every FK column has an index, and every table with `updated_at` has the
  auto-update trigger bound.
- **Given** the schema is applied, **When** a row with an invalid status value is inserted (e.g.,
  `account_status = 'banana'`), **Then** PostgreSQL rejects the insert with a CHECK constraint
  violation.

#### Table: `users`

**CHECK constraints:**

**Indexes:**

- `idx_users_clerk_id` on `clerk_id` (for O(1) Clerk JWT → user lookups)
- `idx_users_email` on `email` (for email-uniqueness queries and account lookups)

**UNIQUE constraints:**

- `UNIQUE (clerk_id)`
- `UNIQUE (email)`

**Trigger:**

```sql
CREATE TRIGGER set_users_updated_at
  BEFORE UPDATE ON users
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
```

**Migration structure:**

```sql
-- migrate:up
BEGIN;
CREATE TABLE IF NOT EXISTS users ( ... );
CREATE UNIQUE INDEX IF NOT EXISTS idx_users_clerk_id ON users (clerk_id);
CREATE UNIQUE INDEX IF NOT EXISTS idx_users_email ON users (email);
CREATE TRIGGER set_users_updated_at
  BEFORE UPDATE ON users
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
COMMIT;

-- migrate:down
DROP TABLE IF EXISTS users CASCADE;
```

> Note: `DROP TABLE ... CASCADE` implicitly drops all triggers and indexes on the table.
> Explicit `DROP TRIGGER` before `DROP TABLE` is acceptable but redundant.

### Migration 3: `20260306000003_create_kyc.sql`

#### Table: `kyc_verifications`

| Column | Type | Nullable | Default | Description |
|--------|------|----------|---------|-------------|
| `id` | UUID | NOT NULL | — | PK (application-generated) |
| `user_id` | UUID | NOT NULL | — | FK → `users.id` (one-to-one via UNIQUE) |
| `status` | TEXT | NOT NULL | — | Verification lifecycle state (see CHECK below) |
| `provider_reference` | TEXT | NULL | — | Veriff session ID (set when verification initiated) |
| `verified_at` | TIMESTAMPTZ | NULL | — | Timestamp of `verified` status transition |
| `expires_at` | TIMESTAMPTZ | NULL | — | Document expiry or 2-year re-verification deadline |
| `failure_count` | INT | NOT NULL | `0` | Count of failed verification attempts |
| `created_at` | TIMESTAMPTZ | NOT NULL | `NOW()` | Row creation timestamp |
| `updated_at` | TIMESTAMPTZ | NOT NULL | `NOW()` | Last modification timestamp |

**CHECK constraints:**

- `CHECK (status IN ('not_verified', 'pending', 'pending_resubmission', 'in_manual_review', 'verified', 'expired', 're_verification_required', 'rejected', 'locked'))`
- `CHECK (failure_count >= 0)`

- `idx_kyc_verifications_user_id` on `user_id`

**UNIQUE constraints:**

- `UNIQUE (user_id)` — one KYC record per user

**Foreign keys:**

- `user_id` → `users(id)` ON DELETE CASCADE

> **KYC cascade-delete rationale**: The `ON DELETE CASCADE` is safe because users are NEVER
> physically deleted from this database.
> GDPR erasure nulls PII fields (`email`, `display_name`, `avatar_url`, `bio`) but the `users`
> row remains with `account_status = 'deleted'`.
> Therefore, the KYC record will never be cascade-deleted in normal operation; the AML/CTF
> retention obligation is satisfied.
> Physical row deletion of a `users` row must never occur — the application enforces this.

**Trigger:**

```sql

  BEFORE UPDATE ON kyc_verifications
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
```

**Migration structure:**

```sql
-- migrate:up
BEGIN;
CREATE TABLE IF NOT EXISTS kyc_verifications ( ... );
CREATE UNIQUE INDEX IF NOT EXISTS idx_kyc_verifications_user_id ON kyc_verifications (user_id);

  BEFORE UPDATE ON kyc_verifications
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
COMMIT;

-- migrate:down
```

### Migration 7: `20260306000007_create_escrow_ledger.sql`

#### Table: `escrow_ledger`

| Column | Type | Nullable | Default | Description |
|--------|------|----------|---------|-------------|
| `id` | UUID | NOT NULL | — | PK (application-generated) |
| `campaign_id` | UUID | NOT NULL | — | FK → `campaigns.id` |
| `entry_type` | TEXT | NOT NULL | — | Ledger entry type (see CHECK below) |
| `amount_cents` | BIGINT | NOT NULL | — | Entry amount in integer cents (always positive) |
| `contribution_id` | UUID | NULL | — | FK → `contributions.id` (NULL for non-contribution entries) |
| `reference` | TEXT | NULL | — | Gateway transfer ID or external reference |
| `created_at` | TIMESTAMPTZ | NOT NULL | `NOW()` | Row creation timestamp |

> `escrow_ledger` has NO `updated_at` column and NO trigger — it is append-only and immutable.
> Corrections are made by inserting a new offsetting entry, never by updating existing rows.
> This satisfies the audit trail requirement in L3-006.

**Sign convention:**
`amount_cents` is ALWAYS a positive value.
The `entry_type` determines the direction of the balance effect:

- `contribution` and `interest` → CREDIT (increases escrow balance)
- `disbursement` and `refund` → DEBIT (decreases escrow balance)

**Escrow balance calculation:**

```sql
SUM(CASE WHEN entry_type IN ('contribution', 'interest')
         THEN amount_cents
         ELSE -amount_cents
    END)
```

**CHECK constraints:**

- `CHECK (entry_type IN ('contribution', 'disbursement', 'refund', 'interest'))`
- `CHECK (amount_cents > 0)` — all amounts are positive; sign is determined by entry_type

**Indexes:**

- `idx_escrow_ledger_campaign_id` on `campaign_id`
- `idx_escrow_ledger_contribution_id` on `contribution_id` (FK index)

**Foreign keys:**

- `campaign_id` → `campaigns(id)` ON DELETE RESTRICT
- `contribution_id` → `contributions(id)` ON DELETE RESTRICT

**Migration structure:**

```sql
-- migrate:up
BEGIN;
CREATE TABLE IF NOT EXISTS escrow_ledger ( ... );
CREATE INDEX IF NOT EXISTS idx_escrow_ledger_campaign_id ON escrow_ledger (campaign_id);
CREATE INDEX IF NOT EXISTS idx_escrow_ledger_contribution_id ON escrow_ledger (contribution_id);
COMMIT;

-- migrate:down
DROP TABLE IF EXISTS escrow_ledger CASCADE;
```

### UUID Generation

All `id` columns are defined as `UUID PRIMARY KEY` with NO `DEFAULT gen_random_uuid()`.
UUIDs are generated by the application domain layer before INSERT.
This is intentional: domain entities know their own ID before persistence, enabling the hexagonal
architecture pattern where entities are created independently of the database.

Physical deletion of `users` rows must NEVER occur; GDPR erasure nulls PII fields only. The cascade is a last-resort safety net, not an intended path | Operational |
| 40 | `dbmate up` then `dbmate down` (×7) then `dbmate up` — final schema must match initial | All migrations have symmetric up/down sections; `CREATE TABLE IF NOT EXISTS` + `DROP TABLE IF EXISTS CASCADE` are reversible — schema after second `up` is structurally identical | Integration |
