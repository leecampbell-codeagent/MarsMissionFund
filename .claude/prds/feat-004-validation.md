## Warnings (Should Fix)

### W-001: Component file paths and names inconsistent between UI spec and design spec

**Documents affected:** `feat-004-spec-ui.md` and `feat-004-design.md`

The UI spec (`feat-004-spec-ui.md`) and design spec (`feat-004-design.md`) disagree on:

1. **Component names:**
   - UI spec: `CampaignCard`, `CampaignStatusBadge`
   - Design spec: `PublicCampaignCard`, `PublicCampaignStatusBadge`
   - `FundingProgressBar` is the same in both — no conflict.

2. **Page file paths:**
   - UI spec: `packages/frontend/src/pages/CampaignDiscoveryPage.tsx`, `packages/frontend/src/pages/CampaignDetailPage.tsx`
   - Design spec: `packages/frontend/src/pages/campaign/CampaignDiscoveryPage.tsx`, `packages/frontend/src/pages/campaign/CampaignDetailPage.tsx`

3. **Component file paths:**
   - UI spec: `packages/frontend/src/components/CampaignCard.tsx`, `packages/frontend/src/components/FundingProgressBar.tsx`, `packages/frontend/src/components/CampaignStatusBadge.tsx`
   - Design spec: `packages/frontend/src/components/campaign/public-campaign-card/PublicCampaignCard.tsx`, etc.

4. **Test file names:** The test files in `feat-004-spec-ui.md` use the UI-spec component names (`CampaignCard.test.tsx`) and flat paths, not the design-spec names and paths.

The design spec file paths and names match the existing codebase convention (confirmed: `packages/frontend/src/components/campaign/campaign-card/CampaignCard.tsx` already exists using the subdirectory pattern). **The design spec is authoritative.** The UI spec file paths and component names should be updated to match the design spec.

**Impact:** An implementor following the UI spec alone would create files at the wrong paths with the wrong names. The `PublicCampaignCard` vs `CampaignCard` naming distinction also matters — an existing `CampaignCard` component already exists in the codebase for authenticated creator views; the new public component must have a distinct name.

### W-002: `limit` minimum value wording contradicts Zod schema

**Documents affected:** `feat-004-spec.md` (AC line 114) and `feat-004-spec-api.md` (Zod schema line 227, query param table line 354)

AC in `feat-004-spec.md` states:
> "Given a request with `limit=-1` When processed Then the response is `400 VALIDATION_ERROR` — limit must be a non-negative integer"

The phrase "non-negative integer" implies `limit=0` is valid (0 is non-negative). However, the Zod schema uses `.min(1)` and the query param table states "Min 1, max 100." `limit=0` would be rejected by the Zod schema but the AC wording implies it would be accepted.

The `min(1)` restriction is the correct implementation choice (returning 0 results per page is nonsensical). The AC wording should be corrected to "limit must be a positive integer (minimum 1)."

### W-003: Route conflict for `/campaigns/:id` not resolved in UI spec

**Documents affected:** `feat-004-design.md` (acknowledges), `feat-004-spec-ui.md` (does not mention)

The existing `routes.tsx` wraps `/campaigns/:id` in `<ProtectedRoute>`. The design spec acknowledges this and provides two migration paths (role-conditional rendering or split components). The UI spec does not acknowledge the existing route or specify which approach to take.

The implementor needs a definitive answer: split into two components (`CampaignDetailPage` for authenticated/creator views and `PublicCampaignDetailPage` for public view) or merge into one component with role-conditional rendering. Given the existing `CampaignDetailPage` component has significant authenticated logic (reviewer actions, rejection feedback, PATCH operations), **splitting is strongly preferable** to avoid a component that handles both public anonymous reads and privileged write operations.

This decision should be made explicit in the spec to avoid implementation ambiguity.

---

### W-004: `fundingPercentage` future computation should use BigInt (documentation gap)

**Documents affected:** `feat-004-spec-api.md` (field notes for `fundingPercentage`)

The spec notes that `fundingPercentage = (totalRaisedCents / fundingGoalCents) * 100`. In feat-004 this is always 0.00 and safe. For feat-005, when real BIGINT values are in play, dividing two large BIGINT string values via JavaScript `Number` division risks float precision loss on values exceeding 2^53. The research document (Section 1, "Funding Progress Percentage") explicitly recommends `BigInt` arithmetic for this computation.

The spec should add a note to the serializer or application service instructing feat-005 implementors to replace the `Number()`-based percentage computation with `BigInt` arithmetic when wiring real contribution data.
