### 2. Migration File — PASS

**File:** `/workspace/db/migrations/20260305000009_create_processed_webhook_events.sql`

The migration file exists and is correctly named in dbmate format (`YYYYMMDDHHMMSS_description.sql`). Contents match the spec exactly:

```sql
-- migrate:up
BEGIN;

CREATE TABLE processed_webhook_events (
  id            UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  event_id      TEXT         NOT NULL UNIQUE,
  event_type    TEXT         NOT NULL,
  processed_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_processed_webhook_events_event_id ON processed_webhook_events (event_id);

COMMIT;

-- migrate:down
BEGIN;

DROP TABLE IF EXISTS processed_webhook_events;

COMMIT;
```

The migration follows all infra rules: `TIMESTAMPTZ` for date columns, `TEXT NOT NULL UNIQUE` for `event_id` (preventing duplicate event processing at the DB level), no `updated_at` (append-only table, no update trigger needed per spec note), both `-- migrate:up` and `-- migrate:down` sections present, wrapped in `BEGIN;...COMMIT;`.

**CI impact:** The current `ci.yml` does NOT run `dbmate up` — there is no migration step in the workflow. This is consistent with the existing CI design (tests use in-memory adapters, not a live database). If CI ever adds a database integration step, `dbmate up` will automatically pick up this migration since it is tracked in the `schema_migrations` table by dbmate's standard mechanism.

**No CI change required** for the migration to be tracked correctly when deployed.

### 3. New Secrets Requirements — PASS (no new secrets needed)

**Analysis of env var usage:**

The `.env.example` documents the following payment-related variables added by feat-009:

```bash
# Mock adapters (all mocked by default for local dev)
MOCK_PAYMENTS=true

# Payment Gateway (Stripe) - only needed when MOCK_PAYMENTS=false
STRIPE_SECRET_KEY=your-stripe-secret-key
STRIPE_WEBHOOK_SECRET=whsec_placeholder
STRIPE_PUBLISHABLE_KEY=pk_test_placeholder
VITE_STRIPE_PUBLISHABLE_KEY=pk_test_placeholder
```

The backend application initialises the payment gateway adapter based on the `MOCK_PAYMENTS` env var. When `MOCK_PAYMENTS=true` (the default), the `MockPaymentGatewayAdapter` is used and no Stripe credentials are read at runtime. The `StripePaymentGatewayAdapter` (which requires `STRIPE_SECRET_KEY` and `STRIPE_WEBHOOK_SECRET`) is only instantiated when `MOCK_PAYMENTS=false`.

CI runs with no injected payment env vars, which means `MOCK_PAYMENTS` is either `undefined` or read as falsy. This could be a latent risk if the app startup code reads `MOCK_PAYMENTS` and defaults to the real Stripe adapter when the variable is absent.

**Recommendation (not blocking):** Verify that the app bootstrap code (`packages/backend/src/app.ts` or the container wiring file) defaults to `MockPaymentGatewayAdapter` when `MOCK_PAYMENTS` is absent or `'true'`, rather than requiring the variable to be explicitly set. If it defaults to mock mode, CI is safe. If it defaults to real mode, CI would fail at startup attempting to instantiate `StripePaymentGatewayAdapter` with empty credentials.

Based on the test evidence (all 127 backend tests pass in CI-equivalent conditions with no env vars set), the mock adapter is the effective default. No new GitHub Actions secrets are required for feat-009.

**STRIPE_* secrets:** Only required in CI/CD if a future pipeline adds real-Stripe integration tests (`MOCK_PAYMENTS=false`). When that pipeline step is added, the following GitHub Actions secrets would be needed:
- `STRIPE_SECRET_KEY` — test-mode Stripe secret key
- `STRIPE_WEBHOOK_SECRET` — webhook signing secret for test mode

Neither is needed today.
