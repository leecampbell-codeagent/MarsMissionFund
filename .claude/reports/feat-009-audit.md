## Warnings (Should Fix)

### WARN-001: Duplicate Submission Idempotency Not Implemented (Edge Case #12)

**Spec:** Edge case #12 — "Same contribution submitted twice: Second request should return existing captured contribution with `200` — no second gateway call."
**Current behaviour:** Each call to `POST /api/v1/payments/capture` creates a new `Contribution` entity and charges the gateway. There is no deduplication by `payment_method_token`.
**Note:** The `ContributionRepository` port has no `findByPaymentMethodToken` method, so this may be an acknowledged scope limitation rather than a bug. Clarify with the spec owner whether feat-009 is expected to implement full token-level deduplication or defer to feat-008.

### WARN-002: Fake TransactionClient in Failed-Capture Event Writes

**File:** `/workspace/packages/backend/src/payments/application/payment-app-service.ts:126-143`
**Issue:** Failed-capture event writes use `{ _brand: 'TransactionClient' } as Parameters<...>` — a fake client object. If the `PgEventStoreAdapter` dereferences `_pgClient`, this will throw at runtime and PAYMENT.CONTRIBUTION_FAILED events will not be recorded.
**Fix:** Update `EventStorePort` to support optional/null transaction client, or use a no-op transaction for event writes on failure path.

### WARN-003: Manual Tasks from feat-009 Spec Not Documented
