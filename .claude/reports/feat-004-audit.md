### FAIL-003 RESOLVED: Shared database transaction via `PgTransactionAdapter`

**Location**: `/workspace/packages/backend/src/shared/adapters/pg/pg-event-store-adapter.ts` lines 44-65

`PgTransactionAdapter.withTransaction()` correctly:
1. Acquires a `PoolClient` from the pool
2. Issues `BEGIN`
3. Wraps the client as a branded `PgTransactionClient` (keeping `pg` out of domain/application code)
4. Calls `fn(txClient)` — all repo and event store operations within share the same client
5. Issues `COMMIT` on success, `ROLLBACK` on error
6. Releases the client in a `finally` block (prevents connection leaks)

Mock adapters (`InMemoryAccountRepository`, `InMemoryEventStore`) accept the optional `txClient` parameter but ignore it, so tests run without a real database.

### WARN-004: Inline `<style>` tags in React components

Both `settings-profile.tsx` and `settings-preferences.tsx` — and several onboarding components — use inline `<style>` tags with raw CSS strings. This is functional but atypical for a production React app and can cause style re-injection on every render. Prefer CSS modules or a global stylesheet.

### WARN-005: wizardRoleToAccountRoles treats 'creator' and 'both' identically

**Location**: `/workspace/packages/frontend/src/pages/onboarding.tsx` lines 55-59

```typescript
function wizardRoleToAccountRoles(role: WizardRole): readonly AccountRole[] {
  if (role === 'creator') return ['backer', 'creator'];
  if (role === 'both') return ['backer', 'creator'];
  return ['backer'];
}
```

Both `creator` and `both` map to `['backer', 'creator']`, which is correct per the spec. However, the function has an unreachable dead branch for `both` (the `creator` branch returns first, making `both` unreachable). This should be simplified to `if (role !== 'backer') return ['backer', 'creator']` or a switch statement.

### WARN-006: No test for 404 ACCOUNT_NOT_FOUND on PATCH /api/v1/accounts/me

The spec lists `404 ACCOUNT_NOT_FOUND` as an error response for `PATCH /api/v1/accounts/me` and `PATCH /api/v1/accounts/me/preferences`. No tests cover this path. The scenario is admittedly difficult to reach (JIT creation ensures accounts exist), but the spec lists it and defensive tests would improve coverage.
