### 5. Financial Data Compliance: ✅ PASS

Verified in both the serializer and adapter layers:

- **`totalRaisedCents`**: hardcoded as string literal `'0'` (not number `0`) in both `searchPublicCampaigns` (adapter line 383: `totalRaisedCents: '0' as string`) and `findPublicById` (adapter line 457: `totalRaisedCents: '0'`), and in `getCategoryStats` (adapter line 487: `totalRaisedCents: '0'`).
- **`fundingGoalCents`**: cast to `::TEXT` in SQL (`c.funding_goal_cents::TEXT AS funding_goal_cents`) — BIGINT returned as string by `pg` driver, then passed through without conversion.
- **`fundingCapCents`**: cast to `::TEXT` in the detail query (`c.funding_cap_cents::TEXT AS funding_cap_cents`).
- **`estimatedCents`** in budget breakdown: typed as `string` in `BudgetItem` interface; `budgetBreakdown` is JSON from PostgreSQL and is passed through without numeric conversion.
- **`serializeCategoryStats`**: passes `stats.totalRaisedCents` directly (already a string from the domain type).
- **No float arithmetic**: `fundingPercentage` is computed as `0` (integer) when `fundingGoalCents !== null`, and `null` otherwise — no division performed in feat-004 (stub).
- **Frontend `PublicCampaignListItem` type**: `fundingGoalCents: string | null`, `totalRaisedCents: string`, `fundingCapCents: string | null` — all annotated as strings with comment "never parse to Number (G-024)".

### 7. Security Status: ✅ PASS

Security review verdict: 0 Critical / 0 High / 2 Medium / 3 Low. Both pre-merge gates (0 critical, 0 high) are satisfied.

**Medium findings acknowledged (non-blocking):**

- **M-001**: `sort=most_funded`/`sort=least_funded` sorts by `funding_goal_cents` (goal) rather than amount raised. This is the correct stub behaviour for feat-004 where `totalRaisedCents` is always `'0'`. Documented as intentional in spec Key Decisions. Will require a column change to a contributions aggregate in feat-005.
- **M-002**: `GET /:id` path parameter has no UUID format validation before the DB call. PostgreSQL silently returns zero rows for non-UUID strings, resulting in correct 404 responses. Confirmed working by Playwright test (`/api/v1/public/campaigns/not-a-valid-uuid` returns 404). Recommend adding `z.string().uuid()` check in feat-005 as the public endpoint scales.

**Low findings acknowledged (informational):**

- **L-001**: No `helmet()` or rate limiting on public endpoints — systemic gap across entire backend, not feat-004-specific.
- **L-002**: `getCategoryStats` scans all campaigns in a category before filtering by status — performance note only, no data leakage.
- **L-003**: `searchPublicCampaigns` in `public-campaign-api.ts` duplicates the fetch-and-error-handle pattern instead of using the `publicFetch` helper. Minor code quality issue with no security impact.

## Warnings (Should Fix)

1. **W-001 (Low): `patterns.md` not updated with feat-004 patterns** — Three new patterns introduced by feat-004 (public router without requireAuth, FTS empty-string guard, URL-state search params) are not documented as `P-026+` entries in `/workspace/.claude/context/patterns.md`. Recommend adding in a follow-up task or as part of feat-005 kickoff.

2. **W-002 (Carry-forward from Security M-001): Sort-by-goal vs sort-by-raised** — `most_funded`/`least_funded` sort by `funding_goal_cents` (target) not `totalRaisedCents` (raised). Correct stub behaviour for feat-004 but must be updated in feat-005 when the contributions table lands.

3. **W-003 (Carry-forward from Security M-002): No UUID format validation on `/:id`** — The path parameter `req.params.id` is not Zod-validated before the database call. Recommend adding `z.string().uuid()` safeParse in feat-005 to eliminate unnecessary DB round-trips from scanner traffic on the public endpoint.

4. **W-004 (Carry-forward from Security L-003): `searchPublicCampaigns` fetch logic duplicated** — The fetch/error-handling pattern in `public-campaign-api.ts` for `searchPublicCampaigns` is inlined rather than using the `publicFetch` helper. This creates a subtle inconsistency: the inline path does not check `Content-Type` of the response. Refactor in feat-005 or as a standalone cleanup task.

## Changelog Entry

### feat-004: Campaign Discovery and Public Campaign Pages

- Added public campaign search API at `/api/v1/public/campaigns` (no authentication required)
- Full-text search via PostgreSQL `tsvector`/`tsquery` with GIN index on `campaigns.search_vector`
- `websearch_to_tsquery` with empty-string guard (G-037) for safe arbitrary user input
- Filter by category (multi-value via `= ANY($1::TEXT[])`, G-039), status (`active`/`funded`/`ending_soon`), sort (`newest`/`ending_soon`/`most_funded`/`least_funded`)
- "Ending Soon" filter returns campaigns with deadline within 7 days of server time
- Pagination: `limit` (1–100, default 20) and `offset` (≥0, default 0)
- Campaign detail endpoint at `GET /api/v1/public/campaigns/:id` with complete campaign data (milestones, teamMembers, riskDisclosures, budgetBreakdown, alignmentStatement, tags, fundingCapCents)
- Category aggregate stats endpoint at `GET /api/v1/public/campaigns/stats?category=X`
- Only `live` and `funded` campaigns publicly accessible; non-public/non-existent campaigns return `404 NOT_FOUND` (no existence disclosure)
- Creator name joined at query time from `users.display_name` (null fallback; G-038)
- Separate public router mounted without `requireAuth` at `/api/v1/public/campaigns` in `app.ts` (G-036)
- Route registration order: `/stats` before `/:id` (G-023)
- Frontend: `CampaignDiscoveryPage` at `/campaigns` (no `ProtectedRoute`)
- Frontend: `PublicCampaignDetailPage` at `/campaigns/:id` (no `ProtectedRoute`)
- New components: `PublicCampaignCard`, `FundingProgressBar`, `PublicStatusBadge`, `CampaignSearchBar`, `CategoryFilter`, `CategoryStatsBar`
- URL-state-backed search/filter parameters via `useSearchParams` with 300ms debounce on search input
- All monetary fields serialised as strings: `fundingGoalCents`, `fundingCapCents`, `totalRaisedCents`, `estimatedCents` (G-024)
- Migration: `db/migrations/20260305160000_campaign_search_vector.sql` — adds `search_vector TSVECTOR` column with GIN index and auto-update trigger; partial indexes on `deadline`, `launched_at`, `category`
- Funding progress stubbed (`totalRaisedCents: '0'`, `contributorCount: 0`, `fundingPercentage: 0 | null`) pending feat-005 contribution flow
- Sort by `most_funded`/`least_funded` currently orders by `funding_goal_cents` (stub); must be updated to contributions aggregate in feat-005 (Security M-001)
