# Security Review: feat-013 — KYC Verification (Mock Provider)

> Security review results. Generated by Security Reviewer.

## Verdict: 0 CRITICAL / 0 HIGH / 1 MEDIUM / 2 LOW

## Summary

The feat-013 KYC implementation demonstrates a sound security posture. All three endpoints require authentication via the `requireAuth` + `enrichContext` middleware chain applied at the router level in `app.ts`. SQL injection risk is absent — all queries use parameterised placeholders. The admin unlock endpoint enforces a role check both at the controller layer and redundantly in the application service. No PII is leaked in error messages, and document references (not file content) are stored in the database per spec requirements.

One medium finding exists: the `front_document_ref` and `back_document_ref` fields accept arbitrary string values with only a `min(1)` length check — there is no format validation or max-length constraint on these reference strings. Two low findings relate to defence-in-depth improvements.

### Medium (Should Fix)

#### MED-001: Document Reference Strings Lack Max-Length and Format Constraints

- **Location:** `packages/backend/src/kyc/api/kyc-router.ts:19-20`
- **Issue:** `front_document_ref` and `back_document_ref` accept any string with `z.string().min(1)`. There is no maximum length constraint (no `.max(N)`) and no format validation (e.g., URL-safe path, no `..`, no null bytes). A malicious user could submit a several-megabyte string or path-traversal tokens that would be stored in the database.
- **Recommendation:** Add `.max(512)` (or an appropriate limit per your storage scheme) and optionally a regex that matches your document ref format (e.g., `mock://...` or a UUID-based path).

### Low / Informational

#### LOW-001: Admin Role Check is Duplicated Across Router and Application Service

- **Location:** `packages/backend/src/kyc/api/kyc-router.ts:142-153` and `packages/backend/src/kyc/application/kyc-app-service.ts:146-149`
- **Note:** The admin role check exists in both the router (using `req.authContext.roles`) and the application service (using a DB lookup). This duplication is defence-in-depth and acceptable, but the router-layer check uses the `authContext.roles` array which is populated from the Clerk JWT claims. Ensure that this roles claim is always populated by `enrichContext` before the route handler executes — which it is (verified in `app.ts:77`). No action required; noted for awareness.

#### LOW-002: Mock KYC Adapter Sessions Are In-Memory and Not Scoped to Request Lifetime

- **Location:** `packages/backend/src/kyc/adapters/mock/mock-kyc-adapter.ts:8`
- **Note:** The `MockKycAdapter` stores sessions in an in-process `Map`. In production, `MOCK_KYC=true` should never be set (enforced by env var docs). Ensure deployment runbooks make this explicit. No code change required; document clearly that `MOCK_KYC=false` in any real deployment.

### Auth & Authz Detail

- All 3 endpoints protected via `requireAuth` + `enrichContext` applied at `app.ts:77` before the KYC router.
- `user_id` is always sourced from `req.authContext.userId` (set by `enrichContext` from the verified Clerk JWT) — never from request body or URL params.
- Admin unlock endpoint: `accountId` target comes from `req.params.accountId` (URL param) — correct. The requester's identity comes from `req.authContext.userId` — correct.
- Role check verified at both controller (`req.authContext.roles`) and application service (DB lookup of roles).
- 401 returned for missing/invalid auth; 403 for insufficient role.

### SQL Injection Detail

All queries in `pg-kyc-repository.ts` use parameterised placeholders:
- `SELECT * FROM kyc_verifications WHERE account_id = $1` — parameterised.
- `SELECT * FROM kyc_verifications WHERE id = $1` — parameterised.
- `INSERT INTO kyc_verifications (...) VALUES ($1, $2, ..., $13)` — parameterised.
- `UPDATE kyc_verifications SET ... WHERE id = $1` — parameterised.

No template literal SQL interpolation detected.
