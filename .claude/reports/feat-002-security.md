# Security Review: feat-002 — Core Database Schema

> Security review results. Generated by Security Reviewer.

## Verdict: CONDITIONAL PASS — 0 CRITICAL / 0 HIGH / 3 MEDIUM / 4 LOW

## Summary

feat-002 introduces seven SQL migration files that constitute the foundational PostgreSQL schema for Mars Mission Fund. The schema is well-constructed: all monetary columns use BIGINT (integer cents), all status/enum columns have CHECK constraints, foreign key relationships enforce data isolation, and the escrow ledger is correctly designed as an append-only structure with no `updated_at` column. Three medium findings require attention before the schema can be considered fully hardened — the most significant is the absence of a database-level UPDATE prevention mechanism on `escrow_ledger`, which relies entirely on application-layer enforcement for immutability. No critical or high severity findings were identified.

### Medium (Should Fix)

#### MED-001: Escrow Ledger Immutability Relies Solely on Application-Layer Enforcement

- **Location:** `db/migrations/20260306000007_create_escrow_ledger.sql`
- **Issue:** The `escrow_ledger` table has no `updated_at` column and no trigger, which is correct design. However, there is no database-level mechanism (e.g., a `BEFORE UPDATE` trigger that raises an exception) to prevent UPDATE or DELETE operations on ledger rows. The spec acknowledges this explicitly in edge case 27: "UPDATE succeeds at DB level — append-only semantics are enforced by application code, not the schema." Any application bug, direct DB access by an operator, or future repository code that issues an UPDATE will silently mutate financial audit records.
- **Recommendation:** Add a `BEFORE UPDATE OR DELETE` trigger on `escrow_ledger` that raises an exception unconditionally. This makes immutability a database invariant rather than a convention:
  ```sql
  CREATE OR REPLACE FUNCTION prevent_escrow_ledger_mutation()
  RETURNS TRIGGER AS $$
  BEGIN
    RAISE EXCEPTION 'escrow_ledger rows are immutable — insert a correcting entry instead';
  END;
  $$ LANGUAGE plpgsql;

  CREATE TRIGGER escrow_ledger_immutable
    BEFORE UPDATE OR DELETE ON escrow_ledger
    FOR EACH ROW EXECUTE FUNCTION prevent_escrow_ledger_mutation();
  ```

#### MED-002: `users.email` Has No Length Constraint — DoS via Oversized Input

- **Location:** `db/migrations/20260306000001_create_accounts.sql:4`
- **Issue:** The `email` column is defined as `TEXT NOT NULL` with no length limit. PostgreSQL `TEXT` has no practical upper bound. While application-layer Zod validation will enforce a maximum length, the database itself will accept arbitrarily large strings if accessed directly or if validation is bypassed. An attacker with direct DB access or exploiting a gap in validation could store multi-megabyte strings in an indexed column, degrading index performance. Similarly, `display_name`, `avatar_url`, `bio`, `title`, `summary`, `description`, `gateway_reference`, `reference`, `provider_reference`, and `verification_criteria` columns have no length constraints.
- **Recommendation:** Add `VARCHAR(n)` or `CHECK (char_length(email) <= 254)` constraints on PII and user-controlled text fields. Email: 254 characters (RFC 5321). Display name: 100 characters. Bio: 2000 characters. This is defence-in-depth alongside application-layer validation. For fields like `description` where rich text is expected, a generous but bounded limit (e.g., 50,000 characters) prevents abuse.

#### MED-003: `campaigns.amount_raised_cents` Is a Denormalised, Mutable Running Total Without Concurrency Guard in Schema

- **Location:** `db/migrations/20260306000004_create_campaigns.sql:12`
- **Issue:** `amount_raised_cents` is a denormalised column that accumulates contributions via additive UPDATE statements. The spec notes (edge case 34) that this is safe provided the application uses `UPDATE campaigns SET amount_raised_cents = amount_raised_cents + $1` (additive) rather than read-modify-write. However, the schema has no CHECK constraint or trigger to detect if the value ever exceeds `funding_cap_cents`, which is a business invariant. A bug that sets `amount_raised_cents` higher than `funding_cap_cents` (e.g., from a race condition or application error) would result in an over-funded campaign with no database-level rejection.
- **Recommendation:** Add a CHECK constraint:
  ```sql
  CONSTRAINT campaigns_amount_raised_lte_cap_check CHECK (amount_raised_cents <= funding_cap_cents)
  ```
  This enforces the business rule at the database layer and prevents the denormalised total from ever exceeding the campaign cap, regardless of application code correctness.

### Low / Informational

#### LOW-001: `user_roles.assigned_by` Nullable FK — No Audit Trail for Bootstrap Roles

- **Note:** `assigned_by` is intentionally nullable for system/bootstrap role assignments. However, there is no `assigned_at` alias or metadata distinguishing bootstrap assignments from operator-assigned roles where `assigned_by` was incorrectly omitted. This is acceptable per the spec, but when building the application service, ensure that non-bootstrap role assignments always supply a non-null `assigned_by`. Consider a future migration adding a `source` or `assignment_reason` column to distinguish system vs. operator vs. user-initiated assignments for audit purposes.

#### LOW-002: No `BEGIN`/`COMMIT` Transaction Wrapper in Migration 1 and Migration 2

- **Note:** Migrations 3–7 and the spec examples show `BEGIN; ... COMMIT;` wrapping. Inspecting the actual migration files, migration 1 (`create_accounts.sql`) and migration 2 (`create_roles.sql`) do not have explicit `BEGIN`/`COMMIT` wrappers. dbmate wraps each migration in a transaction by default, so this is not a functional defect, but it is an inconsistency with the spec's documented structure and with the other migrations. For clarity and to make the transactional intent explicit (matching the spec and other files), add `BEGIN;`/`COMMIT;` to both files.

#### LOW-003: `kyc_verifications` CASCADE Delete Relies on Application Covenant — No DB Enforcement

- **Note:** The spec documents that `ON DELETE CASCADE` on `kyc_verifications.user_id` is safe because users are never physically deleted (GDPR erasure nulls PII fields only). This is correct reasoning, but the safety guarantee is a convention enforced entirely by application code, with no database-level guard. If a future developer issues `DELETE FROM users WHERE id = $1` directly, KYC records will be silently cascade-deleted, violating AML/CTF retention obligations. Consider adding a `BEFORE DELETE` trigger on `users` that raises an exception, enforcing the "soft delete only" invariant at the database layer. This is particularly important for a regulated financial application.

#### LOW-004: No Index on `contributions.gateway_reference` or `escrow_ledger.reference`

- **Note:** `gateway_reference` in `contributions` (the Stripe PaymentIntent ID) and `reference` in `escrow_ledger` (the gateway transfer ID) will likely be used in webhook processing and reconciliation queries to look up rows by external identifier. Without indexes on these columns, webhook handlers processing Stripe events will perform full table scans as the dataset grows. These are not immediate security risks but represent a future DoS/performance vector. Add indexes when the columns begin to be queried in application code.
