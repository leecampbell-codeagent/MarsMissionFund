## Blocking Issues

### BLOCK-001: Missing `npm run build` step in CI workflow

**File:** `/workspace/.github/workflows/ci.yml`

The workflow runs `npm run lint`, `npm run format`, `npm run typecheck`, and `npm test`, but it
does **not** run `npm run build`. A broken build (e.g. import errors, missing exports, Vite config
errors) would not be caught before merge. The build currently passes locally, but the gate is absent.

**Fix:** Add a build step to the workflow after typecheck:

```yaml
      - name: Build
        run: npm run build
```

### BLOCK-002: Pre-existing `npm run lint` failures will fail CI

**307 lint errors and 107 warnings** exist across the codebase, all in pre-existing files from
feat-001/feat-002/feat-003 (account adapters, shared schemas). These are NOT introduced by feat-004 —
confirmed by diffing the set of files with lint errors against the set of files changed in the
feat-004 commit (zero overlap).

However, since CI runs `npm run lint` and it returns a non-zero exit code, the CI pipeline will
**fail on this step** and never reach the test or build steps.

Affected pre-existing files include:
- `packages/backend/src/account/adapters/*.ts`
- `packages/backend/src/account/domain/**/*.ts`
- `packages/shared/src/**/*.ts`
- `biome.json`, `package.json`, `tsconfig.json` files (formatting violations)

**Fix:** These pre-existing violations must be resolved (or the CI lint step introduced with a
baseline exclusion) before feat-004 can pass CI. This is a pre-existing debt that blocks the
entire pipeline, not a feat-004 regression.

### BLOCK-003: Pre-existing `npm run format` failures will fail CI

**256 format errors** in pre-existing files. Same root cause as BLOCK-002. The format check step
runs `npm run format` which calls `biome check --formatter-enabled=true` and fails.

None of the format-failing files are feat-004 files (confirmed by cross-referencing the diff).

## Warnings

### WARN-001: No `dbmate` migration step in CI

The CI workflow has no step to run `dbmate up`. Tests pass because they use in-memory/mock
adapters that do not require a real database. However, a full integration/staging deploy pipeline
would need a `dbmate up` step before starting the application to apply
`20260305160000_campaign_search_vector.sql`.

This is not a blocker for unit/integration tests as configured, but would be a blocker for any
future end-to-end or database-backed test pipeline.

**Recommended addition to a deploy or e2e job:**

```yaml
      - name: Run database migrations
        run: docker run --rm --network host \
          -e DATABASE_URL=${{ secrets.DATABASE_URL }} \
          amacneil/dbmate up
```

### WARN-002: No `npm run build` gate means dead-code imports could reach main undetected

Related to BLOCK-001. TypeScript `noEmit` typecheck catches type errors but will not catch
Vite-specific build failures (e.g., dynamic imports, CSS module issues, asset resolution).
The build currently succeeds, but the gate should be enforced in CI.
