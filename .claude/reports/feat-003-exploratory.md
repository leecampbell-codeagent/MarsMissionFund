## Remaining Known Issue (Non-Merge-Blocking)

### LOW — `linkedInUrl` silently stripped by backend Zod schema

**Where:** PATCH `/api/v1/campaigns/:id` — team member array items

**Behaviour:** The frontend `TeamMember` type has `readonly linkedInUrl: string | null` and `TeamSection` collects and sends this field. The backend `teamMemberSchema` does NOT include `linkedInUrl`. Because `teamMemberSchema` is a plain `z.object()` (not `.strict()`), Zod strips `linkedInUrl` from each team member object before it reaches the application layer. No `400` error is returned — the field is silently dropped.

**Impact:** Any LinkedIn URL entered by the user appears to save but is not persisted in the database. The field is blank when the campaign is reloaded.

**Why not blocking:** The `feat-003-spec-data.md` TeamMember data model does not include `linkedInUrl` as a required field. The feature spec is silent on LinkedIn URLs. The frontend implemented it as a UI convenience field. No acceptance criterion in the spec requires LinkedIn URL persistence. Data loss is scoped to an optional field not in the spec.

**Recommended future fix:** Either add `linkedInUrl: z.string().url().nullable().optional()` to `teamMemberSchema` and the backend `TeamMember` domain interface, or remove the LinkedIn URL field from the frontend form entirely.
