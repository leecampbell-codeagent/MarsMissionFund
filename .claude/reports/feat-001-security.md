### Low / Informational

#### LOW-001: No CORS configuration or security headers yet

- **Location:** `packages/backend/src/server.ts`
- **Note:** No CORS policy or HTTP security headers (X-Content-Type-Options, X-Frame-Options,
  Strict-Transport-Security) are configured on the Express app. This is expected for a scaffold
  feature with no real API endpoints, but these controls must be added before any authenticated
  or data-serving endpoint goes live. Recommend adding `helmet` and a strict CORS origin allowlist
  in the feat-003 Clerk integration feature at the latest.

#### LOW-002: No rate limiting on the health endpoint

- **Location:** `packages/backend/src/health/api/health-router.ts`
- **Note:** The `/health` endpoint has no rate limiting. For a liveness check this is low risk,
  but it represents an unbounded public endpoint. Rate limiting should be applied at the
  infrastructure/load-balancer layer when the service is deployed. No application-level action
  is required in this feature.

#### LOW-003: VITE_CLERK_PUBLISHABLE_KEY is exposed to the browser — correct, but must stay scoped

- **Location:** `/workspace/.env.example` line 31
- **Note:** `VITE_CLERK_PUBLISHABLE_KEY` is intentionally `VITE_`-prefixed because the Clerk
  publishable key is designed to be browser-visible. All other secrets (`CLERK_SECRET_KEY`,
  `CLERK_WEBHOOK_SIGNING_SECRET`, `DATABASE_URL`) correctly lack the `VITE_` prefix and will
  not be exposed to client bundles. This is correctly implemented; recorded here as a reminder
  that no additional secret keys should ever receive the `VITE_` prefix.

## Specific Checks Performed

### Secrets in code

Grep for hardcoded credentials across all new files: **none found**. All environment variable
references use `process.env.*` with safe fallbacks. The `.env.example` file uses only placeholder
values (non-sensitive placeholder strings; no real credentials).

### Error handler stack trace suppression

`server.ts` line 16–24: The error handler logs the full error internally via Pino
(`logger.error({ err }, 'Unhandled error')`) but returns only a generic message to the client:
`{ error: { code: 'INTERNAL_ERROR', message: 'An unexpected error occurred.' } }`. Stack traces
and internal error details are not exposed in API responses. **Correct.**

### console.log absence

Grep across `packages/backend/src/**` for `console.log/error/warn/debug/info`: **none found**.
All backend logging goes through the Pino logger instance. **Correct.**

### .env in .gitignore

`.gitignore` contains `.env` and `.env.*` with `!.env.example` and `!.env.agent.example`
carve-outs. The actual `.env` file is never committed. **Correct.**

### Vite env var exposure

No server-side secrets are prefixed with `VITE_` in `.env.example`. Only `VITE_API_URL` (the
backend URL, not a secret) and `VITE_CLERK_PUBLISHABLE_KEY` (a browser-safe Clerk key by design)
use the `VITE_` prefix. **Correct.**

### NODE_ENV bracket notation

`server.ts` line 6 uses `process.env.NODE_ENV` (dot notation) rather than the spec's recommended
`process.env['NODE_ENV']` (bracket notation). This is functionally equivalent — not a security
issue, but a minor deviation from the spec's style guidance. No finding raised.
