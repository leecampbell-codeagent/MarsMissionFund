# Security Review: feat-001 — Account Registration and Authentication

> Security review results. Generated by Security Reviewer.

## Verdict: 0 CRITICAL / 3 HIGH / 3 MEDIUM / 4 LOW

## Summary

The authentication foundation for feat-001 is well-structured and follows the hexagonal architecture correctly. Clerk JWT verification is applied consistently, `user_id` is never taken from request bodies, and all SQL queries use parameterised placeholders with no interpolation. Three high-severity findings require attention before merge: Zod validation error details are leaked in API responses, domain error messages expose user-supplied input back to callers, and the `PATCH /api/v1/me/profile` schema accepts `onboardingCompleted` and `onboardingStep` fields that are not in the feat-001 spec and widen the attack surface. No critical findings were identified.

---

## Findings

### High (Must Fix Before Merge)

#### HIGH-001: Zod Validation Error Details Leaked to Callers

- **Location:** `packages/backend/src/account/api/account-router.ts:148` and `:211`
- **Issue:** Both PATCH endpoints (`/me/profile` and `/me/notifications`) include `details: parseResult.error.format()` in the 400 error response body. `ZodError.format()` returns the full internal validation tree including field paths, received values, and internal error codes. This exposes the Zod schema structure and in some cases the values the caller submitted.
- **Impact:** Information disclosure. An attacker probing the API learns exact field names, types, and constraints from error responses. For a financial platform this violates the spec requirement (L3-002 Section 7.1) that error responses do not expose internal details. While not immediately exploitable on its own, it materially assists reconnaissance.
- **Fix:** Remove the `details` field from the 400 response bodies in production. If detailed validation feedback is required for client UX, return only a sanitised array of `{ field, message }` tuples with no raw Zod internals. The generic `"Check your input and try again."` message already in the `message` field is sufficient at the API boundary. Field-level detail is noted as a future improvement in the spec itself.

---

#### HIGH-002: Domain Error Messages Expose User-Supplied Input

- **Location:** `packages/backend/src/account/domain/errors/account-errors.ts:11-13` (`InvalidEmailError`), `:31-34` (`InvalidAvatarUrlError`), `:79-88` (`UserNotFoundError`), `:91-95` (`UserAlreadyExistsError`)
- **Issue:** Several domain error constructors embed user-supplied values or internal identifiers in the error message:
  - `InvalidEmailError`: `"Invalid email address: ${email}"` — echoes the submitted email back.
  - `InvalidAvatarUrlError`: `"Invalid avatar URL: ${url}"` — echoes the submitted URL back.
  - `UserNotFoundError(identifier)`: `"User not found: ${identifier}"` — echoes the Clerk user ID.
  - `UserAlreadyExistsError(clerkUserId)`: `"User already exists with Clerk ID: ${clerkUserId}"`.

  The generic `DomainError` catch-all in `error-handler.ts` at line 81 returns `err.message` directly in the response body: `message: err.message`. This means any unhandled `DomainError` subclass (including future additions) will leak its message to the API caller. Even the specifically handled errors could be affected if their handler order changes.
- **Impact:** Information disclosure. User-supplied values reflected back in error messages create a reflected content path (low XSS risk in JSON APIs, higher risk if messages are ever rendered as HTML). Internal identifiers (Clerk user IDs) in error messages violate the principle of not leaking identifiers. For `UserNotFoundError`, leaking the identifier being searched creates an account enumeration vector.
- **Fix:**
  1. Remove user-supplied data from all domain error messages. Error messages should be static strings. Use the `code` field for machine-readable discrimination.
  2. In `error-handler.ts`, the `DomainError` catch-all (lines 76-85) must not return `err.message` to the caller. Replace with a safe generic message: `"An unexpected error occurred. Please try again."` Log the full `err.message` internally via pino.

#### HIGH-003: `PATCH /api/v1/me/profile` Schema Accepts Undocumented Fields

- **Location:** `packages/backend/src/account/api/account-router.ts:24-25`
- **Issue:** The `updateProfileSchema` includes `onboardingCompleted: z.boolean().optional()` and `onboardingStep: z.enum([...]).optional()` as accepted input fields. These fields are not listed in the feat-001-spec-api.md API contract for `PATCH /api/v1/me/profile`, which specifies only `displayName`, `bio`, and `avatarUrl` as accepted fields. The schema comment `// WARN-003: includes onboardingCompleted + onboardingStep` acknowledges this deviation without resolving it.

  This means any authenticated user can set their own `onboarding_completed = true` and `onboarding_step = 'complete'` by sending these fields in the profile PATCH request, bypassing the intended onboarding flow entirely.
- **Impact:** Business logic bypass. Users can self-complete onboarding without going through the defined steps. Depending on what onboarding gates in future features (e.g., role selection, KYC prerequisites), this could allow feature access that should be blocked. It also corrupts audit trail data if onboarding state transitions are expected to be linear and auditable.
- **Fix:** Remove `onboardingCompleted` and `onboardingStep` from the `updateProfileSchema` and from `UpdateProfileInput` if they are frontend-only internal state fields at this stage. If they are legitimately settable, add them to the spec API contract and add appropriate validation (e.g., only certain transitions are valid, step order is enforced).

### Medium (Should Fix)

#### MED-001: No HTTP Security Headers Configured

- **Location:** `packages/backend/src/app.ts` (no security header middleware)
- **Issue:** The Express app does not configure any HTTP security headers. The security spec (L3-002 Section 7.2) requires the following headers on all responses:
  - `Content-Security-Policy`
  - `Strict-Transport-Security`
  - `X-Content-Type-Options: nosniff`
  - `X-Frame-Options: DENY`
  - `Referrer-Policy: strict-origin-when-cross-origin`
  - `Permissions-Policy`
  - `Cache-Control: no-store` (for authenticated responses)

  No CORS configuration is present either, meaning the browser's default same-origin policy applies. For an API backend this may be acceptable if the frontend is on the same origin, but it should be explicitly configured rather than relying on defaults.
- **Recommendation:** Add `helmet` (or equivalent middleware) to the Express app to set security headers. Configure explicit CORS policy using `cors` middleware, allowing only the expected frontend origin(s) rather than relying on browser defaults. Given this is a demo app, mark as a TODO with a reference to the spec requirement if deferring.

#### MED-002: No Rate Limiting Implemented

- **Location:** `packages/backend/src/app.ts` (no rate limiting middleware)
- **Issue:** The security spec (L3-002 Section 7.3) requires rate limiting at:
  - General API endpoints: 100 requests/minute
  - Auth endpoints (`/auth/sync`): 10 requests/minute
  - Financial action endpoints: 20 requests/minute

  No rate limiting is configured anywhere in the Express application. The `/api/v1/auth/sync` endpoint is particularly exposed — it can be called repeatedly to create database load or probe sync behaviour.
- **Recommendation:** For this demo, document as a TODO per the agent instructions. This is acceptable for a local demo. Add a comment in `app.ts` noting the spec requirement and that rate limiting would be implemented at the API Gateway layer (CloudFront/ALB) in production per L3-002 Section 7.3.

#### MED-003: Webhook Body Fallback Undermines Signature Verification

- **Location:** `packages/backend/src/account/api/webhook-router.ts:62-64`
- **Issue:** The webhook handler includes a fallback when `req.body` is not a Buffer:
  ```typescript
  const rawBody =
    req.body instanceof Buffer ? req.body : Buffer.from(JSON.stringify(req.body));
  ```
  If `express.raw()` is not applied correctly (e.g., due to middleware order change or configuration drift), the body would have already been parsed as JSON and then re-serialised with `JSON.stringify`. Re-serialised JSON is NOT the original raw bytes — it will have different whitespace, key ordering, and encoding characteristics, causing signature verification to fail silently (i.e., the `wh.verify()` call would throw and return 400). However, if any body-parsing middleware intercepts the request before `express.raw()` runs and happens to produce a Buffer-like object, the fallback could be used with tampered data.

  The app.ts correctly registers the webhook route BEFORE `express.json()`, but the fallback suggests this invariant may not be relied upon. The fallback should not exist — if the body is not a Buffer, that is a configuration error that should fail loudly, not silently re-serialise.
- **Recommendation:** Remove the fallback. Replace with:
  ```typescript
  if (!(req.body instanceof Buffer)) {
    logger.error({ correlationId }, 'Webhook body is not a Buffer — express.raw() middleware not applied');
    res.status(500).json({ error: { code: 'INTERNAL_ERROR', message: "Something went wrong on our end. We're looking into it." } });
    return;
  }
  const rawBody = req.body;
  ```
  This makes middleware misconfiguration visible immediately rather than producing a subtle signature mismatch.

#### LOW-002: `securityAlerts` Typed as `boolean` in Frontend Interface

- **Location:** `packages/frontend/src/api/account-api.ts:13`
- **Issue:** The `NotificationPrefs` interface declares `securityAlerts: boolean` rather than the literal `true`. This means TypeScript allows the frontend to represent `securityAlerts: false` in local state, even though the server will always return `true` and rejects attempts to set it to `false`. The backend domain model correctly types it as literal `true`.
- **Recommendation:** Update the frontend `NotificationPrefs` interface to type `securityAlerts` as `true` (literal type) to match the backend domain model and make the constraint visible at compile time.

---

#### LOW-003: `clerkUserId` Logged in `handleClerkWebhook` Warning

- **Location:** `packages/backend/src/account/application/account-app-service.ts:285-288`
- **Issue:** The `user.updated` out-of-order delivery warning logs `{ clerkUserId }`:
  ```typescript
  this.logger.warn({ clerkUserId }, 'user.updated received before user.created — creating user');
  ```
  Clerk user IDs (`user_KSUID`) are internal identifiers and generally safe to log. However, the spec (L1-002 Section 1.7) states that "log entries use resource identifiers, not raw personal data." Clerk user IDs are not PII, so this is informational. Email addresses are correctly NOT logged.
- **Note:** This is acceptable. The pattern of logging `clerkUserId` (not email, name, or financial data) is consistent throughout the codebase and appropriate for audit purposes.

---

#### LOW-004: `InvalidEmailError` Message Echoes Submitted Email

- **Note:** This overlaps with HIGH-002 but is called out separately. The `InvalidEmailError` in `account-errors.ts` (line 12) includes the submitted email in its message: `"Invalid email address: ${email}"`. This is caught by the specific handler in `error-handler.ts` (lines 38-51) and returns a generic `VALIDATION_ERROR` message — so the email is NOT currently leaked to the API response. However, it IS logged to pino via the `err` object in the error handler's `logger.warn({ err, correlationId }, ...)` call for other domain errors. If the `InvalidEmailError` were to slip through to the catch-all handler (lines 76-85), the email would be exposed. The fix in HIGH-002 addresses both cases.

## Detailed Checklist Walkthrough

### Auth & Authz — PASS

- [x] Every API endpoint except `/health` requires Clerk JWT authentication via `createRequireAuth()` applied at `app.use('/api/v1', requireAuth, ...)`. The webhook route is authenticated by Svix HMAC, not Clerk JWT, which is correct per spec.
- [x] Auth middleware is applied at the route level in `app.ts` — not optional, not per-controller.
- [x] `user_id` is extracted exclusively from `auth.userId` (which comes from `getAuth(req)` — the verified Clerk JWT). It is never read from `req.body`, `req.params`, or `req.query`. Confirmed by searching all backend source files.
- [x] `clerk_user_id` is stored as `TEXT` in the migration — no UUID casting. No `::uuid` cast present anywhere in the query strings.
- [x] Invalid/expired tokens: Clerk middleware returns 401 via `createRequireAuth()`. Controllers add a second `getClerkAuth(req)` check as defence in depth (returns 401 if `auth.userId` is absent).
- [x] JWT validation (signature, expiry, issuer) is handled by Clerk's `@clerk/express` middleware — standard library behaviour.
- [x] Auth errors return 401 with a generic message. `UserNotFoundError` (no MMF record) correctly returns 404 `USER_NOT_FOUND` rather than 401, which is the correct behaviour per spec US-005.
- [x] Role-based access is not yet gated at the API layer for feat-001 (all endpoints accept any authenticated user). This is correct for feat-001 scope — RBAC enforcement is deferred to later features.

### Input Validation — PARTIAL (HIGH-001, HIGH-003)

- [x] `PATCH /api/v1/me/profile` validates input with Zod schema before processing.
- [x] `PATCH /api/v1/me/notifications` validates input with Zod schema before processing.
- [x] Both schemas use `.strict()` — unknown keys are rejected.
- [x] `displayName` max 255 characters enforced.
- [x] `bio` max 500 characters enforced.
- [x] `avatarUrl` validated as a URL by Zod, and domain model validates protocol (http/https only).
- [x] Boolean fields in notifications schema typed as `z.boolean()` — no coercion, no string-to-boolean.
- [x] Zod schema at `PATCH /api/v1/me/notifications` correctly excludes `securityAlerts` as an accepted field (it is not listed, and `.strict()` rejects unknown keys).
- [ ] **HIGH-001**: `details: parseResult.error.format()` exposed in validation error responses at lines 148 and 211.
