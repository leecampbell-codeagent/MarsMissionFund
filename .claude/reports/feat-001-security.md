# Security Review: feat-001 -- Monorepo Scaffold & Dev Environment

> Security review results.
> Generated by Security Reviewer.

## Verdict: 1 CRITICAL FINDING

## Summary

feat-001 is an infrastructure scaffold with no domain logic, authentication, financial data, or user input handling.
The codebase is minimal and well-structured from a security perspective: no secrets in code, `.env` in `.gitignore`, `no-console` enforced via ESLint, structured logging with Pino, non-root Docker containers, and a health endpoint that exposes no internal details.
One critical dependency vulnerability exists (`happy-dom` VM context escape) that must be resolved before merge.

## Findings

### Critical (Must Fix Before Merge)

#### CRIT-001: Critical vulnerability in happy-dom (VM Context Escape -- Remote Code Execution)

- **Location:** `packages/frontend/package.json:28` -- `"happy-dom": "^16.6.0"`
- **Issue:** `npm audit` reports a critical severity vulnerability (GHSA-37j7-fg3j-429f) in `happy-dom` versions below 20.0.0.
  The vulnerability allows VM context escape that can lead to Remote Code Execution.
- **Impact:** While `happy-dom` is a devDependency used only in tests (not shipped to production), a compromised test environment could allow arbitrary code execution on developer machines and CI runners.
  In a financial application, CI/CD compromise is a supply chain attack vector.
- **Fix:** Update `happy-dom` to `^20.0.0` or later.
  Run `npm audit fix --force` or manually update the version in `packages/frontend/package.json`.
  Verify tests still pass after the upgrade (breaking change between v16 and v20).

### High (Must Fix Before Merge)

No high severity findings.

### Medium (Should Fix)

#### MED-001: Express JSON body parser has no size limit configured

- **Location:** `packages/backend/src/app.ts:28` -- `app.use(express.json())`
- **Issue:** The JSON body parser is used without an explicit size limit.
  Express 5 defaults to 100KB, which is reasonable, but an explicit limit documents the intent and prevents surprises if defaults change.
- **Recommendation:** Set an explicit limit: `app.use(express.json({ limit: '100kb' }))`.
  This becomes more important when endpoints accepting user input are added in future features.

#### MED-002: No CORS configuration present

- **Location:** `packages/backend/src/app.ts`
- **Issue:** No CORS middleware is configured.
  For the scaffold this is acceptable since no API endpoints handle user data, but CORS must be added before any authenticated endpoints are introduced.
- **Recommendation:** Document as a requirement for feat-002 or the first feature that adds authenticated API endpoints.
  When added, configure CORS to allow only the frontend origin (not wildcard `*`).

#### MED-003: No security headers middleware

- **Location:** `packages/backend/src/app.ts`
- **Issue:** No security headers are set (X-Content-Type-Options, X-Frame-Options, Strict-Transport-Security).
  Not immediately exploitable in a scaffold with only a health endpoint, but should be established early.
- **Recommendation:** Add `helmet` middleware before authenticated endpoints are introduced.
  This sets security headers as a baseline defence-in-depth measure.

### Low / Informational

#### LOW-001: PostgreSQL default credentials in .env.example and docker-compose.yml

- **Note:** The `.env.example` and `docker-compose.yml` use default credentials (`mmf`/`mmf`).
  This is standard practice for local development and the values are clearly marked as development defaults.
  The `.env` file is correctly in `.gitignore`.
  No action needed for the scaffold, but production deployment must use secrets management (AWS Secrets Manager / SSM Parameter Store per infra rules).

#### LOW-002: Database port exposed on host

- **Note:** `docker-compose.yml` maps PostgreSQL port 5432 to the host (`"${POSTGRES_PORT:-5432}:5432"`).
  This is expected for local development tooling (database clients, migrations).
  Production deployments should not expose the database port publicly (per infra rules: database in private subnet).

#### LOW-003: Frontend container runs as root

- **Note:** The frontend Docker Compose service (`image: node:22-bookworm-slim`) does not specify `USER node`.
  The backend Dockerfile correctly sets `USER node` for both development and production stages.
  For the frontend service (dev-only Vite server), this is low risk but inconsistent with the backend's security posture.
- **Recommendation:** Add `user: "node"` to the frontend service in `docker-compose.yml` for consistency.

#### LOW-004: `sslmode=disable` on DATABASE_URL

- **Note:** The database connection string uses `sslmode=disable` in both `.env.example` and `docker-compose.yml`.
  This is correct for local Docker networking where TLS adds no security benefit.
  Production deployments must use `sslmode=require` or `sslmode=verify-full`.

## Security Patterns Verified

The following positive security patterns were confirmed in the scaffold:

1. **No `console.log`:** ESLint `no-console: "error"` rule enforced for both backend and frontend source files.
2. **Structured logging:** Pino with request serializer that strips sensitive headers (only logs `id`, `method`, `url`).
3. **`pino-pretty` only in development:** Conditional transport based on `NODE_ENV`.
4. **Non-root Docker containers:** Backend Dockerfile uses `USER node` in both development and production stages.
5. **Multi-stage Docker build:** Production image contains only compiled JS and production dependencies.
6. **`.env` in `.gitignore`:** Environment files excluded from version control; `.env.example` committed with placeholder values only.
7. **No secrets in code:** No API keys, tokens, or credentials found in any source files.
8. **Health endpoint is minimal:** Returns only `{"status":"ok"}` with no version info, hostname, or internal details.
9. **Lock file committed:** `package-lock.json` (8442 lines) is committed for reproducible builds.
10. **TypeScript strict mode:** `strictTypeChecked` and `stylisticTypeChecked` ESLint configs applied.
