## Impact Assessment of Security Fixes

Three security fixes were applied and reviewed:

### Fix 1: HIGH-003 — `onboardingCompleted`/`onboardingStep` removed from PATCH /me/profile schema

**File:** `packages/backend/src/account/api/account-router.ts`

The `updateProfileSchema` Zod object now contains only `displayName`, `bio`, and `avatarUrl` — with `.strict()` rejecting any unrecognized keys. `onboardingCompleted` and `onboardingStep` are no longer in the schema.

**Impact on ACs:** This breaks the onboarding completion flow. The frontend (`onboarding-page.tsx`) still sends these fields:
1. `profileMutation.mutate({ onboardingStep: 'profiling' })` — the skip handler on profile step
2. `completeMutation.mutate()` which calls `updateProfile({ onboardingCompleted: true, onboardingStep: 'complete' })` — step 5 completion

Both calls will receive `400 VALIDATION_ERROR` from the backend because `.strict()` rejects `onboardingStep` and `onboardingCompleted` as unrecognized keys. The onboarding flow will not complete. Users will be stuck in the onboarding wizard and never reach the main app.

**Verified by running Zod schema validation locally:**
```
Test onboardingStep field: FAIL - Unrecognized key(s) in object: 'onboardingStep'
Test onboardingCompleted+onboardingStep: FAIL - Unrecognized key(s) in object: 'onboardingCompleted', 'onboardingStep'
```

**Classification: Major** — Core user flow (onboarding completion) is broken. The spec (feat-001-spec-ui.md) requires that Step 5 sets `onboardingCompleted = true` and Step 3 skip records `onboardingStep = 'profiling'`. The frontend correctly implements the spec; the schema removal was a security fix that was not coordinated with the frontend.

### Fix 2: `error-handler.ts` — Static generic message for unexpected errors

**File:** `packages/backend/src/shared/middleware/error-handler.ts`

The 500 handler now returns: `"Something went wrong on our end. We're looking into it."` — static, no dynamic content. This is correct. No user data, stack traces, or internal error details are leaked.

**Impact on ACs:** None. The error message matches the spec: `feat-001-spec-api.md` table shows `INTERNAL_ERROR` → `"Something went wrong on our end. We're looking into it."`. PASS.

### Fix 3: `account-errors.ts` — Static error messages

**File:** `packages/backend/src/account/domain/errors/account-errors.ts`

All domain error constructors now pass static string literals instead of interpolating the input parameter (e.g. `_email?: string` is accepted but ignored; the message is always `'The email address provided is invalid.'`). This prevents accidental user-supplied data from appearing in error messages.

**Impact on ACs:** None. The error messages returned to clients are the spec-mandated messages defined in the error handler. Domain error messages are internal and are not returned to the API consumer (the error handler maps domain errors to its own static messages). PASS.

## Issues Found

### Issue 1: Onboarding completion flow broken by HIGH-003 schema change — Major

- **Where:** `PATCH /api/v1/me/profile` — backend schema; `/onboarding` — frontend page
- **Security fix applied:** `packages/backend/src/account/api/account-router.ts` — `updateProfileSchema` Zod object now uses `.strict()` and no longer includes `onboardingCompleted` or `onboardingStep`
- **What the frontend sends:**
  - `onboarding-page.tsx` line 96: `profileMutation.mutateAsync({ onboardingStep: 'profiling' })` — skip handler
  - `onboarding-page.tsx` lines 58–62: `updateProfile({ onboardingCompleted: true, onboardingStep: 'complete' })` — step 5 completion
- **Backend response:** `400 VALIDATION_ERROR` with `{ issues: [{ path: '', message: 'Unrecognized key(s) in object: onboardingStep, onboardingCompleted' }] }`
- **User impact:** Users completing onboarding will be stuck at the Step 3 skip action and the Step 5 completion screen. The `onboardingCompleted` flag will never be set to `true`. The user will be redirected to `/onboarding` on every sign-in indefinitely, because `AuthCallbackPage` redirects to `/onboarding` when `onboardingCompleted === false`.
- **Spec reference:** feat-001-spec-ui.md, OnboardingPage section: "Step 5 — Complete: Sets `onboardingCompleted = true` via `PATCH /api/v1/me/profile`" and "Step 3 — Progressive Profiling: Step is skippable"
- **Root cause:** The security fix (HIGH-003) correctly identified that `onboardingCompleted` and `onboardingStep` should not be freely mutable via the profile patch API (to prevent clients from bypassing onboarding). However, the fix was not coordinated with a corresponding frontend change. The frontend still sends these fields and there is no alternative backend endpoint to complete onboarding.
- **Resolution needed:** Either (a) restore `onboardingCompleted` and `onboardingStep` to the backend schema with appropriate server-side business logic guards (e.g., only allow `onboardingCompleted: true` if all required steps have been completed), or (b) create a dedicated `POST /api/v1/me/onboarding/complete` endpoint and update the frontend to call it instead. The backend application service and repository layer already support these fields — only the API schema blocks them.

### Issue 2: US-004 AC wording vs. implementation for `security_alerts: false` — Minor

- **Where:** `PATCH /api/v1/me/notifications` endpoint
- **Expected per US-004 AC:** "When they call PATCH /api/v1/me/notifications with `{ 'security_alerts': false }` Then the response is `400 SECURITY_ALERTS_MANDATORY`"
- **Actual:** Response is `400 VALIDATION_ERROR` with Zod rejection message, because the schema uses `.strict()` and `securityAlerts` is not a recognised key
- **API spec note (feat-001-spec-api.md):** Explicitly notes: "`SECURITY_ALERTS_MANDATORY` — securityAlerts key is present (caught by .strict() before this code is reached, but included for completeness)"
- **Assessment:** The implementation follows the API spec note. The `SecurityAlertsCannotBeDisabledError` domain error and the `SECURITY_ALERTS_MANDATORY` error handler mapping are both correctly implemented for the defence-in-depth path. The US-004 acceptance criterion wording predates the architectural decision to use Zod `.strict()`. This is a documentation inconsistency, not an implementation bug.
- **Severity:** Minor (no blocking, behaviour is intentional and documented)
