### 4. Spec Compliance: PASS

All 9 KYC status states implemented in `KycStatus` type:
- `not_verified`, `pending`, `pending_resubmission`, `in_manual_review`, `verified`, `rejected`, `locked`, `expired`, `reverification_required`

`ALLOWED_TRANSITIONS` map is defined and enforced via `assertTransition()` on every state transition method.

All 3 API endpoints implemented:
- `POST /api/v1/kyc/submit` — creates/updates KYC record, transitions to `pending` (and auto-approves with mock)
- `GET /api/v1/kyc/status` — returns status record with required fields: `id`, `accountId`, `status`, `documentType`, `failureCount`, `verifiedAt`, `submittedAt`
- `POST /api/v1/kyc/admin/unlock/:accountId` — admin role required, unlocks locked accounts

Admin unlock: `accountId` from URL params, requester role validated at both controller and service layer.

Auto-lock after `MAX_KYC_FAILURES = 5` is enforced in `requestResubmission()`.

**Note:** `requestResubmission()`, `sendToManualReview()`, `reject()`, and `requestReverification()` transitions are implemented in the domain entity and application service but are not exposed as API endpoints. These would be used by webhook callbacks from the real KYC provider (Veriff) in production. For the mock scope of feat-013, auto-approve bypasses these paths — this is expected and appropriate.
