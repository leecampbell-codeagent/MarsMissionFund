## Warnings (Should Fix)

### WARN-001: Generic `throw new Error()` in audit repository adapter

**File:** `/workspace/packages/backend/src/campaign/adapters/pg-campaign-audit-repository.adapter.ts:63`

```typescript
throw new Error('Failed to create campaign audit event');
```

This is a defensive null-check for a successful `INSERT RETURNING *` result (practically unreachable). The code standards require no generic `throw new Error()` — all errors should be typed with a domain error class. Low priority given the unreachable nature of the branch, but should use a typed error class in a future cleanup pass.

---

### WARN-002: `campaign_audit_events` table missing `updated_at` column

**File:** `/workspace/db/migrations/20260305151000_create_campaign_audit_events_table.sql`

The infra rule requires "All tables: `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`, `updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`". The audit events table (append-only, immutable) only has `created_at`. This is architecturally correct for an immutable audit log — `updated_at` would be semantically misleading — but technically violates the blanket standard. Either add `updated_at` for blanket compliance, or update the infra rule to explicitly exempt append-only audit tables.

---

### WARN-003: New patterns not documented in `patterns.md` / `gotchas.md`

**File:** `/workspace/.claude/context/patterns.md`

Three reusable patterns introduced in feat-003 are not documented:
1. JSONB array storage for campaign sub-entities with client-side UUID generation (`crypto.randomUUID()`)
2. `emptyBodySchema = z.object({}).strict()` guard for action endpoints accepting no body
3. Atomic optimistic locking: `WHERE id = $1 AND status = $fromStatus` for state transitions

These patterns will recur in feat-004 and feat-005 — documenting them now prevents re-discovery.

---

### WARN-004: MED-003 — Suspended accounts can submit campaigns

**File:** `/workspace/packages/backend/src/campaign/application/campaign-app-service.ts`

`createDraft` checks `user.accountStatus !== 'active'` but `submitCampaign` (and `launchCampaign`) do not. A user suspended after draft creation can still submit. Documented in security report as MED-003. Recommended fix: add `if (user.accountStatus !== 'active') throw new AccountNotActiveError()` as first check in `submitCampaign` and `launchCampaign`.
