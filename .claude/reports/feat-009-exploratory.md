### 2. MockPaymentGatewayAdapter Token Outcomes

**File:** `/workspace/packages/backend/src/payments/adapters/mock/mock-payment-gateway-adapter.ts`

**Spec requirements:**

**Finding: PASS**

`capturePayment` (lines 37-61) checks `token.includes('tok_chargeDeclined')` first, then `token.includes('tok_insufficient_funds')`, then falls through to a default success branch returning `pi_mock_${input.contributionId}`. `parseWebhookEvent` (lines 77-119) implements the sentinel pattern: `signature === MOCK_INVALID_SIGNATURE` throws `InvalidWebhookSignatureError`. Unknown event types map to `'unknown'`.

`MOCK_INVALID_SIGNATURE = 'invalid-sig'` is exported as a named constant (line 16) and correctly imported in the webhook integration test.

## Known Limitations (Pre-existing, Not Blocking)

1. **Duplicate submission idempotency (WARN-001):** Edge case #12 in the spec requires returning the existing captured contribution when the same payment token is submitted twice. The `ContributionRepository` has no `findByPaymentMethodToken` method. Each call creates a new contribution and charges the gateway. This is an acknowledged scope gap — likely deferred to feat-008 when the UI layer provides deduplication before calling the API.

2. **Fake TransactionClient on failed-capture event writes (WARN-002):** `payment-app-service.ts` lines 123-144 pass `{ _brand: 'TransactionClient' }` to the event store for the CONTRIBUTION_FAILED event path. The in-memory mock tolerates this, but the pg adapter would attempt to use `_pgClient` and throw at runtime. Low risk in feat-009 scope (mock-only), but must be resolved before activating the Stripe adapter.
