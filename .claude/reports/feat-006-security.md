# Security Review: feat-006 — Campaign Review Pipeline

> Security review results. Generated by Security Reviewer.

## Verdict: PASS — 0 CRITICAL / 0 HIGH FINDINGS

## Summary

The Campaign Review Pipeline implementation demonstrates strong security posture. Authentication is enforced at every new endpoint, role checks are applied at both the API layer and the application service layer (defence-in-depth), and `reviewer_id` is always sourced from the auth context rather than the request body. All SQL queries against the new `reviewer_id`, `reviewer_comment`, and `reviewed_at` columns use parameterised placeholders. One medium informational note is raised about a hardcoded approval comment in the frontend page.

## Findings

### Critical (Must Fix Before Merge)

None.

### High (Must Fix Before Merge)

None.

### Medium (Should Fix)

#### MED-001: Hardcoded Approval Comment in Frontend Page

- **Location:** `packages/frontend/src/pages/admin-review-queue.tsx:26-28`
- **Issue:** `handleApprove` sends a hardcoded string `'Campaign approved. Meets all curation criteria for Mars Mission Fund.'` rather than collecting reviewer input. The backend requires a non-empty comment; this fulfils that requirement, but the approval rationale is not reviewer-authored.
- **Recommendation:** Implement an approval comment modal analogous to `RejectionReasonModal`, so reviewers provide meaningful, campaign-specific approval notes. The spec (AC-006-003) requires a "non-empty comment" but the design intent is reviewer-authored rationale.

### Low / Informational

#### LOW-001: `findSubmitted` Query Has No Caller-Scoped Filter

- **Note:** `pg-campaign-repository.ts:33` — `SELECT * FROM campaigns WHERE status IN ('submitted', 'under_review')` is intentionally unscoped (all submitted campaigns are visible to reviewers). This is correct per spec. Confirmed the app service enforces `assertReviewerRole` before calling this method, so there is no data isolation risk.

#### LOW-002: Role Check Dual-Layer (Positive)

- **Note:** Both the API layer (`campaign-router.ts`) and application service layer (`campaign-app-service.ts`) enforce the reviewer role check independently. This is defence-in-depth and is a good pattern.

## Detailed Checklist

### Auth & Authz

- [x] All 6 new endpoints check `req.authContext?.userId` and return 401 if missing
- [x] `reviewer_id` extracted from auth context only — never from request body or URL params
- [x] Role check at application service level: `assertReviewerRole(roles)` called before any data access in all review methods
- [x] Role check at API layer as well (defence-in-depth)
- [x] `returnCampaignToDraft` enforces creator ownership: `record.campaign.creatorId !== creatorUserId` → 404 (no info leak about existence)
- [x] Wrong reviewer attempting approve/reject returns 409, not revealing campaign existence separately

- [x] `reviewCommentSchema` applies Zod validation: `z.string().trim().min(1).max(5000)` — non-empty, bounded at 5000 chars

- [x] Campaign ID taken from URL params (no user control over the param binding for domain logic)
- [x] No body required for `/claim`, `/recuse`, `/return-to-draft` — these are body-less endpoints
