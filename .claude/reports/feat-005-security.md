## Summary

The feat-005 Contribution Flow and Payment Processing implementation is broadly sound. Authentication is applied via `requireAuth` middleware at the router-registration level in `app.ts` before the contribution router is mounted, and `user_id` is consistently derived from the Clerk auth context rather than request body. All SQL queries use parameterised placeholders with no interpolation. The primary concerns are: (1) the `payment_token` is stored in the database in plaintext and reconstituted on every DB read, making it part of the in-memory `Contribution` domain object even though it is correctly omitted from API serialisation; (2) an error message leaks internal campaign status to the API consumer; (3) the `amountCents` field lacks an upper-bound constraint, leaving the escrow balance arithmetic vulnerable to integer overflow on pathological inputs; and (4) the idempotency key feature is silently non-functional because the Zod schema does not expose it.

### MEDIUM

#### MED-001: `payment_token` Stored in Plaintext and Reconstituted on All Reads

- **Location:** `/workspace/db/migrations/20260305170000_feat005_contributions.sql:24`, `/workspace/packages/backend/src/payments/adapters/pg-contribution-repository.adapter.ts:25-26`, `/workspace/packages/backend/src/payments/adapters/pg-contribution-repository.adapter.ts:63`
- **Issue:** The `payment_token` column is stored as `VARCHAR(500)` in plaintext in the `contributions` table. Every `SELECT *` query in `findById`, `findByIdForDonor`, `updateStatus`, and `listByDonorForCampaign` returns the token and it is reconstituted into the `Contribution` domain object via `rowToContribution`. This means the token lives in application memory on every read — not just during the capture flow where it is needed. The serializer correctly omits it from the API response, but the token is unnecessarily present in memory for all read paths.
- **Recommendation:** For production (when a real Stripe-like gateway is used), `payment_token` should either: (a) be encrypted at rest in the DB column, or (b) not stored at all after the capture attempt completes (nulled out after gateway call). At minimum, read queries that do not need the token (e.g., `listByDonorForCampaign`, `findByIdForDonor` for status display) should select only the columns they need rather than `SELECT *`. The stub is acceptable for demo purposes, but the pattern must not carry to production.

#### MED-002: `CampaignNotAcceptingContributionsError` Leaks Internal Campaign Status

- **Location:** `/workspace/packages/backend/src/payments/domain/errors/payment-errors.ts:38`, `/workspace/packages/backend/src/shared/middleware/error-handler.ts:274-283`
- **Issue:** `CampaignNotAcceptingContributionsError` embeds the raw campaign `status` value in the error message: `This campaign is not accepting contributions (status: ${status}).` The error handler returns `err.message` verbatim in the 422 response body. This discloses an internal state machine value (`draft`, `pending_review`, `rejected`, `funded`, `archived`, etc.) to any API consumer, including unauthenticated or unauthorised parties who can discover campaigns via the public campaign endpoint.
- **Recommendation:** Remove the `status` interpolation from the error message and return a static message such as "This campaign is not currently accepting contributions." If the specific status is needed for debugging, log it server-side with the correlation ID but do not include it in the API response.

---

#### MED-003: No Upper Bound on `amountCents` — Potential Integer Overflow in Escrow Balance

- **Location:** `/workspace/packages/backend/src/payments/api/schemas/create-contribution.schema.ts:9-26`, `/workspace/packages/backend/src/payments/adapters/pg-escrow-ledger-repository.adapter.ts:43`
- **Issue:** The Zod schema enforces a minimum of 500 cents but no maximum. JavaScript `number` can safely represent integers up to `Number.MAX_SAFE_INTEGER` (9,007,199,254,740,991 cents ≈ $90 trillion), but the escrow running-balance computation `prevBalance + input.amountCents` in the application layer uses JavaScript arithmetic. If a malicious client submits an amount near `MAX_SAFE_INTEGER`, the addition could yield a value that exceeds BIGINT range or produce a nonsensical running balance before the DB constraint catches it. There is also no server-side cap on a single contribution to a reasonable business limit.
- **Recommendation:** Add a maximum constraint to the Zod schema, e.g., `.max(1_000_000_00, { message: 'Maximum single contribution is $1,000,000.' })` (or whatever the business ceiling is). This prevents abuse and aligns with the spec's intent that amounts represent realistic crowdfunding contributions.

#### MED-004: `idempotencyKey` Not Exposed in Zod Schema — Feature Silently Non-Functional

- **Location:** `/workspace/packages/backend/src/payments/api/schemas/create-contribution.schema.ts:3-29`, `/workspace/packages/backend/src/payments/domain/models/contribution.ts:28`
- **Issue:** `CreateContributionInput` in the domain accepts an optional `idempotencyKey`. The database schema defines a UNIQUE constraint on `idempotency_key`. However, the `createContributionSchema` Zod schema does not include an `idempotencyKey` field, and the schema uses `.strict()` which strips unknown fields. As a result, any `idempotency_key` supplied by a client is silently ignored, `idempotencyKey` is always `null`, and the UNIQUE constraint on `(null)` rows in PostgreSQL means the constraint is never enforced. Duplicate requests cannot be idempotency-deduplicated at the database level. While the 60-second window duplicate check provides some protection, a client relying on idempotency keys for safe retry logic will find that behaviour absent.
- **Recommendation:** Either (a) expose `idempotencyKey` in the Zod schema with appropriate UUID or opaque-string validation, or (b) generate the idempotency key server-side (e.g., from `(userId, campaignId, amountCents, timestamp-window)`) and remove the misleading field from the domain input. Also document the intended idempotency design so it is not ambiguous.

### LOW

#### LOW-001: `failureReason` from Gateway Returned Verbatim to Client

- **Location:** `/workspace/packages/backend/src/payments/api/contribution-serializer.ts:19`, `/workspace/packages/backend/src/payments/application/contribution-app-service.ts:122-123`
- **Note:** The `failureReason` string from the payment gateway is stored in the DB and returned via `serializeContribution`. With the stub adapter this is benign. With a real payment gateway (Stripe etc.), the raw decline reason from the gateway could contain processor codes or internal details that should be sanitised before being returned to the end user. Recommend mapping gateway failure reasons to user-safe messages in the application service before persisting.

#### LOW-002: `GET /api/v1/contributions/:id` — Route Parameter Not UUID-Validated

- **Location:** `/workspace/packages/backend/src/payments/api/contribution-router.ts:61`, `/workspace/packages/backend/src/payments/api/contribution-router.ts:75`
- **Note:** The `/:id` route parameter is passed directly to `contributionAppService.getContributionForDonor` without validating that it is a valid UUID format. While PostgreSQL will safely reject a malformed UUID with an error (caught by the global error handler as a 500), it generates unnecessary DB round-trips and exposes an unhandled error path that currently returns a generic 500 rather than a clean 400. Recommend adding Zod UUID validation on `req.params.id` before the service call.

---

#### LOW-003: `dollarsToCentsString` Uses Floating-Point Arithmetic in Frontend

- **Location:** `/workspace/packages/frontend/src/pages/campaign/contribute-to-mission-page.tsx:21-25`
- **Note:** The frontend helper `dollarsToCentsString` converts a dollar string to cents using `parseFloat(amountDollars) * 100` followed by `Math.round`. While `Math.round` mitigates the most common floating-point rounding issues, the result is sent as a string to the backend where the Zod schema re-parses it as an integer. For practical amounts (< $1M) this is not exploitable, but it violates the principle that monetary arithmetic should avoid floating-point entirely. The server-side `min(500)` validation acts as a safety net. This is a low-severity informational note per the financial data integrity rules in `backend.md`.

#### LOW-004: Moderate-Severity npm Vulnerabilities in Dev Dependencies (`vite`/`esbuild`)

- **Location:** `node_modules/vite`, `node_modules/esbuild` (dev only)
- **Note:** `npm audit` reports 4 moderate-severity findings, all in `vite` → `esbuild` (GHSA-67mh-4wv8-2f99, CVSS 5.3). These affect the development server only (esbuild's dev server can accept cross-origin requests) and are not present in production builds. No critical or high vulnerabilities were found. The fix requires a major version bump (`vitest` 4.x). Recommend scheduling this upgrade; it is not merge-blocking.
