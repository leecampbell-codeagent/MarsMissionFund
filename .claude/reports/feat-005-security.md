# Security Review: feat-005 — Campaign Creation (Draft & Submit)

> Security review results. Generated by Security Reviewer.

## Verdict: PASS — 0 CRITICAL / 0 HIGH / 2 MEDIUM / 1 LOW

## Summary

The feat-005 Campaign Creation implementation demonstrates a strong security posture. All five API endpoints require authentication, creator scoping is correctly enforced at the application service layer (creatorId === userId checks), and the KYC gate is applied before any campaign data is processed at submission. All database queries use parameterised placeholders — no SQL injection vectors found. Monetary values are stored as BIGINT in the migration and handled as integer cents throughout the stack.

### Medium (Should Fix)

#### MED-001: `as any` casts in application service for category field

- **Location:** `packages/backend/src/campaign/application/campaign-app-service.ts:104` and `packages/backend/src/campaign/application/campaign-app-service.ts:155`
- **Issue:** The category field is cast with `as any` when passed from the application service into the domain `CreateCampaignInput` / `UpdateCampaignInput`. There is a `biome-ignore` suppression comment explaining this is "type checked at API layer". While Zod validation at the API layer does validate the category enum, the cast bypasses TypeScript's type system in the application service layer.
- **Recommendation:** Define a shared type or narrow the `CreateCampaignServiceInput.category` field from `string` to `CampaignCategory` to eliminate the cast and restore full type safety through the application layer.

#### MED-002: `hero_image_url` validated as `https://` URL prefix but no domain allowlist

- **Location:** `packages/backend/src/campaign/api/campaign-router.ts:53`
- **Issue:** The schema validates `hero_image_url` starts with `https://` and is a valid URL (max 2048 chars). However, there is no domain allowlist. A creator could supply an HTTPS URL to any arbitrary domain (e.g., a tracking pixel, a competitor site). For MVP this is acceptable but worth noting.
- **Recommendation:** In a future iteration, restrict `hero_image_url` to a CDN or allowed origin list.

### Low / Informational

#### LOW-001: `findById` in PgCampaignRepository does not scope by creator_id at DB level

- **Note:** `PgCampaignRepository.findById` queries `SELECT * FROM campaigns WHERE id = $1` without a `creator_id` filter. The scoping check `record.campaign.creatorId !== userId` is performed in the application service, not the DB query. This is architecturally correct per the hexagonal pattern (repositories are dumb data access), and the application-layer check is consistently applied in all three methods that call `findById`. No data leaks in practice. Noted as a defence-in-depth opportunity — adding `AND creator_id = $2` at the DB level would add an extra safety layer.

## Auth & Authz Detail

- All 5 endpoints (POST /campaigns, GET /campaigns/mine, GET /campaigns/:id, PATCH /campaigns/:id, POST /campaigns/:id/submit) check `req.authContext?.userId` and return 401 if absent.
- Creator role check enforced on POST /campaigns (403 if not creator role).
- `userId` extracted exclusively from `req.authContext` — never from request body or URL params.
- `submitForReview` checks `record.campaign.creatorId !== userId` — prevents cross-user submission.
- `updateDraft` and `getCampaign` both check `record.campaign.creatorId !== userId` — returns 404 (not 403) per AC-CAMP-005-009.
- KYC gate checked before campaign data access in `submitForReview`.

## SQL Injection Detail

All queries reviewed in `pg-campaign-repository.ts` and `pg-kyc-status-adapter.ts`:

- `SELECT * FROM campaigns WHERE id = $1` — parameterised
- `SELECT * FROM campaigns WHERE creator_id = $1 ORDER BY created_at DESC` — parameterised
- `SELECT * FROM milestones WHERE campaign_id = $1 ORDER BY target_date ASC, created_at ASC` — parameterised
- `INSERT INTO campaigns (...)` — 17 parameterised values ($1–$17)
- `UPDATE campaigns SET ... WHERE id = $1` — 15 parameterised values
- `DELETE FROM milestones WHERE campaign_id = $1` — parameterised
- `INSERT INTO milestones (...)` — 10 parameterised values ($1–$10)
- `SELECT status FROM kyc_verifications WHERE account_id = $1` — parameterised

No string interpolation or template literals found in any SQL query.

## Financial Data Integrity Detail

- `min_funding_target_cents` and `max_funding_cap_cents` stored as BIGINT in the migration (`20260305000003_extend_campaigns.sql` adds columns; existing `20260304000004_create_campaigns.sql` defines these as BIGINT).
- Serialised as strings in JSON responses: `String(result.minFundingTargetCents)` in `formatCampaignResult`.
- Received as strings in API input: `z.string().regex(/^\d+$/).transform((v) => Number(v))` — converts to integer, no float arithmetic.
- Frontend `campaign-api.ts` types `min_funding_target_cents` and `max_funding_cap_cents` as `string` — never parsed to `number`.
- Domain validates `Number.isInteger()` on all monetary inputs — no floating point accepted.
