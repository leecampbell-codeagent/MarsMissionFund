## Detailed Checklist

### Authentication & Authorisation

- [x] Every API endpoint except `/health` requires authentication ‚Äî `/health` is mounted before `buildClerkMiddleware` and before `createMmfAuthMiddleware`; all `/api/v1/*` routes are behind both middleware layers
- [x] `user_id` is always sourced from the auth context (`req.auth.userId`) ‚Äî never from request body, URL params, or query string
- [x] Invalid or missing JWTs return `401 UNAUTHORIZED` ‚Äî `getAuth(req).userId` is `null` for unauthenticated requests; explicit 401 branch in `createMmfAuthMiddleware` returns before any business logic executes
- [x] Account status gating enforced for all four blocked states: `suspended` ‚Üí 403 `ACCOUNT_SUSPENDED`; `deactivated` ‚Üí 403 `ACCOUNT_DEACTIVATED`; `deleted` ‚Üí 403 `ACCOUNT_DELETED`; `pending_verification` ‚Üí 403 `ACCOUNT_PENDING`
- [x] Auth errors do not reveal resource existence ‚Äî the 401 shape is identical for all invalid token cases (`UNAUTHORIZED` / `Authentication required.`)
- [x] Catch block in `createMmfAuthMiddleware` re-checks for domain error types and maps to 403; all other errors propagate to the global error handler returning generic `INTERNAL_ERROR`

### Mock Auth Containment

- [x] `IS_MOCK_AUTH` is evaluated exactly once at module load in `server.ts` (line 25: `const IS_MOCK_AUTH = process.env.MOCK_AUTH === 'true'`) ‚Äî it is immutable for the lifetime of the process
- [x] `buildClerkMiddleware` and `createMmfAuthMiddleware` accept `isMockAuth: boolean` as a constructor parameter ‚Äî neither function reads `process.env` at request time
- [x] Mock auth requires an explicit `Authorization` header to be present before injecting `user_test_mock` ‚Äî `mockClerkMiddleware` checks `authHeader && authHeader.length > 0`, preserving 401 behaviour for unauthenticated requests even in mock mode
- [x] In production (MOCK_AUTH not set), `buildClerkMiddleware(false)` returns the real `clerkMiddleware()` from `@clerk/express` ‚Äî the mock path is unreachable
- [ ] `_mockClerkUserId` is set via an inline type cast rather than a module augmentation in `express.d.ts` ‚Äî see LOW-002

### SQL Injection

## Findings

### Medium

#### MED-001: Clerk User ID Embedded in Error Message String

- **Location:** `/workspace/packages/backend/src/account/adapters/clerk/clerk-adapter.ts:15`
- **Severity:** Medium (defence-in-depth)
- **Issue:** When no primary email address is found for a Clerk user, the adapter throws:
  ```
  throw new Error(`No primary email address found for Clerk user ${clerkUserId}`);
  ```
  The `clerkUserId` value (e.g. `user_2NNEqL2nrIRdJ194ndJqAHwjfxe`) is concatenated into the error message string. The global error handler in `server.ts` returns only the generic `INTERNAL_ERROR` message to the client ‚Äî the raw error message does not reach the caller today. However, the `clerkUserId` is written to the Pino log via `logger.error({ err }, 'Unhandled error')`, making it part of the `err.message` string field rather than a separately structured field. This conflates a log message with an internal identifier and creates a risk if the error handler is ever changed to surface `err.message` to clients.
- **Client exposure:** None in current code.
- **Recommended fix (post-merge):** Restructure to log the Clerk ID as a structured field and use a generic message in the `Error`:
  ```ts
  // In ClerkAdapter or caller:
  logger.error({ clerkUserId }, 'No primary email address found');
  throw new Error('No primary email address found');
  ```
- **Blocking:** No. Does not block merge.

### Low / Informational

#### LOW-001: No Rate Limiting

- **Location:** `packages/backend/src/server.ts` (no rate-limit middleware present)
- **Issue:** No rate limiting is applied to any endpoint. The auth middleware triggers a DB upsert on first login and a DB lookup on every subsequent request. `GET /api/v1/me` is called on every page load. At scale, both are candidates for abuse causing DB resource exhaustion or artificially inflated Clerk API call volume.
- **Risk for demo scope:** Acceptable ‚Äî this is a workshop application.
- **Recommended fix (before production):** Add `express-rate-limit` with at minimum: 100 req/min general, 10 req/min on auth-triggering paths, per spec L3-002 ¬ß 7.3.
- **Blocking:** No.

#### LOW-002: `_mockClerkUserId` Set via Type Cast Instead of Module Augmentation

- **Location:** `packages/backend/src/shared/middleware/auth.ts:42` and `:59`
- **Issue:** The mock auth path uses an inline type cast `(req as Request & { _mockClerkUserId?: string })` rather than extending the `Request` interface in `express.d.ts`. The project already uses module augmentation for `req.auth` and `req.correlationId`. The cast is test/dev-only and carries no production risk, but it is inconsistent with the established pattern and could mislead future maintainers.
- **Recommended fix (post-merge):** Add `_mockClerkUserId?: string` to the `Request` interface in `packages/backend/src/shared/types/express.d.ts`.
- **Blocking:** No.

#### LOW-003: `isSignedIn` Unused in `useCallback` Dependency Array

- **Location:** `packages/frontend/src/hooks/use-api-client.ts:41`
- **Issue:** `isSignedIn` is destructured from `useAuth()` and appears in the `useCallback` dependency array, but it is not referenced anywhere inside the `fetchWithAuth` function body. This causes unnecessary re-creation of the callback on sign-in/sign-out transitions. There is no security risk, but including it implies to future reviewers that the auth guard inside the hook depends on `isSignedIn` state ‚Äî it does not. The actual guard is the `!token` null-check on the result of `getToken()`.
- **Recommended fix (post-merge):** Remove `isSignedIn` from the `useCallback` dependency array and from the destructuring if not needed elsewhere in the hook.
- **Blocking:** No.
