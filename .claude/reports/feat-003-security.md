# Security Review: feat-003 --- Authentication with Clerk

> Security review results. Generated by Security Reviewer.
> Branch: `feat/003-authentication`
> Reviewed: 2026-03-04

## Verdict: 1 HIGH / 0 CRITICAL FINDINGS

**PASS with conditions** --- 1 high-severity finding must be fixed before merge. No critical vulnerabilities found.

## Summary

The feat-003 authentication implementation is architecturally sound. JWT verification is delegated to Clerk's SDK middleware, webhook signature verification uses the Svix library correctly, all SQL queries use parameterised placeholders, and the hexagonal architecture cleanly separates auth concerns. The primary finding is an environment variable name mismatch between `.env.example`/`docker-compose.yml` and the code that reads the webhook signing secret, which would cause webhook signature verification to silently fail in production (using an undefined value). There are also several medium-severity gaps in HTTP security headers, CORS configuration, and rate limiting that should be addressed before production deployment.

### High (Must Fix Before Merge)

#### HIGH-001: Webhook signing secret environment variable name mismatch

- **Location:** `packages/backend/src/composition-root.ts:107`, `.env.example:25`, `docker-compose.yml:41`
- **Issue:** The code reads `process.env['CLERK_WEBHOOK_SIGNING_SECRET']` but `.env.example` and `docker-compose.yml` define the variable as `CLERK_WEBHOOK_SECRET` (missing `_SIGNING`). When `MOCK_AUTH=false`, the application will throw `"CLERK_WEBHOOK_SIGNING_SECRET is required"` at startup, preventing boot entirely. If the startup check were removed or weakened, webhook signature verification would use `undefined` as the signing secret, potentially accepting any payload.
- **Impact:** In production with `MOCK_AUTH=false`, the backend will fail to start. The fail-fast behaviour is actually safe (it prevents running without a secret), but the name mismatch is a configuration bug that will confuse operators and delay deployment.
- **Fix:** Align the names. Either:
  - Change `.env.example` and `docker-compose.yml` to use `CLERK_WEBHOOK_SIGNING_SECRET`, or
  - Change `composition-root.ts` to read `CLERK_WEBHOOK_SECRET`.
  - The spec uses `CLERK_WEBHOOK_SIGNING_SECRET` --- recommend aligning `.env.example` and `docker-compose.yml` to match.

### Medium (Should Fix)

#### MED-001: No CORS configuration

- **Location:** `packages/backend/src/app.ts`
- **Issue:** There is no CORS middleware configured. The frontend runs on port 5173 and the backend on port 3001 in development. Without CORS, browsers will block cross-origin requests in development. In production, if frontend and backend are on different origins, the same applies.
- **Recommendation:** Add `cors` middleware with an explicit allowlist of origins. Include `Authorization` in `Access-Control-Allow-Headers`. The spec (edge case #15) explicitly calls this out: "Backend CORS config must include `Authorization` in `Access-Control-Allow-Headers`."

#### MED-002: No HTTP security headers (helmet or equivalent)

- **Location:** `packages/backend/src/app.ts`
- **Issue:** No security headers are set: `X-Content-Type-Options`, `X-Frame-Options`, `Strict-Transport-Security`, `Content-Security-Policy`. These are standard defence-in-depth headers.
- **Recommendation:** Add `helmet` middleware or manually set security headers. At minimum: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Strict-Transport-Security` (for HTTPS).

#### MED-003: No rate limiting on any endpoint

- **Location:** `packages/backend/src/app.ts`
- **Issue:** No rate limiting is configured on any endpoint, including the webhook endpoint and the auth/me endpoint. The webhook endpoint in particular could be used for a denial-of-service attack by sending a high volume of requests that trigger Svix signature verification (computationally expensive).
- **Recommendation:** Add rate limiting middleware (e.g., `express-rate-limit`). Apply stricter limits to the webhook endpoint.

#### MED-004: No `Cache-Control: no-store` header on authenticated responses

- **Location:** `packages/backend/src/account/api/auth-router.ts`
- **Issue:** The `/api/v1/auth/me` endpoint (and all future authenticated endpoints) does not set `Cache-Control: no-store`. Per edge case #18 in the spec, this is needed to prevent cached authenticated content from being displayed after sign-out (browser back button).
- **Recommendation:** Add `res.setHeader('Cache-Control', 'no-store')` to authenticated responses, either in the `enrichAuthContext` middleware or per-route.

#### MED-005: Mock auth adapter has no production guard

- **Location:** `packages/backend/src/composition-root.ts:79-81`
- **Issue:** `MOCK_AUTH=true` bypasses all authentication. There is no `NODE_ENV` check to prevent this from being enabled in production. The spec (edge case #24) documents this as a known limitation for workshop scope ("No runtime guard"), but it remains a risk.
- **Recommendation:** Add a warning log if `MOCK_AUTH=true` and `NODE_ENV=production`, or refuse to start. Even a prominent log warning would help.

#### MED-006: Webhook error handler does not use `instanceof` for error class (fragile check)

- **Location:** `packages/backend/src/account/api/webhook-router.ts:39`
- **Issue:** The error handler checks `error.name === 'AuthenticationError'` instead of `error instanceof AuthenticationError`. If the error class name is minified in production builds, or if a different `AuthenticationError` class is thrown, this check could fail, causing authentication errors to be returned as 500 Internal Error instead of 401.
- **Impact:** Would leak a generic 500 error instead of the correct 401 for invalid webhook signatures. Not a direct vulnerability, but reduces error fidelity.
- **Recommendation:** Import `AuthenticationError` and use `instanceof` check: `if (error instanceof AuthenticationError)`.

### Low / Informational

#### LOW-001: `verifyToken()` on `ClerkAuthAdapter` always returns null

- **Note:** `ClerkAuthAdapter.verifyToken()` (line 6-14 of `clerk-auth-adapter.ts`) always returns `null`, with a comment that token verification is handled by middleware. This is fine for the current architecture where `clerkMiddleware()` handles JWT verification, but it means the `AuthPort.verifyToken()` contract is not actually fulfilled. If any code relies on `verifyToken()` in the future, it will silently fail. Consider either implementing it using `clerkClient.verifyToken()` or removing it from the port interface if it's not needed.

#### LOW-002: Webhook handler does not validate webhook event type schema

- **Note:** The `verifyWebhookSignature` method in `ClerkWebhookVerificationAdapter` casts the verified payload to `WebhookEvent` (`as WebhookEvent`) without validating that the payload actually conforms to the `WebhookEvent` interface (line 14). The Svix verification ensures the payload is authentic, but it does not guarantee the structure matches the expected TypeScript type. Consider adding Zod validation on the webhook event payload after signature verification.

#### LOW-003: `@types/pg` is a production dependency (should be devDependency)

- **Note:** In `packages/backend/package.json`, `@types/pg` is listed under `dependencies` rather than `devDependencies`. Type packages should not be in production dependencies as they are only needed at compile time.

#### LOW-004: Email fallback to `'unknown@example.com'` in claims extractor

- **Note:** In `composition-root.ts` lines 35 and 63, when Clerk session claims don't contain an email, the fallback is `'unknown@example.com'`. This value could end up persisted in the database via JIT account creation. Consider using a more clearly invalid placeholder (e.g., `'no-email-in-claims@placeholder.invalid'`) or logging a warning when the fallback is triggered.

#### LOW-005: `pg` is not listed as a direct dependency

- **Note:** `packages/backend/package.json` lists `@types/pg` but not `pg` itself in dependencies. The `pg` module is dynamically imported in `composition-root.ts`. It may be installed as a transitive dependency or may be expected to be added separately. Ensure `pg` is an explicit production dependency.
