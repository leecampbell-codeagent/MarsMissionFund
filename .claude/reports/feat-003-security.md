## Iteration-4 Fix Verification

### Fix 1: `milestoneSchema.targetDate` changed to `.nullable().optional()`

**Status: NO SECURITY CONCERN**

Verified in `/workspace/packages/backend/src/campaign/api/schemas/update-campaign.schema.ts` (line 9):

```ts
targetDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/).nullable().optional(),
```

Security analysis:

- **Injection risk: none.** When `targetDate` is present and non-null, Zod still enforces the `/^\d{4}-\d{2}-\d{2}$/` regex — only the exact format `YYYY-MM-DD` is accepted. A null value bypasses the regex but carries no injected content. The value flows through `JSON.stringify(value)` into a parameterised `$n::JSONB` placeholder in `updateDraftFields` — it is not interpolated into any SQL string, so SQL injection is not possible regardless of the value being null or a date string.

- **Unexpected state: none.** Making `targetDate` both nullable (allows `null` as an explicit value) and optional (allows the key to be absent entirely) is consistent with the domain interface `Milestone.targetDate: string | null` defined in `/workspace/packages/backend/src/campaign/domain/models/campaign.ts` (line 24). A milestone with no target date is a documented valid state — the field is labelled "Target Date" (no asterisk, not required) in the frontend form (`MilestonesSection.tsx` line 194).

- **No business-rule bypass.** The submission validation in `submitCampaign` does not require `targetDate` to be set on any milestone. A null `targetDate` does not allow a campaign to bypass the milestone count (≥2) or the `fundingBasisPoints` total (= 10000) requirements.

- **Zod `strict()` on the parent schema still applies.** Unknown keys in the milestone object are not accepted by `z.object({})` (Zod objects reject extra keys by default). No extra fields can be smuggled in alongside a null `targetDate`.

### Fix 2: `teamMemberSchema.bio` changed to `.nullable()`

**Status: NO SECURITY CONCERN**

Verified in `/workspace/packages/backend/src/campaign/api/schemas/update-campaign.schema.ts` (line 16):

```ts
bio: z.string().max(500).nullable(),
```

Security analysis:

- **Injection risk: none.** When `bio` is present and non-null, Zod still enforces `.max(500)` — an arbitrary-length string cannot be injected. A null value carries no content at all. The `teamMembers` array is serialised via `JSON.stringify(value)` into a parameterised `$n::JSONB` placeholder — no SQL string interpolation occurs. No XSS risk either: the frontend renders `bio` as text content, not as `dangerouslySetInnerHTML`.

- **No type confusion.** The domain interface `TeamMember.bio: string | null` in `campaign.ts` (line 31) already declared `bio` as nullable. The Zod schema now matches the domain type — the previous mismatch (Zod requiring a string, domain allowing null) was the actual bug being fixed. Aligning the validator with the domain type reduces the risk of runtime type mismatches reaching the database layer.

- **The `.max(500)` cap is preserved.** The change from `.max(500)` to `.max(500).nullable()` does not relax the string length constraint — it adds a null branch only. A null `bio` stored in the JSONB column introduces no additional storage or parsing risk.

### Fix 4: `MilestonesSection.addMilestone()` generates `id: crypto.randomUUID()`

**Status: APPROPRIATE — consistent with TeamMember pattern verified in iteration 3**

Verified in `/workspace/packages/frontend/src/components/campaign/campaign-form/MilestonesSection.tsx` (lines 54–63):

```ts
const addMilestone = () => {
  const newMilestone: Milestone = {
    id: crypto.randomUUID(),
    title: '',
    description: '',
    fundingBasisPoints: 0,
    targetDate: null,
  };
  onChange('milestones', [...milestones, newMilestone]);
};
```

Security analysis — identical reasoning to `TeamSection.tsx` `crypto.randomUUID()` (iteration 3):

- **Client-side UUID generation is appropriate.** Milestone IDs are used purely as stable list-item identifiers for optimistic UI management (React keys, add/remove correlation). They are stored in the `milestones` JSONB column alongside other milestone fields. There is no separate `milestones` table with referential integrity — these IDs carry no privilege.

- **The backend validates UUID format server-side.** `milestoneSchema` (line 5: `id: z.string().uuid()`) rejects any milestone whose `id` is not a valid UUID before the update reaches the application service. A malicious actor cannot inject arbitrary strings as milestone IDs.

- **`crypto.randomUUID()` is cryptographically strong.** Web Crypto API generates RFC 4122 v4 UUIDs with 122 bits of entropy — no collision-based attacks are feasible.

- **No ID fixation risk.** Milestone IDs carry no privilege or auth context. A crafted UUID cannot bypass any security control or access another user's data.

- **`targetDate: null` in the new milestone default is safe.** This matches the updated schema (nullable/optional) and the domain type. An empty milestone with `targetDate: null` is correctly rejected by the backend's submission validation if the campaign attempts to submit with fewer than 2 milestones or an incomplete basis points total — the null date does not affect those checks.

**Conclusion:** All four iteration-4 fixes are clean. No new injection vectors, no unexpected state transitions, no privilege changes. The `crypto.randomUUID()` pattern for milestone IDs is identical in risk profile to the team member ID pattern already approved in iteration 3.

### Medium (Should Fix)

#### MED-001: `creatorUserId` Exposed in Review Queue Summary

- **Location:** `packages/backend/src/campaign/api/campaign-serializer.ts` lines 42–52
- **Issue:** `serializeCampaignSummary` (used for the review-queue response) returns `creatorUserId` for every entry in the queue. All reviewers and admins can therefore enumerate MMF-internal creator UUIDs for every submitted campaign. The full `serializeCampaign` also returns `reviewedByUserId` (reviewer's internal UUID). While not high-severity (roles are already checked before the endpoint is reached), it exposes internal identifiers that could aid enumeration or correlation attacks. The spec does not require reviewer identity to be hidden from other reviewers, but the `creatorUserId` in the summary is superfluous.
- **Recommendation:** Remove `creatorUserId` from `serializeCampaignSummary` unless it is needed by a named client requirement. Consider returning only `id`, `title`, `category`, `fundingGoalCents`, `submittedAt`, and `status` in the summary.
- **Severity:** MEDIUM — data minimisation principle; no auth bypass.

#### MED-002: Dynamic SET Clause in `updateDraftFields` Uses Fixed `fieldMap` but No Runtime Key Validation

- **Location:** `packages/backend/src/campaign/adapters/pg-campaign-repository.adapter.ts` lines 190–252
- **Issue:** The `fieldMap` is a static object (defined at module scope in the function body) mapping camelCase schema keys to SQL column names. The loop iterates `Object.entries(fieldMap)` and then reads `(input as Record<string, unknown>)[camelKey]`. This design is safe because the field names used in SET clauses come exclusively from the trusted `fieldMap` values (snake_case column names hardcoded), never from user input. All parameter values are passed via `$n` placeholders. However, the dynamic `${setClauses.join(', ')}` string interpolation inside the SQL template literal could become unsafe if a future developer adds a `fieldMap` entry whose value is not a safe column name (e.g., contains SQL syntax). The current code is not exploitable, but the pattern is fragile.
- **Recommendation:** Add a comment clearly marking the `fieldMap` values as the trusted whitelist for SET clause column names. Optionally validate that every value in `fieldMap` matches a known safe pattern (e.g., `/^[a-z_]+$/`) at startup or test time.
- **Severity:** MEDIUM — not currently exploitable; fragile pattern.

#### MED-003: Inconsistent `accountStatus` Guard Across Creator Workflow

- **Location:** `packages/backend/src/campaign/application/campaign-app-service.ts` lines 64–114 (`createDraft`)
- **Issue:** `createDraft` checks `user.accountStatus !== 'active'` and throws `AccountNotActiveError`. However, `submitCampaign` skips the `accountStatus` check entirely — it only checks the Creator role and KYC. A user whose account was subsequently suspended after obtaining the Creator role can still submit campaigns. The `accountStatus` guard in `createDraft` is not applied consistently throughout the workflow.
- **Recommendation:** Add `if (user.accountStatus !== 'active') throw new AccountNotActiveError()` as the first check in `submitCampaign`, `launchCampaign`, and any other creator-initiated state transitions, mirroring the pattern in `createDraft`.
- **Severity:** MEDIUM — allows suspended/deactivated accounts to advance campaigns already in draft.

#### MED-004: Reviewer Partial ID Leaked in Frontend UI

- **Location:** `packages/frontend/src/pages/campaign/campaign-detail-page.tsx` line 427
- **Issue:** When a campaign is `under_review` and the current viewer is not the assigned reviewer and is not an admin, the frontend renders: `"Claimed by reviewer {campaign.reviewedByUserId.substring(0, 8)}..."`. This leaks the first 8 characters of the assigned reviewer's internal UUID to any other reviewer who views the campaign. While this does not enable a direct attack, it can be used to fingerprint/track internal user identities and confirm account existence.
- **Recommendation:** Do not render any portion of `reviewedByUserId` in the UI. Replace with a generic message such as "Claimed by another reviewer" or surface a display name via a separate, role-gated API call.
- **Severity:** MEDIUM — data exposure; no auth bypass.

### Low (Fix When Convenient)

#### LOW-001: No Rate Limiting on State-Transition Endpoints

- **Location:** All `POST /campaigns/:id/*` endpoints
- **Issue:** None of the campaign state-transition endpoints have explicit rate limiting. This is an infrastructure concern (typically handled at the API gateway or load balancer layer), but it is worth documenting. A reviewer or admin with valid credentials could rapidly spam the `claim` endpoint to exhaust the review queue, or a creator could trigger rapid `submit`/`archive` cycles.
- **Recommendation:** Apply per-user rate limiting at the infrastructure layer (e.g., API Gateway throttling, nginx `limit_req_zone`). No code change required in this PR — document the expectation.
- **Severity:** LOW — no implementation-level controls required in this PR.

#### LOW-002: `heroImageUrl` Validated as HTTPS URL but No Domain Allowlist

- **Location:** `packages/backend/src/campaign/api/schemas/update-campaign.schema.ts` line 46
- **Issue:** The schema validates that `heroImageUrl` starts with `https://` and is a valid URL (≤2048 chars), but does not restrict the domain. A creator could submit an image URL pointing to an arbitrary external host (including attacker-controlled infrastructure), which the frontend renders in an `<img>` tag. This could be used for pixel tracking, exfiltrating browser information, or serving malicious content if the CSP is permissive.
- **Recommendation:** Enforce a domain allowlist in the Zod schema (e.g., only permit URLs from trusted CDN domains such as `upload.cloudinary.com` or an S3 bucket). Alternatively, implement a server-side image proxy that fetches and re-hosts images.
- **Severity:** LOW in current context (no contributions/money movement yet); escalate to MEDIUM before launch.

#### LOW-003: `reviewNotes`, `rejectionReason`, `resubmissionGuidance` Have No Max Length

- **Location:** `packages/backend/src/campaign/api/schemas/approve-campaign.schema.ts` line 5; `packages/backend/src/campaign/api/schemas/reject-campaign.schema.ts` lines 5–6
- **Issue:** `reviewNotes`, `rejectionReason`, and `resubmissionGuidance` have no `max()` constraint in their Zod schemas. These fields are stored in `TEXT` columns (unbounded in PostgreSQL). A reviewer could submit megabytes of text, causing high memory allocation during JSON serialisation and potential DoS via response inflation.
- **Recommendation:** Add `.max(5000)` (or a similar sensible cap) to all three fields.
- **Severity:** LOW — requires a privileged role to exploit.

## Summary

Iteration-3 fixes are clean. All four changes — adding `createdAt`/`updatedAt` to `serializeCampaignSummary`, adding `id: string` to the frontend `TeamMember` interface, using `crypto.randomUUID()` for client-side team member IDs, and updating test fixtures — were verified and introduce no new security issues. The `crypto.randomUUID()` pattern is appropriate: team member IDs are non-privileged list-management identifiers, the backend enforces UUID format validation via `z.string().uuid()` in `teamMemberSchema`, and the Web Crypto API source is cryptographically strong. The `createdAt`/`updatedAt` timestamps added to the review-queue serialiser expose no sensitive information and are already declared in the frontend `CampaignSummary` type.

HIGH-001 (missing body validation on claim/launch/archive endpoints) remains resolved from iteration 2. feat-003 delivers a well-structured campaign pipeline with strong security foundations: consistent auth guards, proper ownership checks, optimistic locking on state transitions, SQL injection protection, correct monetary serialisation, and a thorough error-handler that maps all domain errors to safe HTTP responses without leaking internal details. The 404-for-ownership-checks pattern is correctly applied throughout.

Four medium findings remain open (data minimisation on review queue serialiser, fragile dynamic SQL pattern, inconsistent account-status guard in the creator workflow, reviewer UUID substring leak in the frontend). Three low findings cover rate limiting (infra), an image URL domain allowlist, and unbounded reviewer text fields. None of the open findings are merge-blocking.
