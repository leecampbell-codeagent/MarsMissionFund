# Security Review: feat-004 — Account Registration & Onboarding

> Security review results. Generated by Security Reviewer.
> Date: 2026-03-05

## Verdict: PASS — 0 CRITICAL / 0 HIGH / 3 MEDIUM / 4 LOW

## Summary

The feat-004 implementation demonstrates a solid security posture overall. Authentication is correctly enforced on all account endpoints via a two-layer middleware chain (requireAuthentication + enrichAuthContext), user_id is always sourced from the auth context and never from request bodies, and all SQL queries use parameterised placeholders with no string interpolation. Three medium-severity findings exist: missing HTTP security headers (no helmet/CORS configuration), an avatar URL validation gap that allows non-URL HTTPS strings without further domain validation, and a role logging pattern that writes the full role array to structured logs. No critical or high severity issues were found that would block merge.

### Medium (Should Fix)

#### MED-001: Missing HTTP Security Headers (No helmet / CORS Configuration)

- **Location:** `packages/backend/src/app.ts`
- **Issue:** The Express application mounts no HTTP security headers middleware. There is no `helmet` (for X-Content-Type-Options, X-Frame-Options, Strict-Transport-Security, etc.) and no explicit CORS configuration. The `app.ts` file has no `cors()` middleware call and no custom header-setting code. Express 5 does not add security headers by default. Without explicit CORS configuration, browser preflight behaviour is uncontrolled and the allowed-origin set is undefined.
- **Impact:** Without `X-Content-Type-Options: nosniff`, browsers may MIME-sniff responses. Without `X-Frame-Options`, clickjacking attacks against authenticated pages are possible. Without an explicit CORS `Allow-Origin` policy, the backend may accept cross-origin requests from any origin in some configurations. The overall attack surface is wider than necessary.
- **Recommendation:** Add `helmet` to the backend dependencies and mount `app.use(helmet())` before route registration. Configure `cors()` with an explicit origin allowlist (`VITE_API_URL` / expected frontend domain). These are one-line additions that eliminate an entire class of web security issues.

#### MED-002: Avatar URL Validation Does Not Restrict Domain or Path

- **Location:** `packages/backend/src/account/api/account-router.ts:14` and `packages/backend/src/account/domain/account.ts:217`
- **Issue:** Avatar URL validation accepts any `https://` URL. The Zod schema at the API layer (`z.string().url().startsWith('https://')`) and the domain method (`avatarUrl.startsWith('https://')`) enforce HTTPS and URL format, which is good. However, there is no restriction on the hostname or path. A user can supply a URL pointing to an internal network resource (`https://169.254.169.254/latest/meta-data/` on AWS, or `https://internal-service/`). If any downstream component (e.g. a future avatar rendering service, an email template engine, or a CDN pre-fetching pipeline) follows this URL server-side, it creates a Server-Side Request Forgery (SSRF) vector. Currently, the URL is only stored and returned to the client — no server-side fetch occurs — so this is medium rather than critical. However, as the feature matures, SSRF becomes exploitable if any service ever fetches the avatar URL on behalf of the server.
- **Impact:** Stored SSRF potential if the avatar URL is ever fetched server-side (e.g. for resizing, caching, email embedding). Currently limited to stored data integrity.
- **Recommendation:** Add a domain allowlist or at minimum block private/link-local IP ranges in the URL hostname. Consider validating that the URL hostname does not resolve to RFC-1918 / RFC-3927 addresses at write time, or restrict avatar URLs to known CDN domains (e.g. only `*.cloudinary.com`, `*.gravatar.com`). Document the constraint in the domain layer.

#### MED-003: Role Array Logged at INFO Level May Expose Privilege Information

- **Location:** `packages/backend/src/account/application/account-app-service.ts:120`
- **Issue:** The `updateRoles` method logs the full merged role array at INFO level: `this.logger.info({ accountId, action: 'roles_updated', roles: mergedRoles }, ...)`. While roles are not PII in the traditional sense, logging `['backer', 'creator', 'reviewer', 'administrator']` reveals privilege assignments in structured logs. In a production environment where logs flow to a SIEM, observability platform, or third-party logging service, this exposes the full privilege set of every user whose roles change. This is especially sensitive for admin-role assignments.
- **Impact:** Privilege escalation events and admin role grants are readable by anyone with log access. In a multi-tenant SaaS this is an information disclosure risk.
- **Recommendation:** Log only the count of roles or a non-sensitive summary (e.g. `{ roleCount: mergedRoles.length }`), or move role detail to DEBUG level. Admin role assignments should emit a dedicated security audit event rather than appear in application INFO logs.

### Low / Informational

#### LOW-001: No Rate Limiting on Account Mutation Endpoints

- **Location:** `packages/backend/src/account/api/account-router.ts` — all PATCH endpoints
- **Note:** The three PATCH endpoints (`/api/v1/accounts/me`, `/api/v1/accounts/me/preferences`, `/api/v1/accounts/me/onboarding`) have no rate limiting. An authenticated user can call these endpoints at arbitrarily high frequency. While this doesn't bypass auth, it could be used for database enumeration side-channel attacks or abuse of the event store (flooding events). This is acceptable for the current workshop scope but should be addressed before production. Add `express-rate-limit` with a per-user limiter on mutation endpoints (e.g. 10 req/min per authenticated userId).

---

#### LOW-002: Idempotent Onboarding Step Handler Silently Swallows All `InvalidOnboardingStepError` Errors

- **Location:** `packages/backend/src/account/api/account-router.ts:254-267`
- **Note:** When `advanceOnboardingStep` throws `InvalidOnboardingStepError`, the handler catches it and returns 200 with the current account state (for idempotency). This is correct for the already-at-or-past-step case. However, the same catch block fires if the domain throws `InvalidOnboardingStepError` for a genuinely invalid step value — which is already prevented by Zod's `.enum()` schema. This means any future domain validation that throws `InvalidOnboardingStepError` for a _different_ reason will silently return 200 instead of 400. The pattern is safe today but fragile. Consider checking the specific error message or subtype before treating it as idempotent.

---

#### LOW-003: `update()` Repository Method Does Not Scope WHERE Clause to `clerk_user_id`

- **Location:** `packages/backend/src/account/adapters/pg/pg-account-repository.ts:68-90`
- **Note:** The `UPDATE accounts ... WHERE id = $1` query scopes updates by the internal account UUID. The account UUID is sourced exclusively from `req.authContext.userId` (which is the MMF account ID, set by the enrich-auth-context middleware after authenticating with Clerk). This is correct — the account ID is never user-supplied. However, if the `account.id` value were ever to arrive from an untrusted path, the WHERE clause would allow updating any account row. The current data flow is safe, but adding a defence-in-depth check (e.g. `WHERE id = $1 AND clerk_user_id = $2`) would make the query self-defending against future refactoring that might inadvertently allow a user-supplied ID.

---

#### LOW-004: `withDisplayNameFromWebhook` Bypasses Display Name Validation

- **Location:** `packages/backend/src/account/domain/account.ts:308-318`
- **Note:** The `withDisplayNameFromWebhook` method sets `displayName` from Clerk webhook data without the length/content validation applied by `withProfile()`. A Clerk account could theoretically have a display name exceeding 100 characters or containing leading/trailing whitespace that would not be accepted through the normal profile update path. This is low risk because: (a) the webhook data originates from Clerk's own platform, (b) the field is only set when the current value is null, and (c) display name length is not a security-critical constraint. Nevertheless, applying consistent validation across all mutation paths reduces inconsistency.

## Detailed Checklist Notes

### Authentication & Authorisation — PASS

- All three account endpoints (`GET /api/v1/accounts/me`, `PATCH /api/v1/accounts/me`, `PATCH /api/v1/accounts/me/preferences`, `PATCH /api/v1/accounts/me/onboarding`) are mounted behind the `requireAuth` + `enrichContext` middleware chain in `app.ts:69`.
- `user_id` is always sourced from `req.authContext?.userId` (set by Clerk-verified middleware), never from request body or URL params.
- 401 is returned for unauthenticated requests; 403 for suspended/deleted accounts.
- Webhook endpoint (`POST /api/webhooks/clerk`) is mounted before auth middleware but uses Svix signature verification — correct pattern.
- Health endpoint is correctly public.
- No role-based endpoints are implemented in this feature (roles are self-assigned during onboarding, not elevated privileges). The restriction of onboarding roles to `['backer', 'creator']` in the Zod schema correctly prevents users from self-assigning admin roles.

### Input Validation — PASS (with MED-002 caveat)

- All three PATCH endpoints validate request bodies with strict Zod schemas (`.strict()` mode prevents extra fields).
- String lengths are bounded: `display_name` max 100, `bio` max 500.
- `avatar_url` is validated as a URL and HTTPS-only, but has no hostname restriction (MED-002).
- Onboarding step is validated as `.enum(...)` — welcome step is excluded from client-settable values (correct).
- Role values are restricted to `['backer', 'creator']` in the API schema, preventing privilege escalation.
- Empty body is explicitly rejected on the profile PATCH endpoint.

### SQL Injection — PASS

- All queries in `pg-account-repository.ts` and `pg-event-store-adapter.ts` use parameterised placeholders (`$1`, `$2`, etc.) exclusively.
- No template literals are used in SQL query strings. `grep pool.query(\`` returned zero results.
- JSONB fields are passed as pre-serialised strings via `JSON.stringify()`, not interpolated into SQL.
- Migration file uses static DDL only — no dynamic SQL.

### Financial Data Integrity — N/A

This feature introduces no monetary amounts, payment processing, or financial calculations. Not applicable to feat-004.

### Data Exposure — PASS (with MED-003 caveat)

- Error responses return structured `{ error: { code, message, correlation_id } }` — no stack traces, SQL errors, or file paths.
- `AccountNotFoundError` returns the same 404 message regardless of whether the account exists or belongs to another tenant (no information leak).
- Logs use structured Pino with no PII: `{ accountId, action }` only — email, display name, and bio are not logged.
- Exception: roles are logged at INFO in `updateRoles` (MED-003).
- Event store payloads contain only field names and non-PII identifiers, consistent with the comment in `account-events.ts` ("no PII — IDs and field names only").
- No `console.log` calls found in any reviewed file.
- No environment variables exposed to frontend beyond what is expected (`VITE_CLERK_PUBLISHABLE_KEY`, `VITE_API_URL`).
