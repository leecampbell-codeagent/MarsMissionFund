### Medium (Should Fix)

#### M-001 — `sort` parameter maps to `funding_goal_cents` instead of raised amount

**File:** `/workspace/packages/backend/src/campaign/adapters/pg-campaign-repository.adapter.ts`, lines 311–312

**Detail:** The `most_funded` and `least_funded` sort options are documented as sorting by funding progress but actually sort by `c.funding_goal_cents` (the campaign's target goal), not by the amount raised. Since `totalRaisedCents` is stubbed to `'0'` across all campaigns in feat-004, this is a logic defect rather than an exploitable security issue, but it constitutes a **data accuracy / information integrity issue**: a public user requesting "most funded" will receive results ordered by highest goal, not most contributed to. When the contributions feature lands this silent mis-sort will persist unless fixed.

```sql
-- Lines 311-312 in pg-campaign-repository.adapter.ts
CASE WHEN $4::TEXT = 'most_funded' THEN c.funding_goal_cents END DESC NULLS LAST,
CASE WHEN $4::TEXT = 'least_funded' THEN c.funding_goal_cents END ASC NULLS LAST,
```

**Risk:** Misleading public-facing data ordering; trust issue once contributions are live.

**Recommendation:** Either rename the sort values to `highest_goal`/`lowest_goal` for feat-004 accuracy, or add a placeholder comment noting the column must be changed to a contributions aggregate in feat-005.

#### M-002 — `GET /:id` path parameter has no UUID format validation

**File:** `/workspace/packages/backend/src/campaign/api/public-campaign-router.ts`, line 120

**Detail:** `req.params.id` is forwarded directly to `findPublicById` without any format check (no Zod UUID validation, no regex). The test suite explicitly acknowledges this (test comment: "PostgreSQL will reject and return no rows = 404") and relies on PostgreSQL returning zero rows for malformed UUID strings.

While PostgreSQL's `uuid` column type does reject non-UUID strings silently (no error, zero rows), this has two concerns:
1. **Unpredictable error surface**: if the `id` column type were ever changed to `TEXT` (common in refactors), malformed values would begin matching or behaving unexpectedly.
2. **Unnecessary database round-trip**: every request with a non-UUID `id` (e.g., from scanners probing `/api/v1/public/campaigns/../../../../etc/passwd`) hits the database unnecessarily.

The authenticated campaign router likewise does not validate the path param UUID, but the public endpoint is significantly higher traffic and lower cost to abuse.

```typescript
// public-campaign-router.ts line 120 — no validation before DB call
const campaign = await campaignAppService.getPublicCampaign(req.params.id);
```

**Recommendation:** Add a simple UUID format check at the router level before delegating to the app service:
```typescript
const idSchema = z.string().uuid();
const parsed = idSchema.safeParse(req.params.id);
if (!parsed.success) {
  res.status(404).json({ error: { code: 'CAMPAIGN_NOT_FOUND', message: 'Campaign not found.' } });
  return;
}
```

### Low / Informational

#### L-001 — No `Content-Security-Policy` or rate limiting on public endpoints

**Files:** `/workspace/packages/backend/src/app.ts`

**Detail:** The Express application has no security headers middleware (e.g., `helmet`) and no rate limiting (e.g., `express-rate-limit`). Public unauthenticated endpoints are high-value scraping and enumeration targets. This is a systemic gap across the entire backend, not specific to feat-004, but the new public endpoints raise its priority.

**Risk:** Scraping of all public campaign data; enumeration of valid campaign UUIDs; potential DoS via search endpoint with complex FTS queries.

**Recommendation:** Add `helmet()` for security headers and `express-rate-limit` on the public routes path as a follow-up task. For FTS specifically, the 200-character max on `q` limits query complexity somewhat.

---

#### L-002 — `getCategoryStats` query counts ALL campaigns of a category, not just public ones

**File:** `/workspace/packages/backend/src/campaign/adapters/pg-campaign-repository.adapter.ts`, lines 474–479

**Detail:** The `getCategoryStats` query filters `WHERE category = $1` but then uses `COUNT(*) FILTER (WHERE status IN ('live', 'funded'))` and `COUNT(*) FILTER (WHERE status = 'live')`. This is correct — it only counts live/funded campaigns. However, the outer `FROM campaigns WHERE category = $1` with no status pre-filter means the query scans ALL campaigns in a category (including drafts, submitted, rejected, archived) and then filters in-application via the `FILTER` clause. This is a performance note rather than a security issue (no private data is returned), but it is worth noting.

**Risk:** Informational / performance. No data leakage.

---

#### L-003 — Frontend `publicFetch` helper is duplicated

**File:** `/workspace/packages/frontend/src/api/public-campaign-api.ts`

**Detail:** The `publicFetch` generic helper (lines 28–61) handles `getPublicCampaign` and `getCategoryStats`, but `searchPublicCampaigns` (lines 68–110) duplicates the entire fetch-and-error-handle pattern inline instead of using the helper. This is a code quality issue only, but duplicated fetch logic risks divergence in error handling behaviour between the two code paths (e.g., the inline variant does not check `Content-Type` of the response, while `publicFetch` does on line 56).

**Risk:** Inconsistent error handling surface on the frontend; not a direct security issue.

## Checklist Detail

### 1. Authentication & Authorisation — PASS

- `GET /api/v1/public/campaigns`, `GET /api/v1/public/campaigns/:id`, `GET /api/v1/public/campaigns/stats` are all mounted WITHOUT `requireAuth` at `/api/v1/public/campaigns` in `app.ts` line 80. This is intentional and correct (G-036).
- Clerk middleware (`clerkMiddleware()`) is still global and populates `req.auth` for all requests — public routes simply do not enforce it.
- Existing authenticated endpoints at `/api/v1/campaigns` (line 72–76 in `app.ts`) still require `requireAuth`. Confirmed.
- The public router never calls `getClerkAuth(req)` or reads any auth context — it is purely unauthenticated.

### 2. SQL Injection — PASS

- All queries use parameterised placeholders (`$1`, `$2`, etc.) exclusively. No string interpolation found.
- FTS uses `websearch_to_tsquery('english', $1)` correctly parameterised — `$1` is bound to the validated and trimmed `q` value.
- Dynamic ORDER BY is implemented entirely via SQL `CASE WHEN $4::TEXT = '...' THEN ... END` expressions using the parameter value only for equality comparisons against hardcoded column names. No user input is interpolated into the SQL text itself.
- Category filter uses `$3::TEXT[]` with `= ANY(...)` — no interpolation.
- `getCategoryStats` uses `WHERE category = $1` — parameterised.
- `findPublicById` uses `WHERE c.id = $1 AND c.status IN ('live', 'funded')` — parameterised.

### 3. Input Validation — PASS WITH NOTE (M-002)

- Zod schema (`publicCampaignSearchSchema`) validates all query parameters before use.
- `q` has a 200-character max: `z.string().max(200)`.
- `category` validated against `CAMPAIGN_CATEGORIES` enum.
- `sort` validated against `PUBLIC_SORT_OPTIONS` enum (`['newest', 'ending_soon', 'most_funded', 'least_funded']`).
- `status` validated against `PUBLIC_STATUS_FILTERS` enum.
- `limit` constrained to 1–100 integer.
- `offset` constrained to >= 0 integer.
- `categoryStatsQuerySchema` validates `category` is a known enum value.
- FINDING: `/:id` path parameter is not UUID-validated before being passed to the DB (M-002).

### 4. Data Exposure / Information Disclosure — PASS

- `findPublicById` query has `AND c.status IN ('live', 'funded')` at the SQL level. A campaign in any other status (draft, submitted, rejected, archived, etc.) returns `null` at the repository layer.
- The app service throws `CampaignNotFoundError` for both `null` (non-existent) and non-public (wrong status) campaigns. The error handler maps this to `404 CAMPAIGN_NOT_FOUND` in both cases — there is no `403` branch that would reveal existence of a non-public campaign.
- The search query also enforces `WHERE c.status IN ('live', 'funded')` at the SQL level — draft/submitted/rejected campaigns never appear in results.
- Error responses contain no stack traces, SQL text, or internal details. The error handler (reviewed in `error-handler.ts`) logs errors internally via Pino but returns only `{ code, message, correlation_id }` to clients.

### 5. Financial Data Integrity — PASS

- `fundingGoalCents` and `fundingCapCents` are serialised as strings (BIGINT returned from `pg` as string, passed through without conversion).
- `totalRaisedCents` is hardcoded as string `'0'` (not number `0`) throughout.
- `serializeCategoryStats` serialises `totalRaisedCents` as the string value from the domain type.
- No floating point arithmetic observed anywhere in the financial value path.

### 6. Frontend Security — PASS WITH NOTE (L-003)

- `public-campaign-api.ts` uses raw `fetch` with no Authorization header — confirmed, no Clerk JWT injection.
- No sensitive authenticated state is referenced in the public API module.
- URL construction uses `new URL(path, window.location.origin)` with `url.searchParams.set()` — all parameter values are properly encoded via the URL API; no manual string concatenation risk.
- `encodeURIComponent(category)` is used in `getCategoryStats` URL construction — safe.
