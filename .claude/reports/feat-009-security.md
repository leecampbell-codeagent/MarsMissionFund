# Security Review: feat-009 — Payment Processing (Mock Gateway)

> Security review results. Generated by Security Reviewer.

## Verdict: 0 CRITICAL / 0 HIGH / 2 MEDIUM / 3 LOW

## Summary

The feat-009 payment implementation has a solid security foundation. All SQL queries use parameterised placeholders, monetary values are correctly stored and serialised, webhook idempotency is properly implemented, and no secrets are hardcoded in committed code. There are two medium findings — a missing test for the `400 INVALID_WEBHOOK_SIGNATURE` path and a type-safety hack in the application service for failed-capture event writes — and several low-severity informational items. No critical or high severity issues were found.

### Medium (Should Fix)

#### MED-001: Missing Integration Test for `400 INVALID_WEBHOOK_SIGNATURE`

- **Location:** `/workspace/packages/backend/src/__tests__/payment-webhook.test.ts`
- **Issue:** The feature spec (US-002) requires: "Given a webhook with an invalid or missing signature, then the response is `400` with error code `INVALID_WEBHOOK_SIGNATURE`." The webhook test file contains 6 test cases but none of them test the invalid signature path. The mock adapter's `parseWebhookEvent` ignores the signature parameter (`_signature`), so the invalid-signature rejection is only exercised in production via the Stripe adapter (which is a skeleton). The controller-level `InvalidWebhookSignatureError` catch handler exists and is correct, but it is not tested.
- **Recommendation:** Add an integration test that forces the mock gateway to throw `InvalidWebhookSignatureError` for a clearly invalid signature value. This can be done by either: (a) adding a special sentinel signature value (e.g., `tok_invalid_sig`) that the mock adapter rejects, or (b) subclassing the mock in the test to throw `InvalidWebhookSignatureError`. Without this test the rejection path is untested code.

#### MED-002: Fake `TransactionClient` Object Used for Failed-Capture Event Writes

- **Location:** `/workspace/packages/backend/src/payments/application/payment-app-service.ts:126-143`
- **Issue:** When a gateway capture fails, the application service needs to write `PAYMENT.CONTRIBUTION_FAILED` events to the event store. Since these writes happen outside a real transaction, the service constructs a fake `TransactionClient` using a type assertion: `{ _brand: 'TransactionClient' } as Parameters<...>`. This means the `EventStorePort.getNextSequenceNumber` and `EventStorePort.append` methods receive a non-functional client. If the `PgEventStoreAdapter` ever attempts to use `_pgClient` from this fake object, it will throw a runtime error in production.
- **Recommendation:** The `EventStorePort` should have a separate method signature for non-transactional writes, or the port should accept `TransactionClient | null` with the implementation handling the null case. The current approach is a type-system workaround that silently breaks the contract and could cause production data loss (failed events not recorded).

### Low / Informational

#### LOW-001: Stripe Adapter Skeleton Contains TODO Comments

- **Location:** `/workspace/packages/backend/src/payments/adapters/stripe/stripe-payment-gateway-adapter.ts:25,33,40,47`
- **Note:** The Stripe adapter has four TODO comments marking unimplemented methods. The spec explicitly states the skeleton (class + interface compliance) is the deliverable for feat-009, and the adapter throws `Error` immediately if called. This is expected behaviour. However, the TODO comments will trigger the auditor's automatic FAIL rule. These should be converted to descriptive comments without the TODO keyword (e.g., `// Stretch goal: implement via stripe.paymentIntents.create(...)`) since the spec says this is intentionally incomplete.

#### LOW-002: Webhook Router Accepts `text/plain` and `application/octet-stream` Content Types

- **Location:** `/workspace/packages/backend/src/payments/api/payment-routes.ts:54`
- **Note:** The `express.raw()` middleware is configured to accept `['application/json', 'text/plain', 'application/octet-stream']`. The `text/plain` and `application/octet-stream` entries were added to facilitate testing. In production, Stripe only sends `application/json`. This is a defence-in-depth concern — accepting additional content types is not a vulnerability since signature verification (in the real Stripe adapter) prevents processing of tampered bodies. The mock adapter ignores the signature entirely (by design). Consider removing the extra types or gating them behind the `MOCK_PAYMENTS` flag.

#### LOW-003: Mock Adapter Does Not Validate Signature at All

- **Location:** `/workspace/packages/backend/src/payments/adapters/mock/mock-payment-gateway-adapter.ts:70`
- **Note:** The `parseWebhookEvent` method accepts `_signature: string` but completely ignores it, meaning any signature (or no signature) is accepted in mock mode. This is correct and expected behaviour for a mock — the Stripe adapter correctly validates signatures. Ensure the `MOCK_PAYMENTS=true` flag is never set in production environments.

### Auth & Authz

- `POST /api/v1/payments/capture` — mounted with `requireAuth, enrichContext` middleware in `app.ts:77`. Auth check verified: `req.authContext` checked at line 109 of `payment-routes.ts`. `donor_id` extracted from `ctx.userId` (auth context), never from request body. A test at line 269 of `payment-capture.test.ts` explicitly verifies that an injected `donor_id` in the request body is ignored.
- `GET /api/v1/campaigns/:campaignId/escrow-balance` — mounted with `requireAuth, enrichContext` middleware. Role check enforces `administrator` or `super_administrator` at `payment-routes.ts:180-187`. Tested at integration level with backer vs admin accounts.
- `POST /api/v1/payments/webhook` — correctly excluded from auth middleware; mounted before `requireAuth` in `app.ts:59`. Authentication is via Stripe signature (Stripe adapter) or implicit trust (mock adapter). This is intentional and correct.

### Input Validation

- Zod schema `capturePaymentSchema` validates `campaign_id` as UUID, `amount_cents` as `int().min(100)`, and `payment_method_token` as `string().min(1)`.
- `campaignId` path parameter on escrow-balance endpoint is validated with `z.string().uuid()` before use.
- All validation occurs at the API layer before domain logic.
- Float amounts (e.g., 10.50) are rejected by `z.number().int()`.
- No unbounded inputs — `payment_method_token` has `.min(1)` but no `.max()`. This is low risk (no SQL interpolation) but noted.

### SQL Injection

All queries in pg adapter files use parameterised `$1`, `$2`, ... placeholders. No template literals are used in SQL strings. No string concatenation in query construction. All parameterised:

- `pg-contribution-repository.ts`: INSERT ($1-$8), UPDATE ($1-$4), SELECT ($1)
- `pg-escrow-ledger-repository.ts`: INSERT ($1-$8), SELECT ($1)
- `pg-processed-webhook-event-repository.ts`: SELECT ($1), INSERT ($1-$2) with ON CONFLICT
- Balance query in `pg-escrow-ledger-repository.ts:31-38` uses a static CASE expression with $1 parameter — no user input in SQL structure.

### Financial Data Integrity

- `amount_cents` stored as `BIGINT` in migrations (contributions table, escrow_ledger table).
- `amountCents` is typed as `number` (integer cents) throughout domain code. `Number.isInteger()` validation enforced in both `Contribution.create()` and `EscrowLedgerEntry.create()`.
- JSON serialisation: `amount_cents` is serialised as `String(contribution.amountCents)` in `payment-routes.ts:27`. `balance_cents` is serialised as `String(balanceCents)` in `payment-routes.ts:207`. Both comply with the string-serialisation rule.
- No floating point arithmetic on monetary amounts.
- Escrow ledger is append-only in the application layer (INSERT only, no UPDATE in repository). Database-level protection via BEFORE UPDATE/DELETE triggers in the migration.
- Capture and escrow write are in the same transaction (MED-002 above covers the failure path issue).
- Compensating refund attempted if DB write fails after gateway success (FATAL log + refund attempt).

### Data Exposure

- Error responses do not expose stack traces or SQL details — all errors map to structured `{ error: { code, message, correlation_id } }`.
- Internal details from `error` object are never passed to the API response.
- The `DomainError` message strings are safe (no internal file paths, SQL, or secrets).
- `pinoHttp` serialiser in `app.ts:36-45` redacts URL parameters from logs, keeping only `id`, `method`, `url`.
- No PII in log fields — event payloads logged as structured objects without user-specific financial details at info level.
