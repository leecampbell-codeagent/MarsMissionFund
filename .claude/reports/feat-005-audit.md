## Warnings (Should Fix)

**W-001 (Medium, Security):** `CampaignNotAcceptingContributionsError` includes raw campaign status in error message, which is returned verbatim to API consumers. This leaks internal state machine values. Fix: replace `(status: ${status})` with a static message in `payment-errors.ts`.

**W-002 (Medium, Security):** No upper bound on `amountCents` in the Zod request schema. A malicious client can submit an astronomically large amount. Fix: add `.max(100_000_00)` or equivalent business ceiling to `create-contribution.schema.ts`.

**W-003 (Architecture):** `ContributionRepository` and `EscrowLedgerRepository` port interfaces import `PoolClient` from `pg`, leaking an infrastructure type into the ports layer. This is spec-prescribed for this feature but should be refactored in a future clean-up to use an abstract transaction handle.

**W-004 (Architecture):** Application service imports `CampaignNotFoundError` and `UserNotFoundError` from other bounded contexts' domain error modules. While spec-prescribed, this creates coupling. Consider republishing these as shared errors or having the repositories return typed nulls.

**W-005 (Documentation):** `.claude/mock-status.md` should be updated to reflect that the stub payment gateway (`StubPaymentGatewayAdapter`) is the mock for `PaymentGatewayPort`.

**W-006 (Test Coverage):** No test for "contribution that pushes an existing total over the funding goal" (Minor-003 from exploratory). The production code is correct but the test coverage gap should be addressed.

**W-007 (Test Coverage):** No clock-mock test for the 60-second duplicate detection window boundary (Minor-004 from exploratory).

## Changelog Entry

### feat-005: Contribution Flow and Payment Processing

- New bounded context `payments` at `packages/backend/src/payments/` with full hexagonal architecture: domain (`Contribution` entity, `ContributionStatus` value object, 7 domain errors), ports (4 interfaces: `ContributionRepository`, `EscrowLedgerRepository`, `PaymentGatewayPort`, `ContributionAuditRepository`), adapters (3 `pg` adapters, 3 in-memory test adapters, `StubPaymentGatewayAdapter`), application service (`ContributionAppService`), and API router.
- New endpoints: `POST /api/v1/contributions` (always returns 201, including payment failures), `GET /api/v1/contributions/:id` (donor-scoped), `GET /api/v1/campaigns/:id/contributions` (paginated).
- Stub payment gateway: `tok_fail` triggers failure path, any other token succeeds.
- 60-second duplicate detection window with failed-contribution exclusion.
- Atomic database transaction: contribution captured + escrow ledger entry + campaign totals update in single `BEGIN/COMMIT`.
- Campaign auto-transition to `funded` when `total_raised_cents >= funding_goal_cents` inside same transaction.
- Full audit trail: `contribution.created`, `contribution.captured`, `contribution.failed` events — best-effort, non-blocking.
- New migration: `db/migrations/20260305170000_feat005_contributions.sql` — adds `total_raised_cents`/`contributor_count` columns to `campaigns`, creates `contributions`, `escrow_ledger`, and `contribution_audit_events` tables with BIGINT monetary columns, CHECK constraints, and composite indexes.
- Frontend: `ContributeToMissionPage` at `/campaigns/:id/contribute` (protected route), `useContribute` TanStack Query mutation hook, `contribution-api.ts` API client, `contribution.ts` frontend types with `Intl.NumberFormat` display helper.
