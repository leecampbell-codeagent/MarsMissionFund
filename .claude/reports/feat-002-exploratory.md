## Issues Found

### Minor: Redundant Unique Indexes on `users` and `kyc_verifications`

**Severity:** Minor / cosmetic — no functional impact.

**Affected tables:** `users`, `kyc_verifications`

**Description:** The migration files declare inline `CONSTRAINT ... UNIQUE` on certain columns AND also execute a separate `CREATE UNIQUE INDEX IF NOT EXISTS` on the same columns. PostgreSQL automatically creates a unique index to back a unique constraint, so the explicit index is redundant.

Redundant index pairs:
- `users.clerk_id`: backed by both `users_clerk_id_unique` (from inline constraint) and `idx_users_clerk_id` (from explicit CREATE UNIQUE INDEX)
- `users.email`: backed by both `users_email_unique` and `idx_users_email`
- `kyc_verifications.user_id`: backed by both `kyc_verifications_user_id_unique` and `idx_kyc_verifications_user_id`

**Impact:** Minor storage overhead (each redundant index occupies additional space and adds marginal write overhead). Reads and uniqueness enforcement are unaffected.

**Recommendation:** Remove the standalone `CREATE UNIQUE INDEX` statements from migration files `20260306000001_create_accounts.sql` and `20260306000003_create_kyc.sql` for the columns already covered by inline UNIQUE constraints. Note: since these migrations are already applied and the spec requires append-only migrations, the fix would require a new migration to `DROP INDEX` the redundant indexes, or can be accepted as-is since it does not affect correctness.
